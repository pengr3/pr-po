    ```
    // =============================================
    // billing_requests collection (Phase 99 — BILL-06)
    // =============================================
    // Project-assigned users (operations_user) submit billing requests with
    // supporting doc links from project-detail; Finance reviews inline in the
    // Collectibles tab and approves (pre-fills Create-Collectible) or rejects
    // (with a reason). The billing_request amount is ADVISORY only — the
    // authoritative collectible amount is re-derived by submitCollectible() at
    // approval time, so create being open to any active user is safe (Finance
    // review is the gate). Schema (D-04): project_code/name, tranche_index/
    // label/percentage, amount_requested, billing_type, documents[], notes,
    // status, requested_by_uid/name, requested_at, reviewed_by/at, rejection_reason.
    match /billing_requests/{id} {
      // Read: any active user (Finance banner + project-detail own-requests list both read)
      allow read: if isActiveUser();

      // Create: any active user (operations_user is the intended creator; they
      // lack collectible authority — this is their billing surface). Amount is
      // advisory; Finance re-derives the real collectible amount at approval.
      allow create: if isActiveUser();

      // Update: status transitions only (approve/reject) — Finance / operations_admin / super_admin
      allow update: if hasRole(['super_admin', 'operations_admin', 'finance']);

      // Delete: same finance-authority set (no UI delete path is built this phase, but lock it down)
      allow delete: if hasRole(['super_admin', 'operations_admin', 'finance']);
    }
    ```

    Do NOT copy the collectibles `create: hasRole([...])` line — that is the exact landmine. Create MUST
    be `isActiveUser()`.

    Threat mitigation note (document in SUMMARY): a malicious active user can write arbitrary
    `amount_requested`/`project_code`/`tranche_*` because create is open. This is ACCEPTED because the
    amount is advisory — `submitCollectible()` (finance.js:1838-1839) re-derives the authoritative amount
    from live `contract_cost`×`percentage` at Finance approval. The billing_request is never trusted as
    the source of money. Finance review is the gate (T-99-01).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operations_user → Firestore (`billing_requests` create) | Untrusted client writes an advisory billing request; rules permit any active user to create |
| Finance → Firestore (`billing_requests` update) | Status transition (approve/reject) gated to Finance/operations_admin/super_admin |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-99-01 | Tampering | `billing_requests` create rule (open to any active user) | accept | Amount/code/tranche fields are ADVISORY only. `submitCollectible()` (finance.js:1838-1839) re-derives the authoritative collectible amount from live `contract_cost`×`percentage` at Finance approval — the billing_request is never trusted as the source of money. Finance review is the gate. Documented in the rules block comment. |
| T-99-02 | Elevation of Privilege | `billing_requests` status transition | mitigate | `allow update: if hasRole(['super_admin','operations_admin','finance'])` — operations_user can create but CANNOT approve/reject. Enforced server-side in this rules block. |
| T-99-03 | Information Disclosure | `billing_requests` read open to all active users | accept | Read is `isActiveUser()` (no PII beyond submitter name + project/amount already visible to active users via collectibles/projects). Matches the collectibles read rule precedent. |
</threat_model>
