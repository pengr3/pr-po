## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Finance (client) → Firestore `billing_requests` update | Status transition (approve/reject) + reviewer fields cross into persistence |
| billing_request fields → Create-Collectible prefill / banner render | Untrusted submitter strings (project_name, tranche_label, doc urls, notes) rendered + used to prefill |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-99-07 | Tampering (XSS) | Banner render of submitter strings (project_name, tranche_label, requested_by_name, doc url/label) | mitigate | All user strings escaped via `escapeHTML()` in renderPendingBillingBanner; doc anchors carry `target="_blank" rel="noopener noreferrer"` (D-19). |
| T-99-08 | Elevation of Privilege | approve/reject updateDoc by non-Finance | mitigate | Server-side: Plan 01 rules `allow update: if hasRole(['super_admin','operations_admin','finance'])`. Client-side: approveBillingRequest reuses openCreateCollectibleModal which gates on hasCollectibleWriteAuthority(); the banner Approve/Reject buttons only render in the Finance Collectibles tab (Finance-only surface). |
| T-99-09 | Tampering (advisory amount) | Approve pre-fill uses billing_request.amount_requested | accept (downstream-mitigated) | The prefill does NOT trust the request amount as money — openCreateCollectibleModal pre-selects only dept/project/tranche; submitCollectible() re-derives the authoritative amount from live contract_cost×percentage (finance.js:1838-1839). The request amount is advisory display only. |
| T-99-10 | Repudiation | Who approved/rejected | mitigate | reviewed_by + reviewed_at (serverTimestamp) recorded on every decision; rejection_reason stored and surfaced both sides. |
</threat_model>
