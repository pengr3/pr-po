<objective>
Plan 05 added optimistic local append, but two failure modes break the Excel-style continuous-entry feel:

1. **Snapshot wipe.** The `onSnapshot` callback does `tasks = []; snap.forEach(...)` — completely rebuilds `tasks` from server. If the user presses Enter rapidly (A then B) while A's setDoc is mid-flight, the snapshot for A's write fires and rebuilds tasks WITHOUT B (B not yet on server). B's optimistic row vanishes from the grid. It re-appears later when B's setDoc snapshot fires, but the user already saw the row disappear.

2. **Focus loss.** Each renderTaskGrid replaces `container.innerHTML`, destroying the input the user is typing in. If a snapshot fires while the user is typing the next task name, their in-progress text and focus are lost.

Fix: maintain a `_pendingOptimistic` Map keyed by taskId; the snapshot listener merges any pending entries that aren't yet in the snapshot. renderTaskGrid captures the empty-row input's value+selection before innerHTML replace and restores them after.
</objective>
