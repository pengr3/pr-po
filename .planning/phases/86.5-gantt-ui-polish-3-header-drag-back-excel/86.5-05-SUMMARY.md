# Phase 86.5 Plan 05: Excel-style new task entry — optimistic, silent, instant Summary

UAT item 7 fix — `handleNewRowCommit` rewritten as an optimistic append with no spinner, no per-row success toast, and a client-derived `task_id`. New rows render synchronously before Firestore setDoc resolves, making rapid Enter→type→Enter feel Excel-instant.

## What Was Built

A single function rewrite in `app/views/project-plan.js`:

- **Removed** `showLoading(true)` / `showLoading(false)` (from `handleNewRowCommit` only — other call sites untouched).
- **Removed** `showToast('Task created.', 'success')` — silent commit.
- **Removed** the `generateTaskId` retry loop with per-attempt `getDoc` collision check (~200-400ms per row).
- **Added** a local-state derivation of the next sequence number (reads `tasks[].task_id`, finds max trailing integer after the last dash, increments). Format unchanged: `TASK-{project_code}-{maxSeq+1}`.
- **Added** an optimistic append: build `optimisticTask` (with `Date` placeholders for the `serverTimestamp()` sentinels), `tasks.push(optimisticTask)`, then `renderTaskGrid()` synchronously — all before the `await setDoc` call.
- **Added** a rollback path: on `setDoc` rejection, find by `task_id` in `tasks`, `splice` the optimistic entry, re-render, and surface the existing permission/generic error toast.

Behavior preserved unchanged:

- Empty-input re-entry guard (returns early on second Enter)
- Synchronous input clear before any async work
- Missing-`project_code` warning toast
- DEFECT-6 parent inheritance (last-sorted-task's parent)
- Phase 86.3 D-04/D-05 anchor logic (`anchorStart` from row above; 1-day inclusive `end = start`)
- All `docData` fields (no schema change)
- Permission-denied vs. generic error toast text
- `_focusNewRowAfterRender` flag set in `handleNewRowKeydown` (Plan 02) — because `renderTaskGrid` is now called synchronously after `tasks.push`, the focus block at ~line 402 fires immediately, focusing the empty input on the new row instantly

## Deviations from Plan

### [Rule 1 - Bug] Kept `generateTaskId` import — still required by `gridInsertRowAbove`

- **Found during:** Pre-edit grep for `generateTaskId` usage across the file.
- **Issue:** The plan's `<action>` and acceptance criteria instructed removing the `generateTaskId` import on the basis that it would become unused after the rewrite. In practice, `generateTaskId` is **also** called by `gridInsertRowAbove` at line 813 (right-click "Insert row above" path). Removing the import would have produced a `ReferenceError` the next time a user inserted a row above an existing task, breaking unrelated functionality not in this plan's scope.
- **Fix:** Left the import statement at line 9 intact: `import { generateTaskId } from '../task-id.js';`. The plan's intent — making `handleNewRowCommit` no longer call `generateTaskId` — is fully satisfied; only the import line itself was preserved.
- **Files modified:** `app/views/project-plan.js` (no change to the import line).
- **Acceptance-criterion impact:** Criterion `grep -c "import { generateTaskId } from '../task-id.js'" app/views/project-plan.js` returns 0` is **NOT met** (returns 1). All other criteria met. No functional impact on the plan's must-have truths.
- **Commit:** 9879382 (no separate commit — this is the deliberate omission of one line from the planned diff).
