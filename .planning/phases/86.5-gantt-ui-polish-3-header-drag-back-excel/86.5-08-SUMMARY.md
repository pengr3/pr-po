# Phase 86.5 Plan 08: Optimistic Merge + Focus Preservation ÔÇö True Excel-Style Entry Summary

**One-liner:** Adds `_pendingOptimistic` Map to merge in-flight rows into the snapshot listener and captures/restores empty-row input value + selection across `renderTaskGrid` innerHTML replaces, so rapid EnterÔćĺEnter and mid-typing-during-snapshot both stay glitch-free.

## What Was Built

Plan 86.5-05 established optimistic local append for the new task row, but two failure modes still broke the Excel-style continuous-entry feel:

1. **Snapshot wipe** ÔÇö onSnapshot rebuilds `tasks` from server. If row A's `setDoc` snapshot fires while row B's optimistic add is still in-flight (no server confirmation yet), B vanishes from the grid until B's own snapshot arrives.
2. **Focus loss** ÔÇö `renderTaskGrid` replaces `container.innerHTML`, destroying the input the user is mid-typing in.

This plan closes both:

- **Task 1 (4 surgical changes):** A new module-scope `_pendingOptimistic = new Map()` registers each optimistic write under its `task_id` immediately after `tasks.push(optimisticTask)`. The onSnapshot callback merges any pending entries that aren't yet in the snapshot back onto `tasks` before re-render. A `finally{}` block deletes the pending entry on both success and failure paths. `destroy()` clears the map.
- **Task 2 (1 surgical change in `renderTaskGrid`):** Before the non-empty-path `container.innerHTML = ...`, a new capture block reads `document.activeElement` and stashes `{value, selectionStart, selectionEnd}` if the active element is the empty-row name input (`data-task-id="__new__"`). After the innerHTML replace, the restore block re-applies the value, then schedules `focus()` + `setSelectionRange(...)` via `setTimeout(0)`. `_savedEmptyRow` takes precedence over the existing `_focusNewRowAfterRender` flag ÔÇö the latter still handles the just-pressed-Enter (no-text) case.
