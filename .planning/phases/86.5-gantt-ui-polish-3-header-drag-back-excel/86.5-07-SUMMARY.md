# Phase 86.5 Plan 07: Suppress Transient FS Violation Toast During Drag-Link Write Summary

Module-scope `_suppressFsViolationToast` guard scopes the FS-violation toast suppression to the drag-link write window so the transient state between `updateDoc(deps)` and `applyFsAutoSchedule(...)` no longer surfaces a misleading "now starts before X finishes" toast.

## What Changed

Single file: `app/views/project-plan.js`. Four surgical edits, all aligned with the plan prose:

1. **Declaration (line 1006).** Added `let _suppressFsViolationToast = false; // Phase 86.5-07: skip toast during drag-link write window` directly after the existing `__lastViolationFingerprint` declaration so the two pieces of toast-baseline state live together.

2. **Drag-link write guard (lines 1667–1683).** Inside the existing svg mouseup link-path try/catch, set `_suppressFsViolationToast = true` at the top of the try (immediately before `await updateDoc(...)`), and added a new `finally { _suppressFsViolationToast = false; }` block. The existing `console.error('[ProjectPlan] drag-to-link write failed:', err)` and `showToast('Could not save dependency link.', 'error')` lines were left untouched.

3. **Toast gate in `checkAndToastFsViolations` (line 1703).** The condition that emits new-violation toasts now also requires `!_suppressFsViolationToast`. The fingerprint update on line 1709 stays unconditional so when the flag clears the next snapshot has a correct baseline and won't re-toast prior violations as new.

4. **Reset in `destroy()` (line 172).** Added `_suppressFsViolationToast = false;` next to `__lastViolationFingerprint = '';` so navigation away can never strand the flag in the suppressed state.

## Why This Approach

Snapshots from Firestore can fire between the two awaits in the link-write path. After `updateDoc` lands but before `applyFsAutoSchedule` updates the successor's start_date, the on-disk state genuinely *is* an FS violation (deps written, start_date not yet shifted). `checkAndToastFsViolations` correctly observes this — it just isn't useful information for the user, because the auto-shift is about to fix it.

Two non-fixes were considered and rejected:

- **Move the auto-shift before the deps write.** Auto-shift needs the predecessor relationship to already exist in the data model (otherwise it has no anchor to align to). Reordering would require restructuring `applyFsAutoSchedule`'s contract.
- **Filter the violation by checking if the schedule write is in flight.** That requires either a global "pending writes" registry or wrapping every write site, which is far more invasive than a single boolean flag scoped to one gesture.

The chosen approach — a module-scope boolean toggled in try/finally — keeps the suppression window minimal (only the deps-write + auto-schedule pair, exactly the gesture that produces the false positive) and adds zero coupling to other write paths. The fingerprint-update-while-suppressed pattern preserves the baseline so the next legitimate violation post-window is detected correctly.
