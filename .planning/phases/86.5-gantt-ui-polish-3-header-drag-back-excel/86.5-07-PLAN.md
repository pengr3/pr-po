<objective>
When the user drops a drag-link, the write sequence is `updateDoc(deps)` then `applyFsAutoSchedule(...)`. Snapshots can fire between these two awaits, surfacing a transient violation (deps written but successor's start_date not yet shifted). `checkAndToastFsViolations` reads this transient state and toasts "Task X now starts before Task Y finishes." After auto-schedule lands the violation goes away, but the toast already fired.

Fix: guard the link-write window with `_suppressFsViolationToast`; while true, `checkAndToastFsViolations` updates `__lastViolationFingerprint` (so the post-window baseline is correct) but skips toast emission.
</objective>
