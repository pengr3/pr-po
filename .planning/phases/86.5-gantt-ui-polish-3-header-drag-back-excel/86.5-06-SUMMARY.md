# Phase 86.5 Plan 06: Leftward-Drag Detect — Self-Bar Drop Routing Summary

Routed self-bar blue-dot drops into the existing resize branch so Plan 86.5-04's leftward-drag duration shrink finally fires; the predecessor-link branch is now reached only for genuinely different bars.

## Problem

Plan 86.5-04 added leftward-drag resize logic into the empty-space branch of the gantt drag-link `mouseup` handler, but in practice the leftward-drag almost never lands on empty space — the cursor is still over the source bar's own `.bar-wrapper`. Control fell through to:

```javascript
if (!toTaskId || toTaskId === fromTaskId) return; // self-link — reject silently
```

…which returned silently before reaching the resize code. The leftward-drag feature was effectively dead.

## Fix

Two surgical edits in `app/views/project-plan.js` (`initGanttDragLink()` mouseup handler):

### Change 1 — line 1614 (branch condition)

```diff
-            if (!toWrapper) {
+            if (!toWrapper || toWrapper.dataset.id === fromTaskId) {
+                // self-bar drop routes to resize (Plan 86.5-06): leftward drag almost always lands
+                // on the source bar's own wrapper, not empty space — both should resize, not self-link.
                 // Dropped on empty space — extend (rightward) or reduce (leftward) duration
```

The resize block uses `fromTaskId` (not `toWrapper`) to look up its own geometry, so it works identically whether the cursor is over empty space OR over the source bar's wrapper. No further changes needed inside the block.

### Change 2 — line 1647 (downstream guard)

```diff
-            if (!toTaskId || toTaskId === fromTaskId) return; // self-link — reject silently
+            if (!toTaskId) return; // wrapper without id — reject silently
```

The `=== fromTaskId` case is now caught earlier (Change 1 routes it into the resize branch and returns from there). The remaining `!toTaskId` guard is still needed for wrappers that may not carry a `data-id` (defensive).
