## Warning

### W1 — `setGanttZoom` rebuilds the SVG without re-attaching the mousedown guard

**Confidence: 88**
**Lines:** 1285–1297 (`setGanttZoom`), 1201–1202 (renderGantt step 11)

`gantt.change_view_mode()` causes Frappe to internally call `gantt.refresh()`, which tears down and rebuilds the SVG element. The comment at line 1207 explicitly states: "Frappe rebuilds the SVG on every `gantt.refresh()` / `new Gantt()`, so we must re-attach the listener after every `renderGantt()` call." `renderGantt()` correctly calls `mountGanttBarDragGuard()` as step 11. But `setGanttZoom()` calls `gantt.change_view_mode()` and then only re-runs `renderTodayLine()`, `applyFsViolationStyles()`, `installGanttScrollClamp()`, `renderCustomGanttHeader()`, and `fixGanttContainerScroll()` — `mountGanttBarDragGuard()` is absent.

After any zoom change, `_ganttDragMousedownHandler` is bound to the old, now-detached SVG node. The next bar drag does not fire the mousedown handler, so `_ganttBarDragging` is never set to `true`. `handleGanttDateChange` falls through to the live `updateDoc` path on every Frappe mousemove callback — the rubber-banding bug Phase 86.7 was built to eliminate is fully restored after every zoom switch.

**Fix:** Add `mountGanttBarDragGuard()` at the end of `setGanttZoom()`, after `fixGanttContainerScroll()`.

### W2 — Safety timer discards pending writes without flushing them

**Confidence: 85**
**Lines:** 1224–1228

The 10-second safety timer fires when `mouseup` is missed (browser loses focus mid-drag). The timer body sets `_ganttBarDragging = false`, `_pendingDragWrite = null`, `_pendingProgressWrite = null` — both pending writes discarded without being sent to Firestore. The user's intentional last drag position is silently lost.

The threat model (T-86.7-02) labels this "no phantom write left orphaned," but the pending writes at timeout-fire represent the user's intended drag position, not phantom data. Practical scenario: user drags a task bar, Alt+Tabs, releases mouse 11 seconds later — bar reverts with no feedback.

**Fix (Option A — flush on timeout, recommended):** Replace null assignments with the same flush logic used in the mouseup handler — call `updateDoc` for both pending writes with the same `.catch` + toast error paths, then null them.

**Fix (Option B — discard + toast):** Add `showToast('Drag cancelled — please try again.', 'warning')` inside the timeout to make the data loss visible. Lower implementation cost.

## Info

### I1 — `destroy()` silently discards pending writes mid-drag

**Confidence: 80**
**Lines:** 261–264

If the user navigates away while a drag is in progress, `destroy()` nulls pending writes without flushing. Threat model explicitly accepts this (T-86.7-04). Safety reasoning is sound — a post-destroy async write would reference stale module-scope closures. No code change required. Worth noting if a "unsaved changes" navigation guard is ever added to the view lifecycle.
