## Tests

### 1. Drag a bar after switching zoom levels (post-zoom guard check)
expected: After clicking a zoom pill (Day, Week, or Month), grab a leaf task bar and drag it to a new position. The bar should move smoothly with no rubber-banding during the drag. On release, check the Network tab — exactly one Firestore write should appear per drag gesture. Grid dates update within ~1 s.
result: PASS

### 2. Progress scrub after zoom change
expected: Same as above but using the progress handle. After a zoom switch, drag the progress handle on a leaf task. The scrub should be smooth (no mid-drag re-renders). On release, one Firestore write for progress appears. Parent task weighted progress updates in the grid.
result: PASS

### 3. Navigate away mid-drag — no listener leak
expected: While dragging a Gantt bar (hold the mouse button down), press the browser Back button or click a nav link to leave the plan view. No uncaught promise errors appear in the console. Re-enter the plan view and drag a bar again — dragging works normally (destroy() cleaned up the document mouseup listener correctly).
result: BUG FOUND + FIXED (commit 03258d4) — when mouse was released outside the Gantt pane (e.g. over a nav link), Frappe calculated improbable dates from the out-of-bounds cursor position and the mouseup handler flushed them to Firestore. Fix: (1) mouseup handler now checks if e.target is inside #ganttPane; if not, it nulls pending writes and reverts via renderGantt(). (2) handleGanttDateChange only stores _pendingDragWrite when newStart <= newEnd.
re-test: REQUIRED — drag a bar, release mouse over a nav link, confirm bar reverts to original position and no Firestore write is made with improbable dates.
