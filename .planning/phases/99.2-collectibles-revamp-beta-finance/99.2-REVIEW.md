## Summary

The phase 99.2 source changes are correct on every priority-1 axis the brief flagged.
I traced each focus area against the actual code and the three PLAN threat models, and
the mitigations landed:

- **Urgency math (D-08b):** `getCollectibleUrgency` tier precedence is correct
  (paid → critical ≥30d → overdue 1–29d → near-due ≤7d → partial → healthy). The
  day-granularity overdue diff is sound: both sides are pinned to **local midnight**
  (`today.setHours(0,0,0,0)` and `new Date(due_date + 'T00:00:00')` which parses as local
  time without a `Z`), so the day delta is a clean integer with no DST/timezone drift, and
  a due-today collectible correctly lands in `daysUntilDue = 0` (near-due, NOT overdue).
- **Pagination recompute (D-08c):** both `renderCollectiblesTable` (L1494-1499) and
  `changeCollectiblesPage` (L1818-1822) recompute totals over the identical post-hide
  visible set, and both clamp the page index; `toggleCollShowCompleted` resets to page 1.
  No drift, no out-of-range landing.
- **CSV non-regression (D-07):** the fully-paid hide is applied inside the render path,
  NOT inside `getDisplayedCollectibles()`; `exportCollectiblesCSV` (L2883, untouched) still
  calls `getDisplayedCollectibles()`, so the 13-column CSV keeps exporting fully-paid rows.
- **Reconciliation (D-04/D-09):** `getOrphanApprovedBillingRequests` matches on dept-aware
  code AND `tranche_index` with strict equality. I confirmed cross-file that BOTH writers
  (`project-detail.js` L915/948, `service-detail.js`, and `finance.js submitCollectible`
  L2267/2305) store `tranche_index` as a `parseInt` integer, so `===` is type-consistent —
  no string-vs-number mismatch.
- **XSS (T-99.2-08):** every dynamic Firestore field in the new markup passes through
  `escapeHTML` (project/service names, tranche labels, notes, requester/approver names,
  `coll_id`, last-payment date/method, `req.id` inside onclick attrs), amounts through
  `formatCurrency`, and the progress-bar width is a guarded `Math.min(100, Math.round(...))`
  over `parseFloat||0` numerics — never attacker-controlled text.
- **SPA lifecycle:** all three new window handlers (`toggleCollShowCompleted`,
  `toggleApprovedBanner`, `fileCollectibleFromBanner`) are registered in
  `attachWindowFunctions()` and deleted in `destroy()`. The widened `billing_requests`
  listener remains a single subscription pushed once to `listeners[]`.
- **No regressions:** `deriveCollectibleStatus` return strings are byte-unchanged,
  `approveBillingRequest`/`rejectBillingRequest` stay status-flip-only, and
  `getDisplayedCollectibles` is untouched.

The findings below are quality / robustness only — no HIGH-severity correctness or
security defect. `node --check app/views/finance.js` passes and CSS braces balance.

## Medium

### MD-01: `destroy()` does not reset 3 new Phase 99.2 module-state vars → stale state leaks across Finance remounts

**File:** `app/views/finance.js:4961-4973`
**Issue:** The Collectibles-tab teardown block resets the Phase 99 state
(`pendingBillingRequests = []`, `billingBannerCollapsed = false`, all 5 filters,
`collCurrentPage = 1`) but **omits the three module-state variables introduced in this
phase**: `collShowCompleted` (L245), `approvedBillingRequests` (L234), and
`approvedBannerCollapsed` (L235). Because these are module-level `let`s, they persist after
the Finance view is destroyed and a different view is mounted, then carry into the next
Finance mount.

Impact, in order of severity:
- **`collShowCompleted`** is the meaningful one. If the user clicks "Show N completed",
  navigates away, and returns to Finance, the flag stays `true` and the freshly-mounted
  table shows fully-paid rows by default — directly contradicting the D-05b
  "fully-paid rows hidden by default" contract. Note `collCurrentPage` IS reset to 1 right
  beside it, so the asymmetry is clearly unintended.
- **`approvedBannerCollapsed`** mirrors `billingBannerCollapsed`, which the block DOES reset.
  A collapsed sub-section-2 stays collapsed across remounts — a cosmetic asymmetry with the
  sibling flag that is reset two lines above.
- **`approvedBillingRequests`** is lower impact because the `billing_requests` listener
  re-assigns `approvedBillingRequests = []` (L1795) on its first snapshot after remount, but
  for the brief window before that snapshot fires, `renderPendingBillingBanner()` could read
  the previous mount's orphan set. Every other data array in the block (`collectiblesData`,
  `pendingBillingRequests`, the two Maps) is defensively reset for exactly this reason.

**Fix:** add the three resets alongside the existing ones (mirror the established pattern):
```javascript
    // Phase 99 — billing request banner state
    pendingBillingRequests = [];
    billingBannerCollapsed = false;
    approvedBillingRequests = [];          // Phase 99.2 — reset orphan set on teardown
    approvedBannerCollapsed = false;       // Phase 99.2 — mirror billingBannerCollapsed
    collShowCompleted = false;             // Phase 99.2 — restore D-05b hidden-by-default on remount
    collProjectFilter = '';
```

## Low

### LW-01: `coll.id` interpolated raw (no escapeHTML) into onclick handlers in the desktop table and mobile card

**File:** `app/views/finance.js:1571` (context-menu td), `1597-1604` (action buttons), `1625-1631` (mobile card buttons)
**Issue:** The new 5-column table and the mobile card interpolate the Firestore document id
raw into onclick attributes, e.g.
`oncontextmenu="...window.showCollectibleContextMenu(event, '${coll.id}')..."` and
`onclick="window.openRecordCollectiblePaymentModal('${coll.id}')"`. In the SAME phase, the
banner code escapes the analogous id (`window.approveBillingRequest('${escapeHTML(req.id)}')`),
so the discipline is inconsistent within the diff. In practice Firestore auto-generated doc
ids are `[A-Za-z0-9]` only, so this is not currently exploitable — but it is a brittle
assumption (a `'` in an id would break the attribute / enable injection) and the inconsistency
invites copy-paste of the unescaped form.
**Fix:** wrap for consistency with the banner: `'${escapeHTML(coll.id)}'` in every new onclick
that interpolates `coll.id`. (Note: this pattern predates the phase elsewhere in the file, so
treat as a low-priority hygiene cleanup, not a blocker.)

### LW-02: Duplicated fully-paid hide expression across three call sites (drift risk)

**File:** `app/views/finance.js:1494-1496` (render), `1819-1820` (changeCollectiblesPage), `1492-1493` (completedCount)
**Issue:** The "split allDisplayed into visible vs Fully-Paid" logic
(`allDisplayed.filter(c => deriveCollectibleStatus(c) !== 'Fully Paid')`) is written out
three times. The brief explicitly worried about pagination drift between
`renderCollectiblesTable` and `changeCollectiblesPage`; they agree today, but because the
predicate is duplicated rather than centralized, a future edit to one site (e.g. changing the
hide criterion) can silently desync the page math from the rendered set.
**Fix:** extract one helper, e.g.
```javascript
function getVisibleCollectibles() {
    const all = getDisplayedCollectibles();
    return collShowCompleted ? all : all.filter(c => deriveCollectibleStatus(c) !== 'Fully Paid');
}
```
and call it from both render and page-change paths so the visible set has a single definition.

### LW-03: `deriveCollectibleStatus` now calls `getCollectibleUrgency`, which recomputes `totalPaid` already computed by callers (redundant work, single-source coupling)

**File:** `app/views/finance.js:110` (deriveCollectibleStatus) and `25-48` (getCollectibleUrgency)
**Issue:** Each rendered row now computes `totalPaid` (reduce over `payment_records`) up to
three times for the same collectible: once inline in the row builder (L1526-1528), once inside
`deriveCollectibleStatus` → `getCollectibleUrgency` (L26-28), and once more directly inside
`getCollectibleUrgency` when the row builder also calls it (L1531). Per-doc `payment_records`
arrays are small so this is not a performance concern (out of v1 scope anyway), but it is
duplicated logic: the non-voided-sum rule now lives in 4+ places. This is a maintainability
note — the single-source intent of Plan 01 centralized the *overdue threshold* but left the
*totalPaid* rule duplicated.
**Fix (optional):** have `getCollectibleUrgency` return `totalPaid` in its result object so
callers consume one computed value instead of re-reducing. Low priority; no behavior change.
