2. Approved-not-filed **recovery banner** (027b)

4. **Mobile ≤768px card layout** + a real CSS block (027c)

**In scope:** `app/views/finance.js` (Collectibles tab only) + a new dedicated CSS block (`components.css` and/or `views.css`).

**Out of scope (boundary):**
- Detail-page surface (project-detail.js / service-detail.js / expense-modal.js) — that is Surface 1 / Phase 99.1 (COMPLETE). Do NOT touch it.
- **No Firestore schema change. No firestore.rules change.** (`collectibles` + `billing_requests` already carry every field needed.)
- The billing-request **approve flow** itself is not restructured (see D-04).
- Other Finance tabs (Payables, etc.) — untouched.
</domain>

<decisions>
## Implementation Decisions

### Scope & data backing
- **D-01** — Finance Collectibles tab only; `finance.js` + new CSS. Zero schema/rules change. Data already present in `collectibles` (Phase 85): `amount_requested`, `payment_records[{amount,date,method,status}]`, `due_date`, `department`, `tranche_index`, `project_code`, `deriveCollectibleStatus()`.
- **D-09** — Money-in vocabulary stays consistent with Phase 99.1 / spec §2: **Billed** (`billing_requests.amount_requested` where `status='approved'`), **Invoiced** (`collectibles.amount_requested`, frozen at creation per D-13), **Collected** (sum of non-voided `payment_records[].amount`), **Outstanding** (Invoiced − Collected). Reconciliation key between the two collections is **`project_code + tranche_index`** (client-side `.find()`).

### Scorecard (027a → SC1)
- **D-02** — **4 reactive chips** above the table: **Total Invoiced / Cash Collected / Outstanding / Overdue**. Overdue chip is red and **hidden when ₱0**. Chips reflect the **currently-filtered rows**, NOT the full dataset (recompute from the same displayed-rows source the table uses).

### Approved-not-filed banner (027b → SC2)
- **D-03** — **Unified collapsible banner** with **two independent sub-sections**:
  1. **"Awaiting Your Review"** (amber) — `billing_requests` where `status='pending'`.
  2. **"Approved — File as Collectible"** (blue) — `billing_requests` where `status='approved'` with **no matching `collectibles` doc** on `project_code + tranche_index`. "File COLL-xxx" button → existing **`openCreateCollectibleModal(preselectKey)`**.
  - **This banner is OWNED in Phase 99.2** (user decision — not folded into a Phase 99 closure item). Sub-section #2 is the **only recovery path for the Phase 99 orphan state**.
- **D-04** — **Orphan-state handling = recovery, not prevention** (user-delegated; decided here). Keep `approveBillingRequest()` as a simple `status='approved'` flip — do **NOT** make approve+create atomic. The banner's "Approved — File as Collectible" reconciliation (D-03 #2) guarantees any abandoned-create orphan stays **visible and re-fileable**. Rationale: atomicity would gut the banner's purpose, expand scope past the `finance.js`-only / no-schema boundary into the shared approve flow, and remove Finance's legitimate "approve now, file the COLL later" workflow. Recovery-by-reconciliation is more robust than prevention. **Reconciliation correctness is therefore a must-have** (an orphan must NEVER be silently absent from sub-section #2).

### Table redesign (027c → SC3)
- **D-05** — **10 cols → 5**: `Project · Tranche` (name + tranche label + COLL id + dept chip merged) / `Collection Progress` (6px bar + "₱X of ₱Y · Z%", color tracks urgency) / `Due · Urgency` (relative: "65 days overdue ‼" / "Due in 5 days ⚠" / "Due in 40 days") / `Last Payment` ("₱1M · Jun 1 · Bank Transfer" or italic "No payments yet") / `Actions` (Record Payment when non-paid / View History when paid).
- **D-05a** — **4px urgency left-borders**, 6 tiers: critical (30+ days overdue, red) / overdue (1–29, orange) / near-due (≤7 days, amber) / partial (blue) / healthy (grey) / fully-paid (green, de-emphasized).
- **D-05b** — **Fully-paid rows hidden by default** behind a **"Show N completed"** toggle. Hiding must **recompute pagination totals** (the count and page math use the visible set).

### Mobile (027c → SC4)
- **D-06** — Add a **≤768px card layout** (collectibles is the one Finance table Phase 73.1 skipped) using the **established dual-mode CSS pattern** (table on desktop / cards on mobile). Rich visuals (progress bars, urgency borders) live in a **real CSS block**, not inline styles.

### CSV export
- **D-07** — **CSV keeps ALL discrete fields** (user-delegated; decided here). The redesign is **display-only**; the existing filter-aware **13-column `exportCollectiblesCSV`** (COLL-08) must NOT regress — keep flat, machine-readable columns with **absolute** values (no merged cells, no progress bars, no relative dates). Do NOT collapse the export to the condensed 5-col view. Rationale: CSV is for accounting/spreadsheet consumption; condensed display strings would harm it.

### Build-time correctness (SC5 — cross-cutting invariants)
- **D-08a** — "Last Payment" must filter `status !== 'voided'` (use the same non-voided rule as `Collected` in D-09).
- **D-08b** — **ONE shared "overdue" definition** used by BOTH the scorecard Overdue chip (D-02) and the row urgency tiers (D-05a). Single helper, single source of truth — do not duplicate the threshold logic.
- **D-08c** — Hiding fully-paid rows (D-05b) recomputes pagination totals.

### Claude's Discretion
- Exact CSS class names, file split between `components.css` vs `views.css`, helper function names, and where in the existing `renderCollectiblesTable()` / `getDisplayedCollectibles()` pipeline the new logic hooks in — follow existing `finance.js` conventions and the dual-mode pattern from Phase 73.1.
- Banner collapse/expand state persistence (session-only vs none) — pick the lightest approach consistent with existing Finance UI.
</decisions>
