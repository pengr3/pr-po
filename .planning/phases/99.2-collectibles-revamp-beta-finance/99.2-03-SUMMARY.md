# Phase 99.2 Plan 03: Two-section recovery banner Summary

**Unified approved-not-filed recovery banner — one collapsible banner with two independent sub-sections (pending review + reconciled approved orphans), the only recovery path for the Phase 99 silent-orphan state, with a File-as-Collectible bridge to the existing prefilled create modal.**

## Performance

- **Duration:** ~15 min
- **Tasks:** 2
- **Files modified:** 1 (app/views/finance.js)

## Accomplishments
- **Widened listener (Task 1):** the single `brUnsub` subscription drops the `status=='pending'` filter and splits client-side into `pendingBillingRequests` + `approvedBillingRequests` — still exactly one entry in `listeners[]`.
- **Reconciliation (D-04):** `getOrphanApprovedBillingRequests()` returns every approved request whose dept-aware code (`service_code`/`project_code`) + `tranche_index` has no matching `collectiblesData` doc. The collectibles `onSnapshot` now re-renders the banner so a freshly-filed COLL drops its orphan immediately.
- **Two-section banner (SC2/Task 2):** `renderPendingBillingBanner` is one outer container with "Awaiting Your Review" (amber, pending, Approve/Reject unchanged) + "Approved — File as Collectible" (blue, orphans). Each sub-section collapses independently; the whole banner auto-hides only when both are empty.
- **File bridge:** `fileCollectibleFromBanner(reqId)` builds the exact 3-segment preselectKey used by `approveBillingRequest` and calls the existing `openCreateCollectibleModal(preselectKey)` — no new modal path (D-03). `window.toggleApprovedBanner` + `window.fileCollectibleFromBanner` registered and torn down.
- **D-04 invariant:** `approveBillingRequest`/`rejectBillingRequest` bodies are byte-unchanged (verified via diff) — approve stays a simple status flip, recovery is by reconciliation.
