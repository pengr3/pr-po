# Phase 84.1 Plan 01: Procurement-Side Notification Triggers (NOTIF-14 + NOTIF-18) + Creator-UID Schema Fields Summary

## Decisions Made
- **PO creator inheritance from PR creator (not Finance approver):** The PO is created by Finance via `generatePOsForPRWithSignature`, but the procurement actor who owns the work is the PR creator (`pr.pr_creator_user_id` from Phase 84-01). NOTIF-18 must route to the procurement actor, not the approver — so `po_creator_user_id` deliberately reads `pr.pr_creator_user_id ?? null` and Finance's identity is captured separately in `finance_approver_user_id`.
- **`window.getCurrentUser?.()` accessor for all creator-UID stamps (including TR submitTransportRequest where currentUser is not in scope):** Plan stated "Both functions already have `currentUser` in scope" but `submitTransportRequest` does not declare a local `currentUser`. Used the canonical optional-chain accessor (matching Phase 84-01 D-01 pattern) consistently across all 6 write sites — keeps the stamp pattern uniform and removes a hidden dependency on local-variable presence.
- **`poDataFresh` local variable name in NOTIF-18 block (not `poData`):** Inside the `if (newStatus === 'Delivered' && !isSubcon)` try-block, a `const poData = poDocFresh.data() || {};` would shadow the module-scope `poData` array (line 25 of procurement.js: `let poData = []`). The next sibling block (`const currentPO = poData.find(...)`) is OUTSIDE the if-block so it correctly resolves to the module-scope array, but the shadow is a foot-gun for future maintainers. Renamed to `poDataFresh` for clarity.
- **firestore.rules Option B (relax users.list to isActiveUser()):** Approved by user during planning. Aligns with every other operational collection's read pattern. Field exposure (full_name / email / role / status) is not PII-sensitive; write rules are unchanged. BONUS: retroactively unblocks Phase 84 NOTIF-08 fan-out from non-admin actors.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug Prevention] Renamed local `poData` to `poDataFresh` in NOTIF-18 block to prevent module-scope variable shadowing**
- **Found during:** Task 2 Step 2.5 (NOTIF-18 wiring in `updatePOStatus`)
- **Issue:** The plan-supplied snippet used `const poData = poDocFresh.data() || {};` inside the `if (newStatus === 'Delivered' && !isSubcon)` block. This shadows the module-scope `let poData = []` (line 25). While the shadow is scope-limited to the if-block (the next-sibling `const currentPO = poData.find(...)` correctly resolves to the outer array), future maintainers reading the function might assume both `poData` references are the same variable, leading to subtle bugs when extending the block.
- **Fix:** Renamed the local variable to `poDataFresh` and updated the four downstream references inside the same block (`poDataFresh.po_creator_user_id`, `poDataFresh.mrf_id` x2, `poDataFresh.po_id` x2). Also added a comment documenting the rename rationale.
- **Files modified:** `app/views/procurement.js`
- **Verification:** `node --check` passes; manually verified that the outer `currentPO = poData.find(...)` line continues to reference the module-scope array (unchanged).
- **Committed in:** `ec35c21` (Task 2 commit)

**2. [Rule 3 - Blocking] Used `window.getCurrentUser?.()` for tr_creator_user_id in submitTransportRequest (function does not declare local currentUser)**
- **Found during:** Task 1 Step 1.4 (TR creator stamp at line 5760)
- **Issue:** Plan stated "Both functions already have `currentUser` in scope" for the two TR write sites, but the first one (`submitTransportRequest`, line 5639) does NOT declare a local `currentUser` before line 5760 (the addDoc call). Only `generatePR` (line 5841) and `generatePRandTR` (line 6172) declare `const currentUser = window.getCurrentUser();` locally. Using `currentUser` directly inside `submitTransportRequest` would throw a ReferenceError at runtime.
- **Fix:** Used the `window.getCurrentUser?.()?.uid ?? null` and `window.getCurrentUser?.()?.full_name || ... || 'Unknown User'` optional-chain pattern at BOTH TR write sites for consistency. This matches the canonical pattern already established in this file by Phase 84-01 D-01 (the `requestor_user_id` field uses the same accessor). The same pattern is also explicitly required by the plan for the 3 RFP write sites in Step 1.5, so applying it uniformly to the 2 TR sites also keeps the stamp shape consistent across all 5 procurement.js creator-UID stamps.
- **Files modified:** `app/views/procurement.js`
- **Verification:** `node --check app/views/procurement.js` passes; manual inspection confirms no `ReferenceError` path remains in `submitTransportRequest`.
- **Committed in:** `251a6ee` (Task 1 commit)
