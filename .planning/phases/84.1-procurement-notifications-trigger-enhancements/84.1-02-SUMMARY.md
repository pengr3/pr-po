## Decisions Made
- **PROJECT_COST_CHANGED placement adjacent to PROJECT_STATUS_CHANGED in TYPE_META:** Per plan suggestion, placed the new entry right after `PROJECT_STATUS_CHANGED` in both `TYPE_META` maps so all project-domain badges sit together visually. Used `'$'` icon (matches RFP_PAID's color-coded but distinct from RFP_PAID's green by using brand blue `#1a73e8` to signal "project field changed" rather than "money received").
- **Same audience as NOTIF-11 (personnel_user_ids only):** No expansion to procurement / Finance roles — cost changes are an operational concern for the project's personnel, not a procurement-actor concern. Future audience expansion can be done by appending a separate `createNotificationForRoles` call without disturbing this one.
- **WR-03 pre-capture in service-detail.js for NOTIF-19:** The existing NOTIF-11 capture block is positioned outside the `try` to avoid stale `currentService` after the `await updateDoc` lets the onSnapshot listener replace it. NOTIF-19 inherits the same risk and the same mitigation — captures placed in the same scope (immediately after the NOTIF-11 captures, before the `try`).
- **`PHP <formatted>` / `(not set)` display convention:** When the field is being introduced (null → value) or cleared (value → null), the personnel reading the bell needs to see explicitly that "no value was set previously" (or "no value is set now"). Empty string or `PHP 0.00` would falsely imply the value was zero rather than absent. The convention chosen here (`'(not set)'`) matches the existing UI pattern used in the input-field placeholder text on lines 389/393 of project-detail.js (`placeholder="(Not set)"`).
