    ```javascript

The fallback to the unchanged Phase 84 body (no reason suffix) handles theoretical cases where the function is invoked through a future code path that bypasses the prompt validation.

    No new fields, no new imports, no schema changes. The MRF doc already persists `rejection_reason` per Phase 23 (Rejection Reason Passthrough) and Phase 62.2 (MRF rejection timeline event) — this task only surfaces the reason in the in-app notification body.

    Note that the rejection reason in the MRF doc is canonically stored at `rejection_reason` (set inside the updateDoc at line ~4217). The `reason` local variable used here is the same trimmed value that was just persisted. Either source is correct.
