## Warning

### W1 — `gridToggleMilestone` available on parent tasks; silently destroys Gantt summary bar

**File:** `app/views/project-plan.js` lines ~1339–1344 (context menu) and ~1948–1950 (`appendNode`)
**Confidence:** 85%

The single-task context menu renders "Mark as Milestone" for every task including parent tasks (tasks that have children). When a user marks a parent as a milestone:

- `appendNode` evaluates `t.is_milestone` **before** `isParent`. A parent with `is_milestone = true` receives `customClass = 'milestone-marker'`.
- The CSS for `milestone-marker` applies `display: none` to `.bar` and `.bar-progress`. The parent's grey summary envelope bar disappears from the Gantt. Child bars are unaffected.
- In the grid, `tg-row-milestone` is applied (amber tint + ◆ prefix) but `statusClass` is correctly suppressed for parents — so the grid row looks reasonable, but the corresponding Gantt bar is gone.

Data is not corrupted (dates, children, dependencies untouched), but the visual is broken with no warning.

**Fix:** Guard the milestone menu item with `!hasChildren` (already computed at line ~1300 in `showTaskContextMenu`):

```javascript
${!hasChildren ? `
    <div style="border-top:1px solid #f1f5f9;margin:4px 0;"></div>
    <div style="padding:8px 16px;cursor:pointer;font-size:0.875rem;color:#f59e0b;"
         onmouseenter="this.style.background='#fffbeb'" onmouseleave="this.style.background='transparent'"
         onclick="document.getElementById('taskGridContextMenu')?.remove(); window.gridToggleMilestone('${escapeHTML(taskId)}')">
        ${t.is_milestone ? 'Remove Milestone' : 'Mark as Milestone'}
    </div>` : ''}
```

Defensive fallback: also swap branch order in `appendNode` to check `isParent` before `is_milestone` so any existing parent-milestone docs in Firestore render correctly:

```javascript
if (isParent) customClass = 'parent-summary-bar';
else if (t.is_milestone) customClass = 'milestone-marker';
else { /* status branch */ }
```

## Verified Correct

**CSS specificity — milestone overrides status tints.** `.tg-row.tg-row-milestone` and `.tg-row.tg-row-overdue` have identical specificity (0,2,0). The milestone rule appears later in the file, so it wins via cascade order on both `background` and `box-shadow` when both classes are present. `:hover` variants follow the same order. Correct.

**XSS in context menu `onclick` handlers.** All `taskId` values injected into `onclick` attribute strings use `escapeHTML(taskId)`. The new milestone item correctly uses `escapeHTML(taskId)`. No XSS surface.

**`window.gridToggleMilestone` lifecycle.** Registered in `init()`, deleted in `destroy()`. Symmetric — no leak on navigation.

**`gridToggleMilestone` Firestore payload.** `updateDoc` writes `{ is_milestone: newVal, updated_at: serverTimestamp() }`. `newVal = !t.is_milestone` reads from live `tasks[]`. No spurious fields. Error handling covers permission-denied and generic failure.

**`computeStatus` logic and null dates.** ISO string comparison is correct. Priority order (complete → overdue → not-started → in-progress) is unambiguous. Null/undefined dates fall through to `in-progress` (JS: `null < "2026-05-29"` → `false`). Acceptable.

**Gantt `appendNode` customClass for `in-progress`.** `map[status] || ''` yields empty string for in-progress, preserving Frappe default blue. Correct.

**Gantt bar CSS selectors.** `.gantt-container .bar-wrapper.bar-status-overdue .bar` correctly targets Frappe's SVG `<rect class="bar">`. Three fill rules (red/green/gray) are correct, non-overlapping with existing rules.
