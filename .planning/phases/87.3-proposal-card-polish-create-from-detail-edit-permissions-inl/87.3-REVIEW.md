# Phase 87.3 Code Review

**Files reviewed:** `firestore.rules`, `app/proposal-modal.js`, `app/views/project-detail.js`, `app/views/service-detail.js`, `styles/components.css`

## Overall Assessment

The Firestore rules for Phase 87.3 are correctly structured. BRANCH B (ops/services user create) and the expanded BRANCH 2 field-mask are logically sound: `created_by` pinning, `parent_collection` alignment, `personnel_user_ids` membership checks, and `created_at`/`created_by` immutability are all in place.

However, the JS implementation has two critical bugs that make "Start Proposal" completely non-functional for services_user (and all roles in a service context), plus two important issues in the inline card helpers.

---

## Critical

### CR-1: `saveProposal()` CREATE payload omits `parent_collection` — services_user "Start Proposal" always receives permission-denied

**File:** `app/proposal-modal.js` — saveProposal() CREATE path  
**Confidence:** 100

`docPayload` never sets `parent_collection`. The Firestore BRANCH B rule for `services_user` evaluates:

```
request.resource.data.get('parent_collection', 'projects') == 'services'
```

Because the field is absent, the `get()` default fires and returns `'projects'`, making the condition `false`. Every "Start Proposal" click from `service-detail.js` by a `services_user` (or `services_admin`) is rejected with permission-denied.

**Secondary consequence:** `_applyProposalStateTransition` reads `proposal.parent_collection` to determine which parent collection to update. An absent field defaults to `'projects'`, causing all state transitions (submit, approve, reject, loss) on service proposals to target the `projects` collection rather than `services` — silently corrupting or failing all lifecycle writes.

**Fix:** Add `parent_collection` to `docPayload`. Extend `openCreateProposalModal` to accept a third `parentCollection` parameter (default `'projects'`), stash it in a module-level variable (e.g. `_createModalParentCollection`), and write it into the payload. Callers in `service-detail.js` must pass `'services'` at the call site.

---

### CR-2: `_loadModalDropdownData()` only loads `projects`; service doc ID set into a projects-only select silently fails validation and breaks `projectCode` resolution

**File:** `app/proposal-modal.js` — `_loadModalDropdownData()`, preselect block, `saveProposal()` `projectCode` resolution  
**Confidence:** 100

`openCreateProposalModal` is called from `service-detail.js` with the Firestore service doc ID as `preselectedProjectId`. After `_loadModalDropdownData()` populates `_modalProjectsData` with only project docs, the preselect code does `projectSelectEl.value = preselectedProjectId`. No `<option>` with that service doc ID exists in the list, so the select silently stays at `''`. Validation fires "Please select a project" and submission is blocked.

Even if validation were bypassed, `_modalProjectsData.find(p => p.id === projectId)` returns `undefined`, leaving `projectCode` as `null`. The Firestore BRANCH B personnel-lookup at `projects/${request.resource.data.project_id}` then targets a non-existent document.

"Start Proposal" is fully non-functional from `service-detail.js` for all roles.

**Fix:** `openCreateProposalModal` needs the `parentCollection` parameter from CR-1. The simplest path for context-locked creates (select is already `disabled`): accept `project_id`, `project_code`, and `parent_collection` as direct parameters and bypass the dropdown lookup in `saveProposal()` when they are provided. The service detail page already holds `currentServiceDocId` and `currentService.service_code` at the call site.

## Warning

### WR-1: `_proposalStageLabel` and `_proposalStatusDotColor` map key `'approved'` instead of `'client_approved'`; client-approved proposals show "Proposal" label and wrong gray dot

**Files:** `app/views/project-detail.js`, `app/views/service-detail.js`  
**Confidence:** 95

Both `_proposalStageLabel` dictionaries contain `approved: 'Client Approved'`. The canonical status written by `submitClientApproved()` in `proposal-modal.js` is `status: 'client_approved'` (CLAUDE.md schema). The key `'client_approved'` is absent from both maps:

- `_proposalStageLabel('client_approved')` → fallback `'Proposal'`
- `_proposalStatusDotColor('client_approved')` → gray `#94a3b8` instead of green `#059669`

A client-approved proposal renders with the generic "Proposal" label and a gray dot — visually misleading.

**Fix:** Replace `approved:` with `client_approved:` in both functions in both files.

---

### WR-2: `service-detail.js` listener registration uses `if (!existing)` guard instead of detach-then-reattach; stale handler closure leaks on re-init

**File:** `app/views/service-detail.js`, permission and assignment listener registration  
**Confidence:** 88

`project-detail.js` unconditionally removes the old handler first, then always overwrites the stored reference — the stored reference is always current. `service-detail.js` guards with `if (!window._serviceDetailPermissionHandler)`, preventing the stored reference from being updated on a second `init()` call.

If `init()` is called without `destroy()` first (possible per CLAUDE.md same-view re-initialisation), a new anonymous closure is added to `addEventListener` but not stored. `destroy()` removes the old handler via the stale stored reference, leaving the new closure permanently attached and firing `renderServiceDetail()` with stale `currentService` state.

**Fix:** Mirror project-detail.js exactly — unconditionally remove first, always overwrite the stored reference for both the permission and assignment handlers.
