## Critical

### CR-3: `closeProposalModal()` deletes `window.openCreateProposalModal`, breaking Start Proposal CTA after detail modal close

**File:** `app/proposal-modal.js` ‚Äî `closeProposalModal()` line 1674  
**Confidence:** 82

`closeProposalModal()` executes `delete window.openCreateProposalModal` as cleanup (line 1674). This was added in Phase 87.1 when `openProposalModal()` began re-registering `window.openCreateProposalModal` at line 1637 for the Edit flow inside the detail modal. Plans 04+05 added `window.openCreateProposalModal = openCreateProposalModal` to both views' `attachWindowFunctions()` calls ‚Äî establishing the assumption that this binding persists for the view session. But the delete in `closeProposalModal()` destroys it whenever the user closes a Proposal Detail modal while the view is still active.

Concrete reproduction:
1. service-detail for a service at `For Proposal` with no proposals ‚Äî Start Proposal CTA renders.
2. User opens any proposal detail modal (e.g. via notification), which re-registers `window.openCreateProposalModal` at line 1637.
3. User closes the detail modal ‚Äî `closeProposalModal()` deletes `window.openCreateProposalModal` at line 1674.
4. Start Proposal button is broken ‚Äî onclick calls `undefined()`, no modal opens, no error toast.

**Fix:** Remove `window.openCreateProposalModal` from lines 1637 and 1674 in `proposal-modal.js`. The detail modal Edit button calls `window.openEditProposalModal`, not `window.openCreateProposalModal`. Removing it from the detail-modal register/delete cycle has no functional impact. View-level `attachWindowFunctions()` is the correct owner.

## Warning

### WR-3: `loadProposalCard` `canDrive` includes cross-domain admin roles

**Files:** `app/views/project-detail.js` line ~1505; `app/views/service-detail.js` line ~1258  
**Confidence:** 80

Both `loadProposalCard` functions define `adminRoles = ['super_admin', 'operations_admin', 'services_admin']`. A `services_admin` viewing a project-context card gets `canDrive = true` and sees the Submit for Approval button. The actual write is blocked by Firestore security rules and the modal's `_isCallerAttachedToProposalParent` gate ‚Äî but the user sees an action button that produces an error toast rather than being hidden. A mirror issue affects `operations_admin` on service-context cards.

**Fix:** Scope `adminRoles` by `parentCollection`:
```js
const adminRoles = parentCollection === 'services'
    ? ['super_admin', 'services_admin']
    : ['super_admin', 'operations_admin'];
```
