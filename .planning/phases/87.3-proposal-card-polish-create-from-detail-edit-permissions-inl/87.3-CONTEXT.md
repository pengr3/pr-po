# Phase 87.3: Proposal Card Polish — create from detail, edit permissions, inline card redesign - Context

**Gathered:** 2026-05-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 87.3 polishes the inline proposal card in project-detail.js and service-detail.js and fixes two permission gaps that block the proposal lifecycle for non-admin users. Three areas:

1. **Create from detail** — When a project/service is in `'For Proposal'` status and no proposal exists, show a "Start Proposal" button on the inline card that opens the existing Create Proposal modal pre-filled with the current project. Phase 87.1 deferred this; this phase ships it.

2. **Edit permissions** — Two Firestore rules gaps: (a) the 87.2 field-mask for operations_user is too narrow and blocks `saveProposal()` edits; (b) there is no `create` rule allowing assigned operations_user / services_user to start a proposal. Both are fixed here.

3. **Inline card redesign** — The card header label changes from static "Active Proposal" to the human-readable proposal lifecycle stage; version number is added to the body; visual layout is polished. Card stays minimal (Submit + View buttons only — no new action shortcuts).

### In scope

- `app/views/project-detail.js` — `loadProposalCard`, `renderInlineProposalCard`, `isInProposalRange` gate, "Start Proposal" card path
- `app/views/service-detail.js` — full parity with project-detail.js
- `app/proposal-modal.js` — `openCreateProposalModal` must accept a pre-selected/locked project ID
- `firestore.rules` — proposals block: expand ops_user update field-mask + add ops_user create rule
- `styles/components.css` — `.proposal-inline-card` refinements

### Out of scope

- Any additional action buttons on the inline card (Mark Sent, Request Revision quick actions)
- Edit button shortcut on the inline card — editing stays inside the full modal
- Legacy proposal backfill
- Email/push notifications for proposal creation or editing
- Any change to the proposal Firestore schema

</domain>

<decisions>
## Implementation Decisions

### Create from detail

- **D-01:** When `project_status === 'For Proposal'` AND no proposal is linked, `loadProposalCard()` renders a "Start Proposal" card in place of the "No proposal linked yet." text. The card contains a single `"Start Proposal"` CTA button.
- **D-02:** "Start Proposal" button is gated by the same `canDrive` role set as Phase 87.2 D-05 (super_admin, operations_admin, services_admin, assigned operations_user, assigned services_user). Roles outside this set see the "No proposal linked yet." placeholder; the button is not rendered.
- **D-03:** After the Create Proposal modal closes (cancel or save), the inline card refreshes via `loadProposalCard(parentDocId, parentCollection)`. If a proposal was saved, the card now shows it via `renderInlineProposalCard`. No page navigation.
- **D-04:** When opened from the detail page, the Create Proposal modal has the current project/service **pre-selected and the project dropdown disabled** (cannot change). Target client auto-populates from `project.client_id` per 87.2 D-04 behavior. Implementation: pass the `projectId` as a parameter to `openCreateProposalModal(projectId)` so the modal selects + disables the dropdown.
- **D-05:** "Start Proposal" CTA only appears when `project_status === 'For Proposal'`. For other statuses with no proposal, the card container is hidden.
- **D-06:** Full parity with service-detail.js — the same "Start Proposal" behavior applies when a service is in `'For Proposal'` status with no linked proposal. `parent_collection` is set to `'services'` automatically.
- **D-07:** **Proposal card persists for the full project lifetime once any proposal is linked.** The `isInProposalRange` gate changes: `loadProposalCard()` fires unconditionally in `init()` (not gated on project status). Within `loadProposalCard()`, the rendering logic is:
  - Proposal exists → `renderInlineProposalCard(proposal)` (always, regardless of current project status)
  - No proposal AND `project_status === 'For Proposal'` → render "Start Proposal" card
  - No proposal AND NOT `'For Proposal'` → render nothing (hide the container / return early)

### Edit permissions

- **D-08:** No "Edit" button is added to the inline card. Editing stays inside the full proposal modal (accessed via the "View Proposal" button). The inline card stays minimal.
- **D-09:** **Expand the operations_user / services_user `update` field-mask** in `firestore.rules` proposals block. The 87.2 allowlist (`['comms_log', 'audit_log', 'status', 'current_status_since', 'updated_at']`) is expanded to also include: `title`, `description`, `amount`, `target_client_id`, `target_client_name`, `version`, `attachment_kind`, `attachment_url`, `attachment_filename`. Assignment check and department alignment stay unchanged.
- **D-10:** **Add a `create` rule** for operations_user / services_user in `firestore.rules` proposals block. Conditions mirror the update rule: assignment check (caller UID in parent project/service `personnel_user_ids` via `get()`) + department alignment (`parent_collection == 'projects'` for operations_user; `'services'` for services_user). Without this, the "Start Proposal" button fails for non-admin users.
- **D-11:** The expanded rules block must include a comment explaining the rationale for both the create and update branches (mirrors the Phase 84 NOTIF-12 comment pattern and 87.2 D-20 convention).

### Inline card redesign

- **D-12:** **Replace the static "Active Proposal" header label** with the human-readable proposal lifecycle stage. Mapping:
  - `draft` → "Draft Proposal"
  - `pending_internal` → "For Internal Approval"
  - `pending_client` → "Under Client Review"
  - `for_revision` → "Revision Requested"
  - `approved` (client-approved) → "Client Approved"
  - `loss` → "Loss"
  The colored status badge (`getProposalStatusBadge`) is **removed** from the header since the label communicates the same information. A status badge alongside the label would be redundant.
- **D-13:** **Add version number** to the card body. Display as `v{n}` inline with the proposal ID line (e.g., "PROP-2026-001  v2"). Reads from `proposal.version` field (defaults to `1` if absent on legacy docs).
- **D-14:** **Card buttons unchanged** — only "Submit for Approval" (for canDrive users at draft/for_revision) + "View Proposal" (always). No additional action shortcuts.
- **D-15:** Visual/layout polish — Claude's discretion on exact values. Must match the aesthetic of other detail-page cards in the project/service detail view.

### Claude's Discretion

- **Exact styling** for the redesigned card header (font weight, color, size) — must match the visual language of the existing card body typography
- **Target client name and submitter name** in the card body — these were not explicitly decided. Claude may show `target_client_name` if present (adds useful context without clutter). Submitter name is lower priority; omit if it complicates the data fetch pattern.
- **"Start Proposal" card layout** — whether the CTA is a standalone card (same `.proposal-inline-card` shell with a centered button) or a minimal banner. Must fit visually within the project detail section.
- **Status label → dot color mapping** — if the header label replaces the badge, a small colored dot before the label could retain visual status distinction. Use the existing `AUDIT_ACTION_DOT_COLORS` / `getProposalStatusBadge` color logic as a palette reference.
- **Exact Firestore rules helper naming** — whether to extract `isProposalWriterAssigned()` or inline the predicate; match the style of the existing 87.2 update branch.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`loadProposalCard(parentDocId, parentCollection)`** (`app/views/project-detail.js:1472`) — the function already handles the no-proposal empty state; D-07 expands its logic to (1) always fire unconditionally, (2) render the "Start Proposal" card path, (3) keep the existing `renderInlineProposalCard` path
- **`renderInlineProposalCard(proposal)`** (`app/views/project-detail.js:1357`) — current card HTML builder; D-12/D-13 modify it: replace the `"Active Proposal"` label + remove badge, add version number, apply visual polish
- **`openCreateProposalModal()`** (`app/proposal-modal.js:664`) — currently takes no parameters; D-04 requires it to accept an optional `preselectedProjectId` param; the downstream `showCreateModal(null)` call reads `createModalEditingId` state
- **`getProposalStatusBadge(status)`** (`app/views/proposals.js`) — exported utility; use its color logic as palette reference for D-12 status-label coloring
- **`PROPOSAL_RANGE_STATUSES`** (`app/views/proposals.js`) — currently imported by project-detail.js for the `isInProposalRange` check; this import stays but usage changes per D-07 (gate removed from render path, only used in `loadProposalCard` to decide whether to show the Start Proposal path)
- **`_isCallerAttachedToProposalParent(proposal)`** (`app/proposal-modal.js:161`) — synchronous gate that reads `_parentDocCache`; returns `{ canApprove, canDrive }`. The "Start Proposal" button visibility check in `loadProposalCard` cannot use this (no proposal object yet, no cached parent doc). It must do its own role + assignment check using the current user + the parent project/service doc already loaded in the detail view.
- **Existing Firestore rules update branch** for operations_user — the 87.2 `allow update` predicate with `affectedKeys().hasOnly([...])` + `get()` assignment check is the template for D-09 (expand allowlist) and D-10 (create branch mirrors same structure)

### Established Patterns
- **`loadProposalCard` fire-and-forget**: called inside `renderProjectDetail()` after innerHTML swap, not awaited. D-07 keeps this pattern but removes the `isInProposalRange` guard so it always fires.
- **inline card container as a DOM patch target**: `document.getElementById('proposalInlineCard')?.innerHTML = ...` — safe guard for navigated-away state. Pattern unchanged.
- **`openCreateProposalModal` → `showCreateModal(null)` flow**: `createModalMode = 'create'`, `createModalEditingId = null`, then `_loadModalDropdownData()` (one-shot cache), then `showCreateModal(null)`. D-04 inserts a pre-selection step after dropdown load: if `preselectedProjectId` is passed, set `#proposalCreateProject.value = preselectedProjectId` and `disabled = true`, then trigger the `change` event to auto-fill target client (per 87.2 D-04's existing change listener).
- **Sub-modal confirmation pattern**: existing `openRequestRevisionModal` shape in `app/proposal-modal.js:920+` — reference for any future sub-modal; not needed in this phase.
- **Firestore rules `get()` for parent doc**: Phase 84 notifications block and 87.2 proposals update branch both use `get(/databases/$(database)/documents/$(collection)/$(docId))` — same syntax applies to D-10 create rule.

### Integration Points
- **`project-detail.js` render cycle**: `renderProjectDetail()` sets `isInProposalRange` and conditionally calls `loadProposalCard()`. D-07 removes the conditional so the container always renders and `loadProposalCard()` always fires. The `proposalInlineCard` div must always be in the rendered HTML.
- **`service-detail.js`**: mirrors project-detail.js. Same changes apply; the `loadProposalCard(id, 'services')` call is already in service-detail. Gate removal and Start Proposal card path apply identically.
- **`openCreateProposalModal` window registration**: currently registered during `init()` of the views that use it (home.js, project-detail.js, service-detail.js). The Start Proposal button's onclick calls this same function — no new window function registration needed if the existing one is already available.

### Pitfalls to watch
- **No `_parentDocCache` when rendering "Start Proposal"**: The `canDrive` check in `loadProposalCard` (for whether to show the "Start Proposal" button) cannot call `_isCallerAttachedToProposalParent()` because there is no proposal object and no parent doc cache. The check must be done inline: read `window.getCurrentUser?.()?.role` + compare against the parent doc's `personnel_user_ids` (the parent doc is already loaded into `currentProject` / `currentService` in the detail view's module scope — accessible there).
- **`PROPOSAL_RANGE_STATUSES` gate removal from render**: removing the guard means the `proposalInlineCard` container div is ALWAYS rendered in the project/service detail HTML, even for projects in `'For Inspection'` or `'Completed'` status with no proposal. `loadProposalCard` returning early (no proposal, not `'For Proposal'`) leaves the div empty — verify the container has `display:none` or `height:0` behavior when innerHTML is empty, or explicitly hide it in the early-return path.
- **`openCreateProposalModal` param extension**: adding `preselectedProjectId` must be backward-compatible — all existing callers (home.js Engagements sub-tab) call it with no args. The param should be optional (`function openCreateProposalModal(preselectedProjectId = null)`).
- **Firestore rules `create` vs `update`**: the `create` rule and `update` rule are separate allow blocks. The assignment check `get()` call in both must use consistent path resolution — `resource.data.parent_collection` is available in `update` (the doc exists); in `create` it's `request.resource.data.parent_collection`. Use `request.resource.data.get('parent_collection', 'projects')` for safe access with default.
