# Phase 87.3 Plan 04: CR-1/CR-2 Fix — parent_collection payload and service context preselect Summary

**One-liner:** Propagate parent_collection='services' into Firestore CREATE payload and inject synthetic option to preselect service doc ID in the Create Proposal modal.

## What Was Built

**Task 1 — proposal-modal.js:**

- Added two module-level vars: `_createModalParentCollection = 'projects'` and `_createModalLockedProjectCode = null`
- Extended `openCreateProposalModal` signature: `(preselectedProjectId, onClose, parentCollection='projects', lockedProjectCode=null)`
- Stores both new params immediately after `_createModalOnClose = onClose`
- In the `if (preselectedProjectId)` block: when `lockedProjectCode` is truthy, creates a synthetic `<option>` element (`syntheticOpt`) with `value = preselectedProjectId` and text `lockedProjectCode + ' (Service)'`, then calls `projectSelectEl.insertBefore(syntheticOpt, projectSelectEl.firstChild)`. Then unconditionally sets value, disables, and dispatches change.
- `saveProposal` projectCode resolution changed to: `project?.project_code || _createModalLockedProjectCode || null`
- `saveProposal` CREATE docPayload: added `parent_collection: _createModalParentCollection` after `project_id`
- `closeCreateProposalModal`: resets `_createModalParentCollection = 'projects'` and `_createModalLockedProjectCode = null` before firing the onClose callback

**Task 2 — service-detail.js:**

- Updated Start Proposal button onclick from 2-arg to 4-arg call:
  `window.openCreateProposalModal('${parentDocId}', window._startProposalCallback, 'services', '${service_code}')`
- `project-detail.js` callers untouched — they use 2-arg form which defaults `parentCollection` to `'projects'`
