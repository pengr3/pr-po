<objective>
Three coordinated changes to `project-detail.js`, `service-detail.js`, and `styles/components.css`:

**Area 1 — isInProposalRange gate removal + proposalInlineCard always rendered (D-07):**
Remove the `isInProposalRange` conditional from the HTML template so the `proposalInlineCard` div is ALWAYS present in the rendered output. Remove the `if (isInProposalRange)` guard from the `loadProposalCard` fire-and-forget call so it fires unconditionally. Move the branching logic inside `loadProposalCard` itself.

**Area 2 — loadProposalCard: Start Proposal path (D-01, D-02, D-05, D-07):**
When `snap.empty` (no proposal exists), `loadProposalCard` now checks the parent doc's `project_status`:
- `'For Proposal'` + `canDrive` user → render a "Start Proposal" card (CTA button calling `window.openCreateProposalModal(parentDocId, () => loadProposalCard(...))`)
- `'For Proposal'` + non-canDrive user → render the "No proposal linked yet." placeholder
- Any other status, no proposal → hide the container (`el.style.display = 'none'`) and return

The canDrive check reads `window.getCurrentUser?.()?.role` + checks if the caller's UID is in `currentProject.personnel_user_ids` (for project-detail) or `currentService.personnel_user_ids` (for service-detail). canDrive is true if role is `super_admin`, `operations_admin`, or `services_admin`; also true if role is `operations_user` or `services_user` AND uid is in the parent doc's `personnel_user_ids`.

**Area 3 — renderInlineProposalCard redesign (D-12, D-13, D-14, D-15):**
- Replace `"Active Proposal"` label with a `_proposalStageLabel(status)` helper that maps status strings to human-readable stage names.
- Remove `${getProposalStatusBadge(proposal.status)}` from the header.
- Add a small colored dot before the label (color sourced from `getProposalStatusBadge`'s CSS class lookup — use a lookup map for dot color per status).
- Add version number inline with the proposal ID: `PROP-2026-001 · v${proposal.version || 1}`.
- Gate the Submit for Approval button by `canDrive` (pass as parameter to `renderInlineProposalCard(proposal, canDrive)`).
- Apply visual polish per D-15 (card border, header font weight, label sizing).

Full parity applied to service-detail.js for all the above.

Purpose: Ships the "create from detail" user story and makes the proposal card useful as a status indicator at a glance.

Output: Modified `app/views/project-detail.js`, `app/views/service-detail.js`, and `styles/components.css`.
