## Critical

### CR-1: Month view `xPerDay` places all week-range labels off-screen

**File:** `app/views/project-plan.js` ÔÇö `renderGanttHeaderSvg()`, lines ~1401-1404 and ~1534

In Frappe v1.2.2, Month view sets `config.step = 1` (meaning 1 month unit) and `config.column_width = 120` (px per month column). The formula `xPerDay = configColWidth / configStep = 120 / 1 = 120` treats this as 120 px/day instead of ~4 px/day (~120px ├À 30 days). Week-range labels land ~120x further right than the visible SVG area. Day view and Week view are unaffected (their `config.unit = 'day'` so the formula is correct).

**Fix:**
```javascript
// Guard on config.unit before computing xPerDay:
const xPerDay = (gantt.config.unit === 'day')
    ? configColWidth / configStep
    : configColWidth / 30; // Month view: approximate 30d/month
```

## Warning

### WR-1: D-SCROLL destroy() re-queries DOM for element refs instead of caching at bind time

**File:** `app/views/project-plan.js` ÔÇö `destroy()` lines ~214-225, `bindScrollSync()` lines ~1337-1368

`destroy()` calls `document.getElementById('taskGridRail')` and `document.querySelector('.gantt-pane')` to find elements to remove listeners from. If the router replaces the DOM before calling `destroy()`, these queries return `null` and `removeEventListener` is silently skipped. Ultimately safe (orphaned listeners are GC'd with the DOM node) but fragile.

**Fix (low urgency):** Cache element refs in `_syncRailEl` / `_syncPaneEl` at bind time, mirroring the `_ganttScrollClampPane` pattern. `destroy()` then uses the stored refs directly.
