## Accomplishments

- Deleted `renderGanttHeaderOverlay()` (168-line DOM overlay function, Phase 86.3 D-03) — was never wired in and broke on horizontal scroll due to `position: absolute`
- Added `renderGanttHeaderSvg()` — injects `<text class="gantt-header-svg-label">` elements directly into Frappe's `<svg>`; labels scroll with the timeline because they are children of the SVG

- Month view: week-range sub-labels ("03 May - 09", "10 - 16", "31 - 06 Jun") centered per 7-day column
- Week view: skipped — Frappe native "11 May" / "18 May" labels confirmed sufficient (Week view HTML injection added in debug session for day-of-week row)

- Debug session (bb89717, de32df9): fixed `infinite_padding`, `gantt.config.*` vs `gantt.options.*`, requestAnimationFrame scroll-sync, gantt=null empty-tasks guard, gridInsertRowAbove date defaults

## Decisions Made

- SVG injection (not DOM overlay) is the correct approach for Gantt header labels — DOM overlay at `position:absolute` inside `.gantt-pane` stays pinned to viewport left while the SVG content scrolls, causing all 9 D-03 UAT failures. SVG children scroll automatically with their parent SVG viewport.
- `labelY` is derived at runtime from the max y-value of existing Frappe `<text>` elements + 14px, rather than a hardcoded constant — this adapts to Frappe's header geometry.
- `infinite_padding: false` is required in the Gantt constructor — Frappe's infinite-padding mode offsets all x-coordinates by a large constant, causing label positions to be wildly incorrect.
- `gantt.config.step` and `gantt.config.column_width` are the correct runtime properties; `gantt.options.*` was being used in 4 functions but returns `undefined` at runtime.
- Some UAT items remain (scroll jank, floor regression, label alignment) and are tracked for a follow-up fix session.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] infinite_padding: false missing from Gantt constructor**
- **Found during:** Debug session after checkpoint (commit bb89717)
- **Issue:** Frappe infinite-padding mode offset all SVG x-coordinates by a large constant, rendering labels at wrong positions
- **Fix:** Added `infinite_padding: false` to the Gantt constructor options object in `renderGantt()`
- **Files modified:** `app/views/project-plan.js`
- **Committed in:** bb89717

**2. [Rule 1 - Bug] gantt.options.step / gantt.options.column_width used instead of gantt.config.*  **
- **Found during:** Debug session after checkpoint (commit bb89717)
- **Issue:** `gantt.options.step` and `gantt.options.column_width` return `undefined` at runtime in Frappe Gantt v1.2.2; the correct live properties are `gantt.config.step` and `gantt.config.column_width`. This affected x-coordinate calculation in renderGanttHeaderSvg, renderTodayLine, computeGanttFloorX, and scrollToToday — all producing NaN or 0 positions.
- **Fix:** Replaced `gantt.options.*` with `gantt.config.*` across 4 functions
- **Files modified:** `app/views/project-plan.js`
- **Committed in:** bb89717

**3. [Rule 1 - Bug] _syncingScroll reset without requestAnimationFrame caused ping-pong scroll**
- **Found during:** Debug session after checkpoint (commit bb89717)
- **Issue:** Synchronous `_syncingScroll = false` reset allowed rapid successive scroll events to re-enter before the browser had settled, causing scroll ping-pong jank
- **Fix:** Wrapped `_syncingScroll = false` reset in `requestAnimationFrame()`
- **Files modified:** `app/views/project-plan.js`
- **Committed in:** bb89717

**4. [Rule 2 - Missing Critical] gantt = null missing from empty-tasks guard in renderGantt()**
- **Found during:** Debug session after checkpoint (commit bb89717)
- **Issue:** When the snapshot returned zero tasks, renderGantt() returned early but left the stale `gantt` object in place; subsequent calls to renderGanttHeaderSvg() and renderTodayLine() would attempt to read from the stale object
- **Fix:** Added `gantt = null` to the empty-tasks early return in renderGantt()
- **Files modified:** `app/views/project-plan.js`
- **Committed in:** bb89717

**5. [Rule 1 - Bug] gridInsertRowAbove created tasks without start_date/end_date**
- **Found during:** Debug session after checkpoint (commit bb89717)
- **Issue:** Newly inserted rows above had no date fields, causing Frappe "start can't be after end" error when the Gantt tried to render them
- **Fix:** gridInsertRowAbove now sets `start_date` and `end_date` on the newly inserted task document
- **Files modified:** `app/views/project-plan.js`
- **Committed in:** bb89717
