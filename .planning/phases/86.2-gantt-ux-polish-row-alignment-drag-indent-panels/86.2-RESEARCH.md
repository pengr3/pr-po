# Phase 86.2: Gantt UX Polish — Research

**Researched:** 2026-05-06
**Domain:** Frappe Gantt v1.2.2 SVG layout, CSS grid height synchronisation, HTML5 drag-and-drop isolation, resizable panel dividers, keyboard UX
**Confidence:** HIGH

---

## Summary

Phase 86.2 fixes 10 discrete UX defects discovered during Phase 86.1 UAT. All defects are isolated to a single file (`app/views/project-plan.js`) plus its companion style block in `styles/views.css`. No Firestore schema changes, no new collections, no security rule changes are required — this is purely a rendering and event-handling polish pass.

The most technically complex item is the **row alignment fix** (defect 1): Frappe Gantt v1.2.2 renders each task row at exactly `bar_height + padding = 42px` (using the current `bar_height: 24, padding: 18` options). The grid's `.tg-row` elements have no explicit height, so they collapse to ~27px from input line-height. Setting `.tg-row { height: 42px }` and aligning the thead height to Frappe's `header_height` (85px by default) fully resolves misalignment.

The **resizable panel divider** (defect 8) requires a pure CSS/JS drag handle between `.task-grid-rail` and `.gantt-pane`. No library is needed — a `<div class="plan-divider">` with `mousedown`/`mousemove`/`mouseup` handlers updates `grid-template-columns` in real time. This is the only structural HTML change.

The **drag isolation fix** (defect 3) is already partially solved by the existing `isDescendant` guard. The remaining issue is that HTML5 drag's `dragstart` fires even when a user clicks an input cell — the current guard `if (e.target.tagName === 'INPUT')` prevents this but has an edge case when the user clicks outside an input but within the row. Tightening the drag handle to only fire from a dedicated handle element (the `#` column or a drag icon) fully isolates it.

**Primary recommendation:** Plan the 10 defects in 3 sequential plans: Plan 01 for structural/CSS fixes (row alignment, Back button, full-width Gantt, monthly header), Plan 02 for interaction fixes (drag isolation, Enter key, smart indent, delete modal), Plan 03 for resizable panels. Each plan is self-contained with no cross-plan dependencies other than the shared file.

### Pattern 1: Row Height Synchronisation
**What:** Frappe Gantt v1.2.2 calculates each task bar's y-position as:
```
y = config.header_height + options.padding/2 + task._index * (bar_height + padding)
```
With default `bar_height: 24, padding: 18`, each row occupies **42px** in the SVG.
The grid's `.tg-row` must be exactly 42px tall for bars to align with their labels.

**When to use:** Whenever Frappe Gantt options `bar_height` or `padding` change.

**Fix — CSS:**
```css
/* Source: Frappe Gantt v1.2.2 source (verified via CDN) */
.tg-row {
    height: 42px;  /* MUST match bar_height(24) + padding(18) = 42px */
    border-bottom: 1px solid var(--gray-200, #dadce0);
}
```

**Fix — align thead:** The Frappe header height is `upper_header_height(45) + lower_header_height(30) + 10 = 85px`. The grid thead does not need to match exactly (it shows column names, not dates), but the top of the first task row in both panes must be visually aligned. The simplest approach: give `.task-grid thead` an explicit `height: 85px` so it matches the SVG header region. [VERIFIED: Frappe v1.2.2 CDN source]

### Pattern 3: Drag Isolation — Moving B Must Not Shift C
**What:** HTML5 drag-to-reorder currently fires `dragstart` on the full `<tr>` row element (`draggable="true"`). When B is dragged, any row below B that happens to be in drag-over state may receive drag events. The existing `_gridDragOverHandler` correctly guards against descendant drops, but the visual "shift" the user sees is likely the `dragover` visual hint being applied incorrectly.

**Root cause:** The `draggable="true"` attribute is on the full `<tr>`, which means any click (including on inputs) can accidentally initiate a drag. The drag-start guard `if (e.target.tagName === 'INPUT' || e.target.classList.contains('tg-input')) { e.preventDefault(); return; }` is present but only fires after `dragstart` has already triggered the OS-level drag cursor.

**Fix:** Restrict `dragstart` to only fire when the user drags the row-number `<td class="tg-rn">` cell (acts as a drag handle). Move `draggable="true"` off the `<tr>` and onto just the `#` cell, or add a `⣿` drag handle icon to the row-number column and check `e.target.closest('.tg-rn')` in `_gridDragStartHandler`. This ensures dragging input cells never triggers a row reorder.

```javascript
// In _gridDragStartHandler:
_gridDragStartHandler = function(e) {
    const handle = e.target.closest('.tg-rn');
    if (!handle) { e.preventDefault(); return; }  // only drag from # cell
    const row = handle.closest('.tg-row');
    if (!row || row.dataset.taskId === '__new__') { e.preventDefault(); return; }
    _draggedTaskId = row.dataset.taskId;
    row.classList.add('tg-dragging');
    e.dataTransfer.effectAllowed = 'move';
    e.dataTransfer.setData('text/plain', _draggedTaskId);
};
```
[ASSUMED — exact drag target is a design decision, but this is the standard drag-handle approach]

### Pattern 4: Restrict Gantt Interactions to Allowed Set
**What:** The current Frappe Gantt instance shows hover handles for bar-extend (left/right handles) and progress-extend on all bars. The ROADMAP restricts interactions to: drag-predecessor (SVG overlay from 86.1), bar-extend, bar-drag, progress-extend. **Fix:** Set `popup: false` in the Gantt constructor options to suppress any click-triggered popup. The `on_click` handler is already a no-op. No other Frappe interactions need to be disabled — bar-drag, bar-extend, and progress-extend are the native Frappe interactions and are already the only ones active.     lower_text: "MMMM",      // Full month name — "January", "February", etc.

MS Project shows month names large at the top with smaller dates below. The Frappe Month mode is functionally equivalent but the visual hierarchy is inverted from MS Project.

**Fix options:**
1. Accept Frappe's native Month view as-is (months in lower row, years above) — no code change
2. Override via custom `view_modes` option passed to Gantt constructor to swap upper/lower text

The ROADMAP says "monthly header format matching MS Project style". MS Project Month view shows: upper row = year (e.g., "2026"), lower row = month names (e.g., "January"). This is EXACTLY what Frappe Month mode already produces. So the defect may have been a misread during UAT — verify with user. ### Pattern 6: Smart Indent Inheritance on New Row
**What:** When the user types a name in the empty trailing row and presses Enter, `handleNewRowCommit` creates the task with `parent_task_id: null` (root level). The user expects the new row to inherit the `parent_task_id` of the row immediately above it (so if row 5 is indented under task A, typing a new row below it should also be under task A).

**Current code (project-plan.js line 510):** `parent_task_id: null`

**Fix:**
```javascript
// In handleNewRowCommit — compute parent from the last visible (non-empty) row
const sorted = tasks.slice().sort((a, b) => (a.row_order ?? Infinity) - (b.row_order ?? Infinity));
const rowAbove = sorted[sorted.length - 1];  // last task in order
const inheritedParent = rowAbove?.parent_task_id ?? null;

const docData = {
    // ...
    parent_task_id: inheritedParent,  // inherit from row above instead of null
    // ...
};
```

**For `gridInsertRowAbove`:** Already correctly inherits `parent_task_id` from the row being displaced (line 707: `parent_task_id: t.parent_task_id || null`). No change needed there.

### Pattern 7: Delete Confirmation — Modal vs Inline Tooltip
**What:** The ROADMAP says "delete-confirmation modal for indented tasks" but Phase 86.1 CONTEXT.md D-05 says "inline confirm — no modal". These conflict.

**Resolution analysis:**
- Phase 86.1 CONTEXT.md D-05 was the design decision at plan time: "no separate modal for delete confirmation — keep everything inline"
- Phase 86.2 ROADMAP explicitly says "delete-confirmation modal for indented tasks" — this is the UAT feedback that the inline tooltip was too easy to dismiss/miss for high-stakes subtree deletes

**Decision:** The ROADMAP as the latest user expression overrides the Phase 86.1 design decision. Use a proper modal overlay (not a full-screen backdrop, but a centered confirm dialog) for deletes that affect subtrees. ### Pattern 8: Resizable Panel Divider
**What:** A drag handle between `.task-grid-rail` and `.gantt-pane` that lets the user resize the split. On `mousedown` on the divider, `mousemove` events update the `grid-template-columns` value on `.plan-split-pane`.

**HTML change in render():**
```html
<div class="plan-split-pane" id="planSplitPane">
    <div class="task-grid-rail" id="taskGridRail" ...></div>
    <div class="plan-divider" id="planDivider"></div>
    <div class="gantt-pane"><div id="ganttPane"></div></div>
</div>
```

**CSS:**
```css
.plan-split-pane {
    display: grid;
    grid-template-columns: var(--plan-grid-split, 35%) 4px 1fr;  /* divider is 4px column */
    flex: 1 1 auto;
    min-height: 0;
}
.plan-divider {
    background: var(--gray-200, #dadce0);
    cursor: col-resize;
    user-select: none;
}
.plan-divider:hover, .plan-divider.dragging { background: var(--primary, #1a73e8); }
```

**JS in init() — attach divider drag:**
```javascript
function initPanelResize() {
    const divider = document.getElementById('planDivider');
    const splitPane = document.getElementById('planSplitPane');
    if (!divider || !splitPane) return;
    let dragging = false;
    divider.addEventListener('mousedown', (e) => {
        dragging = true;
        divider.classList.add('dragging');
        e.preventDefault();
    });
    document.addEventListener('mousemove', (e) => {
        if (!dragging) return;
        const rect = splitPane.getBoundingClientRect();
        const pct = Math.max(15, Math.min(75, ((e.clientX - rect.left) / rect.width) * 100));
        splitPane.style.gridTemplateColumns = `${pct}% 4px 1fr`;
    });
    document.addEventListener('mouseup', () => {
        if (!dragging) return;
        dragging = false;
        divider.classList.remove('dragging');
    });
}
```

**Cleanup in destroy():**
```javascript
// Remove the mousemove/mouseup listeners added in initPanelResize()
// Use a module-scope flag or named function references for cleanup
```

**Key pitfall:** The `mousemove` and `mouseup` listeners are on `document`, not on the divider. They must be removed in `destroy()` to avoid memory leaks. Store them as module-scope function references.

### Pattern 10: Enter Key Commits Duration and Resources Cells
**What:** Currently `handleNewRowKeydown` fires only for the `__new__` empty row's name input. Duration and Resources inputs have no Enter key handler — the user must Tab or click away to commit.

**Fix:** In `bindGridEvents`, add a `keydown` handler (delegated, same as other handlers):
```javascript
_gridKeydownHandler = function(e) {
    if (e.key !== 'Enter') return;
    const input = e.target;
    if (!input.classList.contains('tg-input')) return;
    const taskId = input.dataset.taskId;
    const col = input.dataset.col;
    if (taskId === '__new__') return; // handled by handleNewRowKeydown already
    if (col !== 'duration' && col !== 'resources') return; // only these two per spec
    e.preventDefault();
    input.blur(); // blur triggers the existing handleGridCellBlur via capture-phase listener
};
container.addEventListener('keydown', _gridKeydownHandler);
```

**Cleanup:** Add `_gridKeydownHandler` to module-scope vars and remove in `destroy()`.

**Note:** The spec says "Enter key commits Duration and Resources cells". It does NOT say Enter commits Name, Start, End, or Predecessors. Those remain blur-only. This is deliberate — Name commits on the `__new__` row via `handleNewRowKeydown`; Start/End date pickers commit via `change` event.

### Anti-Patterns to Avoid
- **Forking Frappe Gantt:** All these fixes are achievable via CSS overrides and JS wrappers. Never modify `frappe-gantt.umd.js` directly.
- **Setting `pointer-events: none` on all Gantt bars:** Would disable bar-drag and bar-extend. Only apply `pointer-events: none` to specific child elements (e.g. `.parent-summary-bar .handle`).
- **Using `window.resize` for panel sync:** Don't re-render the Gantt on resize events — Frappe handles its own scroll. Only update `gridTemplateColumns`.
- **Adding document-level handlers without cleanup:** All `document.addEventListener` calls in panel resize MUST have corresponding `removeEventListener` in `destroy()`.
- **Making Delete modal fire for leaf tasks:** Only show the modal for tasks that have children. Leaf tasks delete immediately (consistent with the existing pattern and user expectation).

## Common Pitfalls

### Pitfall 1: Grid Row Height Must Be Pixel-Exact
**What goes wrong:** Setting `.tg-row { height: 42px }` is correct, but `td` elements inside the row may have padding that pushes the row taller. The `height` property on `<tr>` sets a minimum, but padding on `<td>` still adds to it.
**Why it happens:** `height` on `<tr>` is a minimum height in CSS — cell padding can exceed it.
**How to avoid:** Use `box-sizing: border-box` on all `<td>` elements. Remove bottom padding from `<td>` cells and rely on the row height: set `td { padding-top: 0; padding-bottom: 0; }` and let input padding handle vertical spacing.
**Warning signs:** Row heights vary as task count grows.

### Pitfall 2: Grid-Template-Columns with a 4px Divider Column
**What goes wrong:** The current `.plan-split-pane` has `grid-template-columns: 35% 1fr` (2 children). Adding a divider `<div>` as a third child without updating the CSS means the divider renders as an extra `1fr` column, breaking the layout.
**How to avoid:** Change to `grid-template-columns: var(--plan-grid-split, 35%) 4px 1fr` in the same commit as adding the `<div class="plan-divider">` to render().
**Warning signs:** Gantt pane appears very narrow after adding divider.

### Pitfall 3: Document mousemove Handler Leaks on Tab Switch
**What goes wrong:** The panel resize attaches `mousemove` and `mouseup` to `document`. The router does NOT call `destroy()` between tab switches within the same view — but there is no tab switching in project-plan.js (it's a standalone view). However, if the user navigates away and comes back, `init()` runs again and re-attaches, while the old handlers from the previous visit may not have been cleaned up if the route change triggered `destroy()` correctly.
**How to avoid:** Store handler refs in module-scope variables (`_resizeMouseMoveHandler`, `_resizeMouseUpHandler`) and null them after `removeEventListener` in `destroy()`. Pattern: mirror `_gridInputHandler` / `_gridDragStartHandler` cleanup in destroy().
**Warning signs:** Resizing works on first visit, broken on second visit.

### Pitfall 4: Frappe Gantt Snapshot Re-Fires After Each Firestore Write
**What goes wrong:** Every `updateDoc` triggers the `onSnapshot` listener, which calls `renderTaskGrid()` + `renderGantt()`. The `renderGantt()` calls `gantt.refresh()` (not `new Gantt()`), which preserves scroll/zoom. But the panel resize state (CSS variable on `.plan-split-pane`) is on the DOM element's `style` attribute — `renderGantt()` does NOT modify `.plan-split-pane`, so the panel width is preserved. No issue here.
**Warning signs:** Panel width resets to 35% after every task edit.

### Pitfall 5: Smart Indent Inheritance for gridInsertRowAbove vs handleNewRowCommit
**What goes wrong:** The user may expect "Insert Row Above" (right-click menu) to also inherit the parent of the row above, but the current implementation correctly inherits from the displaced row's parent (which IS the row above). `handleNewRowCommit` (empty trailing row) is the ONLY broken case.
**How to avoid:** Only fix `handleNewRowCommit`. Do not change `gridInsertRowAbove`.

### Pitfall 6: Back Button Destination Ambiguity
**What goes wrong:** `href="#/projects"` navigates to the projects LIST page. The ROADMAP says "Back button navigates to project list" — so this is the INTENDED destination. If the user expected to go back to project DETAIL, the href would be `#/projects/detail/${projectCode}`.
**How to avoid:** The ROADMAP language "navigates to project list" is the specification. Use `href="#/projects"` as-is. The bug during UAT may have been that the `‹ Back` link was not visible or not styled as a link. Verify the link renders correctly in the toolbar.
**Warning signs:** Back button present in code but invisible in UI (check `.plan-back-link` CSS).

### Panel Resize JS (in init(), called after DOM mount)
```javascript
let _resizeMouseMoveHandler = null;
let _resizeMouseUpHandler = null;
let _resizeDragging = false;

function initPanelResize() {
    const divider = document.getElementById('planDivider');
    const splitPane = document.getElementById('planSplitPane');
    if (!divider || !splitPane) return;

    divider.addEventListener('mousedown', (e) => {
        _resizeDragging = true;
        divider.classList.add('dragging');
        e.preventDefault();
    });

    _resizeMouseMoveHandler = function(e) {
        if (!_resizeDragging) return;
        const rect = splitPane.getBoundingClientRect();
        const pct = Math.max(15, Math.min(75, ((e.clientX - rect.left) / rect.width) * 100));
        splitPane.style.gridTemplateColumns = `${pct.toFixed(1)}% 4px 1fr`;
    };

    _resizeMouseUpHandler = function() {
        if (!_resizeDragging) return;
        _resizeDragging = false;
        divider.classList.remove('dragging');
    };

    document.addEventListener('mousemove', _resizeMouseMoveHandler);
    document.addEventListener('mouseup', _resizeMouseUpHandler);
}
```

### Destroy cleanup for panel resize
```javascript
// In destroy() — add after existing cleanup:
if (_resizeMouseMoveHandler) document.removeEventListener('mousemove', _resizeMouseMoveHandler);
if (_resizeMouseUpHandler)   document.removeEventListener('mouseup',   _resizeMouseUpHandler);
_resizeMouseMoveHandler = null;
_resizeMouseUpHandler   = null;
_resizeDragging = false;
```
