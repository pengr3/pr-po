**DEFECT 3 fix:** The current `_gridDragStartHandler` fires for any click on the `<tr>` (since `draggable="true"` is on the row element). The INPUT guard is insufficient â€” clicking the row number area or any non-input cell still initiates a drag, and the OS drag cursor flickers even when the handler calls `e.preventDefault()`.

The fix: restrict drag to only begin when the user drags the `<td class="tg-rn">` cell (the row number column). This is the standard "drag handle" pattern.

**Step 1 â€” Update `_gridDragStartHandler` in `bindGridEvents`:**

Find the `_gridDragStartHandler = function(e) {` block (lines ~360â€“373). Replace the entire handler body with:

```javascript
_gridDragStartHandler = function(e) {
    // DEFECT-3 fix: only allow drag from the row-number (.tg-rn) cell (drag handle)
    const handle = e.target.closest('.tg-rn');
    if (!handle) { e.preventDefault(); return; }
    const row = handle.closest('.tg-row');
    if (!row || row.dataset.taskId === '__new__') { e.preventDefault(); return; }
    _draggedTaskId = row.dataset.taskId;
    row.classList.add('tg-dragging');
    e.dataTransfer.effectAllowed = 'move';
    e.dataTransfer.setData('text/plain', _draggedTaskId);
};
```

The old INPUT guard lines are removed â€” they are no longer needed because the `.tg-rn` check already excludes all input clicks.

**Step 2 â€” Verify row HTML template still has `draggable="true"` on `<tr>`:**
The `<tr class="tg-row" data-task-id="..." draggable="true">` line (around line 256) MUST keep `draggable="true"`. Do NOT move it to the `<td>`. HTML5 drag events always originate from the `draggable` element, but `e.target` will be the innermost clicked element â€” so `e.target.closest('.tg-rn')` correctly identifies whether the drag started on the # column.

No change needed to the row HTML template itself.
