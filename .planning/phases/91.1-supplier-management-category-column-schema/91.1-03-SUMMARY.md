## Decisions Made

- **Categories column at position 2, not last.** Matches the inline-edit row column position Plan 02 already shipped. If this plan had placed the column last (after Phone), Plan 02's edit-row template would have been out-of-position relative to the header — forcing a follow-up plan to re-shuffle one of the two branches. Plan 02's SUMMARY explicitly flagged "coordinate now, avoid re-work later" as the rationale; this plan honors that contract.
- **Read-only display pills reuse `.personnel-pill` class without `.pill-remove`.** Visual parity with the inline-edit row's pill control gives users a single mental model ("these things are categories") regardless of whether they're viewing or editing. Omitting the `.pill-remove` button child explicitly signals non-interactive, and reuses the existing CSS verbatim — no new `.category-pill-readonly` class introduced. Consistent with Plan 01's CSS-reuse decision.
- **`renderCategoryDisplay` hoisted locally inside `renderSuppliersTable`, not module-scope.** The helper's sole call site is the display-row branch of this function. Module-scope would have polluted the namespace, and the closure overhead is zero since the helper captures nothing it doesn't already see via parameter. The trade-off is rebuilding the function on every `renderSuppliersTable` call, which is negligible (~ns) at supplier-table scale (hundreds of rows max).
- **Em-dash literal `<span style="color: #94a3b8;">—</span>` is a static HTML entity reference; no `escapeHTML` call.** The em-dash character itself comes from a JavaScript string literal — no user data flows through it. The threat model (T-91.1-10) explicitly accepts this as safe. Per CONTEXT.md D-06, the color must match existing muted-text usage (`#94a3b8`).
- **`applySupplierSearch` restructured into a two-stage pipeline rather than a single inline predicate.** The original single-stage version inlined all match conditions into one `.filter(s => ...)` predicate; adding the categories branch + uncategorized filter would have produced a >10-line single-expression predicate that's hard to reason about. The two-stage form (named `result` accumulator with Stage 1 + Stage 2 comment headers) makes the composition semantics explicit: AND, search narrows first, uncategorized further narrows. Easier to extend in the future (e.g., a multi-select category filter dropdown deferred per CONTEXT.md "Deferred Ideas" would slot in as Stage 3).
- **`typeof c === 'string'` guard inside the `.some(...)` callback even though Plan 01 already filters non-strings.** Defense-in-depth. Plan 01's `getKnownCategories` filter operates on the read side of the dropdown population path; this guard sits on the read side of the search path. The two paths are independent; a future bug that bypasses Plan 01's filter (e.g., direct Firestore manipulation by a power user) shouldn't crash search on `c.toLowerCase()` if `c` happens to be a number or `null`. Threat T-91.1-12 in the plan's threat register explicitly calls this out as a mitigate-via-guard item.
- **Filter-bar `align-items: flex-end`.** Without this, the new checkbox column (which has a single-line label and no separate "Search" heading) would have aligned to the top of the flex row while the search-input column extends taller (label + input). Adding `flex-end` aligns both columns at the bottom — visually consistent without per-element margin hacks.
- **D-08 verified by grep, not by edit.** The plan explicitly required a verify-only check at sub-step 2.3: confirm the `match /suppliers/{supplierId}` block has no `categories` reference and that `super_admin` + `procurement` roles are both present. The check passed (one `match /suppliers/{supplierId}` at line 354, zero `categories` references in the entire `firestore.rules` file, both roles present at lines 359-361). No edit, no deploy required for this phase. Documented as a permanent record so future readers don't have to re-derive the verification when extending the categories feature.
