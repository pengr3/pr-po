<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Add form Categories pill control → Firestore | New category strings (typed or seed-selected) written to `suppliers/{id}.categories` |
| Edit form Categories pill control → Firestore | Existing or new category strings updated on `suppliers/{id}.categories` |
| Edit form clientside validation | Bypassable via DevTools (per ASVS L1 — server-side authority is Firestore rules) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-91.1-06 | Tampering (validation bypass) | addSupplier client-side ≥1 check | accept | Per D-08, Firestore rules at firestore.rules:354-362 do not enforce field-level constraints; a malicious procurement-role user could bypass the ≥1 check via DevTools and write an empty categories array. Documented business risk — they could do the same by clicking "Edit" → save with empty. Not a security issue, just a process one. No rules update planned. |
| T-91.1-07 | Injection | Saved categories rendered later via Plan 03's display row | mitigate | All read-side rendering goes through escapeHTML in Plan 03; pill render in inline-edit row already uses Plan 01's escapeHTML-guarded renderCategoryPillControl |
| T-91.1-08 | Data integrity | Firestore writes from this plan include `updated_at` / `created_at` | mitigate | Existing patterns preserved (created_at: new Date().toISOString() on Add; updated_at on Edit); categories field added alongside without disturbing timestamps |
| T-91.1-09 | DoS (state leak) | Per-row pill state in `_categoryPillState` Map keyed by supplier.id | mitigate | cancelEdit and saveEdit both call clearCategoryPillState; even if the user rapidly toggles edit rows the Map grows at most by O(rows-edited-this-session) and is reset on view destroy() (Map exists at module scope; gets garbage-collected when the procurement view module is replaced) |

</threat_model>
