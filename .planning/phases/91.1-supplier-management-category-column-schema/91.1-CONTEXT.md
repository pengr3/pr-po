# Phase 91.1: Supplier Management — Category Column Schema Addition — Context

**Gathered:** 2026-05-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 91.1 extends the `suppliers` schema with a multi-category tagging capability and surfaces it across the Supplier Management sub-tab (Procurement → Suppliers, inserted by Phase 91). Suppliers can belong to many categories (e.g., a contractor doing both Electrical and Lighting), and the picker self-grows: a hardcoded seed list of 20 trade categories plus any new value the user types via "Add new..." becomes available on the next render. Legacy supplier records remain untagged at deploy time — the team will encode them manually post-deploy.

### In scope
- Schema: add `categories: string[]` (array of strings) to `suppliers` documents on create/update
- Add Supplier form: replace today's 4 fields (Name / Contact / Email / Phone) with a 5th field — a tag-pill input control for Categories; **required (≥1 pill) on new submissions**
- Edit Supplier form: same tag-pill control; **legacy/blank rows may be saved without a category** (gradual cleanup, not forced)
- Suppliers table: add a "Categories" column right after "Supplier Name"; pills render inline; blank rows show a dim em-dash (`—`)
- Dropdown population: derived from `distinct(suppliers.flatMap(s => s.categories || []))` ∪ `SEED_CATEGORIES`, sorted A-Z, deduped
- "Add new..." in dropdown reveals a small text input; on save, the typed value becomes a pill on the supplier — no separate collection write
- Toolbar: existing search input extended to also match category strings; new control "Show uncategorized only" filters the table to suppliers with empty/missing `categories`
- Firestore Security Rules: NO update needed — existing `match /suppliers/{supplierId}` rules (firestore.rules:354-362) have no field-level write restrictions, so `categories` writes are already permitted under `super_admin` and `procurement` roles
- Procurement sub-tab permission gate: continue using existing `canEditTab('procurement')` check OR align with Phase 91's `procurement_suppliers` sub-tab key — planner to decide based on what existing Phase 91 suppliers UI uses

### Out of scope
- No migration script to bulk-tag existing suppliers — they remain blank until manually encoded post-deploy
- No category management UI (rename, merge, delete categories) — categories are derived, not authored
- No per-category metadata (notes, default tax %, color) — `categories` is plain `string[]`
- No multi-select category filter dropdown in the toolbar (deferred — search extension only)
- No new Firestore collection (`supplier_categories`) — derivation pattern only
- No changes to supplier search pagination or the Edit-mode 4-field inline editor layout (only adds the new field)
- Field naming on existing `suppliers` documents (the ROADMAP success criterion said `category` singular; superseded by `categories` plural array — see Deferred Ideas section if a singular shim is ever needed elsewhere)

</domain>

<decisions>
## Implementation Decisions

### D-01 — Categories model: array of strings, multi-tag

The schema is `categories: string[]` on `suppliers` documents (array, plural). A supplier can carry zero, one, or many category strings. Examples:

```js
// New supplier — required ≥1
{ supplier_name: 'ACME', categories: ['Electrical', 'Lighting'], ... }

// Legacy supplier — array may be absent or empty
{ supplier_name: 'Old Co', /* no categories field */ ... }
{ supplier_name: 'Old Co', categories: [], ... }
```

Reads should treat `undefined`, `null`, and `[]` as equivalent "uncategorized" — defensive: `const cats = supplier.categories || [];`

### D-02 — UI: tag-pill input with "Add new..." overflow

The Add and Edit forms render a single tag-pill control in place of a traditional dropdown:

```
Categories *
[ Materials × ] [ Subcon × ] [ + Add... ▾ ]
                                 Materials
                                 Subcontractor
                                 Electrical
                                 ...
                                 ─── Add new... ───
```

Behavior:
- Existing pills show with an `×` to remove
- Clicking `+ Add...` opens a dropdown of all known categories sorted A-Z (see D-04 for the source)
- Picking a category adds it as a pill
- Picking "Add new..." reveals a small inline text input; user types a new category name + commits (Enter or a small "Add" button), which becomes a new pill
- Add Supplier: **must have at least one pill to submit**; client-side validation blocks empty submissions with a toast
- Edit Supplier: **no minimum on save** — a legacy supplier with zero categories may still be saved (e.g., user only updates phone)

### D-03 — Seed list (20 categories, hardcoded constant)

A `SEED_CATEGORIES` constant lives in `app/views/procurement.js` (or `app/utils.js` if reused elsewhere later). Initial contents — **exact strings, exact order in source, displayed sorted A-Z in the UI**:

```js
const SEED_CATEGORIES = [
  'Civil / Structural',
  'Architectural Works',
  'Flooring',
  'Ceiling Systems',
  'Wall Finishes',
  'Doors, Windows & Glass',
  'Joinery / Carpentry',
  'Plumbing / Sanitary',
  'Bathroom Fixtures & Accessories',
  'Electrical',
  'Lighting',
  'Mechanical / HVAC',
  'Fire Protection',
  'Security Systems',
  'Smart Technology & Automation',
  'Landscaping',
  'Roofing Systems',
  'Hardware & General Construction Supplies',
  'Construction Service',
  'Professional Service',
];
```

Spacing, slashes, commas, ampersands — preserve verbatim. The UI sorts these alphabetically on render.

### D-04 — Dropdown options: derived (no new collection)

The full pickable list for the dropdown is computed on each render of the form (or once per supplier-data refresh):

```js
function getKnownCategories(suppliersData) {
  const fromSuppliers = suppliersData.flatMap(s => Array.isArray(s.categories) ? s.categories : []);
  const all = [...new Set([...SEED_CATEGORIES, ...fromSuppliers])];
  return all.sort((a, b) => a.localeCompare(b));
}
```

This means "Add new... = Welding" on supplier ACME → next time any supplier's Add/Edit form opens, "Welding" appears in the dropdown automatically because ACME's `categories` array now contains it. **No separate `supplier_categories` collection is created.** No new Firestore Security Rules are written.

Sort: `localeCompare` (case-insensitive alphabetical). Dedupe via `Set`.

### D-05 — Suppliers table: new "Categories" column, position 2

Column order after this phase: **Supplier Name | Categories | Contact Person | Email | Phone | Actions**.

Rendering:
- Each category becomes a pill (reuse an existing pill/badge style if `components.css` already has one; otherwise a small new style class scoped to the suppliers table)
- Blank/missing categories: dim em-dash (`—`) in `color: #94a3b8` or similar to match existing muted-text style
- Inline edit row (`editingSupplier === supplier.id`): the existing 4 `<input>` cells gain a 5th tag-pill input in the same control style as the Add form

### D-06 — Legacy UX: em-dash + "Show uncategorized only" toolbar control

To support the manual post-deploy encoding pass:

- Blank `categories` cells render as `—` (em-dash, muted color)
- The supplier toolbar (`<div class="filter-bar">` at procurement.js:1817) gains one new control next to the search input — a checkbox or compact toggle labeled **"Show uncategorized only"**
- When checked: filter the table to suppliers where `(supplier.categories || []).length === 0`
- When unchecked: standard search/pagination behavior
- The control composes with the search input: search term AND uncategorized filter both apply

### D-07 — Search: extended to match categories

`applySupplierSearch()` (procurement.js:4609) currently matches `supplier_name` and `contact_person` against a lowercased search term. Extend to also match if **any** category in `supplier.categories` includes the search term (case-insensitive):

```js
filteredSuppliersData = !term
  ? [...suppliersData]
  : suppliersData.filter(s =>
      (s.supplier_name && s.supplier_name.toLowerCase().includes(term)) ||
      (s.contact_person && s.contact_person.toLowerCase().includes(term)) ||
      (Array.isArray(s.categories) && s.categories.some(c => c.toLowerCase().includes(term)))
    );
```

No new filter dropdown is added to the toolbar — only the existing search input and the new "Show uncategorized only" control from D-06.

### D-08 — Firestore Security Rules: no change

`firestore.rules:354-362` already permits `create` / `update` / `delete` on `/suppliers/{supplierId}` for `super_admin` and `procurement` roles, with no field-level restrictions. The new `categories` field is automatically writable under existing rules. **No rules update, no deploy required for this phase.** Planner should still call this out explicitly in the plan as a verified-not-needed item.

<specifics>
## Specific Ideas

- **Seed list — exact strings**: see D-03. Display them sorted A-Z but keep the source-of-truth array ordered as the user provided (so future readers see the intent grouping).
- **"Add new..." flow**: dropdown shows known categories; the LAST entry in the dropdown is a separator + "Add new..." action. Picking it reveals a small inline text input under the pill row; pressing Enter or clicking a small "+ Add" button commits a new pill. ESC or clicking elsewhere cancels.
- **Em-dash style**: a dim em-dash matching existing muted-text usage (e.g., `#94a3b8` per CLAUDE.md text colors), not the full-color em-dash.
- **"Show uncategorized only" label**: keep this exact wording — short, action-oriented, matches the existing toolbar tone.
- **Field naming clarification**: ROADMAP success criterion #1 says `category` (singular). The actual schema is `categories: string[]` (plural array). Planner should not "fix" criterion #1 — treat it as superseded by D-01.

</specifics>

- **Per-category metadata** (notes, default tax %, color) — would also require a dedicated collection. Defer.
- **Migration script** to bulk-tag legacy suppliers based on heuristics (name keywords, etc.) — out of scope; team will encode manually. Could be revisited if the manual pass stalls.
- **Singular-`category` field shim** if any downstream consumer expects the old wording — unlikely (the field is brand new in this phase); document as an internal note only.
