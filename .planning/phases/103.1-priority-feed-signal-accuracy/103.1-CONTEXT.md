# Phase 103.1: Priority Feed Signal Accuracy — Context

**Gathered:** 2026-06-13 (DRAFT — pending operator review of the open decisions below)
**Status:** Draft for review → then `/gsd-plan-phase 103.1`
**Source:** Post-103 signal-coverage review (this session) + code verification of `updated_at` write sites, journal subcollections, stage timestamps, and `firestore.rules` projects-update branches.

<domain>
## Phase Boundary

Phase 103 shipped the Priority Feed, but its urgency rests entirely on **"days in stage = days
since `updated_at`"** — and the code review found that proxy is wrong in the dangerous direction
for the flagship signal, plus three statuses get no signal at all. This phase makes the existing
feed **trustworthy** without expanding what it surfaces.

**In scope:**
1. Fix the On-going "gone quiet" false-urgent (projects) by giving the portfolio a real
   last-activity clock.
2. Make the For Mobilization signal accurate (projects) using a timestamp we already store.
3. Fill the three silent statuses (For Proposal, Proposal for Internal Approval, Client Approved)
   in both views.
4. Tame the On-going-quiet signal for services (which have no activity data) so it stops mis-firing.
5. (Optional) a "DLP expiring soon" watch for lead time before retention goes overdue.

Touches: `app/views/projects.js` + `app/views/services.js` (`getProjectSignal`/`getServiceSignal`,
`URGENCY_THRESHOLDS`), `app/views/project-detail.js` (3 journal submit handlers), and
`firestore.rules` (one allow-list addition). **This phase DOES change `firestore.rules`** (unlike
Phase 103) → dev deploy for UAT + joins the standing prod-rules-deploy debt.

**Out of scope:**
- **Cash/collection-risk signal** (the inert ≥86%-billed hook, "final billing due", money owed) —
  the highest *business* value gap, but it needs a billing_requests/collectibles read the portfolio
  is forbidden from doing. Separate, larger phase.
- **Threshold re-tuning against production data** — thresholds here stay spike/draft defaults as
  tunable constants; real tuning needs the feed in live use first.
- **Services journal/lifecycle parity** (services have no Phase 101 journal / Phase 100 stage
  timestamps) — not built here; it's why services get the conservative treatment (D-05).
- Option E swimlane; any new portfolio Firestore listener; the W-2 empty mini-bar (belongs to the
  cash-risk phase).
</domain>

<verified_facts>
## Code facts this scope depends on (verified 2026-06-13)

- **Journal writes do NOT bump the parent project `updated_at`.** `submitProgressUpdate`
  (project-detail.js:3203), `submitNewIssue` (:3392), `_addActivityEntry`/`postActivityEntry`
  (:3046/:3014) all `addDoc(collection(db,'projects',id,'<subcollection>'), …)` — no `updateDoc` on
  the parent. So `updated_at` tracks "record-field edits," not "activity." This is the root cause of
  the On-going false-quiet.
- **Stage-entry timestamps already exist on projects:** `project_started_at`,
  `mobilization_started_at`, `project_completed_at`, `dlp_expires_at`, `retention_released_at` are
  written on the lifecycle gate transitions (project-detail.js:3779/3790/3802 …) and are already in
  the operations_user `affectedKeys` allow-list (firestore.rules:229).
- **`firestore.rules` projects `allow update` has 3 branches** (rules:214–240): (1)
  super_admin/operations_admin/finance + code-lock preservation; (2) **operations_user assigned to
  the project**, gated by `affectedKeys().hasOnly([… project_status, updated_at, lifecycle fields …])`
  (:225–235); (3) finance retention-release field-mask. Writing a new `last_activity_at` field as an
  assigned operations_user requires adding it to branch (2)'s allow-list.
- **Services have neither journal nor stage timestamps.** service-detail.js has zero
  `progress_updates`/`activity_entries`/`issues` subcollections and no `*_started_at`; service
  `updated_at` is bumped only by field edits. So the projects accuracy fixes have no service analog.
- `getProjectSignal` (projects.js) and `getServiceSignal` (services.js) are identical logic; both
  read only the doc's `updated_at` via `normalizeUpdatedAt`.
</verified_facts>

<decisions>
## Implementation Decisions

### D-01 — Asymmetric scope: projects get accuracy, services get conservatism (LOCKED — forced by data availability)
Projects have the Phase 101 journal + stage timestamps; services have neither (verified). So
activity-dependent accuracy fixes (D-02, D-03) are **projects-only**; services receive signal-tuning
only (D-05). Pre-contract status rules (D-04) and the DLP-soon watch (D-06) apply to both (they use
data both views already have).

### D-02 — On-going accuracy via a denormalized `last_activity_at` (projects) (LOCKED — proper fix)
The three journal submit handlers in project-detail.js additionally
`updateDoc(projectRef, { last_activity_at: serverTimestamp() })` (fire-and-forget; see landmine).
`getProjectSignal` computes On-going staleness from `last_activity_at ?? updated_at` (falls back for
legacy docs that predate the field). `firestore.rules` branch (2) allow-list gains `last_activity_at`
(rules:~226, next to `updated_at`). **No new portfolio listener** — the feed reads the one extra
denormalized field on the project doc it already loads (Phase 103 D-05 preserved).
- **LANDMINE:** the parent `updateDoc` only passes for assigned operations_users (branch 2 requires
  `request.auth.uid in personnel_user_ids`) or admins (branch 1). A non-team active user may post a
  journal entry (subcollection create is open to active users) but their parent write will be denied.
  That's the correct semantic ("team activity refreshes the clock"), so the write must be
  **fire-and-forget with the permission error swallowed** — never block the journal post on it. Do
  NOT batch it atomically with the subcollection addDoc (a denied parent write would roll back the
  legitimate journal entry).

### D-03 — For Mobilization accuracy via existing `mobilization_started_at` (projects) (LOCKED)
The "Contract signed, not mobilized" watch should measure days waiting in For Mobilization.
`getProjectSignal` uses `mobilization_started_at ?? updated_at` for that branch. Zero new data/rules
(the field already exists and is already rules-allowed).

### D-04 — Fill the three silent statuses, both views (LOCKED; tiers/thresholds = Claude's discretion)
Add staleness rules for **For Proposal**, **Proposal for Internal Approval**, **Client Approved**
(today all silently On Track). The `updated_at` proxy is acceptable here — no subcollection activity
happens during proposal/handoff stages. New `URGENCY_THRESHOLDS` constants. Recommended defaults
(tunable, to confirm in plan):
- For Proposal > 10d → watch ("Proposal not yet drafted/sent").
- Proposal for Internal Approval > 5d → watch ("Awaiting internal sign-off" — our bottleneck;
  consider urgent).
- Client Approved > 7d → watch ("Won — not yet mobilized"; revenue/relationship risk).

### D-05 — Services On-going "quiet": conservative, not accurate (LOCKED; specifics = discretion)
Services have no activity signal and recurring services are quiet by design. So for services, the
On-going-quiet rule must not scream false-urgent. Recommended (confirm in plan): demote to **watch**
and **only for one-time** services (suppress entirely for `service_type === 'recurring'`), or raise
the threshold substantially. No data/rules change.

### D-06 — "DLP expiring soon" watch, both views (LOCKED-as-include; thresholds = discretion)
When `getDlpState === 'in-dlp'` and `dlp_expires_at` is within N days (default 14), emit a **watch**
("Retention release due in Nd") so Finance gets lead time before it flips to the urgent overdue
signal. Reads existing `dlp_expires_at`; no new data.

### Claude's Discretion
- Exact threshold values and whether each silent-status rule is watch vs urgent.
- Services On-going-quiet specifics (demote-and-one-time-only vs raise-threshold).
- `last_activity_at` field type (serverTimestamp Timestamp — `normalizeUpdatedAt` already handles
  both shapes, so consistency with the mixed-write reality is automatic).
- Whether to also stamp `last_activity_at` from the lifecycle gate transitions (cheap, optional).
- Plan/wave split (likely: rules+journal+projects.js signal as one stream; services.js tuning as a
  second; both reuse the same threshold constants).
</decisions>

<success_criteria>
## Success Criteria
- **SC-1:** An On-going project that received a progress update / issue / journal entry within the
  threshold window no longer shows "gone quiet" (reads On Track). Verified by posting a progress
  update and confirming the feed re-tiers after reload.
- **SC-2:** A For Mobilization project's "not mobilized" days count matches time since it entered
  For Mobilization (not since last edit).
- **SC-3:** A project sitting in For Proposal / Proposal for Internal Approval / Client Approved past
  its threshold surfaces in Worth Watching (or Needs Attention) with a reason, instead of silently
  On Track.
- **SC-4:** A recurring service that is On-going and untouched does NOT appear as Needs Attention.
- **SC-5:** An in-DLP project within N days of `dlp_expires_at` shows a "release due soon" watch.
- **SC-6:** No new portfolio Firestore listener; the feed still reads only the project/service docs
  (+ the one denormalized `last_activity_at` field). `node --check` clean both views.
- **SC-7:** `firestore.rules` change is the single `last_activity_at` allow-list addition; deployed
  to dev; assigned-ops-user journal posts succeed and refresh the clock; non-team posts still succeed
  (journal entry written) without erroring on the denied parent write.
</success_criteria>
