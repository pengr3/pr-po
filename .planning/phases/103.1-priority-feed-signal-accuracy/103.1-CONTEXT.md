<domain>
## Phase Boundary

Phase 103 shipped the Priority Feed, but its urgency rests entirely on **"days since `updated_at`"**,
which the code review found is the wrong clock — corruptible by incidental edits, blind to real
activity, and absent for three statuses entirely. This phase makes the existing feed **trustworthy**
without surfacing anything new.

**In scope:**
1. Introduce a real **stage-duration clock** (`status_changed_at`) — the spine; makes every
   "stuck in a stage" signal accurate in BOTH views (D-02).
2. Add an **activity-recency clock** (`last_activity_at`) for On-going projects only, where
   stage-duration is meaningless (D-03).
3. Fill the three silent statuses + give the whole funnel a coherent **two-tier** signal matrix
   with scoped ambient day-counts (D-04 — the resolved design).
4. Tame the On-going-quiet signal for **services** (no activity data; recurring is quiet by design)
   (D-05).
5. (Optional) a "DLP expiring soon" watch (D-06).

Touches: `app/views/projects.js` + `app/views/services.js` (signal functions, `URGENCY_THRESHOLDS`),
`app/views/project-detail.js` + `app/proposal-modal.js` + `app/views/service-detail.js` (status-
transition write sites + the 3 journal handlers), and `firestore.rules` (two allow-list fields).
**This phase DOES change `firestore.rules`** → dev deploy for UAT + joins the prod-rules-deploy debt.

**Out of scope:** cash/collection-risk signal (needs billing read — separate, higher-value phase);
threshold re-tuning against production data; Option E swimlane (needs full `status_history`, not just
the current-stage `status_changed_at`); any new portfolio Firestore listener; the W-2 empty mini-bar.
</domain>

## Code facts this scope depends on (verified 2026-06-13)

- **No stage-entry timestamp exists for the funnel statuses.** The intermediate statuses are set in
  `proposal-modal.js` (`newProjectStatus: 'Proposal for Internal Approval'` :981, `'Proposal Under

  `*_at`. Grep for `status_changed_at` / `client_approved_at` / any stage timestamp across `app/`
  returns nothing. So today the ONLY clock for these is `updated_at`.
- **Journal writes don't bump the parent `updated_at`.** `submitProgressUpdate` (project-detail.js
  :3203), `submitNewIssue` (:3392), `postActivityEntry`/`_addActivityEntry` (:3014/:3046) `addDoc`
  to subcollections only. So `updated_at` ≈ "last field edit," not "last activity."
- **Some specific stage timestamps already exist on projects** (`project_started_at`,
  `mobilization_started_at`, `project_completed_at`, `dlp_expires_at`) from the lifecycle gates
  (project-detail.js :3779/3790/3802) and are already rules-allowed (firestore.rules :229). These
  stay for their existing purposes; `status_changed_at` is the new generic "current-stage-entry"
  field the feed reads.
- **`firestore.rules` projects `allow update` (rules:214–240)** has 3 branches: (1) super_admin/
  operations_admin/finance + code-lock preservation; (2) assigned operations_user gated by
  `affectedKeys().hasOnly([…])` (:225–235); (3) finance retention field-mask.
- **Services have no journal and no stage timestamps** (service-detail.js: 0 subcollections, no
  `*_started_at`) — but they DO have status transitions (service-detail.js :1636…, proposal-modal.js
  shared). So `status_changed_at` is writable for services; `last_activity_at` is not.
- `getProjectSignal` / `getServiceSignal` are identical and read only `updated_at` today.

<decisions>
## Implementation Decisions

### D-01 — Two clocks, asymmetric only where forced (LOCKED)
The feed needs two different clocks for two different questions:
- **Stage-duration** (`status_changed_at`) — "stuck in this stage too long?" — serves For Proposal,
  Internal Approval, Proposal Under Client Review, For Revision, Client Approved, For Mobilization,
  For Inspection. **Both views** (both have status transitions).
- **Activity-recency** (`last_activity_at`) — "has the team gone quiet?" — serves On-going only
  (the stage legitimately lasts months). **Projects only** (needs the journal; services have none).

### D-02 — `status_changed_at` stage-duration backbone (LOCKED — the spine)
Stamp `status_changed_at: serverTimestamp()` on **every** status transition: the lifecycle gates
(project-detail.js), the proposal transitions (`proposal-modal.js`), the status dropdown/manual sets,
and the service equivalents (`service-detail.js`). Prefer a single shared setter so no transition
site is missed. The signal functions read `status_changed_at ?? updated_at ?? created_at` for all
stage-duration branches (legacy docs degrade gracefully until their next transition).
`firestore.rules` branch (2) operations_user allow-list gains `status_changed_at`; the proposal-
transition write path's allow-list must include it too. **No new portfolio listener** — the feed
reads one extra field on the doc it already loads. This single field also makes the EXISTING Phase 103
signals (For Inspection, Proposal Under Client Review, For Revision) accurate, and `mobilization_started_at`
becomes just a special case (the feed can use `status_changed_at` uniformly).

### D-03 — `last_activity_at` activity clock for On-going (projects) (LOCKED)
The three journal submit handlers (project-detail.js) additionally
`updateDoc(projectRef, { last_activity_at: serverTimestamp() })`. On-going staleness reads
`last_activity_at ?? updated_at`. `firestore.rules` branch (2) allow-list gains `last_activity_at`.
- **LANDMINE:** the parent write only passes for assigned operations_users / admins; a non-team
  active user can post a journal entry but their parent write is denied. That's the correct semantic
  ("team activity refreshes the clock"). So it must be **fire-and-forget, error swallowed, NOT
  batched** with the subcollection addDoc (a denied parent write must not roll back the journal entry).

### D-04 — Two-tier funnel signal matrix + scoped ambient day-counts (LOCKED — resolved 2026-06-13)
Non-uniform, two-tier (each stage has its own watch AND urgent point), reading `status_changed_at`.
Days-in-stage `d = now − status_changed_at`. Thresholds are tunable `URGENCY_THRESHOLDS` constants —
**first-guess defaults below; pending operator sanity-check against CLMC's real stage cadence.**
★ = new in D-04; the rest are existing Phase 103 signals retuned to two-tier on the accurate clock.

| Stage | 🟠 Watch | 🔴 Urgent | Fuse rationale |
|-------|---------|----------|----------------|
| For Proposal ★ | 10d | 25d | proposal prep is real work |
| Proposal for Internal Approval ★ | **2d** | **5d** | work done, one click — loudest, shortest fuse |
| Proposal Under Client Review | 7d | 14d | follow-up nudge → "client gone dark" |
| For Revision | 5d | 12d | 3–5d expected turnaround |
| Client Approved ★ | 7d | 18d | some legit lag; idle-alarm ~2.5wk |
| For Mobilization | 3d | 10d | should move fast post-contract |
| For Inspection | 14d | 30d | scheduling lag tolerated; overdue at 1mo |

**Signal text (the three new D-04 statuses):**
- For Proposal — 🟠 "Proposal not yet sent — {d}d" · *"Client awaiting our quote"* · 🔴 "Proposal stalled — {d}d" · *"No proposal sent in {d} days"*
- Internal Approval — 🟠 "Awaiting internal sign-off — {d}d" · *"Proposal ready, pending approval"* · 🔴 "Sign-off overdue — {d}d" · *"Finished proposal blocked internally"*
- Client Approved — 🟠 "Won — not yet mobilized — {d}d" · *"Client expects kickoff"* · 🔴 "Kickoff overdue — {d}d" · *"Won work idle {d} days"*

**Scoped ambient subtext (chosen 2026-06-13):** while a stage-duration row is still On-Track (below
its watch threshold), show grey "In {stage} · {d}d" subtext (e.g. "In proposal stage · 4d",
"Won · awaiting kickoff · 3d"). Applies ONLY to the stage-duration statuses above — **never** On-going
(activity clock; would show huge numbers) or terminal Completed/Loss. Once it crosses watch, the flag
text replaces the subtext. This gives "about to tip" awareness without flooding the feed.

### D-05 — Services On-going "quiet": conservative, not accurate (DRAFT — pending nod; specifics = discretion)
Services have no activity clock and recurring services are quiet by design. Recommended: demote the
On-going-quiet rule to **watch** and **only for one-time** services (suppress for `service_type ===
'recurring'`), or raise the threshold substantially. No data/rules change. (Services still benefit
from `status_changed_at` for all their funnel signals via D-02/D-04.)

### D-06 — "DLP expiring soon" watch, both views (DRAFT — pending include/defer; threshold = discretion)
When `getDlpState === 'in-dlp'` and `dlp_expires_at` within N days (default 14), emit a watch
("Retention release due in {n}d") for Finance lead time before it flips to urgent overdue. Reads
existing `dlp_expires_at`.

<success_criteria>
## Success Criteria
- **SC-1:** Every status transition (gates, proposal flow, manual, services) writes `status_changed_at`;
  the feed's stage-duration days reflect time since the stage began, not since last edit.
- **SC-2:** Editing a field on a stalled For Proposal / Client Approved project does NOT reset its
  staleness (clock survives incidental edits).
- **SC-3:** Each funnel stage shows its watch flag at the watch threshold and escalates to the urgent
  flag at the urgent threshold (two-tier verified for at least Internal Approval + Client Approved).
- **SC-4:** Below threshold, stage-duration rows show the grey "In {stage} · {d}d" subtext; On-going
  and terminal rows do not.
- **SC-5:** An On-going project that received a progress update within the window reads On Track (not
  "gone quiet"); a recurring service that is On-going and untouched is NOT Needs Attention.
- **SC-6:** No new portfolio Firestore listener; `node --check` clean both views.
- **SC-7:** `firestore.rules` adds exactly `status_changed_at` + `last_activity_at` to the allow-lists;
  deployed to dev; assigned-ops journal posts refresh `last_activity_at`, non-team posts still succeed
  without erroring on the denied parent write.
</success_criteria>
