## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-103.1-01 | Tampering | projects operations_user write path (firestore.rules :225-235) | mitigate | The two new fields (status_changed_at, last_activity_at) are added to the EXISTING `affectedKeys().hasOnly([...])` field-mask, which is itself gated by `request.auth.uid in resource.data.personnel_user_ids`. So only an operations_user already assigned to that specific project can write them, and only as part of a masked key set — they cannot use these keys to smuggle a write to project_code/client_id/client_code (those remain absent from the allow-list and code-locked). No privilege expansion: the writable surface grows by exactly two timestamp fields on projects the user already controls. |
| T-103.1-02 | Information Disclosure | status_changed_at / last_activity_at values | accept | Both fields are coarse activity timestamps on a project the actor is already assigned to; they expose no PII and no cross-project data. Low-value target; no read-rule change (existing project read access unchanged). |
| T-103.1-03 | Repudiation | proposal funnel project write (parentCollection batch.update) | accept | status_changed_at is a denormalized convenience clock for the feed, not an authoritative audit record. The canonical transition audit trail remains the proposal audit_log + project audit entries (unchanged). A wrong/forged status_changed_at only mis-times a UI signal; it cannot alter the lifecycle record. |

Note: services need NO rules change — the services update rule (:533-534) is role+assignment only (no
field-mask), so status_changed_at is already permitted there; no new trust surface is opened on services.
