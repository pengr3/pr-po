<objective>
Add the `last_activity_at` activity-recency clock (D-03) to the three project journal submit handlers in
project-detail.js. On-going projects legitimately sit in one stage for months, so stage-duration is
meaningless there — the feed instead needs "has the team gone quiet?" which means the last journal
activity, not the last field edit. Each handler additionally bumps `last_activity_at` on the parent
project doc, fire-and-forget, so the On-going staleness signal (Plan 03) can read it.

Purpose: Make the On-going "gone quiet" signal trustworthy (SC-5) — a project that received a progress
update within the window reads On Track instead of falsely "quiet". Honors D-03's landmine: the parent
write must be fire-and-forget and error-swallowed, NOT batched with the subcollection addDoc, because a
non-team active user can legitimately post a journal entry while their parent write is correctly denied
by the rules — and that denial must never roll back the journal entry.
Output: last_activity_at refreshed on every journal post by an assigned team member.

CRITICAL — the postActivityEntry asymmetry (must resolve): the trustworthiness goal of this phase means a
FALSE clock refresh is as harmful as a missed one. `submitProgressUpdate` and `submitNewIssue` each
`await addDoc(...)` DIRECTLY inside their try block, so a failed addDoc THROWS past the bump — the bump is
naturally success-path-only. But `postActivityEntry` persists via `await _addActivityEntry(...)`, and
`_addActivityEntry` wraps its OWN addDoc in a try/catch that swallows the error and does NOT rethrow — so
a failed addDoc returns normally and control reaches the bump even though nothing persisted, falsely
refreshing the On-going quiet clock. Resolution (this plan): make `_addActivityEntry` RETURN a success
boolean (true after its addDoc resolves, false in its catch) and gate the postActivityEntry bump on that
boolean (`if (ok) updateDoc(...)`). This restores the same success-only semantics the other two handlers
get for free, while keeping all three bumps fire-and-forget / not-awaited / not-batched.

CRITICAL — ownership boundary (orphan-ownership rule): this plan and Plan 01 BOTH edit
app/views/project-detail.js, in DISJOINT functions.
- Plan 02 (this plan) OWNS ONLY: the 3 journal handlers — `postActivityEntry` (~3014),
  `submitProgressUpdate` (~3203), `submitNewIssue` (~3392) — plus the shared `_addActivityEntry` (~3046)
  write primitive (adding its success-boolean return is part of this plan's journal-handler scope and
  does NOT collide with Plan 01's lifecycle gates / saveField). Add the last_activity_at write to each
  handler.
- Plan 01 OWNS: the 4 lifecycle gates + saveField status write. Do NOT touch those.
This plan is Wave 2 (depends_on Plan 01) because (a) the rules allow-list for last_activity_at ships in
Plan 01 and must be dev-deployed before an assigned operations_user's parent write succeeds in UAT, and
(b) sequencing avoids a same-file same-wave conflict with Plan 01.
</objective>

`await addDoc(collection(db,'projects',currentProject.id,'issues'), {...})` inside try/catch,
then toast + clear fields. Same direct-await shape — bump after the await is already success-path-only.

LANDMINE (D-03): the parent updateDoc passes ONLY for assigned operations_users / admins. A non-team
active user can post a journal entry (subcollection create is allowed for active users per the Phase 101
rules) but their parent project write is DENIED. That denial is correct ("team activity refreshes the
clock"). So the parent write must be fire-and-forget, error swallowed, NOT awaited-into-the-same-catch
that would let it bubble, and NOT batched with the subcollection addDoc (a denied parent write must not
roll back the journal entry). Use `.catch(() => {})` (or `.catch(err => console.debug(...))`) on a
non-awaited updateDoc, AFTER the subcollection write has already succeeded.

ASYMMETRY (must resolve — see objective): `_addActivityEntry` swallows its own addDoc failure, so
postActivityEntry cannot tell "persisted" from "failed-and-swallowed". Give `_addActivityEntry` a boolean
return (true on success, false on its swallowed catch) and gate the postActivityEntry bump on it, so the
clock only refreshes when an entry actually persisted. The other two handlers throw on failure, so their
bump-after-await is inherently success-only — leave their structure as-is.
