## D-10 Diagnosis Note

Plan called for live diagnosis in Chrome DevTools, which is not possible during overnight batch execution.
Applied PATTERNS.md Candidate A — the most-likely cause per the pattern map.

**Candidate A hypothesis:** `.task-grid-rail` had `overflow: auto` + descendant `.task-grid` had `min-width: 640px`. At default 35% rail width on a 1440px viewport (~504px), the inner grid overflowed horizontally and the rail rendered a stray horizontal scrollbar at its bottom — appearing roughly mid-screen, just left of the divider.

**Before (line 2446):**
```css
.task-grid-rail {
    overflow: auto;       /* both axes — grid may be wider than the rail */
    background: #ffffff;
    border-right: 1px solid var(--gray-200, #dadce0);
}
```

**After:**
```css
.task-grid-rail {
    /* Phase 86.3 D-10: kill stray horizontal scrollbar (Candidate A — .task-grid min-width: 640px > narrow rail width).
       At default 35% split on a 1440px viewport (~504px rail), the 640px min-width forced a stray horizontal scrollbar
       at the bottom of the left rail (appearing mid-screen, just left of the divider).
       Vertical scroll preserved for many-tasks projects.
       NOTE: Diagnosis assumption per CONTEXT.md D-10 batch-authorization — user verifies in morning UAT. */
    overflow-x: hidden;
    overflow-y: auto;
    background: #ffffff;
    border-right: 1px solid var(--gray-200, #dadce0);
}
```

**Constraints verified:**
- `.gantt-pane` still has `overflow: auto` — timeline horizontal scroll preserved
- `.plan-split-pane` — no overflow added
- `.plan-view-surface` — no overflow added
- `.plan-divider` — not touched (Phase 86.2 verified)

**If morning UAT (Test 7 in the punch list) shows the stray scrollbar is STILL present,** the cause is Candidate B or C from PATTERNS.md §D-10 — investigate via DevTools and revisit. The candidates are documented in PATTERNS.md.
