**In scope (9 defects):**

Right-panel (Gantt timeline):
1. **Date floor at today (soft)** — leftmost visible date is today UNLESS earlier task data exists (then floor at the project's earliest task start_date). User cannot scroll left of the floor.
2. **Scroll jank investigation** — diagnose and fix buggy horizontal scroll on the timeline.
3. **Header rendering by zoom** — Day-of-week / Monday / week-range labels per zoom level (see D-03).
4. **Bar drag row lock** — bars never change rows when dragged; horizontal drag only repositions in time on the same row.
5. **Drag-semantics unification** — horizontal bar drag = move/extend in time on its own row; drop-on-bar = create FS predecessor (existing 86.1 behavior, preserved); cross-row vertical drag of bars = removed.
6. **Stray middle-of-screen scrollbar** — diagnose source and remove.

Left-panel (inline grid):
7. **New-task default 1d duration** — newly inserted task renders a 1-day Gantt bar immediately without requiring Duration entry (see D-04 for date anchor).
8. **New-task row position preservation** — newly inserted task stays on the row where the user inserted it; no auto-resort by start date.
9. **Predecessors cell — Enter commit** — pressing Enter inside the Predecessors cell commits the value (extend the 86.2 `_gridKeydownHandler` blur-delegation pattern).

**Out of scope (deferred):**
- Cross-row drag-to-reorder of bars (rejected; row order is owned by the left-rail # cell drag handle, per 86.2 DEFECT-3)
- Critical path highlight, baseline view, resource view, export — separate future phases (already deferred from Phase 86)
- Auto-cascade of dependent task dates when a predecessor is rescheduled (rejected in 86 D-10; remains visual-warn-only)
- New auto-chain default ("start = previous task's end + 1") — rejected in this discussion
- Hard floor at today that hides historical task bars — rejected (would orphan data on in-flight projects)

<decisions>
## Implementation Decisions

### Date Axis Floor (defect #1)
- **D-01 — Soft floor at min(today, earliest task start_date):** Leftmost visible/scrollable date = `min(today, min(tasks.start_date))`. For new or future projects with no past tasks, the floor is today and you cannot scroll left of today. For in-flight projects with tasks scheduled in the past, the floor extends back to the earliest task so historical bars remain visible. Rationale: hard floor at today would permanently hide past tasks on any in-flight project — orphaning real data behind a UI rule.
- **D-02 — Floor enforced at the scroll-bound level, not by clamping data:** Tasks keep their actual `start_date` / `end_date` as written in Firestore. The floor only restricts what the timeline scroll can render. No write to `project_tasks` happens as a side effect of date-floor enforcement. Auto-scroll-to-today on init (Phase 86 D-09) still applies — today is where the timeline lands, not the leftmost reachable point.

### Calendar Header Rendering (defect #3)
- **D-03 — Per-zoom header rendering:**
  - **Day view (zoom: Day):** show a day-of-week row (M T W Th F S S) under the date numbers. Each column = 1 day; weekday letter labels each column.
  - **Week view (zoom: Week):** column major label = the Monday date of each week (e.g., "03 May", "10 May", "17 May"). The day-of-week M T W Th F S S row is not required in this view (the Monday-date label is the affordance).
  - **Month view (zoom: Month):** column major label = week range in the form `DD MMM - DD` for in-month ranges and `DD - DD MMM` (or `DD MMM - DD MMM`) when the range crosses a month boundary, e.g. `03 May - 09`, `10 - 16`, `24 - 30`, `31 - 06 Jun`, `07 Jun - 13`. Above the week-range row, a parent month-name row (`May`, `June`) spans the weeks belonging to that month — see attached image (project plan calendar mockup).
  - **No M T W Th F S S row in Month view.**
  - **"Today" pill in the top-right of the header is preserved** (already present in the mockup).

### New-Task Default Duration & Anchor (defects #7, #8)
- **D-04 — New task gets default 1d duration on insert:** `handleNewRowCommit` (`app/views/project-plan.js:541`) currently writes a task without `start_date` / `end_date`, leaving the Gantt bar invisible until the user enters Duration. Phase 86.3 changes this to write both fields at insert time so the 1-day bar renders immediately on the next snapshot.
- **D-05 — Anchor for new-task start_date:** Use the start_date of the row immediately above the newly inserted row (same parent_task_id inheritance reuses Phase 86.2 DEFECT-6 logic — `sortedForInherit[last]`). If the row above has no `start_date`, OR the new row is the first row in the project, default to today (`formatDateISO(new Date())`). End_date = start_date (1 day, inclusive — matches 86.1 D-03 `addDays(start, days - 1)` convention). Rationale: tasks naturally cluster near the user's current work area; in-progress projects don't snap new rows back to today; new/empty projects still anchor sensibly.
- **D-06 — Row order preserved (no auto-resort):** Newly inserted task stays on the row where the user inserted it. The Gantt does not reorder rows by start_date on insert. `row_order` continues to drive display order in both left rail and Gantt (existing 86.1 PM-02 behavior). Confirms current grid behavior; defect #8 investigation should identify what's currently re-sorting and remove it.

### Bar Drag Behavior (defects #4, #5)
- **D-07 — Bars are row-locked during drag:** Horizontal drag of a Gantt bar repositions it in time on its own row only. Cross-row vertical drag of bars is removed entirely. Investigation should determine whether the row-shift is coming from (a) Frappe's own drag handlers, (b) custom code added in Phase 86 / 86.1, or (c) DOM-reorder during `gantt.refresh(tasks)` — and fix at the source.
- **D-08 — Drop-on-bar still creates FS predecessor (preserved from 86.1):** No change to the Phase 86.1 SVG drag-handle-on-bar-edge → release-on-target-bar gesture. That gesture creates an FS dependency arrow with cycle detection + toast, exactly as today. Only the unintended "drag bar body across rows" behavior is removed.

### Predecessors Cell — Enter Commit (defect #9)
- **D-09 — Extend the 86.2 Enter handler to Predecessors:** The `_gridKeydownHandler` in `app/views/project-plan.js:454` currently checks `col !== 'duration' && col !== 'resources'` and returns early. Phase 86.3 changes this to also accept `col === 'predecessors'`. The blur-delegation pattern then routes through the existing `handleGridCellBlur` predecessors branch (`app/views/project-plan.js:501`) which already parses comma-separated row#s, dedupes, and runs cycle detection. No new commit path needed.

### Stray Middle Scrollbar (defect #6)
- **D-10 — Investigate and fix at source:** The middle-of-screen scrollbar is most likely an unintended overflow on a wrapper between the resizable panel divider (86.2 DEFECT-8) and the Gantt scroll container. Researcher should diagnose the offending CSS rule (`overflow-x` / `overflow-y`) and fix without breaking the legitimate scroll containers (timeline horizontal scroll, grid vertical scroll).

### Scroll Jank (defect #2)
- **D-11 — Diagnose first; no premature optimization:** Root-cause investigation before fix. Likely candidates: re-render on every snapshot without throttling, layout thrash from `gantt.refresh()` on each change, missed `gantt-today-line` cleanup. Researcher pulls a profile, identifies the cause, and fixes the actual hot path. No speculative virtualization or library swap.

### Claude's Discretion
- Exact CSS class names / selectors for the new header rows
- Where the soft date-floor logic lives (Frappe option, custom scroll guard, or post-render DOM hook) — researcher decides based on Frappe's API
- Whether the day-of-week letter is `M T W T F S S` or `Mon Tue Wed Thu Fri Sat Sun` (lean on the shorter form to match Phase 86.2's 42px row sizing)
- Exact toast / error copy for any invalid Predecessors entries on Enter
