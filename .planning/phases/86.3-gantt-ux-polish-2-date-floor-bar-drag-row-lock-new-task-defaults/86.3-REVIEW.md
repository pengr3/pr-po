## High Severity

### H-01: Month-view week-range labels always include month name — violates D-03 spec

**File:** `app/views/project-plan.js:1394-1402`

**Issue:** For same-month week-range cells, the code unconditionally formats the label as `startDD + ' ' + startMMM + ' - ' + endDD`, producing `"03 May - 09"`, `"10 May - 16"`, `"17 May - 23"`, etc. CONTEXT.md D-03 explicitly states the visual contract with the examples `03 May - 09, 10 - 16, 17 - 23, 24 - 30` — only the first week of each month bears the month name; subsequent weeks are `DD - DD` only. The VERIFICATION.md plan 05 UAT test (line 248) also calls out `"10 - 16"` (without month) as the expected value. The parent month-name row already displays the month, so repeating it on every cell is redundant and contradicts the spec. This will be visible as a mismatch in the first morning UAT run.

**Fix:** Track whether the current week is the first week within its month run and suppress `startMMM` on subsequent weeks:

```javascript
} else if (mode === 'Month') {
    // ...existing setup...
    for (let w = 0; w < totalWeeks; w++) {
        const monday = new Date(firstMonday);
        monday.setDate(monday.getDate() + w * 7);
        const sunday = new Date(monday);
        sunday.setDate(sunday.getDate() + 6);
        const startDD = String(monday.getDate()).padStart(2, '0');
        const startMMM = monthShort[monday.getMonth()];
        const endDD = String(sunday.getDate()).padStart(2, '0');
        const endMMM = monthShort[sunday.getMonth()];

        // Is this the first week in its month run?
        const prevMonday = w === 0 ? null : new Date(firstMonday);
        if (prevMonday) prevMonday.setDate(prevMonday.getDate() + (w - 1) * 7);
        const isFirstWeekOfMonth = w === 0 || prevMonday.getMonth() !== monday.getMonth();

        let label;
        if (monday.getMonth() === sunday.getMonth()) {
            // Same month: show "DD MMM - DD" for first week of month, "DD - DD" for subsequent
            label = isFirstWeekOfMonth
                ? `${startDD} ${startMMM} - ${endDD}`
                : `${startDD} - ${endDD}`;
        } else {
            // Cross-month: "31 - 06 Jun"
            label = `${startDD} - ${endDD} ${endMMM}`;
        }
        // ...rest of cell creation unchanged...
    }
```

### H-02: `renderGanttHeaderOverlay` null-guard on `gantt_end` is defeated by `Invalid Date`

**File:** `app/views/project-plan.js:1316-1317`

**Issue:** The guard `if (!endDate) return;` fires only for falsy values. When `gantt.gantt_end` is `undefined` (e.g., on first render before Frappe has fully initialized), the ternary evaluates to `new Date(undefined)` which produces an `Invalid Date` object. An `Invalid Date` is truthy, so the guard never triggers. Subsequent operations (`endDate.setHours(0,0,0,0)`, `endDate - startDate`) silently produce `NaN`. `Math.round(NaN)` is `NaN`, `Math.ceil(NaN / 7)` is `NaN`, and `for (let w = 0; w < NaN; w++)` never executes. The overlay is appended as an empty `div` with no cells — the wrong result, produced silently with no console warning (because the catch block is never reached — no exception is thrown). The existing `try/catch` does not help here.

**Fix:** Replace the guard with an `isNaN` check:

```javascript
const endDate = gantt.gantt_end instanceof Date ? gantt.gantt_end : new Date(gantt.gantt_end);
// new Date(undefined) is Invalid Date (truthy but isNaN) — must check explicitly
if (!endDate || isNaN(endDate.getTime())) return;
endDate.setHours(0, 0, 0, 0);
```

The identical pattern applies to `startDate` on line 1311, though `gantt.gantt_start` being undefined is less likely once `new Gantt(...)` has been called. Adding the check to `endDate` is the critical fix.

## Medium Severity

### M-01: Scroll-clamp listener registered as `passive: true` — causes snap-back jitter

**File:** `app/views/project-plan.js:1294`

**Issue:** The scroll-clamp handler that enforces the date floor is registered with `{ passive: true }`:

```javascript
pane.addEventListener('scroll', _ganttScrollClampHandler, { passive: true });
```

Passive scroll listeners run after the browser has already composited the scroll frame. When the user attempts to scroll left past the floor, the sequence is: (1) browser composites the over-scrolled position, (2) the handler fires and sets `pane.scrollLeft = floorX`. The user sees the over-scrolled position for one composited frame before the snap-back. On low-end devices or when compositing is slow, this manifests as visible jitter — a one-frame flash of the forbidden region. Setting `scrollLeft` from a passive listener is allowed by the spec (it is not `preventDefault()`), but the visual result is inconsistent floor enforcement. The comment in `renderGantt` calls this a "smooth scroll" fix, but passive listeners are the wrong tool for position clamping.

**Fix:** Remove `passive: true` to use the default `{ passive: false }`:

```javascript
pane.addEventListener('scroll', _ganttScrollClampHandler);
// or explicitly:
pane.addEventListener('scroll', _ganttScrollClampHandler, { passive: false });
```

The handler body is O(1) (a single comparison and a conditional assignment), so the performance cost of a non-passive handler is negligible. The Gantt container's horizontal scroll is driven by the user, not `wheel` events — the performance-sensitive context for `passive: true` is `touchmove` and `wheel` on mobile, not `scroll` on a desktop Gantt pane.

### M-02: `gridInsertRowAbove` does not apply D-04/D-05 start_date defaulting — inconsistent UX

**File:** `app/views/project-plan.js:786-801`

**Issue:** The D-04/D-05 fix (default 1-day bar anchored on row above) was applied only to `handleNewRowCommit` (the trailing-row path). The `gridInsertRowAbove` function, which creates a task via right-click "Insert Row Above", writes the Firestore document without `start_date` or `end_date`:

```javascript
await setDoc(doc(db, 'project_tasks', newTaskId), {
    task_id: newTaskId,
    // ...
    name: '',
    // NO start_date, NO end_date
    row_order: targetOrder,
    // ...
});
```

A task inserted via "Insert Row Above" still produces no Gantt bar until the user manually enters Duration, directly contradicting the D-04 goal: "New task renders a 1-day Gantt bar immediately on the next snapshot; no Duration entry required." The CONTEXT.md D-04 paragraph references only `handleNewRowCommit` by name, so this may be an intentional scope limit — but the resulting behavior is a visible inconsistency between the two insertion paths.

**Fix:** Apply the same anchor logic in `gridInsertRowAbove`. The task at `targetOrder` (the row below the insertion point) provides the date anchor, or fall back to today:

```javascript
// In gridInsertRowAbove, after computing targetOrder and before setDoc:
const sorted = tasks.slice().sort((a, b) => (a.row_order ?? Infinity) - (b.row_order ?? Infinity));
const rowAboveInsert = sorted.filter(x => (x.row_order ?? 0) < targetOrder).pop(); // last row before insertion point
const anchorStart = rowAboveInsert?.start_date || formatDateISO(new Date());

await setDoc(doc(db, 'project_tasks', newTaskId), {
    // ...
    start_date: anchorStart,
    end_date: anchorStart, // 1-day inclusive
    // ...
});
```

If the CONTEXT.md scope limit is intentional, this finding should be documented as a known inconsistency, not silently left to UAT discovery.

### M-03: Week/Month overlay cells can extend past `gantt_end` — last column overflows visible Gantt range

**File:** `app/views/project-plan.js:1354-1368` (Week mode), `1384-1410` (Month mode)

**Issue:** Both Week and Month mode compute `totalWeeks = Math.ceil(totalDays / 7)`. When `firstMonday` lands 1-5 days after `startDate` (due to the day-alignment offset), the last week cell `w = totalWeeks - 1` can extend past `endDate`. For example, with `startDate = Thu May 1`, `endDate = Fri May 30`, `totalDays = 29`, `totalWeeks = ceil(29/7) = 5`: `firstMonday = Mon May 5`; week 4 is `Mon Jun 2` (past endDate). The cell is absolutely positioned with `left = (dayDiffFromStart * xPerDay)px` and `width = (7 * xPerDay)px`, placing it visibly outside the Gantt's `gantt_end` boundary. Since the overlay has `pointer-events: none`, there is no click interception, but cells visually extend past the timeline's right edge — partially hidden by the pane's `overflow: auto` scroll container if the user hasn't scrolled right, and visible as stray labels if they have.

This is cosmetic rather than a crash, but it can produce spurious labels (e.g., a "Jun" week cell in a May-only project). The fix should clip cells whose `monday >= endDate`:

```javascript
for (let w = 0; w < totalWeeks; w++) {
    const monday = new Date(firstMonday);
    monday.setDate(monday.getDate() + w * 7);
    if (monday >= endDate) break; // do not render cells beyond gantt_end
    // ...rest of cell creation...
}
```

Apply the same guard in the Month-mode week-range loop and the Month-mode month-name sweep loop.

## Low Severity

### L-01: `applyFsViolationStyles` function body removes violation class but never adds it — dead code path

**File:** `app/views/project-plan.js:1171-1191`

**Issue:** `applyFsViolationStyles` queries arrows, calls `el.classList.remove('violation')` on every arrow, then exits. The function computes `violations` (the array of `{from, to}` pairs with FS constraint breaches) but never applies the `violation` class to any element. The comment at line 1185 explains: "Frappe arrows don't carry id attrs out of the box, so we cannot reliably tag specific arrows as violations without a Frappe fork." This explanation is accurate and the decision not to fork is correct per Phase 86 D-05.

However, the `violations` array computation (lines 1174-1183) runs on every snapshot even though its output is never used. This is dead code: the `violations` array is populated and immediately discarded. The entire `tasks.forEach(b => { deps.forEach(aId => { ... violations.push(...) }) })` block serves no purpose today.

**Fix:** Remove the dead `violations` array computation and simplify the function to its actual effect:

```javascript
function applyFsViolationStyles() {
    // Frappe arrows have no stable id attrs — cannot target specific arrows without a fork.
    // Clear any stale violation classes (safety net for future Frappe version upgrade).
    document.querySelectorAll('#ganttPane .arrow').forEach(el => el.classList.remove('violation'));
}
```

This makes the intent clear: the function is a forward-compatible stub that clears stale state. The `checkAndToastFsViolations()` function (line 1587) handles the actual violation detection and toasting; the two are separate responsibilities and should remain separate.
