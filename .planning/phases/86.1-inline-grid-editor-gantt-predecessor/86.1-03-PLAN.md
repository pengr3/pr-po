<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → pill click | User clicks toggle pills; each click triggers an updateDoc to project_tasks.assignees |
| picker DOM → document.body | Picker is appended to body, outlives the grid container; needs cleanup on view destroy |
| Firestore write payload | assignees array is constructed client-side from the union/diff of currentAssignees toggles; written without server-side validation of UID membership |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-86.1.3-01 | Tampering / XSS | openAssigneePicker interpolates personnel names + uids into innerHTML | mitigate | escapeHTML on every interpolation per PATTERNS.md template — `data-uid="${escapeHTML(uid)}"` and `${escapeHTML(name)}`. Names come from currentProject.personnel which is server-controlled (registered users); uids are Firebase auth uids (alphanumeric + dashes). Both are low-risk inputs but escapeHTML is mandatory per project XSS posture. |
| T-86.1.3-02 | Tampering | User could console-tamper currentAssignees array to include arbitrary UIDs not in the project's personnel list | mitigate | The existing project_tasks update rule (Phase 86 Plan 01 tier 1) does NOT validate that assignee UIDs are members of the project's personnel — that's a server-side gap shared with Phase 86 modal-based assignment. Disposition: same as Phase 86 D-16 (project-personnel-only is a UX guard, not an authority guard). Server-side enforcement deferred to v4.1 hygiene phase. |
| T-86.1.3-03 | Information Disclosure | normalizePersonnel(currentProject) reads from project doc which is already in scope; no new disclosure | accept | currentProject was already loaded by Plan 01 init() via getDocs. Plan 03 reuses the existing data. |
| T-86.1.3-04 | DoS | User clicks pills rapidly → many concurrent updateDoc calls | accept | Each click awaits its own write before the next can fire (UX-bound, not race-bound). Even fully-concurrent clicks last-write-wins per Firestore single-doc semantics. Same disposition as Phase 84.1-01. |
| T-86.1.3-05 | Repudiation | Pill toggle has no audit trail | accept | Same disposition as Phase 86 task-level edits — assignment changes are not in an audit collection. v4.1 candidate if needed. updated_at serverTimestamp captures the write moment. |
| T-86.1.3-06 | DoS | Stale picker DOM accumulates if user opens many pickers without clicking outside | mitigate | openAssigneePicker first removes any existing `#taskAssigneePicker`. Outside-click dismissal removes it on first outside click. Net: at most one picker in the DOM. |
| T-86.1.3-07 | Tampering / Sandbox | The 10ms-delayed `document.addEventListener('click', handler)` could race with another modal/menu's click handlers | accept | Same async-after-bind pattern as Phase 86 modal CRUD — proven in production. The 10ms delay prevents same-tick mousedown→click double-fire. Outside-click handlers do not stack because each picker open removes the prior one first. |
</threat_model>
