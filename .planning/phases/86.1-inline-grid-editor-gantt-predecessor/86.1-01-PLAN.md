<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → DOM (grid input) | User-typed task names, duration strings, predecessor row# strings flow through HTML inputs into JS handlers and onto Firestore writes |
| DOM → Firestore (project_tasks) | Each cell blur triggers an updateDoc; new-row commit triggers a setDoc; payload assembly is in handleGridCellBlur and handleNewRowCommit |
| renderTaskGrid → DOM (innerHTML) | Render writes user-controlled task name, dates, dep row#s, assignee names back into the table via innerHTML — escapeHTML must be present on every interpolation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-86.1.1-01 | Tampering / XSS | renderTaskGrid innerHTML interpolation of t.name, t.assignees-derived names, predecessor row# strings, t.start_date, t.end_date | mitigate | Wrap every user-controlled string in `escapeHTML(...)` per PATTERNS.md template — `value="${escapeHTML(t.name || '')}"`, `${escapeHTML(durStr)}`, `${escapeHTML(t.start_date || '')}`, etc. The PATTERNS.md excerpt already encodes this — execute it verbatim. Acceptance criterion: `grep -c "escapeHTML" app/views/project-plan.js` increases by ≥10 over post-Task-1 baseline. |
| T-86.1.1-02 | Tampering | handleGridCellBlur duration branch — user-supplied duration string parsed by parseDuration regex | mitigate | parseDuration uses a strict regex `^(\\d+)\\s*([dw])$/i` that rejects anything other than `Nd` / `Nw`. parseInt is bounded by the captured digit group. Returns null on no-match → handler short-circuits. T-86.1.1-02 disposition: mitigate by validation, no Firestore write occurs on invalid input. |
| T-86.1.1-03 | Tampering | handleGridCellBlur predecessors branch — user types row#s, JS resolves to task_ids, writes to dependencies[] | mitigate | Resolution path: `parseInt(n, 10)` then reverse-lookup in rowOrderCache. Unresolvable row#s are filtered out (`.filter(Boolean)`). Cycle detection (detectDependencyCycle) runs before write — cycle path → toast + revert, no write. Dedupe via Set. Final dependencies[] is always a subset of valid existing task_ids in this project. |
| T-86.1.1-04 | Tampering | handleGridCellBlur start/end branches — user types ISO date strings | mitigate | The `<input type="date">` browser-side restricts input to YYYY-MM-DD. JS handler additionally rejects writes where end < start (`return` without write). No further sanitization needed — the date string is stored as-is and never injected into HTML without escapeHTML. |
| T-86.1.1-05 | Information Disclosure | rowOrderCache reverse lookup leaks all task_ids in the project to the user via Predecessors column | accept | Tasks already visible in the grid — same task_ids are already in `tasks[]` array client-side. No new disclosure surface; rowOrderCache is just a render-time index built from data the user is already authorized to read (Phase 86 Plan 01 isActiveUser predicate). |
| T-86.1.1-06 | Elevation of Privilege | Client-side cycle detection bypassable; user could call updateDoc directly via console with a cycle | accept | Server-side rules don't enforce cycle detection (would require recursive get() calls — Firestore rules don't support recursion). Same disposition as Phase 86 Plan 04 D-13: cycle detection is a UX guard, not an authority guard. The graph is still consistent because Frappe Gantt's renderer tolerates cycles (just renders both arrows); detectDependencyCycle prevents the inadvertent-cycle UX, console-tampering not in threat scope. |
| T-86.1.1-07 | Repudiation | Inline writes overwrite each other on rapid typing (last-write-wins) | accept | updated_at serverTimestamp is captured on every write. Firestore single-doc updates are atomic. Concurrent same-doc edits from two sessions are out of scope (single-user-per-project pattern in this app). |
| T-86.1.1-08 | DoS | User pastes a 10MB string into a Name cell and triggers updateDoc | accept | Firestore rejects docs >1MB at the rule layer. Client-side noop here is consistent with all other inputs in the codebase (procurement.js, finance.js never length-cap user inputs). |
