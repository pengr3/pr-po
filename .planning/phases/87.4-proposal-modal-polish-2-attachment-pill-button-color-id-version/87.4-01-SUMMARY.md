# Phase 87.4 Plan 01: Attachment-Required Submission Gate Summary

**Three-layer defense-in-depth blocking proposal submission to `pending_internal` when `attachment_kind` is null: disabled Submit button + inline hint (UI), early-return guard with toast (handler), and a top-level Firestore rules conjunction (server, applies to both admin and assigned-user branches).**

- **Completed:** 2026-05-24

## Accomplishments

- **Task 1 — UI layer (D-01/D-02):** In `renderProposalActionButtons` (`app/proposal-modal.js:391-409`), when `proposal.status` is `'draft'` or `'for_revision'` and `proposal.attachment_kind` is falsy, the Submit for Internal Approval button renders with `disabled` and a muted hint reading exactly `Add an attachment to submit for approval.` directly beneath. Edit Proposal stays enabled regardless of attachment state. When `attachment_kind` is set, Submit renders enabled byte-identically to the prior behavior.
- **Task 2 — Handler layer (D-05):** In `submitProposalForApproval` (`app/proposal-modal.js:955-963`), an early-return guard fires after the existing draft/for_revision status check and before `showLoading(true)`. When `proposal.attachment_kind` is falsy, the handler toasts `Add an attachment before submitting for approval.` (`error` severity) and returns without calling `_applyProposalStateTransition`. Catches stale-state clicks (DevTools bypass, attachment removed between render and click).
- **Task 3 — Server layer (D-03/D-04):** In `firestore.rules` proposals `allow update` rule (lines 611-659), wrapped the existing `BRANCH 1 || BRANCH 2` disjunction in a top-level conjunction enforcing `attachment_kind != null` whenever the doc is transitioning INTO `pending_internal`. Applies to both admin (BRANCH 1) and assigned-user (BRANCH 2) callers per D-04. D-13 anchor preserved.

## Decisions Made

- **Global `.btn:disabled` was found** at `styles/components.css:179` with the rule `opacity: 0.5; cursor: not-allowed;`. Per the plan's "Claude's Discretion" clause (CONTEXT line 70 and PLAN line 136), the inline `opacity:0.5;cursor:not-allowed;` override on the disabled Submit button was **omitted** in favor of relying on the global rule. The plan explicitly stated: "unless a `.btn:disabled` rule was found in components.css — in which case rely on the global rule and omit the inline override; document the choice in the SUMMARY." This is that documentation.
- **Top-level conjunction (D-03/D-04 server gate):** Rather than duplicating the `attachment_kind != null` check inside each of BRANCH 1 and BRANCH 2, wrapped the existing branch disjunction in a single top-level conjunction. Reads as: "the new status is NOT pending_internal" OR "the old status was already pending_internal (not a transition)" OR "attachment_kind is present on the new doc". Admins do not bypass — data integrity, not authorization.
- **Toast copy intentionally distinct from inline hint:** UI hint is "Add an attachment to submit for approval." (preventive); handler toast is "Add an attachment before submitting for approval." (reactive after stale-state click). Users can tell which defense layer fired.
