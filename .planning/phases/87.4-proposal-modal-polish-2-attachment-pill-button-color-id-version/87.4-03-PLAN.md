<objective>
Fix the audit/comms-pill background-fill bug observed in the proposal modal â€” pills currently render with their background extending the full row width instead of sizing to their text content (CONTEXT.md Item 5, decisions D-15..D-20).

This plan is DIAGNOSIS-FIRST per memory `feedback_root_cause_first` and D-20: the fix must address the actual CSS mechanism, NOT paper over with `overflow:hidden`, fixed widths, or `max-width`. Per revision 1 (checker WARNING #3, WARNING #4): `max-width` is REMOVED from the default rule template (opt-in only with explicit DevTools justification), and Task 1 must produce a verifiable file artifact (SUMMARY findings section) so Task 2 cannot proceed if diagnosis was skipped.

Working hypothesis based on code-context review (CONTEXT.md "Established Patterns" line 128):
- The audit-action label at `proposal-modal.js:345+` is a plain `<div>` with no pill class â€” likely NOT the culprit
- The comms-child pill at `proposal-modal.js:328+` is a `<span class="${meta.cls}">` placed inside a `display:flex` parent
- `.badge-primary` (line 600 of components.css) has NO `display` declaration â€” inherits default `inline` BUT inside a `display:flex` parent, flex children stretch to fill the cross-axis by default; combined with no `width: fit-content`, a long-pill-text + flex-row context produces the row-width fill
- `.status-badge` (line 541) has `display: inline-block` but NOT `width: fit-content` â€” flex children with `inline-block` can still stretch under certain flex-direction/align-items combinations

The fix adds `width: fit-content` (and reinforces `display: inline-block` where missing) at the class level so every current and future caller of the same class is covered.

Output: Updated `styles/components.css` (defensive rule at the class level) + 87.4-03-SUMMARY.md with `## Task 1 Diagnosis Findings` section.
