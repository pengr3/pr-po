<domain>
## Phase Boundary

Five polish items observed on the proposal modal during a super_admin walkthrough after Phase 87.3 shipped. All five touch `app/proposal-modal.js` (and a small Firestore rules tweak); no schema changes, no new collections, no new routes.

### In scope

1. **Attachment-required submission gate** — Submit for Internal Approval is hard-blocked when `proposal.attachment_kind` is unset. Client-side block + Firestore rules block.
2. **PROP-ID header surface decision** — Decision-record only. PROP-YYYY-NNN stays visible in the modal H2; zero code change for this item.
3. **Request Revision button color** — Recolor from red (`btn-danger`) to orange `#f59e0b` (matches `AUDIT_ACTION_DOT_COLORS.REVISION_REQUESTED` set in 87.2-04 D-13). Red becomes uniquely Mark as Loss.
4. **Dead Version field removal** — Delete the hardcoded `<div>v1</div>` at `app/proposal-modal.js:484`. Adjust the `buildProposalDetailsBlock` grid so Target Client and Project span the freed column. No schema change, no version concept introduced.
5. **Audit pill background-fill bug** — Root-cause CSS fix; sweep all `AUDIT_ACTION_LABELS` keys and all `COMMS_TYPE_META` classes so no current or future pill exhibits the same row-width overflow.

### Out of scope

- Any new proposal schema fields (e.g., a real `version` field) — deferred
- Surfacing PROP-YYYY-NNN on home queue table or stage cards — deferred (separate phase if wanted)
- Notification/audit changes
- Adding/removing action buttons beyond the Request Revision color change
- Touching unrelated proposal-modal surfaces (Edit modal layout, comms log section, attachment section UX) beyond the four items above
- Any change to file-upload / attachment-storage path (Blaze upgrade still pending — link-only attachments are the production reality)

<decisions>
## Implementation Decisions

### Item 1 — Attachment-required submission

- **D-01: Disable button + inline hint UX.** The "Submit for Internal Approval" button in `renderProposalActionButtons` is rendered with `disabled` attribute (and a muted style) when `proposal.attachment_kind` is unset. A small inline helper hint renders directly beneath the disabled button: `"Add an attachment to submit for approval."` The user never gets to the failure path.
- **D-02: Block applies to BOTH starting statuses.** The same disable + hint behavior fires whenever the Submit button is eligible to render — i.e., `status === 'draft'` (initial submit) AND `status === 'for_revision'` (resubmit after revision). Both transitions land at `pending_internal`; both require an attachment.
- **D-03: Firestore rule enforces server-side.** Extend the existing proposals `allow update` block: when `request.resource.data.status == 'pending_internal'` AND `resource.data.status != 'pending_internal'` (i.e., a transition INTO pending_internal), require `request.resource.data.attachment_kind != null`. Mirrors the existing field-mask pattern; defense-in-depth against console bypass.
- **D-04: Apply to both BRANCH 1 (admin) and BRANCH 2 (assigned ops/services user)** of the update rule. Admins do not get a bypass — the rule is about data integrity, not authorization.
- **D-05: No change to the Submit lifecycle handler (`submitProposalForApproval`)** beyond an early-return guard that toasts and bails if `proposal.attachment_kind` is null. This guard catches any path that bypasses the disabled button (e.g., a stale state where the button rendered enabled then attachment was removed before click). Belt-and-suspenders with the visual disable.

### Item 2 — PROP-ID header surface

- **D-06: Keep PROP-YYYY-NNN in the modal H2.** Zero code change. The investigation confirmed: `generateProposalId()` (`app/proposal-id.js:16`) is called on every Create at `app/proposal-modal.js:887`; `proposal_id` is written to the doc payload at `:897`; the modal H2 at `:429` renders `${proposal.proposal_id || proposal.id} — ${title}`. The ID is also used by NOTIF-09/10 `source_id`, queue links, and audit references. This item collapses to a decision record — no plan task generated.
- **D-07: PROP-ID visibility on home queue / stage cards** is explicitly deferred. The home Approval Queue table (`home.js:235`) currently shows only `p.title`; surfacing PROP-YYYY-NNN there would be additive scope and is filed as a deferred idea.

### Item 3 — Request Revision button color

- **D-08: Recolor via inline style on the button**, not a new CSS class. The existing button uses `class="btn btn-danger"` at `app/proposal-modal.js:411`. Drop `btn-danger`, keep `btn` base, add `style="background:#f59e0b;color:#fff;border-color:#f59e0b;width:100%;"` (or equivalent override that wins over `.btn` base rules). The exact final form is Claude's discretion — what matters is that the rendered color is `#f59e0b` and the button is visually consistent with the existing `btn-success` / `btn-danger` shape (radius, padding, font-weight).
- **D-09: Scope of recolor is the Request Revision button only** in `renderProposalActionButtons`. Do NOT touch the audit-trail REVISION_REQUESTED dot (already `#f59e0b` per 87.2-04 D-13) and do NOT touch comms-log REVISION_REQUESTED pill styling (handled separately by Item 5's broad sweep).
- **D-10: Red (`btn-danger`) remains exclusively for Mark as Loss** after this change. No other action button shares it. Verify by visual scan of all status branches in `renderProposalActionButtons` post-change.

### Item 4 — Version field removal

- **D-11: Remove the dead `<div>v1</div>` field entirely.** Delete the entire grid cell at `app/proposal-modal.js:482-485` (the label + value div pair labeled "Version"). No schema work, no write logic, no history concept introduced.
- **D-12: Adjust the grid layout** so the freed column is reclaimed. Current grid in `buildProposalDetailsBlock` is `grid-template-columns:1fr 1fr` with five cells: Title (span 2), Description (span 2), Amount (1), Target Client (1), Project (1), Version (1). After removal: Title (span 2), Description (span 2), Amount (1), Target Client (1), Project (1). Either let Project occupy its existing column and leave Amount alone (cleanest), or restructure to three columns in the bottom row. Claude's discretion on the exact post-removal layout — must look intentional, not like a hole.
- **D-13: Backward-compat note.** The Firestore rules field-mask added in 87.3 D-09 includes `'version'` as one of the 14 allowed update keys. We do NOT remove it from the field-mask in this phase — removing the visual surface doesn't justify a rules deploy, and a future "real versioning" phase would want the slot pre-allocated. Field-mask cleanup deferred to v4.1+.
- **D-14: No data migration.** Any existing proposal doc with a `version` field stays as-is in Firestore. Read paths simply never reference the field anymore.

### Item 5 — Audit pill background-fill bug

- **D-15: Broad sweep, root-cause CSS fix.** Researcher diagnoses the exact CSS cause. Fix applies a defensive rule (`width: fit-content` AND/OR `display: inline-block` — whichever the diagnosis says is missing) at the class level so every pill class used in the audit trail (`badge-primary`, `status-badge pending`, `status-badge rejected`, plus any future `COMMS_TYPE_META` entries) is covered.
- **D-16: Audit ALL `AUDIT_ACTION_LABELS` keys** rendered by `renderAuditTrail` (`proposal-modal.js:345`): CREATED, SUBMITTED, APPROVED, REJECTED, ATTACHMENT_REPLACED, SENT_TO_CLIENT, CLIENT_APPROVED, REVISION_REQUESTED, LOSS_RECORDED. Note: as of current code, these labels render in a plain `<div>` (no pill class) — so if the bug is on the audit-action LABEL itself, the fix involves either confirming the label has no background or restructuring how it's rendered. The researcher must pinpoint which element actually has the row-width background.
- **D-17: Audit ALL `COMMS_TYPE_META` classes** rendered by `_renderNestedCommsChild` (`proposal-modal.js:328`): `sent` (badge-primary), `feedback_received` (status-badge pending), `revision_requested` (status-badge rejected). The fix must be visually verified across all three pill types.
- **D-18: Visual smoke-test target.** A proposal with audit entries spanning CREATED → SUBMITTED → APPROVED → SENT_TO_CLIENT → CLIENT_APPROVED → REVISION_REQUESTED → SENT_TO_CLIENT (resend) → CLIENT_APPROVED or → LOSS_RECORDED, plus comms entries of all three types (sent / feedback_received / revision_requested), must render with every pill sized to its text content. PROP-2026-001 on CLMC-SPI-2026004 (the 87.2 walkthrough proposal) is a candidate smoke-test target if its history covers the necessary states; otherwise the researcher constructs one in dev.
- **D-19: Fix location preference — `styles/components.css`** (a defensive rule on the relevant class) over inline styles on the `<span>` elements. CSS rule survives future inline-style refactors and applies to any future caller of the same class outside the proposal modal. If the diagnosis requires per-call inline style, that's acceptable but document why in the plan.
- **D-20: Per `feedback_root_cause_first`** — do NOT paper over with `overflow:hidden` / explicit pixel widths / forced `max-width`. The fix must address the actual mechanism (likely a missing `width: fit-content` or `display: inline-block` on a class that's being placed in a block-level or full-width-flex-child context).

### Claude's Discretion

- Exact disabled-button styling for D-01 (opacity, `cursor: not-allowed`, color) — must match any existing disabled-button pattern in the codebase (`.btn:disabled` or equivalent); if none exists, use `opacity: 0.5; cursor: not-allowed;` as the conservative default.
- Exact inline hint copy and placement for D-01 — guide is `"Add an attachment to submit for approval."` placed directly under the disabled button as a small muted line (`font-size: 12px; color: #64748b; margin-top: 4px; text-align: center;` or similar).
- Exact post-removal grid layout for D-12 — must look intentional. Three-cell bottom row (Amount / Target Client / Project) with `grid-template-columns:repeat(3, 1fr)` for that row, OR keep two-column and let Project occupy the second column of its row. Pick what reads cleanest.
- Exact final form of the Request Revision orange button (D-08) — inline style vs a new `btn-warning` class. If `btn-warning` already exists in `styles/components.css` and matches the visual intent, use it; otherwise inline style is fine. Do NOT introduce a single-use class.
- Per-pill verification methodology for D-18 — manual visual scan in browser DevTools acceptable; automated test not required.

## Existing Code Insights

### Reusable Assets
- **`AUDIT_ACTION_DOT_COLORS.REVISION_REQUESTED = '#f59e0b'`** (`app/proposal-modal.js:98`) — the exact color value to reuse for D-08 button recolor (visual consistency between the action button and the resulting audit-trail dot).

- **`_isCallerAttachedToProposalParent(proposal)`** (`app/proposal-modal.js:164`) — returns `{ canApprove, canDrive }`; unchanged by this phase. Both items 1 and 3 sit inside its existing gates.
- **`_fetchProposalDoc(proposalDocId)`** (`app/proposal-modal.js:129`) — used by `submitProposalForApproval` to revalidate state at click time; D-05's early-return guard fits naturally before the existing status check.
- **`COMMS_TYPE_META`** (`app/proposal-modal.js:114`) — keys: `sent`, `feedback_received`, `revision_requested`. Classes referenced: `badge-primary`, `status-badge pending`, `status-badge rejected`. D-17 verification target.
- **`AUDIT_ACTION_LABELS`** (`app/proposal-modal.js:102`) — 9 keys. D-16 verification target.
- **Existing Firestore rules `allow update` BRANCH 1/BRANCH 2 structure** (`firestore.rules:613–642`) — the template for adding the attachment-presence sub-condition. Add it as a top-level conjunction so both branches inherit it.

### Established Patterns
- **Inline-style buttons in `renderProposalActionButtons`** — every button uses `class="btn ..." style="width:100%;"`. D-08's inline-style override follows the same pattern. New CSS classes are NOT introduced for one-off color changes.
- **Disabled-state visual** — codebase does not appear to have a global `.btn:disabled` rule that styles disabled buttons distinctly. Researcher confirms and, if missing, D-01 applies inline `opacity:0.5;cursor:not-allowed;` defensively.
- **Audit-pill rendering** — audit-action label currently renders as `<div style="font-size:13px;font-weight:600;color:#1e293b;">${label}</div>` (no pill class, no background). Comms-child pill renders as `<span class="${meta.cls}" style="font-size:11px;padding:1px 6px;border-radius:4px;">${label}</span>` inside a `display:flex` parent. The reported bug ("REVISION REQUESTED type-pill fills full row width") most plausibly targets the comms-child pill, NOT the audit-action label — researcher confirms via DOM inspection.
- **Firestore rules field-mask + state-transition gate** — Phase 87.2 introduced status-transition checks; the attachment-presence gate (D-03) follows the same `request.resource.data.status` vs `resource.data.status` comparison pattern.

### Integration Points
- **`renderProposalActionButtons` is called from `buildProposalDetailModalHtml`** (line 445). Re-render happens via `_refreshDetailModalAfterTransition` after any lifecycle action. D-01's disabled state therefore refreshes whenever an attachment is added or removed (`saveAttachment` / `removeAttachment` both call refresh).
- **Firestore rules attachment-presence check** (D-03) is the ONLY rule change in this phase. No new rule file, no new helper function. Production deploy required after merge (per memory `project_phase841_pending_followups` — confirm the prod rules deploy is current before adding new rules).
- **`styles/components.css` pill rule** (D-15/D-19) — fix lives in the same file as the existing `.status-badge` block (lines 541–588). Add a defensive base rule above all pill classes; do NOT scatter inline fixes across `proposal-modal.js`.

### Pitfalls to watch
- **PROP-ID surface visibility (Item 2)** — the user's initial belief was that PROP-IDs aren't generated. Verified: they ARE generated and stored, but only visible in the modal H2. If anyone in the future asks "why don't we see PROP-IDs?" — the answer is they're invisible on the queue and stage cards by design (D-07 deferred surfacing).
- **Disabled-button vs reactive-validation drift (D-01)** — if any future lifecycle handler skips the modal flow (e.g., a batch script), the Firestore rule (D-03) is the only line of defense. Don't remove the rule even if the visual disable proves bulletproof.
- **Audit-action label is NOT a pill in current code (D-15/D-16)** — researcher must NOT assume the label has a background to fix. The bug is most likely in `_renderNestedCommsChild`'s comms-pill `<span>`, OR the user is seeing something I haven't located. Diagnosis comes first; fix shape depends on what's actually rendering as row-width.
- **Field-mask preservation (D-13)** — the `'version'` key in the proposals update rule field-mask STAYS even though the visual surface is removed. Do not delete it from `firestore.rules:622`.
- **`btn-danger` reuse** — after D-08, only Mark as Loss should use `btn-danger`. Approve button uses `btn-success`. If a future phase adds a second danger-action it must justify whether it really wants red.
- **Firestore rules deploy timing** — D-03/D-04 require a production rules deploy. Memory `project_phase841_pending_followups` notes a prior pending deploy — verify current deploy state before adding more rule changes, to avoid an unintentional dual-deploy bundle.
