## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Firestore | Authenticated user write; client controls payload shape — rules are the only server-side gate |
| Modal UI → Lifecycle handler | `submitProposalForApproval` re-fetches the doc but trusts the click — disabled-state can be bypassed by DevTools |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-87.4-01 | Tampering | submitProposalForApproval (client) | mitigate | Early-return guard checks `proposal.attachment_kind` after refetch — disabled-button bypass via DevTools is caught here (D-05) |
| T-87.4-02 | Tampering | Direct Firestore client write bypassing modal | mitigate | Server-side rule in `firestore.rules` requires `attachment_kind != null` on `pending_internal` transitions for BOTH admin and assigned-user branches (D-03/D-04). Defense-in-depth even against console attacks. |
| T-87.4-03 | Elevation of Privilege | Admin bypass of data-integrity rule | accept | D-04 explicitly chose to apply the gate to admins too — rationale is data integrity, not authorization. Admins still have all other update privileges. |
| T-87.4-04 | Repudiation | Audit log of failed attachment-less submit attempts | accept | Failed rule-rejected writes are not logged to `audit_log` (they never commit). Acceptable: the user sees the toast and corrects; no compliance requirement to log denied attempts. |
| T-87.4-05 | Information Disclosure | Disabled-button reveals state to unauthorized viewer | accept | Read access to proposals is already gated by `isActiveUser()` (rules line 588); disabled-button state leaks no additional info beyond what the read grant already provides. |
</threat_model>
