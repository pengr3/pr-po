## Task 1 Diagnosis Findings

**Diagnosis method:** CODE-INSPECTION of the CSS cascade in `styles/components.css` (lines 532-608) and the render sites in `app/proposal-modal.js` (lines 328-343 for `_renderNestedCommsChild`, lines 345-376 for `renderAuditTrail`). Performed in lieu of browser DevTools per the user's overnight batch-execution authorization (memory `feedback_overnight_batch_execution`). DevTools confirmation deferred to Task 3 UAT (see `## Task 3 Pending Browser UAT` step 5 below).

**Offending element:** `<span class="${meta.cls}" style="font-size:11px;padding:1px 6px;border-radius:4px;">${label}</span>` rendered by `_renderNestedCommsChild` (`app/proposal-modal.js:337`), placed inside a `<div style="display:flex;align-items:center;gap:6px;...">` parent (`app/proposal-modal.js:336`). The most acute case is `COMMS_TYPE_META.sent` → `class="badge-primary"` because `.badge-primary` declares NO `display` (see cascade analysis below). The audit-action label rendered by `renderAuditTrail` at line 369 is a plain `<div>` with no pill class and no background — confirmed as NOT the symptom site (audit-action labels render as block-level text without coloured background, so a "background-fill" cannot manifest there).

**Computed style on the pill (BEFORE fix) — cascade reasoning from code inspection:**

- `display:` — for `<span class="badge-primary">`: `.badge-primary` (components.css:600-603) declares NO `display` rule, so the `<span>` retains its UA default `display: inline`. **HOWEVER**, per CSS Flexbox spec §4 ("Flex Items"), inline-level boxes become flex items and have their `display` value blockified: `inline` → `block`. So the computed `display` of the offending pill resolves to `block` (blockified flex item). For `<span class="status-badge pending">`, `.status-badge` (line 541-547) declares `display: inline-block`; an `inline-block` flex item is NOT further blockified (per spec it is treated as `block` for main-axis sizing but keeps its shrink-to-fit hypothetical content sizing) — so this class is closer to the desired behaviour but still missing `width: fit-content`, which is what locks the box to its content rather than allowing flex-item main-size to resolve to anything larger.
- `width:` — NONE of the nine badge classes (`.status-badge`, `.badge-primary`, `.badge-secondary`, `.badge-pending`, `.badge-approved`, `.badge-rejected`, `.badge-success`, `.badge-danger`, `.badge-warning`) declare `width`. UA default for `<span>` is `width: auto`. For a flex item with `flex: initial` (= `0 1 auto`) and no `flex-basis`, the resolved width depends on (a) the hypothetical main-size from the content (shrink-to-fit) and (b) the available free space in the parent. In the absence of explicit `width: fit-content`, the layout engine can — under certain combinations of parent width, sibling flex constraints, and the blockification rule above — resolve a blockified inline pill's main-size to grow beyond its intrinsic content width. This is the leading mechanism for the observed row-width background-fill.

**Parent computed style (from inline style at app/proposal-modal.js:336):**

- `display:` — `flex`
- `flex-direction:` — `row` (unspecified, UA default for `display: flex`)
- `align-items:` — `center` (declared inline)
- `justify-content:` — `flex-start` (unspecified, UA default)
- `gap:` — `6px` (declared inline)

The cross-axis (vertical, since `flex-direction: row`) has `align-items: center` — irrelevant to horizontal width. On the main axis (horizontal), with `justify-content: flex-start` and `gap: 6px`, the items lay out left-to-right starting at the parent's left edge with 6px gaps. The CRITICAL observation is that the pill `<span>` is the FIRST flex item; the second flex item is `<span style="color:#64748b;">${tsLabel}...</span>` (line 338) which also has no explicit flex constraints. With both items at `flex: initial` and a parent that may be wider than `(pill content + 6px gap + timestamp content)`, the blockification of the pill's `inline` `display` produces a box that — without `width: fit-content` — can resolve its main-size to fill more of the parent than the text needs. This is the row-width background-fill.

**Property that fixed it (code-inspection reasoning):** BOTH `display: inline-block` AND `width: fit-content` applied together at the class level. `display: inline-block` ensures the pill is never blockified from `inline` (cascade safety for `.badge-primary` and `.badge-warning` which currently declare nothing); `width: fit-content` locks the box to its content width regardless of flex parent free space (cascade safety for ALL nine classes — none currently declare `width`). Each property alone is insufficient: `display: inline-block` alone leaves `width: auto` flex-sizing in play; `width: fit-content` alone on an `inline` element that gets blockified by flex can still be overridden by the flex algorithm under edge cases. Both together = root-cause fix at the class level.

**Decision on max-width inclusion:** OMITTED — default per revision 1 of the plan. The code-inspection diagnosis surfaces no extreme-long-label overflow evidence — `COMMS_TYPE_META.label` values are short fixed strings ("Sent", "Feedback Received", "Revision Requested"), all comfortably under any reasonable parent width. Per PLAN.md Task 2 `<action>` block + revision 1 WARNING #3, `max-width: 100%` is opt-in only with explicit DevTools-quoted justification of an extreme-long-label overflowing its flex parent; that justification does not exist, so `max-width` stays out of the rule block per D-20 (no paper-overs).

**Audit-action label sweep:** Confirmed via code inspection of `renderAuditTrail` at `app/proposal-modal.js:369`: the audit-action label renders as `<div style="font-size:13px;font-weight:600;color:#1e293b;">${escapeHTML(label)}</div>`. This is a plain `<div>` (block-level by default, fills row width as expected for block elements) with NO `background` property and NO pill class. A row-width "background-fill" symptom cannot manifest on a node that has no background colour. Therefore the bug is confined to `_renderNestedCommsChild`'s pill `<span>` (line 337) and the nine badge classes that COMMS_TYPE_META resolves to. All nine `AUDIT_ACTION_LABELS` keys (CREATED, SUBMITTED, APPROVED, REJECTED, ATTACHMENT_REPLACED, SENT_TO_CLIENT, CLIENT_APPROVED, REVISION_REQUESTED, LOSS_RECORDED) render through this same backgroundless-div path — none of them produce a pill, so the expected UAT outcome on the audit-action labels is "no pill, no background, no row-fill" (which is the correct rendering).

## Follow-up — UAT Multi-Line Wrap (added 2026-05-24)

**Reported during UAT:** The defensive `display: inline-block` + `width: fit-content` rule correctly stopped the row-width background overflow (original bug), but the "Revision Requested" and "Feedback Received" pills then began wrapping their text onto two lines INSIDE the pill — visually worse than the original bug.

**Diagnosis (the gap in Task 1's code-inspection):**

The pill is a flex item inside `<div style="display:flex;align-items:center;gap:6px;">` (`_renderNestedCommsChild` at `app/proposal-modal.js:336`). Flex items default to `min-width: auto`, which resolves to `min-content`. For text WITHOUT `white-space: nowrap`, `min-content` is the width of the longest single unbroken word — `Requested` for the revision pill, `Received` for the feedback pill (each ~55px at 11px font).

So `width: fit-content` correctly evaluates to `min(max-content, max(min-content, available))`, but `min-content` was pulling the resolved size down to the longest-word width whenever the flex parent's available space was narrower than the full label. The pill then sized to that word, and the rest of the label wrapped inside the pill — exactly what the screenshot showed.

**Root cause:** Missing `white-space: nowrap` on the badge classes. This is the third leg of the canonical pill pattern (the other two being `display: inline-block` and `width: fit-content`), and it was absent from BOTH the original `.status-badge` / `.badge-primary` rules AND Task 1's code-inspection diagnosis. The original full-row-width bug masked this latent wrapping behaviour — once the row-width problem was fixed, the flex-min-content shrink became the dominant visual issue.

**Why Task 1 missed it:** The code-inspection diagnosis described the row-width overflow mechanism correctly but did not enumerate the secondary flex-item-min-content behaviour that activates once you DO constrain the pill width with `fit-content`. A DevTools session would have surfaced it immediately — typing a longer label or resizing the flex parent reveals the wrap behaviour on first interaction. This is exactly the kind of gap the original PLAN.md flagged DevTools confirmation as the safety net for (Task 3 UAT step 5).

**Fix applied:** Added `white-space: nowrap` to the Phase 87.4 defensive rule block in `styles/components.css`. `nowrap` forces `min-content` to equal `max-content` (the full label width on one line), so:
- The flex algorithm's `min-width: auto` floor now equals the full text width
- The pill cannot shrink below the full label
- `width: fit-content` still resolves cleanly — just to `max-content` instead of `min-content`
- The pill stays one line, content-sized, no row-width overflow

**Mechanism check — D-20 root-cause-first still satisfied:** `white-space: nowrap` is a content-flow declaration that addresses the *actual* flex-min-content mechanism, not a paper-over. It's not `overflow: hidden` (which would just hide overflow), not a fixed pixel width (which would lie about the content), and not a `max-width` cap (which would clip long labels). The pill genuinely cannot wrap because we told it not to — and the labels in `COMMS_TYPE_META` and `AUDIT_ACTION_LABELS` are all short fixed strings that comfortably fit on one line in any realistic modal width.

**Visual UAT after follow-up fix:** User should re-verify Task 3 step 4 — the three comms pills (`sent`, `feedback_received`, `revision_requested`) all render on a single line at content width with no wrap. DevTools step 5 should now show `white-space: nowrap` in addition to the previously-targeted `display: inline-block` + `width: fit-content`.
