### Human Verification Required

All automated checks pass. The following items require browser testing before the phase can be marked UAT-complete.

#### 1. Overlay Appears Immediately on First Baseline Save

**Test:** Open a project plan with at least 3 tasks that have start and end dates. Click "Set Baseline". Observe the Gantt immediately after the success toast.
**Expected:** Dashed grey outline rects appear overlaid on each task bar at the task's current date range. Slip summary row above the chart shows "0 behind / 0 ahead / N on track" (matching number of tasks with dates).
**Why human:** SVG DOM injection into a live Frappe Gantt canvas; SVG coordinate math output (x/width of rects) requires visual confirmation that rects align with bars.

#### 2. Behind (Late) Task Badge

**Test:** After setting a baseline, drag a task bar's right edge to a later date (or edit the end date field). Wait for the Gantt to re-render.
**Expected:** A red pill badge (e.g. "+3d") appears to the right of that task's bar. Slip summary "behind" count increments by 1.
**Why human:** Requires live Gantt interaction, badge pixel positioning (`curX2 + 4`), and correct diffDays calculation — cannot be verified without a browser.

#### 3. Ahead (Early) Task Badge

**Test:** After setting a baseline, drag a task bar's right edge to an earlier date.
**Expected:** A green pill badge (e.g. "-2d") appears beside that bar. Slip summary "ahead" count increments by 1.
**Why human:** Same as above — negative diffDays branch requires browser state with loaded baseline.

#### 4. On-Track Task Shows Outline Only — No Badge

**Test:** Leave at least one task end date unchanged from when the baseline was set.
**Expected:** That task bar shows only a dashed grey outline rect. No colour pill badge appears beside it.
**Why human:** Confirms `diffDays === 0` early-exit in `injectBaselineOverlay()` produces a clean visual result.

#### 5. Overlay Persists After Zoom Change

**Test:** With a baseline loaded and badges visible, switch Gantt zoom from Day → Week → Month and back.
**Expected:** Dashed outline rects and slip badges re-appear at visually correct positions after each zoom change. Slip summary counts remain correct.
**Why human:** `setGanttZoom()` triggers Frappe SVG rebuild; x-coordinate correctness under each `column_width`/`step` combination (ganttXPerDay) requires visual inspection.

#### 6. Phase 86.11 Bar Fill Colours Unaffected

**Test:** Observe bar colours for overdue tasks (red), complete tasks (green), not-started tasks (grey) before and after setting a baseline.
**Expected:** No bar fill colour changes after clicking "Set Baseline". The overlay is additive (dashed outlines + badges only).
**Why human:** CSS class mutation or fill attribute overwrite would only be detectable via live DOM inspection or visual comparison.

#### 7. Baseline Overlay Present on Page Reload

**Test:** After saving a baseline, reload the project plan page.
**Expected:** Without any user interaction, the slip summary row appears (with the saved baseline counts) and dashed outlines are visible on task bars.
**Why human:** Depends on `init()` → `loadBaseline()` → Firestore read → `project_tasks` onSnapshot triggering `renderGantt()` which calls `injectBaselineOverlay()`. Timing of Firestore snapshot delivery is live-only.
