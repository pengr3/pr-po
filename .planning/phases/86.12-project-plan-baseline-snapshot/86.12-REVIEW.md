## Findings

### WR-01 — Overlay does not render immediately after first baseline save

**Severity**: Warning | **Confidence**: 90
**File**: `app/views/project-plan.js`
**Function**: `saveBaseline()`

After `saveBaseline()` calls `await loadBaseline()` (which correctly populates `_baselineData`), the function shows the success toast and returns. Neither `injectBaselineOverlay()` nor `renderSlipSummary()` is called. These are only reached via `renderGantt()` (triggered by `project_tasks` onSnapshot) or `setGanttZoom()`. Writing to the `baselines` subcollection does not trigger the `project_tasks` listener, so no re-render fires. On the first-ever baseline save — when `_baselineData` was previously `null` — the slip summary row stays hidden and no dashed outlines appear. The user sees a success toast but no visual change until they drag a Gantt bar, edit a task, or reload.

**Fix** — add overlay calls in `saveBaseline()` after `loadBaseline()` returns:

```javascript
await loadBaseline();
injectBaselineOverlay();
renderSlipSummary();
showToast(`Baseline "${label}" saved`, 'success');
```

### WR-02 — `dateToX` mutates `gantt.gantt_start` in-place on every loop iteration

**Severity**: Warning | **Confidence**: 80
**File**: `app/views/project-plan.js`
**Function**: `dateToX()`

```javascript
const anchor = gantt.gantt_start instanceof Date ? gantt.gantt_start : new Date(gantt.gantt_start);
anchor.setHours(0, 0, 0, 0);
```

When `gantt.gantt_start` is already a `Date`, `anchor` holds the same reference — not a clone. `anchor.setHours(0,0,0,0)` mutates Frappe's internal state. The same pattern exists in `renderTodayLine` (one-shot call), but `dateToX` is called in a loop — once per task in `injectBaselineOverlay`. In practice `gantt_start` is already midnight-normalized, making the mutation idempotent. However if Frappe ever produces a non-midnight anchor (DST edge case, future version), the first call corrupts it and all subsequent loop iterations silently use the corrupted value, producing incorrect x-coordinates for every task after the first.

**Fix** — clone before mutating:

```javascript
const anchor = new Date(
    gantt.gantt_start instanceof Date ? gantt.gantt_start.getTime() : gantt.gantt_start
);
anchor.setHours(0, 0, 0, 0);
```

### IN-01 — `saveBaseline` label derivation is non-atomic (benign in single-user SPA)

**Severity**: Info | **Confidence**: 85
**File**: `app/views/project-plan.js`
**Function**: `saveBaseline()`

Label number is derived from `countSnap.size + 1` via `getDocs` before the `addDoc`. Two concurrent saves would read the same `size` and both write the same label string (e.g., `Baseline 2` twice). In this single-user SPA the scenario is unrealistic and the label is cosmetic only (doc ID is auto-generated Firestore ID). No action required unless multi-user collaborative editing is introduced.
