## What Was Done This Session

### Bug 1 — MRF visibility: operations_user / services_user saw all MRFs ✅ FIXED
**Root cause (two independent gaps):**
- `operations_user`: predicate `!mrf.project_code` was falsy-checked, so service-type MRFs with `project_code: ''` (empty string) always passed the filter.
- `services_user`: `getAssignedProjectCodes()` returns `null` for all roles except `operations_user`, so the filter block was never entered — zero filtering applied. `getAssignedServiceCodes()` existed in utils.js and was on `window` but was **never called in procurement.js**.

**Fix applied:** `298ad05`
- Changed `!mrf.project_code` → `mrf.project_code == null` at all 5 project-filter call-sites.
- Added a parallel service-scope filter block (`getAssignedServiceCodes()`) at all 5 call-sites.
- Call-sites: `loadMRFs` snapshot (~2672), `reFilterAndRenderMRFs` (~2745), `reFilterAndRenderPRPORecords` (~2788), `loadPRPORecords` cache-hit (~5025), `loadPRPORecords` fresh-fetch (~5095).

### Bug 2 — `loadServicesForNewMRF` permission error ✅ FIXED
**Root cause:** Unscoped `where('active', '==', true)` query on `services` collection. `operations_user` has no `list` access to `services`. `services_user` per-document rule fails for non-assigned services in an unscoped query.

**Fix applied:** `22140f0`
- `operations_user`: return early (project-scoped, no service types in their MRF form).
- `services_user`: scope query to `where('service_code', 'in', assignedServiceCodes)` — mirrors `mrf-form.js:loadServices` pattern.

## Bug 3 — Scoreboards show unfiltered PO counts ⏳ DIAGNOSED, NOT YET IMPLEMENTED

**Root cause (from `.planning/debug/scoreboard-unfiltered-counts.md`):**
- `loadPRPORecords` fetches the entire `pos` collection independently (~line 5108–5114) as a local `allPOData` variable.
- `updatePOScoreboards(allPOData)` receives all POs with no scope constraint (~line 5117).
- `reFilterAndRenderPRPORecords` never calls `updatePOScoreboards` at all — scoreboards are stale on assignment change too.

**Agreed fix approach (explained to user, ready to implement):**
1. Add a module-level `cachedAllPOData = []` variable.
2. In `loadPRPORecords`: after fetching POs, save to `cachedAllPOData`. Then derive visible MRF IDs from the already-filtered `allPRPORecords` and pass only scoped POs to `updatePOScoreboards`:
   ```js
   const visibleMrfIds = new Set(allPRPORecords.map(m => m.mrf_id || m.id));
   updatePOScoreboards(cachedAllPOData.filter(po => visibleMrfIds.has(po.mrf_id)));
   ```
3. In `reFilterAndRenderPRPORecords`: add the same scoped scoreboard call after re-scoping `allPRPORecords`.

**Key data model note:** POs have a `mrf_id` field linking them to their MRF. MRFs use `m.id` (the Firestore doc ID) not `m.mrf_id` for their own ID. The Set should use `m.id` when building from MRF records, and match against `po.mrf_id`.
