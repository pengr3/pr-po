| services_user role → other teams' service_tasks | A field-team user must not write tasks for services they are not assigned to |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-105-01 | Elevation of Privilege | service_tasks create/delete | mitigate | create/delete gated on `hasRole(['super_admin','services_admin'])` OR `isRole('services_user') && isAssignedToService(service_code)` — field team cannot author tasks for unassigned services |
| T-105-02 | Tampering | service_tasks progress-only update | mitigate | Tier 2 `affectedKeys().hasOnly(['progress','updated_at'])` + `request.auth.uid in resource.data.assignees` — non-assignees cannot edit progress; finance/procurement/services-non-assignee are read-only even if added to assignees |
| T-105-03 | Elevation of Privilege | new rules block widening other collections | mitigate | block is scoped to `match /service_tasks/{taskId}` only; project_tasks + all other blocks byte-untouched (verified by git diff); short-circuit `isRole('services_user') &&` guard prevents dereferencing `assigned_service_codes` on non-services_user docs |
| T-105-04 | Information Disclosure | service_tasks read scope | accept | `allow read: if isActiveUser()` mirrors project_tasks (JS-layer scoping). Task names/dates are low-sensitivity internal scheduling data, no PII/financials; matches the shipped project plan posture (D-06 default) |
