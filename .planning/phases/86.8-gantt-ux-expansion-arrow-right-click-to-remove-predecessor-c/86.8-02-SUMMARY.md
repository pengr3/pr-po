- **Feature 4 — Critical-path highlight.** Added `computeCriticalPath(taskList)` (forward+backward pass CPM, leaf-only participation, cycle protection via iteration cap), `_criticalPathSet` module state recomputed every renderGantt, `applyCriticalPathStyles()` DOM tag pass that adds `.critical` to bars (`.bar-wrapper`) and to arrow paths (`gantt.arrows[].element`), and `toggleCriticalPath(checked)` toolbar handler. Parents inherit visual state from any critical descendant — a critical leaf inside a collapsed subtree still flags its parent so the user sees risk before expanding.
- **Feature 5 — Progress % column + weighted rollup.** Added `computeWeightedProgress(parentId)` at module scope (consumed by both renderTaskGrid for the rollup span AND renderGantt for the Frappe `progress` field). - **Feature 6 — Task search/filter bar.** Added `matchesSearch(t)` and `getVisibleByFilter(t)` (three-rule visibility: direct match / descendant match shows parent greyed / ancestor match keeps children for context). Sticky search bar with `type="search"` input + X clear button renders above the grid in BOTH branches (empty-state + populated). Search composes with `_collapsedParents`: a row is visible iff it passes BOTH filters. Same predicate chain in `renderGantt` (filteredFrappeTasks) so left rail and right pane filter in lockstep. Caret survives re-renders via `_savedSearch` capture/restore (mirrors the Phase 86.5-08 empty-row pattern). Clear button restores all rows and re-focuses the input. `.tg-row-context` greys (opacity 0.55) parents kept around because a descendant matched.
- **destroy() cleanup.** Every new state variable cleared, every new listener removed: `_criticalPathSet`/`_showCriticalPath`/`_searchQuery` reset; `_searchInputHandler` removed from the search input; `window.toggleCriticalPath` deleted.

## Decisions Made

See `key-decisions` in frontmatter. Most consequential:

1. **CPM operates on LEAVES ONLY** — parents are envelope-computed; including them would create duplicate critical edges through every envelope. Parents inherit visual via the DOM tag pass.
2. **Arrow `.critical` lives on the path, not the layer** — Frappe v1.2.2 puts every arrow path under one `<g class="arrow">`, so `.arrow.critical` (which the plan suggested) would never match. Used `gantt.arrows[].element` (Plan 01's source of truth for arrow data) and CSS selector `g.arrow path.critical`. This is the single intentional CSS-selector deviation from the plan, called out in the orchestrator's drift notes.
3. **Predicate composition** — collapse and search use the SAME chain in both panes so the rail and Gantt filter together. Empty query short-circuits to visible so the no-filter state has zero overhead.
4. **filteredFrappeTasks replace-in-place** — `frappeTasks.length = 0; Array.prototype.push.apply(frappeTasks, filteredFrappeTasks)` mutates the array in place so the rest of `renderGantt` (refresh, init, scroll-clamp, today-line, etc.) reads from the same `frappeTasks` reference without further changes.

## Deviations from Plan

**1. [Plan deviation — CSS selector correctness, called out in orchestrator drift notes]**
The plan's CSS instructed `.gantt-container .arrow.critical { stroke: #ef4444; ... }` and the JS pass said to iterate `svg.querySelectorAll('.arrow')`. Frappe v1.2.2 layers all arrow paths under a single `<g class="arrow">` — `.arrow.critical` would only match the layer group, not the individual paths. Used `gantt.arrows[].element` (Plan 01's authoritative arrow handle) for tagging and `.gantt-container g.arrow path.critical` for the stroke. Identical visual outcome; no behavioral change beyond actually working.

**2. [Plan deviation — search ancestor-walk cycle guard]**
The plan's `getVisibleByFilter` ancestor walk had no termination guard. Added `let guard = 0; while (cur && guard < 64)` to defend against any future schema bug that creates a parent_task_id cycle. Cycle would already be toasted by `detectDependencyCycle`, but a 64-iteration cap is cheaper than an infinite loop. Belt-and-braces.

Otherwise, plan executed as written. The drift items the orchestrator flagged (renderGantt line ~1290 / appendNode line ~1300 / colspan exact-2 occurrences / etc.) all matched reality and required no action beyond reading the current source and applying surgical edits.
