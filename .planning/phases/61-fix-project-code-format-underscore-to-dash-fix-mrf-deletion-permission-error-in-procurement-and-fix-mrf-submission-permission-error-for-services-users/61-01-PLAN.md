In `app/utils.js`, update BOTH `generateProjectCode()` and `generateServiceCode()` to use dash separators instead of underscores:

In `generateProjectCode()`:
- Change `rangeMin = \`CLMC_${clientCode}_${currentYear}000\`` → `rangeMin = \`CLMC-${clientCode}-${currentYear}000\``
- Change `rangeMax = \`CLMC_${clientCode}_${currentYear}999\`` → `rangeMax = \`CLMC-${clientCode}-${currentYear}999\``
- Change `codeRegex = /^CLMC_.+_\d{4}(\d{3})$/` → `codeRegex = /^CLMC-.+-\d{4}(\d{3})$/`
- Change return `\`CLMC_${clientCode}_${currentYear}${...}\`` → `\`CLMC-${clientCode}-${currentYear}${...}\``

In `generateServiceCode()` — same four changes (identical pattern):
- `rangeMin`, `rangeMax`, `codeRegex`, and return value

Also update the JSDoc comments in both functions:
- `@returns {Promise<string>} Generated project code (e.g., CLMC_ACME_2026001)` → `(e.g., CLMC-ACME-2026001)`
- `@returns {Promise<string>} Generated service code (e.g., CLMC_ACME_2026003)` → `(e.g., CLMC-ACME-2026003)`
- The description comment `Generate composite project code: CLMC_CLIENT_YYYY###` → `CLMC-CLIENT-YYYY###`

Do NOT change anything else in utils.js. The range queries use Firestore lexicographic ordering — `CLMC-ACME-2026000` to `CLMC-ACME-2026999` works correctly because `-` (ASCII 45) sorts before digits, just as `_` (ASCII 95) does. Ordering is preserved.

<task type="auto">
  <name>Task 2: Fix Firestore rules — procurement delete MRF, services_user list MRFs</name>
  <files>firestore.rules</files>
  <action>
Make three targeted changes to `firestore.rules`:

**Change 1 — mrfs delete rule:** Add `procurement` to the allowed roles so procurement users can delete MRFs.

Current:
```
// Delete: super_admin, operations_admin
allow delete: if hasRole(['super_admin', 'operations_admin']);
```

Change to:
```
// Delete: super_admin, operations_admin, procurement
allow delete: if hasRole(['super_admin', 'operations_admin', 'procurement']);
```

**Change 2 — deleted_mrfs create rule:** Add `procurement` to the allowed roles so the audit trail document can be written when procurement deletes an MRF. The `deleteMRF()` function writes to `deleted_mrfs` before deleting the MRF.

Current:
```
// Create: super_admin, operations_admin (append when MRF is soft-deleted)
allow create: if hasRole(['super_admin', 'operations_admin']);
```

Change to:
```
// Create: super_admin, operations_admin, procurement (append when MRF is soft-deleted)
allow create: if hasRole(['super_admin', 'operations_admin', 'procurement']);
```

**Change 3 — mrfs list rule:** Add `services_user` to the unrestricted list branch so `generateMRFId()` can call `getDocs(collection(db, 'mrfs'))` without a per-document permission failure. The current scoped list rule `(isRole('services_user') && isAssignedToService(resource.data.service_code))` causes Firestore to deny the query when any MRF doc (e.g., a project-type MRF with service_code='') doesn't pass the per-document check.

Current mrfs list rule:
```javascript
allow list: if isActiveUser() && (
  hasRole(['super_admin', 'operations_admin', 'services_admin', 'finance', 'procurement']) ||
  (isRole('operations_user') && isLegacyOrAssigned(resource.data.project_code)) ||
  (isRole('services_user') && isAssignedToService(resource.data.service_code))
);
```

Change to:
```javascript
// List: cross-dept roles unrestricted; operations_user project-scoped; services_user unrestricted
// NOTE: services_user was previously scoped to assigned services only, but this blocked
// generateMRFId() in mrf-form.js from doing an unscoped getDocs(mrfs) query needed to
// generate the next MRF sequence number. services_user list is now unrestricted since
// they can already 'get' any individual MRF doc and MRF metadata is not sensitive.
allow list: if isActiveUser() && (
  hasRole(['super_admin', 'operations_admin', 'services_admin', 'services_user', 'finance', 'procurement']) ||
  (isRole('operations_user') && isLegacyOrAssigned(resource.data.project_code))
);
```

Do NOT change the read/get rules or any other collection rules. Do NOT change the `prs`, `pos`, `transport_requests` list rules for services_user (those are separate and not broken). Only modify the `mrfs` collection list rule and the delete rule, and the `deleted_mrfs` create rule.
  </action>
