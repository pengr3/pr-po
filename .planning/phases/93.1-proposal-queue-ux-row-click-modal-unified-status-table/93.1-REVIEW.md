## Critical Issues

### CR-01: NaN sort comparator — approval queue not sorted oldest-first

**File:** `app/views/home.js:573-574`

**Issue:** The timestamp extraction for the approval-queue sort uses chained `??` operators, but arithmetic (`* 1000`) has higher precedence than `??`. When `current_status_since` is a plain Firestore-style object with a `seconds` field, the expression evaluates as:

```
(a.current_status_since?.seconds * 1000) ?? 0
```

If `current_status_since` is absent entirely, `a.current_status_since?.seconds` is `undefined`, and `undefined * 1000` is `NaN`. Because `??` only short-circuits on `null`/`undefined` — not `NaN` — `NaN ?? 0` returns `NaN` rather than `0`. A sort comparator that returns `NaN` produces implementation-defined (browser-dependent) ordering, silently breaking the "oldest-first" guarantee documented above the function.

**Fix:** Parenthesise the multiplication branch and add an explicit `isNaN` guard:

```javascript
const tsA = a.current_status_since?.toMillis?.()
    ?? (a.current_status_since?.seconds != null ? a.current_status_since.seconds * 1000 : 0);
const tsB = b.current_status_since?.toMillis?.()
    ?? (b.current_status_since?.seconds != null ? b.current_status_since.seconds * 1000 : 0);
```

This ensures the fallback `0` is used when `seconds` is absent rather than propagating `NaN`.

### WR-02: Approval-queue amount missing currency symbol

**File:** `app/views/home.js:242`

**Issue:** The approval queue renders `amount` via `formatCurrency(p.amount)` (no peso symbol), while the unified proposals table at line 489 prepends `'₱'`. A user looking at the approval queue sees bare numbers like `1,500.00` with no indication of currency, while the same proposal appears as `₱1,500.00` in the table below. Both surfaces are visible on the same tab.

**Fix:** Apply the same `'₱' + formatCurrency(p.amount)` pattern used in `_renderHomeProposalTable`:

```javascript
// line 242 — approval queue row builder
const amount = typeof p.amount === 'number' ? '₱' + formatCurrency(p.amount) : '—';
```

### WR-03: Unified table shows "0 days" for same-day proposals; queue shows "Today"

**File:** `app/views/home.js:484-501` (via `renderAgeBadge` at `proposals.js:140-147`)

**Issue:** The unified proposals table calls `renderAgeBadge(p)` which formats `days === 1 ? '1 day' : \`${days} days\`` — producing "0 days" when a proposal was created or transitioned today. The approval queue directly above it uses `ageDays < 1 ? 'Today' : ...` (line 244) so a just-submitted proposal shows "Today" in the queue but "0 days" in the table on the same page. The inconsistency is user-visible when the Proposals sub-tab loads.

**Fix:** The simplest fix is to pass a `showToday` flag through `renderAgeBadge`, or replicate the same ternary inline:

```javascript
// Option A — inline in _renderHomeProposalTable row builder:
const ageDays = getAgeInStageDays(p);
const ageDisplay = ageDays < 1 ? 'Today' : ageDays === 1 ? '1 day' : `${ageDays} days`;
// then use ageDisplay in the <td> instead of renderAgeBadge(p)
```

The underlying `renderAgeBadge` helper in `proposals.js` also has this gap (it has no "Today" branch) but that is an issue in proposals.js, not this file.

## Info

### IN-01: Race condition window between scorecard click and `_loadHomeProposalsTab` re-render

**File:** `app/views/home.js:526-541` / `app/views/home.js:652-654`

**Issue:** `_loadHomeProposalsTab` is `async` and resets `_homeProposalStatusFilter = null` at line 557 before the Firestore `getDocs` resolves. If a user clicks a scorecard tile (which writes to `_homeProposalStatusFilter` via `handleHomeProposalScorecardClick`) while a post-action `_loadHomeProposalsTab` call is in-flight and has cleared the DOM but not yet re-injected `homeProposalTableSection`, `_rerenderProposalTable` returns early on `if (!existing) return` (line 531). The filter state (`_homeProposalStatusFilter`) is now set to the clicked key, but the UI will show an unfiltered table once `_loadHomeProposalsTab` finishes (because it always renders with `null` filter). The state and display are silently out of sync until the next interaction.

This window is narrow (the `getDocs` round trip) and recoverable with another click, but it is worth noting for a view that re-fetches on every approve/reject action.

**Fix:** Set `_homeProposalStatusFilter = null` after `getDocs` resolves (just before building HTML), or guard `handleHomeProposalScorecardClick` with a loading flag that disables scorecard clicks while a fetch is in flight.
