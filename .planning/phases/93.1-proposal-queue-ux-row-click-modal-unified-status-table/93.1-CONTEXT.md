# Phase 93.1: Proposal Queue UX — Row-Click Modal + Unified Status Table with Filterable Scorecards - Context

**Gathered:** 2026-05-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Two UX improvements to the **Home → Proposals sub-tab** (`app/views/home.js`):

1. **Queue row-click** — Every row in the Proposal Approval Queue becomes clickable. Clicking anywhere on a row (except the Approve/Reject buttons) opens the proposal detail modal via `window.openProposalModal`. Approve/Reject buttons remain in the last column and stop click propagation so they still work independently.

2. **Unified proposals table with filterable scorecards** — The current per-status stage group cards (`renderStageGroupCard` per stage) are replaced by a **single unified table**. Five scorecard tiles above the table (one per proposal stage) filter the table rows when clicked. The queue section remains as a separate card above the unified table.

### In scope

- Add full-row click handler to queue table rows (`_renderHomeApprovalQueueHtml`) → `window.openProposalModal(p.id)`
- Replace `renderStageGroupCard` loop in `_loadHomeProposalsTab` with a single unified table
- Add 5 scorecard tiles above the unified table (Pending Internal, Pending Client, For Revision, Client Approved, Loss)
- Scorecard click = filter table to that status; click active scorecard again = reset to show all
- Unified table includes **all 5 proposal stages** (both active and terminal: client_approved, loss)
- Table columns: Status | PROP-ID | Title | Project | Client | Amount | Age
- All proposals still scoped through `filterProposalsForUser` (role-based filter unchanged)

### Out of scope (deferred)

- Changing the Approval Queue component itself (it stays above the table as today, with Approve/Reject buttons per row)
- Adding "For Proposal" project-scope items to the proposals table — those are project/service records, not proposal documents
- Inline approve/reject actions in the unified table (terminal actions remain in the queue)
- Any change to proposal data schema, `_applyProposalStateTransition`, or audit trail
- Mobile-specific layout changes beyond what falls out of the existing CSS patterns

</domain>

<decisions>
## Implementation Decisions

### Queue row-click

- **D-01:** Full `<tr>` row click opens the proposal modal — cursor: pointer on hover, subtle row highlight on hover (e.g. `background:#f8fafc` on `:hover` via inline `onmouseenter/onmouseleave` or a style block). Clicking anywhere on the row fires `window.openProposalModal(p.id)`.
- **D-02:** Approve/Reject buttons in the Actions column call `event.stopPropagation()` so they do NOT trigger the row-click modal. Their existing behavior is fully preserved.
- **D-03:** `window.openProposalModal` is already registered in `home.js` `init()` from `proposal-modal.js` — no new registration needed.

### Queue + unified table layout

- **D-04:** The Approval Queue card **stays as a separate section** above the unified table. Both components coexist on the Proposals sub-tab. Approvers see: Queue card (top) → Scorecard tiles → Unified table. Non-approvers see: Scorecard tiles → Unified table.
- **D-05:** The queue is unchanged structurally — only the row-click behavior is added (D-01/D-02).

### Unified table

- **D-06:** The stage group card loop (`for (const stage of STAGE_ORDER) { ... dashboardHtml += renderStageGroupCard(...) }`) is replaced by a single `_renderHomeProposalTable(proposals, activeFilter)` function that renders one `<table>` with all rows.
- **D-07:** Table scope: **all proposals from `filterProposalsForUser`** — no pre-filter for active-only. All 5 statuses (draft if present, pending_internal, pending_client, for_revision, client_approved, loss) are shown. The scorecard tiles give users the ability to filter by any one status.
- **D-08:** Table columns (left → right): **Status** | **PROP-ID** | **Title** | **Project** | **Client** | **Amount** | **Age**. Status column renders a badge using the existing `getProposalStatusBadge()` — no new badge code.
- **D-09:** Each table row is **clickable** — `onclick="window.openProposalModal && window.openProposalModal('${p.id}')"` on the `<tr>`. Cursor: pointer. Same guard pattern as `renderStageGroupCard` rows. (The unified table also gains the row-click pattern from D-01, not just the queue.)
- **D-10:** `_renderHomeProposalTable` is a **module-private function** in `home.js` (not exported, not added to `proposals.js`) — it lives alongside the other local home helpers.

### Scorecard tiles

- **D-11:** Five scorecard tiles rendered above the unified table, one per proposal stage in `STAGE_ORDER` order: Pending Internal Approval, Pending Client Review, For Revision, Client Approved, Loss. Each tile shows the count of proposals in that status (from the scoped + filtered list). Count = 0 is shown (card still renders, greyed count).
- **D-12:** Scorecard colors derived from existing `getProposalStatusBadge()` palette:
  - Pending Internal Approval → amber `#f59e0b` (left border)
  - Pending Client Review → blue `#3b82f6`
  - For Revision → red `#ef4444`
  - Client Approved → green `#059669`
  - Loss → gray `#6b7280`
- **D-13:** Filter behavior: **single-select, click-to-toggle**. Click a tile = filter table to that status. Click the active tile again = reset (show all statuses). No explicit "All" tile. Active tile is highlighted via `.project-scorecard-card--active` class (same as Phase 91.2/92 pattern).
- **D-14:** Filter state is **local to `_loadHomeProposalsTab`** — a module-level variable `_homeProposalStatusFilter = null` (null = show all, string = active status key). Reset to null whenever `_loadHomeProposalsTab` is called fresh (e.g., after a queue action).
- **D-15:** Scorecard tiles use the existing `.project-scorecard-card` CSS class + `data-status` attribute. Click handler: `window.handleHomeProposalScorecardClick(statusKey)` registered in `init()`, deleted in `destroy()`.
- **D-16:** The scorecard tiles container uses the same layout as Phase 91.2 (`display: flex; flex-wrap: wrap; gap: 0.5rem; margin-bottom: 1rem`) — no new CSS class needed.

### Claude's Discretion

- Row hover highlight implementation: inline `onmouseenter`/`onmouseleave` toggling `style.background` vs. a CSS class. Either approach is acceptable — whichever is cleaner given the template-literal pattern in `home.js`.
- Whether to inline the tile rendering in `_loadHomeProposalsTab` or extract to `_renderHomeProposalScorecards(proposals, activeFilter)` — extract is cleaner; inline is acceptable for a short block (< 30 lines).
- Whether `draft` proposals (if any exist) appear in the table — `STAGE_ORDER` does not include `draft`, so drafts are not scorecard-grouped but may appear in "show all" mode if they exist. Researcher should confirm whether any `draft` proposals exist in Firestore; if not, this is moot.

</decisions>

- **`_homeProposalsCache`** (home.js module scope) — already stores the scoped proposals from last `_loadHomeProposalsTab` call. Re-use for scorecard click re-render without re-fetching.
- **`.project-scorecard-card`** (components.css) + `--active` variant — existing scorecard CSS. Apply `data-status` for click routing.

- **`window.openProposalModal`** — already registered in `home.js` `init()`. Queue rows and unified table rows both use it.

### Established Patterns

- **Scorecard single-select toggle** (procurement.js ~line 448): `card.classList.toggle('project-scorecard-card--active', card.dataset.status === activeFilter)` — same pattern for proposal scorecards.
- **Window function lifecycle** — register in `init()`, delete in `destroy()`. Pattern: `window.handleHomeProposalScorecardClick = (statusKey) => { ... }`.
- **Row-click with button stopPropagation** — `<tr onclick="...">` with `<button onclick="event.stopPropagation(); ...">` inside. Standard DOM pattern used elsewhere in the app.
- **Template-literal table rows** — all tables in home.js and proposals.js use template literals with `escapeHTML()`. No DOM builder needed.
- **`_homeProposalsCache` re-render path** — after a scorecard click, re-render the table section from `_homeProposalsCache` (no Firestore re-fetch). After queue actions, `_loadHomeProposalsTab` re-fetches and resets filter state.

### Integration Points

- **`_loadHomeProposalsTab` is the mount point** — it sets `mount.innerHTML`. The scorecard tiles + unified table HTML replace only the `dashboardHtml` portion; `queueHtml` stays above as before.
- **Scorecard click handler** fires synchronously (no async) — filters `_homeProposalsCache` by `statusKey` and re-renders only the table section (not the queue or scorecard tiles). Count on active scorecard does not change on filter (counts reflect total per status, not filtered subset).
- **No Firestore listener** — `_loadHomeProposalsTab` is one-time getDocs, re-called manually after queue actions. Scorecard filter is client-side only.
- **No firestore.rules changes** — pure UI transformation.

<specifics>
## Specific Ideas

- **Row-click visual affordance for queue**: subtle hover effect on `<tr>` (e.g., `background: #f8fafc`) makes the row feel interactive. The cursor already needs `cursor: pointer` on the `<tr>` to signal clickability.
- **"No proposals" empty state**: when `filterProposalsForUser` returns zero proposals, show a single "No proposals in your scope" card — same empty state as today. When a status filter is active but has 0 matches, show "No proposals with status [X]. Click the tile again to clear filter."
- **`_homeProposalsCache` re-render**: on scorecard click, set `_homeProposalStatusFilter` then call a lightweight `_rerenderProposalTable()` that reads from cache — avoids Firestore round-trip on filter toggle.
- **Draft proposals**: `STAGE_ORDER` omits 'draft'. If drafts exist and show up in the getDocs snapshot, they won't match any scorecard. Researcher should check — if any exist, add a guard so they appear in the "show all" view but don't break scorecard counts. If none exist, ignore.

</specifics>

<deferred>
## Deferred Ideas

- **Inline Approve/Reject in the unified table** — user chose to keep the queue section separate. If a future phase collapses the queue into the table, the action buttons would move into the unified table's Action column, and the queue card would retire.
- **Multi-select scorecard filter** (click multiple statuses simultaneously, like Phase 91.2 cross-group) — user chose single-select for this phase. Revisit if users want to compare two statuses side-by-side.
- **"For Proposal" project scope in proposals tab** — not applicable here; "For Proposal" is a project/service status, not a proposal document status. If a future phase wants to show pending-proposal projects on the home Proposals sub-tab, that's a new feature.
- **Draft proposals scorecard tile** — if draft proposals become a regular workflow state visible to users on the home tab, add a Draft tile. Currently STAGE_ORDER omits draft and it's unclear if drafts are exposed here.

</deferred>
