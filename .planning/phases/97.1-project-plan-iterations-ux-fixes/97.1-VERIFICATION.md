### Key Implementation Details

**destroy() confirmed clean:** Lines 594–622 examined — no `localStorage.removeItem('baselineCleared_')` call. The cleared flag intentionally survives page refresh and cross-view navigation. Confirmed by grep: only 2 `removeItem` calls exist (lines 3207 and 3242 — both in the correct re-enable paths).

**STEP 2 and STEP 3 unconditional:** `restoreIteration()` batch-write loop (line 3488) and `loadIterations()` / `renderIterRail()` calls (lines 3507–3508) are outside the `if (!iter.auto)` guard — they always run regardless of the iteration type.

**Fallback toast present:** When `iter.auto` is true, `showToast('Auto-save restored.', 'success')` fires at line 3514 in the `else` branch, ensuring user feedback even when the undo toast is suppressed.

**selectBaseline vs clearBaseline separation:** `selectBaseline()` removes the localStorage flag when the user actively picks a baseline from the dropdown (line 3242). This correctly allows a future page refresh to auto-select again. `clearBaseline()` sets the flag (line 3254). These are the two opposing operations — correctly wired.

### Human Verification Required

All automated checks pass. The following require browser execution to confirm runtime behavior.

#### 1. Baseline Cleared State Persists Across Page Refresh

**Test:** Navigate to a project plan that has a saved baseline. Clear the baseline (click "Set Baseline" when active, or use the toggle to clear). Refresh the page (F5) and navigate back to the project plan.
**Expected:** No baseline overlay appears on the Gantt. The baseline dropdown shows available baselines but none is auto-selected. The overlay remains absent until the user explicitly picks a baseline from the dropdown.
**Why human:** localStorage persistence across a real page refresh cannot be verified by static analysis. The guard logic is correct in code (confirmed), but the actual browser `localStorage.getItem()` call result requires live execution.

#### 2. Auto-Snapshot Flood Stopped for Auto Iterations

**Test:** Open the iteration history rail. Identify an auto-save entry (labeled "auto", has the 'auto' tag chip). Click to load it. Watch the rail after the restore completes.
**Expected:** The iteration count does NOT increase. No new "Auto-save before …" entry appears. The toast message reads "Auto-save restored." (success variant). The undo button/option is NOT shown.
**Why human:** Firestore write suppression at runtime (the addDoc call inside `if (!iter.auto)` being skipped) requires live execution to verify no Firestore document was written.

#### 3. Non-Auto Iteration Still Creates Auto-Snapshot (Regression Check)

**Test:** In the iteration history rail, identify a manually-saved iteration (no 'auto' tag). Click to load it. Watch the rail and toast.
**Expected:** A new "Auto-save before …" entry appears in the rail (count increases by 1). The undo toast appears: "Loaded '…'. Previous state auto-saved." Clicking Undo restores the pre-load state.
**Why human:** Verifies the `if (!iter.auto)` guard did not accidentally suppress auto-snapshots for non-auto iterations. Requires live browser execution to confirm the addDoc fires and undo works.

#### 4. Toolbar Visual Grouping Renders Correctly

**Test:** Open any project plan. Observe the toolbar.
**Expected:** Four visually distinct clusters separated by thin (1px) gray vertical lines: (1) Day/Week/Month zoom pills, (2) Export PDF + Critical Path checkbox, (3) baseline dropdown + Set/Clear button, (4) Save Iteration + History. Search input floats rightmost with clear visual separation from the Iterations cluster. All controls function identically to before.
**Why human:** CSS flex layout and visual divider rendering require a browser to confirm. The DOM structure and CSS rules are correct (confirmed), but pixel-level rendering (e.g., divider height, gap alignment, 56px toolbar height not broken) requires visual inspection.
