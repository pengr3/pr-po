### WR-01 — Stale undo toast after auto-restore interleaved with manual restore

**Severity:** Warning
**File:** `app/views/project-plan.js` — `restoreIteration()`
**Confidence:** 82

**Problem:**

When restoring an auto iteration (`iter.auto === true`), STEP 1 is skipped so `_autoSnapId` retains whatever value it had from any prior manual restore. The `dismissUndoToast()` call is also not made, so any undo toast that was displayed for the prior manual restore remains visible in the DOM.

Concrete scenario:
1. User restores manual iteration A → `_autoSnapId = <snapId>` → undo toast appears with "Undo" button.
2. Within the toast's display window (~5 seconds), user restores auto iteration B.
3. `iter.auto === true` → STEP 1 skipped → `_autoSnapId` is still `<snapId>`.
4. STEP 4 skipped → `dismissUndoToast()` not called → old undo toast from step 1 persists.
5. User sees the "Auto-save restored." success toast AND the old "Undo" toast simultaneously.
6. User clicks "Undo" on the old toast → `undoIterRestore()` fires → restores from `<snapId>` (the pre-A state, not the pre-B state they just loaded).

**Recommended Fix:**

Add `dismissUndoToast()` unconditionally at the top of `restoreIteration()` (before `iter.auto` check), so any stale undo toast from a previous restore is always cleared on a new restore:

```javascript
async function restoreIteration(iterationId) {
    const iter = _iterations.find(i => i.id === iterationId);
    if (!iter) return;
    dismissUndoToast();   // always clear stale undo from any prior restore
    try {
        if (!iter.auto) {
            // STEP 1: auto-snapshot current state before restore
            ...
        }
        // STEP 2: batch-write restore (always runs)
        ...
        // STEP 3: refresh rail (always runs)
        ...
        if (!iter.auto) {
            // STEP 4: show undo toast
            showUndoToast(...);
        } else {
            showToast('Auto-save restored.', 'success');
        }
    } catch (e) {
        ...
    }
}
```
