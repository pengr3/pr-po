## What Was Built

**Bug 1 — Baseline persistence (Task 1):**

Four surgical edits to `project-plan.js`:

- `clearBaseline()`: after setting `_activeBaselineId = null`, writes `localStorage.setItem('baselineCleared_' + currentProject.id, '1')` so the cleared state outlives the page session.
- `loadBaselines()`: after building `_baselines`, reads the flag via `localStorage.getItem(...)`. If the flag is set (`wasCleared === true`), keeps `_activeBaselineId` and `_baselineData` null instead of auto-selecting `_baselines[0]`. The existing auto-select runs only when the flag is absent.
- `selectBaseline()`: in the successful-selection branch (after finding the baseline), calls `localStorage.removeItem(...)` so a future refresh will auto-select again — user re-enables the behaviour by picking from the dropdown.
- `saveBaseline()`: before calling `await loadBaselines()`, calls `localStorage.removeItem(...)` — saving a brand-new baseline signals the user wants it active.

`destroy()` was not modified — the flag intentionally survives page refresh and navigation.

**Bug 2 — Auto-snapshot flood (Task 2):**

One structural edit to `restoreIteration()`:

- STEP 1 (auto-snapshot addDoc + `_autoSnapId` assignment) wrapped in `if (!iter.auto) { ... }` block.
- STEP 4 (undo toast via `showUndoToast`) wrapped in the same `if (!iter.auto)` conditional; an `else` branch calls `showToast('Auto-save restored.', 'success')` so users still get feedback when loading an auto-snapshot.
- STEP 2 (batch-write restore) and STEP 3 (`loadIterations()` + `renderIterRail()`) remain unconditional.
- `undoIterRestore()`, `confirmIterLoad()`, and `openIterConfirm()` were not touched.
