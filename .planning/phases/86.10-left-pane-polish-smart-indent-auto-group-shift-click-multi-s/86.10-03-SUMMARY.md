## What Was Built

### _clipboardTasks Array
Module-scope `let _clipboardTasks = []` declared alongside other Phase 86.10 state vars. Cleared in `destroy()`. Persists across paste calls so paste is repeatable.

### gridCopyRows(taskIds)
- Accepts JSON string or array of task ID strings
- Reads matching task objects from module-scope `tasks[]`
- Saves shallow copies of relevant fields (task_name, parent_task_id, start_date, end_date, assignees, resources, progress, description, is_milestone, dependencies)
- Does NOT copy task_id or row_order — regenerated on paste
- Shows "N task(s) copied" toast via existing `showToast()`

### gridPasteRows(afterTaskId)
- Guards: empty clipboard → return early; no project_code → toast warning
- Determines insert position via `flattenTreeDepthFirst(tasks)` visual order
- Inherits `parent_task_id` from `afterTaskId`'s task (same indent level)
- Generates new task IDs via `generateTaskId(currentProject.project_code)` with collision-retry (up to 5 attempts)
- Appends " (copy)" to task_name for all pasted tasks
- Uses `setDoc` for each new task, then `commitRowOrderReorder` to normalize row_order integers
- Clipboard not cleared after paste (repeatable paste)

### gridGroupIndent / gridGroupOutdent
- Parse JSON array of task IDs from onclick string
- gridGroupIndent: processes bottom-to-top (reverse visual order) to prevent parent-shift mid-loop
- gridGroupOutdent: processes top-to-bottom
- Both call existing `gridIndentTask` / `gridOutdentTask` per ID sequentially

### gridGroupDelete
- Shows single `confirm()` dialog: "Delete N selected tasks? Tasks with subtasks will also remove their subtasks."
- If confirmed, calls `deleteTaskNow(id)` directly (bypasses per-row confirm in `gridDeleteRow`)
- This avoids double-modal UX for tasks with children

### showTaskContextMenu() Branch
Branch on `isGroupMenu = _selectedRowIds.size > 1 && _selectedRowIds.has(taskId)`:

**Group menu:**
- Header: "N tasks selected"
- Indent Selection / Outdent Selection / Insert Row Above / Delete Selection
- Separator + Copy Selection / Paste (disabled-greyed when clipboard empty)
- Insert Row Above targets `topmostSelectedId` (first selected in flattenTreeDepthFirst order)

**Single menu:**
- Preserves all existing items (Indent, Outdent, separator, Insert Row Above, Delete Row + child count)
- Adds separator + Copy / Paste items at bottom
- Paste disabled (grey, no onclick) when `_clipboardTasks.length === 0`

### Window Registration / Cleanup
- `init()`: `window.gridCopyRows`, `window.gridPasteRows`, `window.gridGroupIndent`, `window.gridGroupOutdent`, `window.gridGroupDelete`
- `destroy()`: `delete window.grid*` for all 5 + `_clipboardTasks = []`

## UAT Defect Fixes

Four bugs found and fixed during human-verify UAT:

**UAT-1 — Group indent chaining (commit 1bcb7ea)**
Original `gridGroupIndent` called `gridIndentTask(id)` for each selected row, making each row a child of the row above it — producing a deeply nested chain A→B→C→D. Fixed: compute "root rows" (rows whose parent is not also in the selection), find a single anchor above the topmost root, then reparent all root rows to that one anchor only.

**UAT-2 — (copy) suffix + clipboard lacked hierarchy data (commit bacea61)**
Pasted tasks had "(copy)" appended to their names. Also, `gridCopyRows` was not saving `_orig_id` or `row_order`, which was needed for hierarchy remapping on paste. Fixed: removed the " (copy)" append; added `_orig_id` and `row_order` fields to each clipboard entry.

**UAT-3 — Duplicate task IDs on multi-item paste (commit 341f371)**
`generateTaskId()` does a full Firestore `getDocs` scan each call; N calls before any `setDoc` commits all return the same max+1 ID. Fixed: compute `maxSeq` once from the in-memory `tasks[]` array (kept fresh by `onSnapshot`) and increment locally for each new ID — zero Firestore round-trips, no duplicates.

**UAT-4 — Paste depth wrong when pasting to an indented row (commit 9ac11ea)**
When pasting a parent+children group after an indented row, clipboard root items were being placed as children of the paste target's parent (e.g. pasted Demo2 became a child of the original Demo2). Fixed: compute `hasInternalHierarchy` (whether any clipboard item has its parent also in `origIdSet`); for root items, use `item.parent_task_id ?? null` (original depth) when `hasInternalHierarchy=true`, otherwise use `pasteParentId` (flat selection landing at target's level).

## Deviations from Plan

**1. [Rule 1 - Bug] gridGroupDelete calls deleteTaskNow() instead of gridDeleteRow()**
- **Found during:** Task 1 implementation
- **Issue:** `gridDeleteRow()` is synchronous and shows a modal for tasks with children; `await gridDeleteRow(id)` would not wait for modal confirmation, creating undefined behavior (proceeds without waiting). Double-confirm UX also poor.
- **Fix:** Call `deleteTaskNow(id)` directly since the single upfront `confirm()` in `gridGroupDelete` already covers the entire selection including subtasks.
- **Files modified:** app/views/project-plan.js (gridGroupDelete function)
- **Commit:** 29a6c5d (same task commit)

**2. [Rule 2 - Missing Critical Functionality] Paste disabled state uses grey color + no onclick**
- **Found during:** Task 1 — plan spec mentions "disabled/greyed if _clipboardTasks.length === 0"
- **Fix:** Applied to BOTH group menu Paste and single menu Paste items. Grey color `#9ca3af`, `cursor:not-allowed`, no `onmouseenter`/`onmouseleave`/`onclick` when clipboard empty.
- **Files modified:** app/views/project-plan.js (showTaskContextMenu)
- **Commit:** 29a6c5d
