## DOM ID Distinction (Plan 03 vs Plan 04)

Confirmed: services.js wrapper ID is `collTrancheBuilderWrapperService` (line 187). projects.js wrapper ID is `collTrancheBuilderWrapper` (no suffix). Distinct DOM IDs — no collision risk if both forms ever render concurrently.

scopeKey is `'serviceForm'` here vs `'projectForm'` in projects.js — also distinct, so DOM IDs generated by the shared tranche-builder module (e.g., row containers keyed by scopeKey) are also distinct.

## Window Function Names (Shared with Plan 85-03)

`window.addTranche`, `window.removeTranche`, `window.recalculateTranches` — same names as Plan 85-03 (projects.js). This is intentional: the shared `app/tranche-builder.js` module provides a single implementation, both views attach it during their own `init()`, and the SPA only mounts one view at a time so there is no double-attach. If both views were ever to render concurrently, they would still resolve to the same module-level functions.

## Deviations from Plan

### Plan-text vs Plan-03-mirror choice (intentional, D-01 parity)

The plan-text in 85-04-PLAN.md showed `showLoading(false); return;` inside the validation blocks, implying validation runs AFTER `showLoading(true)`. Plan 85-03's actually-landed projects.js placed validation BEFORE `showLoading(true)` with bare `return;`. To satisfy the D-01 parity requirement (services.js must mirror projects.js exactly), I followed the actually-landed projects.js pattern: validation block sits above `showLoading(true)` and uses bare `return;` (no `showLoading(false)` needed because loading was never shown). This is functionally equivalent but cleaner — no loading-spinner flicker on validation errors.

### Race with parallel Plan 85-07 executor (Task 1 commit)

While Task 1's services.js diff was staged for an explicit `feat(85-04)` commit, the parallel Plan 85-07 executor (which was committing service-detail.js + project-detail.js Financial Summary cells) staged `app/views/services.js` as part of its own commit unintentionally. The result was that Task 1's exact 19-line diff landed in commit `5c09c03` (titled `feat(85-07): add Collected/Remaining Collectible cells to project-detail`), as shown by `git show --stat 5c09c03`:

```
 app/views/project-detail.js | 37 +++++++++++++++++++++++++++++++++++++
 app/views/services.js       | 19 +++++++++++++++++++
 2 files changed, 56 insertions(+)
```

The 19-line services.js diff in 5c09c03 IS Task 1 (verified by `git show 5c09c03 -- app/views/services.js` showing the import, state var, form-group, window-function attach, and detach). No re-commit is needed because the changes are already in HEAD on `v3.3`. The commit-message attribution is incorrect (says 85-07 instead of 85-04), but the code is correct. Logged for historical accuracy.

**Why this happened:** Parallel executors share the same git working tree on `v3.3`; if one executor stages files matching its own glob/path before another committer finishes, the staged changes become part of whichever commit lands first. Task 2 was committed cleanly to `ca22b44` because by that point the parallel work had moved to other files.

**Mitigation for future parallel executors (already standard practice):** stage by explicit file paths (`git add path/to/file`), inspect `git diff --cached --stat` immediately before commit, and commit immediately after staging without intervening filesystem polls.

### No tranche-builder reset in toggleAddServiceForm (matches Plan 85-03 behavior)

Plan 85-03 did not reset the tranche-builder DOM in projects.js's `toggleAddProjectForm` when transitioning from edit-mode back to add-mode. To preserve D-01 parity, services.js follows the same convention. The shared module's renderTrancheBuilder runs at render-time with `[]`, so a fresh page load shows an empty tranche row. After editing then toggling-cancel-toggle to add a fresh service in the same session, the tranche-builder may show the previously-edited tranches as stale UI. This is a minor UX issue captured here for future polish; both views share it.
