# Phase 91.3: RFP Fees — Add Transfer / Cash-Out / Miscellaneous Fees at RFP Creation - Context

**Gathered:** 2026-06-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Add supplementary fee controls to the RFP (Request for Payment) creation flow so users can attach, at creation time, a **Transfer Fee**, a **Cash-Out Fee**, and any number of arbitrary **Miscellaneous Fee** lines (label + amount) that are NOT part of the original RFP base amount. The added fees are folded into the persisted RFP total so payables / remaining-payable calculations reflect the true disbursement cost, while the base amount stays distinct from the fee breakdown on the RFP document. The fee breakdown is surfaced everywhere the RFP amount appears (RFP modal, Finance → Payables tables/cards, expense/financial breakdown). The RFP creation modal is overhauled into a sectioned, mobile-friendly layout and ships regression-free.

**In scope:** fee-entry controls in the RFP creation modals; fee-inclusive total persisted on the RFP doc; fee breakdown surfaced in all RFP-amount display sites; sectioned mobile-friendly modal restructure; backward-compatible handling of legacy RFPs.

**Out of scope (new capabilities → own phase):** preset/standard fee amount management, discounts/negative adjustments, per-fee tax handling, fee approval workflows, editing fees after RFP creation (fees are frozen at creation like the rest of the RFP).
</domain>

<decisions>
## Implementation Decisions

### Fee Scope & Attachment
- **D-01:** Supplementary fee controls apply to **all three RFP creation paths** — standard PO-tranche RFP, Delivery-Fee RFP, and Transport-Request (TR) RFP. All three creation entry points get the fee UI and persist fees.
- **D-02:** Fees attach **per-RFP, independently**. Each RFP (each tranche) carries its own fee set, entered when that RFP is created. This matches the existing frozen-at-creation, per-tranche model — a 3-tranche PO may have, e.g., a transfer fee on each disbursement. No PO-level aggregate fee store.

### Fee Controls & Validation
- **D-03:** **Transfer Fee** and **Cash-Out Fee** are **free-entry and optional** — both default to empty/off; the user types an amount only when it applies. No preset/standard amounts to maintain.
- **D-04:** **Miscellaneous fees** are repeatable lines, each with a **label + amount** (per roadmap). User can add zero or more.
- **D-05:** **Positive amounts only.** No negative/discount fee lines. Validation: a fee amount that is entered must be `> 0`. Empty/omitted fee = not applied (treated as 0, not persisted as a line).

### Fee Display
- **D-06:** **Dense Finance Payables tables/cards show a collapsed grand total** (base + fees) — not each fee line — to keep the dense tables and mobile cards readable with no horizontal overflow.
- **D-07:** **Full itemization** (Transfer / Cash-Out / each Misc line) lives in the **RFP detail modal** (and the creation modal's running total), satisfying SC #3's "breakdown surfaced."
- **D-08:** Add a **subtle "incl. fees" cue** (small badge / info icon / note) next to any displayed total that contains fees, so a reviewer understands the number is larger than the bare tranche % amount.

### Modal Overhaul (UI)
- **D-09:** **Restructure the RFP creation modal into clear sections**: Reference → Base Amount → Fees (add-fee controls + running total) → Payment details. Gives fees a natural home and reads cleanly on mobile. This is the "overhaul" SC #4 calls for — not a minimal inline slot-in.
- **D-10:** Show a **live running grand total** in the modal that updates as fees are typed (base [auto from tranche %] + entered fees), so the user sees the real disbursement before submitting.

### Persisted Total & Payables (locked by success criteria)
- **D-11:** The fee-inclusive **total** is what payables / remaining-payable / partial-payment math tracks against. The **base RFP amount stays distinct** from the fee breakdown on the RFP document (SC #2).
- **D-12:** **No regressions.** Existing RFP create / cancel / mark-paid flows, PO-scoped RFP ID generation, and RFP notifications must continue to work; `node --check` passes on every modified module (SC #5).

### Design Previews (validated via sketches — `.planning/sketches/`, 2026-06-03)
Throwaway HTML mockups in the live CLMC look, decisions picked by the operator:
- **D-13 (sketch 001 → B):** RFP creation modal restructured into 4 sections (Reference → Base Amount → Fees → Payment) with **progressive-disclosure fees** — the Fees section shows only `+ Transfer fee` / `+ Cash-out fee` / `+ Misc fee` chips until a fee is added (each removable); keeps the common no-fee RFP clean. Live "Base + Fees = Grand total" appears once a fee exists.
- **D-14 (sketch 002 → A):** Dense Payables tables show a collapsed fee-inclusive total + a subtle **"incl. fees" pill** with a hover tooltip breakdown; full itemization in the RFP detail modal. **⚠ Touch fallback required** (hover doesn't fire on mobile; SC #4) — on touch, pill should tap-toggle the tooltip or open the detail modal. (Resolve in UI planning.)
- **D-15 (sketch 003 → 4-section):** The no-tranche **Delivery-Fee and TR modals** reuse the *same* 4-section shell as the PO modal (Base section is a single read-only line); compact 3-section variant rejected — cross-modal consistency preferred.
- **D-16 (sketch 004 → A):** PO Payment Summary / Financial Breakdown shows fee-inclusive **Total / Paid / Remaining** with an "incl. fees" note + the sketch-002 pill on fee rows (no dedicated Fees column) — least disruptive, mobile-friendly, consistent with D-14.
- **D-17 (sketch 005 → approved):** Invalid fee (≤ ₱0 or non-numeric) → inline field error + footer alert + **disabled Submit** until fixed (enforces D-05); empty fee = ignored (not applied); new controls coexist with the existing one-per-PO and all-tranches-used blocked-RFP guards without layout breakage.

See `.planning/sketches/MANIFEST.md` and each `NNN-*/README.md` for the winning variants and "what to look for" notes.

### Claude's Discretion
- **Field strategy (research/plan decides):** whether to keep `amount_requested` as the base-only value and add a new computed/persisted total field (e.g. `total_amount` or `fees_total` + base), OR repurpose `amount_requested` as the grand total with base stored separately. **Constraint:** all consumers that currently read `amount_requested` (`derivePOSummary`, `deriveRFPStatus`, `buildRFPCard`, `renderRFPTable`, `renderPOSummaryTable`, `submitPaymentRecord`/`openRecordPaymentModal`) must end up tracking against the fee-inclusive total, and the base must remain recoverable for display. This is the primary regression-risk surface — research must resolve it before planning the edits.
- **Legacy-RFP handling:** existing RFP docs have no fee fields. All display/calc paths must treat absent fee fields as zero/none so legacy RFPs render exactly as today (per D-12).
- **Exact fee-data shape** (e.g. `transfer_fee`, `cash_out_fee` numbers + `misc_fees_json` / array of `{label, amount}`) and where the "incl. fees" cue renders — left to planner using existing patterns.
</decisions>

<specifics>
## Specific Ideas

- Modal section order requested: **Reference → Base Amount → Fees (controls + running total) → Payment details.**
- "incl. fees" reviewer cue is explicitly wanted on dense tables/cards where only the collapsed total shows — so a reviewer doesn't mistake the larger number for a miscalculated tranche %.
- Delivery-Fee RFP (itself a fee-type RFP) still gets the supplementary-fee controls — confirmed via "all 3 RFP types."
</specifics>

<deferred>
## Deferred Ideas

- **Preset / standard fee amounts** (e.g. a maintained "standard bank transfer fee" that pre-fills) — explicitly declined in favor of free-entry; would be its own config/admin feature.
- **Negative fees / discounts / adjustments** — declined for this phase (positive-only). A discount/adjustment concept would be a separate capability.
- None of the above are blockers; captured so they're not lost.

None — discussion otherwise stayed within phase scope.
</deferred>
