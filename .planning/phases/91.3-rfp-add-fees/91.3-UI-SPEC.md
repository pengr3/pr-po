## Spacing Scale

The live app uses **rem-based spacing on a 4px grid** (`0.25rem`=4, `0.5rem`=8, `0.75rem`=12, `1rem`=16, `1.5rem`=24). 12px (`0.75rem`) is an established in-system value (form-control padding, sketch gaps) and is retained as an allowed value for this phase.

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px (0.25rem) | Pill/cue inner gap, `√ó`/`i` icon padding, label‚Üíinput gap (`margin-bottom:4px`) |
| sm | 8px (0.5rem) | Chip gap (`#feeChips` `gap:8px`), fee-row grid gap, modal-footer button gap |
| (in-system) | 12px (0.75rem) | `.form-control` padding (`.5rem .75rem`), tinted-card padding, fee-card inner padding |
| md | 16px (1rem) | Default field stack gap, section internal padding, modal-body padding base |
| lg | 24px (1.5rem) | Section vertical rhythm (`.section margin-bottom` ‚âà `1.15rem`‚Üíround to 24px target), modal-header/footer horizontal padding (`1rem 1.5rem`) |

**Exceptions (intentional, in-system):**
- `0.875rem` (14px) modal-body label/input font ‚Äî this is a **font-size**, not spacing.
- Touch-target minimum **44px** (`min-height:44px`) on mobile interactive controls ‚Äî required by Phase 73/74 mobile contract, not part of the spacing scale.
- Misc-fee amount column fixed width **150px** (sketch `.fee-row` / `.fee-fixed-row` `grid-template-columns: 1fr 150px 32px`); remove-button column **32px**. These are layout track sizes, not spacing tokens.

## Color

The fee UI is **content inside white modals/cards over the app's existing surfaces** ‚Äî it inherits the 60/30/10 split already established in the app; it does not introduce new dominant or secondary surfaces.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` modal/card surface; page bg `#f8f9fa`/`#f1f5f9` | Modal body, input backgrounds, table rows |
| Secondary (30%) | `#f8fafc` reference-grid / tinted panels; borders `#e5e7eb`, `#e2e8f0` | Reference card, section rules, table headers, readonly input bg `#f1f5f9` |
| Accent (10%) ‚Äî app primary | `#1a73e8` (hover `#1557b0`) | Section-number badges, primary **Submit RFP** button, input focus ring, RFP-ID links ‚Äî **unchanged from app** |
| **Fee accent (NEW ‚Äî sketch-defined)** | `#0ea5e9` family: text `#0369a1`, bg `#f0f9ff`, border `#bae6fd` | **Reserved exclusively for fee-additive affordances** (see reserved-for list) |
| Destructive | `#ef4444` (live `#ea4335` for required-asterisks) | Invalid-fee field error border + ring + message, footer alert text `#991b1b` on `#fef2f2`, remove-`√ó` hover |
| Status pills (reuse existing) | paid `#dcfce7`/`#166534`, partial `#fef3c7`/`#92400e`, pending `#e0e7ff`/`#3730a3`, overdue `#fee2e2`/`#991b1b` | Payables status ‚Äî **untouched**; the fee cue must NOT compete visually with these |

**Accent reserved for (app primary `#1a73e8`):** section-number badges, the Submit RFP primary button, input focus ring/border, RFP-ID hyperlinks. Not applied to fee controls.

**Fee accent (`#0ea5e9` / `#0369a1` / `#f0f9ff` / `#bae6fd`) reserved EXCLUSIVELY for:**
1. The `+ Transfer fee` / `+ Cash-out fee` / `+ Misc fee` **add-chips** (`.btn-ghost-fee`: `bg:#f0f9ff`, `color:#0369a1`, `1px dashed #bae6fd`).
2. The **fees tinted card / running-total block** dashed divider and accents (`border:1px ... #bae6fd`, `bg #f0f9ff`).
3. The **"incl. fees" cue pill** (downstream tables/cards) and its hover/tap tooltip accent.
4. The **"incl. fees" pill** in the modal footer/detail-modal grand-total row.
5. The PO-summary **"incl. ‚Ç±X fees" sub-note** and fee-bearing sub-row cue.

Fee accent must **never** be used for errors, for required-field asterisks, for primary actions, or as a status color. It signals *"this number is additive / larger than the bare tranche amount"* ‚Äî nothing else. (Distinct from red error color by design ‚Äî sketch rationale: fees read as additive, not as a problem.)

## Component Inventory & Interaction Contract

Classes below are the **target names** for new fee UI. They mirror the validated sketches and must coexist with existing inline-styled modal markup. Prefix new classes with `fee-`/`rfp-` to avoid collisions.

### A. Sectioned RFP creation modal (sketch 001-B, 003 ‚Äî all 3 RFP types)

- **Structure (locked, D-09/D-13/D-15):** four numbered sections in order ‚Äî **1 Reference ‚Üí 2 Base Amount ‚Üí 3 Fees ‚Üí 4 Payment Details** ‚Äî reused identically across `openRFPModal`, `openDeliveryFeeRFPModal`, `openTRRFPModal`. The no-tranche modals render section 2 as a **single read-only base line** (compact 3-section variant rejected).
- **Section header (`.section-head`):** `[20px circular primary badge with number] + [uppercase 0.72rem/700 muted title] + [1px flex rule]`. Badge: `width:20px;height:20px;border-radius:9999px;background:#1a73e8;color:#fff;font-size:11px;font-weight:700`.
- **Reference card (`.ref-grid`):** 2-col grid (`1fr 1fr`, gap `0.85rem`) on `#f8fafc` with `1px #e2e8f0` border, radius 8px, padding `0.85rem`. Department spans 2 cols. Read-only detail-label/detail-val pairs (label `0.72rem/600 uppercase #64748b`, value `0.9rem/600 #1e293b`).
- **Base Amount:** Tranche `<select class="form-control">` (PO only) + read-only **Amount Requested** input (auto from tranche %; `background:#f1f5f9;cursor:not-allowed`). Delivery-Fee/TR: single read-only base line, no tranche select.
- **Fees section ‚Äî progressive disclosure (sketch 001-B, LOCKED):**
  - Default state = **three add-chips only** (`.btn-ghost-fee`, fee-accent, `1px dashed #bae6fd`) + muted hint *"No fees added‚Ä¶"*. The running-total block is **hidden**.
  - Tapping `+ Transfer fee` / `+ Cash-out fee` reveals that single removable row (grid `1fr 150px 32px`) and hides the chip; `√ó` removes it, clears the value, and restores the chip.
  - `+ Misc fee` appends a removable **label + amount + √ó** row; repeatable (zero-or-more, D-04). New row auto-focuses its label input.
  - Amount inputs: `.fee-amount-wrap` with `‚Ç±` prefix pseudo-element, right-aligned, `inputmode="decimal"`, placeholder `0.00`.
  - Once **any** fee row exists, reveal the **running-total block** (`Base amount` / `Fees` / **Grand total**) with the fee-accent dashed divider, and show the **"incl. fees"** pill in the footer.
- **Live running total (D-10):** recompute on every `oninput`/`onchange`/reveal/remove: `grand = base + Œ£(visible, valid fee amounts)`. Only **visible** rows count (removed/hidden rows excluded). Footer echoes `Total ‚Ç±X` + conditional `incl. fees` pill.
- **Footer:** `.modal-footer` with `Total ‚Ç±X [incl. fees pill]` left (`margin-right:auto`), then **Discard** (`.btn-outline`) + **Submit RFP** (`.btn-primary`).

### B. Validation & blocked states (sketch 005, APPROVED ‚Äî implement verbatim)

- **Positive-only (D-05):** a fee amount that is **entered and ‚â§ 0 or non-numeric** ‚Üí add `.err` to the input (`border-color:#ef4444; box-shadow:0 0 0 3px rgba(239,68,68,.12)`) + show inline `.field-err` message + show footer alert + **disable Submit** (`disabled; opacity:.5; cursor:not-allowed`). Live-validate on input; clearing/fixing the value removes all three and re-enables Submit (unless a blocked-guard lock is set).
- **Empty fee = ignored:** an empty amount field is **not** an error and **not** persisted as a line (treated as 0). The remove-`√ó` is the way to drop an unwanted revealed field.
- **Coexistence with existing blocked guards (must not break layout):**
  - *Delivery-fee already exists* (one-per-PO): red banner at top of body (`bg #fef2f2; color #991b1b; 1px #fecaca; radius 6px`), form dimmed (`opacity:.45; pointer-events:none`), Submit disabled+locked.
  - *All tranches used*: same banner copy, tranche select disabled, Submit disabled+locked.
  - The new fee chips/section sit **inside** the dimmable form region so a blocked guard dims them too; no separate handling.

### C. Downstream "incl. fees" cue + tooltip (sketch 002-A) ‚Äî **TOUCH FALLBACK RESOLVED**

- **Dense Payables tables/cards (D-06/D-14):** show only the **collapsed fee-inclusive total**. On fee-bearing rows append a small **"incl. fees" cue pill** (`.cue`: fee-accent, `0.62rem/700`, `border-radius:9999px`, `margin-left:6px`).
- **Desktop:** hovering the cue reveals a dark tooltip (`bg #0f172a; color #e2e8f0; radius 8px; shadow-md; width ~200‚Äì210px`) listing `Base / Transfer fee / Cash-out fee / each Misc ‚Üí Total`.
- **MOBILE / TOUCH FALLBACK (resolves sketch-002 open item; SC #4):** hover does not fire on touch, so on viewports `<768px` (and for any touch interaction) the cue is **tap-to-toggle**: tapping the cue toggles the same breakdown tooltip open/closed (tap elsewhere or tap again dismisses). The cue is a **real focusable/tappable control** with `min-height:44px` hit area (use padding/`::before` expansion so the visible pill stays small but the touch target meets 44px), `role="button"`, `aria-expanded`, `tabindex="0"`, and keyboard `Enter`/`Space` toggle. **Canonical fallback:** the RFP-ID link in the same row still opens the **RFP detail modal** with full itemization (locked D-07), so the breakdown is always reachable even if the tooltip is missed. Implementation note for planner: implement as tap-toggle on the cue **and** keep the detail-modal path ‚Äî both satisfy the fallback; do not rely on hover alone.
- **RFP detail modal (D-07, locked ‚Äî full itemization):** **Amount breakdown** card (fee-accent tint, `1px #bae6fd`, radius 10px) listing `Base amount (label)` ‚Üí indented fee lines (`Transfer fee`, `Cash-out fee`, each Misc) ‚Üí **Grand total** + `incl. fees` pill (`1.25rem/800` tabular). Below: `Paid` (green `#166534`) / `Remaining` (amber `#92400e`).

### D. PO Payment Summary / Financial Breakdown roll-up (sketch 004-A)

- **Total / Paid / Remaining** stat tiles become **fee-inclusive**. Under "Total payable" add a muted fee-accent sub-note **"incl. ‚Ç±{amount} fees"** (`0.7rem; color #0369a1`). **No dedicated Fees column** (explicit Base/Fees/Total variant B rejected).
- Per-RFP sub-rows reuse the **same "incl. fees" cue pill** + tooltip from section C (consistency).
- Footer/aggregate row: `Total payable [incl. ‚Ç±X fees cue]`.
- Not-yet-raised tranche rows render **muted** (`#64748b`/`#cbd5e1`) with a projected base only ‚Äî no fee cue (no fees exist yet).

## Assumptions (where files did not give a literal answer)

> Written as best-reasoned defaults per instructions; planner may refine but should not contradict the sketches.

1. **Touch-fallback firmness (sketch-002 open item):** specified as **tap-to-toggle the tooltip on the cue, with the RFP-detail-modal path always available** (both, not either/or). Chosen because the detail modal is already the locked canonical itemization home (D-07), so the cue tap-toggle is a progressive enhancement and the modal is the guaranteed fallback. This is the explicit resolution the orchestrator requested.
2. **Spacing scale presentation:** the live system is rem-based on a 4px grid rather than a strict 8-point token set; 12px (`0.75rem`) is retained as an in-system value because `.form-control` padding and sketch gaps already use it. Flagged so the checker reads this as intentional, not a violation.
3. **Discard label:** kept per-modal to avoid a regression (PO modal currently says "Discard RFP"); new sketch shorthand "Discard" is acceptable where no existing label exists.
4. **Icon set:** the app ships no icon library; fee UI uses the same inline text glyphs the sketches use (`+`, `√ó`, `i`, `‚Ç±`). No new dependency.
5. **Field-data shape and `amount_requested` vs new total field** are explicitly **Claude's Discretion / planner-owned** (CONTEXT.md) ‚Äî out of scope for this visual contract. This UI-SPEC only requires that whatever is displayed as the fee-inclusive grand total is what payables math tracks against, and that the base remains recoverable for the breakdown.
