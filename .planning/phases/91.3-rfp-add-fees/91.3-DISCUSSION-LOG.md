## Fee scope — which RFP types get fee controls

| Option | Description | Selected |
|--------|-------------|----------|
| Standard PO RFP only | Only the regular PO-tranche RFP gets fee controls; smallest, safest surface. | |
| PO + TR RFPs | Standard PO RFPs and Transport-Request RFPs get fees; Delivery-Fee RFP excluded. | |
| All 3 RFP types | PO-tranche, Delivery-Fee, and TR RFPs all get fee controls; broadest coverage. | ✓ |

**User's choice:** All 3 RFP types
**Notes:** Transfer/cash-out costs apply to any disbursement regardless of RFP type, so all three creation paths (`openRFPModal`, `openDeliveryFeeRFPModal`, `openTRRFPModal`) get the controls.

## Fee attach — how fees attach across tranches

| Option | Description | Selected |
|--------|-------------|----------|
| Per RFP (independent) | Each RFP carries its own fees, added at creation; matches frozen-at-creation per-tranche model. | ✓ |
| Once per PO | Fees entered once and apply to the whole PO; requires tracking fees outside the RFP doc. | |

**User's choice:** Per RFP (independent)
**Notes:** Aligns with the existing per-tranche, frozen-at-creation architecture; a multi-tranche PO can carry fees on each disbursement's RFP.

## Fee controls — how Transfer/Cash-Out amounts are entered

| Option | Description | Selected |
|--------|-------------|----------|
| Free-entry, optional | Default empty/off; user types an amount only when it applies; no preset. | ✓ |
| Preset default, editable | Pre-fill with a standard amount the user can override; requires maintaining standard amounts. | |
| Free-entry, both required | Must always be entered (even as 0); more friction, fewer accidental omissions. | |

**User's choice:** Free-entry, optional
**Notes:** No preset/standard fee management in scope — declined and noted as deferred.

## Negatives — can a fee line be negative (discount/adjustment)?

| Option | Description | Selected |
|--------|-------------|----------|
| No — positive only | Every fee adds to total; simplest validation (amount > 0). | ✓ |
| Yes — Misc can be negative | A Misc line may reduce the total; adds adjustment use-case but complicates math/display. | |

**User's choice:** No — positive only
**Notes:** Discounts/adjustments deferred to a possible future capability.

## Fee display — granularity in dense Payables tables/cards

| Option | Description | Selected |
|--------|-------------|----------|
| Collapsed total; itemize in modal | Tables/cards show one grand total; full itemization in the RFP detail modal. | ✓ |
| Total + combined 'Fees' line | Show base + a single combined Fees subtotal + grand total (not each fee). | |
| Itemize every fee everywhere | Each fee line rendered in the Payables tables too; risks crowding/overflow. | |

**User's choice:** Collapsed total; itemize in modal
**Notes:** Keeps the dense tables and mobile cards readable; itemization lives in the detail/creation modal.

## Fee flag — surface that totals now include fees?

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — subtle 'incl. fees' cue | Small badge/info icon/note next to totals containing fees. | ✓ |
| No — show total silently | Just show the larger total; breakdown available by opening the RFP. | |

**User's choice:** Yes — subtle 'incl. fees' cue
**Notes:** Prevents a reviewer from mistaking the fee-inflated total for a miscalculated tranche %.

## Live total — show running grand total in modal?

| Option | Description | Selected |
