# Phase 86.9 Code Review

## Critical

### C-01: Wrong field name `t.task_name` in `exportGanttPDF` — all task names render as "(unnamed)"

**File:** app/views/project-plan.js
**Confidence:** 100

The `tasks[]` array field for task name is `t.name` everywhere in the file (renderGrid, renderGantt, showContextMenu, matchesSearch, all Firestore write paths). `exportGanttPDF` reads `t.task_name` which is always `undefined`, so every row in the PDF summary table falls through to the `'(unnamed)'` fallback. The primary deliverable of the export feature (task names) is entirely broken.

**Fix:** Change `t.task_name` → `t.name` in exportGanttPDF.

---

## Warning

### W-01: Mobile CSS — `.task-grid-rail` hidden by first @media block, never restored

**File:** styles/views.css
**Confidence:** 90

Two `@media (max-width: 768px)` blocks target `.task-grid-rail`. The first sets `display: none`. The second sets layout properties (`position`, `flex`, `min-width`) but never overrides `display`. Net result on mobile: both gantt-pane AND task-grid-rail are hidden — the entire split pane is blank on mobile.

**Fix:** Delete the first `@media (max-width: 768px)` block that hides `.task-grid-rail`. The second block handles the complete mobile layout correctly.

## Info

### I-01: No guard against duplicate `#gantt-print-frame` on rapid double-click

**File:** app/views/project-plan.js
**Confidence:** 80

`exportGanttPDF` appends a new `div#gantt-print-frame` without first removing any pre-existing one. If clicked twice within the 800ms cleanup window, two frames exist simultaneously — both visible in print dialog.

**Fix:** Add `document.getElementById('gantt-print-frame')?.remove();` at the top of `exportGanttPDF`.
