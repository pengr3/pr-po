<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated client → Firestore (services subtree) | Any signed-in active user issues subcollection writes; Firestore rules are the ONLY server-side authority (zero-build SPA, no backend) |
| Finance role → service retention field | Finance is granted a narrow field-masked write to `retention_released_at` it does not otherwise have on services |
| non-team active user → parent service doc | A user not in `personnel_user_ids` can post journal entries but cannot update the parent service doc |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-104-01 | Elevation of Privilege | services Finance branch (`retention_released_at`) | mitigate | Field-mask with `affectedKeys().hasOnly(['retention_released_at','updated_at'])` so Finance cannot use this branch to mutate any other service field (mirror of projects:238-242 / 102 D-25). |
| T-104-02 | Tampering | `activity_entries`/`audit_log` append-only logs | mitigate | `allow update: if false` on activity_entries and audit_log; delete restricted to `['super_admin','services_admin']`. Audit/feed entries cannot be silently rewritten. |
| T-104-03 | Denial of Service / Repudiation | D-14 cross-tenant journal-vs-parent write | mitigate | Journal `create: if isActiveUser()` is intentionally decoupled from the parent-doc update rule; the fire-and-forget `last_activity_at` bump (Plan 02/03) is un-batched + `.catch`-swallowed so a denied parent write never rolls back the entry, and a non-team poster's denied bump never blocks their entry. |
| T-104-04 | Elevation of Privilege | service-doc lifecycle/DLP fields via role-only update rule | accept | Per 103.1, the services update rule is role-only (no affectedKeys mask); services_admin / assigned services_user can already write any service field. Adding lifecycle/DLP fields introduces no NEW field surface (same roles already had full-doc write). Accepted — matches the existing services authorization model; no PII beyond what these roles already control. |
| T-104-05 | Information Disclosure | journal/audit read scope | accept | `allow read: if isActiveUser()` matches the projects journal precedent and the existing services read model; service journal content is operational, not sensitive PII. Accepted (consistent with Phase 101). |

No HIGH-severity unmitigated threat. T-104-01/02/03 are mitigated; T-104-04/05 are accepted with rationale consistent with the shipped projects/services authorization model.
