# Phase 97.2 Plan 02: Dismiss Undo Toast + Firestore Cleanup Summary

**async dismissUndoToast() deletes hidden auto-snapshot from Firestore on timeout or dismiss; WR-01 fixed by unconditional await at top of restoreIteration()**

## Accomplishments

- `dismissUndoToast()` is now async and deletes the Firestore `project_iterations` auto-snapshot doc when `_autoSnapId` is set, then clears `_autoSnapId = null` before the await to prevent re-entrance
- The redundant `_autoSnapId = null` inside `showUndoToast`'s `setTimeout` callback was removed — `dismissUndoToast` is now the sole owner of that state transition
- `restoreIteration()` calls `await dismissUndoToast()` unconditionally after the `if (!iter) return` guard and before the `try` block, fixing WR-01: any stale undo toast (and its Firestore auto-snapshot) from a prior restore is always cleaned up before a new restore begins
- Undo toast copy updated from "Previous state auto-saved." to "Undo to revert." (D-04)

## Decisions Made

- Used `await dismissUndoToast()` (not fire-and-forget) in `restoreIteration()` so the prior auto-snapshot is deleted before STEP 1 creates a new one — prevents a brief window where both old and new auto-snapshots exist simultaneously in Firestore
- Race prevention: `snapId = _autoSnapId; _autoSnapId = null;` before the `await deleteDoc` call so a second synchronous dismiss call skips the delete (the first caller owns the cleanup)
- `try/catch` in `dismissUndoToast` is silent — auto-snapshot cleanup is best-effort; worst case is an orphan doc in Firestore, never user-visible data loss (T-97.2-02 accepted)
