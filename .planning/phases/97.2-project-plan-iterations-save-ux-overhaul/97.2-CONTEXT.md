# Phase 97.2: Project Plan Iterations — Save Model Overhaul - Context

**Gathered:** 2026-06-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Rework the iteration save/restore UX in `app/views/project-plan.js`:
1. Remove auto-snapshots from the visible rail (hidden in Firestore, never rendered)
2. Clean up the auto-snapshot Firestore doc when the undo window closes
3. Redesign rail cards: per-card delete, re-sequenced numbering, relative timestamps
4. Replace `window.prompt()` with a styled modal for naming iterations

This phase does NOT change: diff panel, baseline overlay, `project_iterations` Firestore schema, restore batch-write mechanic, or right-rail toggle behavior.

</domain>

<decisions>
## Implementation Decisions

### Auto-snapshot fate
- **D-01:** Auto-snapshots (`iter.auto === true`) are hidden from the rail — `renderIterRail()` must filter them out. They remain in Firestore as the undo safety net but are never rendered as timeline cards.
- **D-02:** Existing auto-snapshot docs in Firestore are silently excluded (not bulk-deleted). No migration — just change the render filter.
- **D-03:** Remove the `iter-auto-tag` chip rendering (`${iter.auto ? ' <span class="iter-auto-tag">auto</span>' : ''}`) and the `.auto` class on the dot. These become dead code after D-01. Remove corresponding CSS too.

### Undo safety net
- **D-04:** Update undo toast copy to: `'Loaded "X" — Undo to revert.'` (removes the phrase "Previous state auto-saved." which implied the auto-snapshot would appear in the rail).
- **D-05:** When `dismissUndoToast()` fires (on timeout or manual close), delete the hidden auto-snapshot from Firestore: `deleteDoc(doc(db, 'project_iterations', _autoSnapId))`. Clear `_autoSnapId = null` after delete. If `_autoSnapId` is null when dismiss fires, skip silently.
- **D-06 (WR-01 fix):** Add `dismissUndoToast()` unconditionally at the very top of `restoreIteration()` — before the `iter.auto` check — so any stale undo toast from a prior manual restore is always cleared on a new restore.

### Rail card redesign
- **D-07:** Add a per-card delete button (× icon) to each timeline card. Clicking shows a brief inline confirm ("Delete iteration 'X'? This cannot be undone.") then calls `deleteDoc(doc(db, 'project_iterations', iter.id))` and reloads/re-renders the rail.
- **D-08:** Re-sequence dot numbering to count only visible (non-auto) iterations. Since auto-saves are filtered, `idx + 1` in the filtered render loop gives clean sequential numbers (1, 2, 3…). The `_iterSeq` counter already tracks non-auto count correctly.
- **D-09:** Relative timestamps for saves < 24 hours old ("2 hours ago"), absolute date for older ("Jun 2, 2026"). Helper: compare `saved_at.toDate()` to `Date.now()` — if diff < 86400000ms, use `Math.floor(diff / 3600000)` + "h ago" (or "< 1h ago" for under an hour); else `toLocaleDateString()`.

### Save trigger UX
- **D-10:** Replace `window.prompt()` in `saveIteration()` with a styled inline modal. Style it to match `openIterConfirm()`: `position:fixed;inset:0;display:flex;align-items:center;justify-content:center;background:rgba(0,0,0,0.3)` + inner white card `background:#fff;border-radius:8px;padding:28px 24px;min-width:320px;max-width:420px`. Input prefilled with `Iteration ${_iterSeq + 1}`. Save + Cancel buttons.
- **D-11:** Both entry points (toolbar "Save Iteration" button + rail "+ Save current state" button) trigger the same modal. No behavioral difference between them.
- **D-12:** Silent 60-char truncation at save time (`label.substring(0, 60)`) preserved. No counter or maxlength enforcement needed.

### Claude's Discretion
- Exact placement of D-06 `dismissUndoToast()` call: before or after the `if (!iter) return` early exit — either is valid. Planner decides.
- Delete confirmation UX detail: inline text below button, or a separate modal. Keep lightweight — brief inline text is fine given the pattern of `openIterConfirm()`.

</decisions>
