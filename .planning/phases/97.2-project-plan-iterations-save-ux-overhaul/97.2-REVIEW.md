## Critical

### CR-01 — XSS: `iter.id` unescaped in `onclick` attributes in `renderIterRail()`

**File:** `app/views/project-plan.js` (renderIterRail render loop)
**Confidence:** 90

Three button `onclick` attributes embed the Firestore document ID without `escapeHTML()`:

```javascript
onclick="window.toggleIterDiff('${iter.id}')"
onclick="window.openIterConfirm('${iter.id}')"
onclick="window.deleteIteration('${iter.id}')"
```

The project pattern wraps every Firestore-derived ID in `escapeHTML()` before inserting into `innerHTML`. Firestore auto-IDs are alphanumeric only today, but this is a latent injection surface and a direct deviation from the project's own consistency rule.

**Fix:** Replace `${iter.id}` with `${escapeHTML(iter.id)}` on all three `onclick` attributes in `renderIterRail()`.

### CR-02 — Async race: `_autoSnapId` may be nulled by auto-dismiss timer during `undoIterRestore()` batch write

**File:** `app/views/project-plan.js` (undoIterRestore)
**Confidence:** 82

`undoIterRestore()` uses `_autoSnapId` directly after a multi-step `await` chain (getDocs + batch writes). During that async gap the 5-second `_undoToastTimer` can fire `dismissUndoToast()`, which sets `_autoSnapId = null` and issues its own `deleteDoc`. When control returns to `undoIterRestore`, `_autoSnapId` is `null` — `deleteDoc(doc(db, 'project_iterations', null))` throws, the `catch` fires "Undo failed." even though the data was successfully restored. Easily triggered on slow connections (batch commit > 5s).

**Fix:** Capture `_autoSnapId` in a `const localSnapId` at function entry and use `localSnapId` throughout, matching the defensive pattern already used in `dismissUndoToast()`.

---

### CR-03 — `dismissUndoToast()` not awaited in `showUndoToast()` — undocumented ordering invariant

**File:** `app/views/project-plan.js` (showUndoToast)
**Confidence:** 85

`showUndoToast()` is synchronous and calls `dismissUndoToast()` (now `async`) without `await`. The current code is safe because `restoreIteration` always `await dismissUndoToast()` before calling `showUndoToast`. But this creates an undocumented invariant: any future caller that calls `showUndoToast` without first awaiting `dismissUndoToast` will overlap async operations on the same auto-snapshot document.

**Fix:** Convert `showUndoToast` to `async` and `await dismissUndoToast()` inside it. Update its call site in `restoreIteration` to `await showUndoToast(...)`.

## Warning

### WR-01 — Modal leak: `iterSaveModal`, `iterDeleteModal`, `baselineSaveModal` not cleaned up in `destroy()`

**File:** `app/views/project-plan.js` (destroy)
**Confidence:** 80

`destroy()` removes `iterConfirmModal` but not the three modals added in Phase 97.2. If a user opens any of these modals and navigates away, the modal remains in the DOM. Clicking the stale button after `destroy()` has removed the `window.*` function produces `TypeError: window.X is not a function`.

**Fix:** Add to `destroy()`:
```javascript
document.getElementById('iterSaveModal')?.remove();
document.getElementById('iterDeleteModal')?.remove();
document.getElementById('baselineSaveModal')?.remove();
```

---

### WR-02 — Pattern deviation: `openIterConfirm()` Load button uses inline `onclick` string instead of `addEventListener`

**File:** `app/views/project-plan.js` (openIterConfirm)
**Confidence:** 80

Every modal added in Phase 97.2 (`deleteIteration`, `saveIteration`, `saveBaseline`) wires confirm buttons via `addEventListener('click', ...)` after `appendChild`. `openIterConfirm` uses an inline `onclick` attribute string with `window.confirmIterLoad`. If `destroy()` runs while the modal is open, `window.confirmIterLoad` is deleted, leaving a stale onclick that silently fails.

**Fix:** Wire the Load button with `addEventListener` capturing a direct closure reference to `confirmIterLoad`.

## Info

### IN-01 — `_formatIterTimestamp` < 1h bucket too coarse for same-session iterations

**File:** `app/views/project-plan.js` (_formatIterTimestamp)
**Confidence:** 80

Spec-compliant (D-09 requires `'< 1h ago'`). However, multiple iterations saved in the same session all display `'< 1h ago'` — the most common usage pattern provides no relative ordering via the timestamp. The dot number provides ordering, but the timestamp is uninformative for sub-hour saves.

**Optional fix:** Add a minutes branch:
```javascript
if (diff < 60000) return 'Just now';
if (diff < 3600000) return Math.floor(diff / 60000) + 'm ago';
```
