**Expected:** Only manually-named iterations appear; the badge count matches; no "Auto-save before..." entries visible; no green "auto" chips present.

#### 2. Relative Timestamp Display

**Test:** Save a new iteration, then immediately check the rail card's timestamp. Also find a card from a prior session (more than 24h old).
**Expected:** New card shows "< 1h ago". A 2-hour-old card shows "2h ago". A card from yesterday shows a locale date string like "6/1/2026".
**Why human:** Wall-clock delta cannot be simulated with static analysis; requires browser and real `saved_at` Firestore timestamps.

#### 3. Delete Button and Confirm Modal

**Test:** Click the × delete button on a named iteration card. Then: (a) click Cancel — verify modal closes, no deletion; (b) click the button again, click Delete — verify the card disappears from the rail.
**Expected:** Confirm modal shows "Delete Iteration" title and the iteration's name (HTML-escaped). Cancel closes without side effect. Delete removes the Firestore doc and re-renders the rail. If Firestore fails, an error toast appears.
**Why human:** Requires Firestore write permission and live DOM interaction.

#### 4. Undo Toast Copy

**Test:** Load a named iteration via the Load button (which triggers `openIterConfirm` → `confirmIterLoad` → `restoreIteration`). Observe the undo toast.
**Expected:** Toast reads exactly: `Loaded "MyIterationName" — Undo to revert.` — no reference to "auto-saved".
**Why human:** Toast is rendered in the DOM from a live restore flow; cannot verify without browser interaction.

#### 5. Auto-snapshot Firestore Cleanup on Toast Expiry

**Test:** Restore a named iteration, then wait 5 seconds without clicking Undo. After the toast disappears, open the Firebase console and query `project_iterations` for `auto: true` docs on the current project.
**Expected:** The auto-snapshot document created by the restore (STEP 1 in `restoreIteration`) is deleted from Firestore within seconds of the toast dismissing.
**Why human:** Requires Firebase console access to verify Firestore-side deletion.

#### 6. Save Iteration Modal UX

**Test:** Click "Save Iteration" from the toolbar and also from the rail "+ Save current state" link. In the modal: (a) press Enter — iteration saves; (b) clear the input and press Save — modal stays open; (c) enter a 70-char name and save — rail shows name truncated to 60 chars; (d) click Cancel — no iteration saved.
**Expected:** Both entry points show the same `iterSaveModal` with pre-filled "Iteration N" where N is the next sequence number. All edge cases behave as described.
**Why human:** Modal UX, Enter-key behavior, and truncation visibility require browser interaction.
