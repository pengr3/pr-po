## Component Inventory

All new components must reuse existing CSS classes. No new class names unless strictly required.

### Payables Tab (Finance view, 3rd tab)

**Tab button:** `.tab-btn` / `.tab-button` with `.active` state — existing pattern. Label: "Payables"

**Filter bar:** Two `<select>` dropdowns side-by-side above the table:
- Status filter: All | Pending | Partially Paid | Fully Paid | Overdue
- Department filter: All | Projects | Services
- Same styling as existing filter dropdowns in Procurement view (`.form-control` at 14px)

**Payables table columns (in order):**
RFP ID | Supplier | PO Ref | Project / Service | Tranche | Amount | Paid | Balance | Due Date | Status | Actions

- "Actions" column: single "Record Payment" button (`.btn .btn-sm .btn-primary`) — Finance role only; disabled/hidden for non-Finance
- Overdue rows get background-color: #fef2f2 applied inline
- Payment history: expandable section below each RFP row, toggled by clicking a chevron/expand icon on the row. Uses `<tr class="payment-history-row">` with `display: none` toggled to `display: table-row`. History rows get 8px padding top/bottom and a 4px solid left border (--gray-200) to visually nest under the parent RFP row.

**Void button (in payment history):** `.btn .btn-sm` with `color: var(--danger); border: 1px solid var(--danger); background: white` — outline danger style. Label: "Void"

### RFP Creation Modal (Procurement — right-click on PO row)

**Trigger:** `oncontextmenu` on `<tr>` in PO Tracking — same pattern as existing proof modal (~line 3584 of procurement.js)

**Context menu:** Small floating `<div>` with one option: "Request Payment" — same positioning and style as existing context menu

**Modal structure:** `.modal` → `.modal-content` → `.modal-header` / `.modal-body` / `.modal-footer` — window style, existing pattern

**Modal title:** "Create Request for Payment"

**Pre-filled read-only fields (shown as `.modal-detail-item` pairs, not inputs):**
- Supplier name
- PO ID
- Project / Service

**Editable fields:**
- Tranche selector: `<select>` populated from PO's tranches array — shows "Label (X%)" per option. Amount auto-fills on change.
- Amount requested: read-only `<input>` that auto-fills from tranche percentage × PO total (displayed in currency format)
- Invoice number: `<input type="text">` — required
- Due date: `<input type="date">` — required

**Footer buttons:** "Discard RFP" (`.btn .btn-outline`) | "Submit RFP" (`.btn .btn-primary`)

### Record Payment Modal (Finance — one per RFP row)

**Modal title:** "Record Payment — [RFP ID]"

**Pre-filled read-only:** Supplier, RFP amount (full tranche amount)

**Editable fields:**
- Amount: `<input type="number">` — read-only, pre-filled with tranche amount (tranches are not partially payable)
- Payment date: `<input type="date">` — required, defaults to today
- Method: `<select>` — Bank Transfer | Check | Cash | GCash/E-Wallet | Other
- "Other method" text input: hidden `<input type="text">` revealed only when Method = "Other" (inline display: none → block toggle)
- Reference number: `<input type="text">` — optional

**Footer buttons:** "Discard Payment" (`.btn .btn-outline`) | "Record Payment" (`.btn .btn-success`)

### Tranche Builder (PO creation/edit modal — replaces free-text payment_terms)

**Replaces:** Single `<input type="text" name="payment_terms">` in PO modal

**New structure:**
```
[ Label input ]  [ % input ]  [ Remove row button ]
[ + Add Tranche button ]
Running total: XX% / 100%   (red text if not 100%, green if exactly 100%)
```

- Each row: `.form-group` row with two inputs side by side (label 65% width, percentage 25% width, remove button 10%)
- Remove button: `.icon-btn` with "×" character — disabled when only one row remains
- "Add Tranche" button: `.btn .btn-outline .btn-sm` — adds a blank row
- Running total line: 14px / weight 600 — red (#ea4335) if total ≠ 100%, green (#34a853) if total = 100%
- Validation: save blocked with inline error if total ≠ 100%
- Default state (new PO): one row pre-filled as label="Full Payment", percentage=100

### PO ID Payment Fill (PO Tracking table, Procurement view)

**Container:** The existing PO ID `<td>` cell becomes `position: relative; overflow: hidden`

**Fill element:** `<div>` absolutely positioned behind the PO ID text:
- `position: absolute; left: 0; top: 0; height: 100%; width: {pct}%`
- `transition: width 0.4s ease` — smooth sweep on load and data change
- No text on the fill div itself; PO ID text sits in a `<span style="position: relative; z-index: 1">`

**Fill color rules (from CONTEXT.md decisions):**
- No RFPs on this PO: #ea4335 at 20% opacity (red tint fills full 100% width — signals "payment not started")
- At least one RFP but not fully paid: #fbbc04 at 35% opacity (orange tint, width = % paid of total PO amount)
- All tranches fully paid: #34a853 at 35% opacity (green tint, width = 100%)

**Hover tooltip:**
- `title` attribute on the `<td>` for simplicity — no custom tooltip library needed
- Format: "Paid: ₱X,XXX.00 | Balance: ₱X,XXX.00 | XX% complete"
- When no RFPs: "No payment requests submitted"

## Interaction Contracts

### Right-click → RFP Creation (Procurement)

1. User right-clicks a PO row in PO Tracking
2. Browser default context menu suppressed (`event.preventDefault()`)
3. Floating context menu appears at click coordinates with option "Request Payment"
4. Click "Request Payment" → opens RFP Creation Modal
5. Modal pre-populates supplier, PO ID, tranche selector from PO data
6. User selects tranche → Amount field auto-updates (no submit required)
7. User fills invoice number + due date → clicks "Submit RFP"
8. On success: modal closes, toast "RFP [ID] submitted successfully", PO ID cell fill updates
9. On error: inline alert in modal footer "Failed to submit RFP. Try again."

### Finance Payables tab — Record Payment

1. Finance user clicks "Record Payment" on an RFP row
2. Record Payment modal opens, amount pre-filled (read-only)
3. If Method = "Other" selected: "Specify method" text input slides in below dropdown
4. User fills date, method, reference → clicks "Record Payment"
5. On success: modal closes, RFP row updates immediately (status badge, Paid/Balance columns), toast "Payment recorded"
6. On error: inline alert in modal "Failed to record payment. Try again."

### Payment History Expand/Collapse

1. Each RFP row has a chevron icon (▶) in the leftmost column
2. Click chevron → expands hidden `<tr class="payment-history-row">` below
3. Chevron rotates 90° (▼) — CSS transform, no JS animation library
4. Payment history shows chronological list: Date | Method | Reference | Amount | Status | (Void button)
5. Voided records show strikethrough text, "Voided" status label, no Void button
6. Click chevron again → collapses row, chevron returns to ▶

### Void Payment

1. Finance clicks "Void" on a payment record in the history expansion
2. Browser `confirm()` dialog: "Void this payment record? This cannot be undone."
3. If confirmed: payment record gets `status: 'voided'`, amount removed from running total, row updates to show strikethrough
4. No modal — confirm() is sufficient for this action (low frequency, matches existing void pattern in system)

### Tranche Builder Validation (PO modal)

1. User edits percentage inputs — running total updates on every `input` event
2. Total turns green when sum = 100, red otherwise
3. Save button disabled when total ≠ 100
4. Inline error below total line when save attempted at ≠ 100: "Tranches must total exactly 100%. Current total: XX%"

## Phase-Specific Notes for Executor

1. **No new CSS files.** All new styles go in styles/views.css (view-specific) or inline where they are one-off overrides. Do not create a new stylesheet.

2. **PO ID fill effect** — implement using `position: relative` on `<td>` and an absolutely-positioned child `<div>` for the fill. The PO ID text must sit in a `<span style="position: relative; z-index: 1">` so it renders above the fill layer. Transition: `width 0.4s ease`.

3. **Payment fill opacity choice** — red fill uses 20% opacity (lower) because 100%-width red is visually alarming; orange and green use 35% because they're partial fills at lower widths and need more visibility.

4. **Expandable payment history rows** — implement as a hidden `<tr>` immediately after each RFP row. The `<td>` inside spans all columns (`colspan="11"`). Content is a nested `<table>` or `<div>` list. Toggle via `style.display = row.style.display === 'table-row' ? 'none' : 'table-row'`. History rows use 8px padding top/bottom (sm token) and a 4px solid left border to provide visual nesting under the parent row.

5. **Tranche builder** — rendered inside the existing PO edit modal body. The running-total display is a `<div>` at the bottom of the tranche section, updated on every `input` event via `addEventListener('input', recalculateTranches)` on the container.

6. **RFP ID generation** — do NOT use `generateSequentialId()`. Use custom inline: query `rfps` where `project_code == X`, find max sequence number, pad to 3 digits. Format: `RFP-[PROJECT_CODE]-001`.

7. **Security rules** — `rfps` collection rules MUST ship in the same commit as the first `addDoc` call. Do not split into two plans.

8. **Finance Payables tab** — added as the 3rd tab in finance.js, after "Purchase Orders". Use same `.tab-btn` / `.section` pattern already in the file.

9. **Status auto-derivation** — never write `status` field to Firestore. Derive in JS at render time: `sum(payment_records.filter(r => r.status !== 'voided').map(r => r.amount))` vs `amount_requested`. If sum === 0: Pending. If 0 < sum < amount_requested: Partially Paid. If sum >= amount_requested: Fully Paid. If due_date < today and not Fully Paid: Overdue (overrides other statuses in display only).
