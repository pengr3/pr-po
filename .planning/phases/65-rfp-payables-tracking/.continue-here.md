<decisions_made>

### RFP Submission Flow
- Trigger: Right-click on specific PO in POs column opens RFP creation
- Pre-fill: Supplier, PO amount, payment terms from PO doc
- Multiple RFPs per PO: YES (progress billing supported)
- Payment terms: STRUCTURED, not free-text. Tranches defined as percentage + label (e.g., [50% | Downpayment], [50% | Upon Completion])
- Tranches defined during PO creation/edit (replaces current free-text payment_terms field)
- User picks which tranche when creating RFP; amount auto-fills from tranche percentage
- Simple terms (COD/lump sum): Default to single 100% tranche
- Tranches must total exactly 100% (validation enforced)

### Payables Tab Layout
- New 3rd tab in Finance view (alongside Pending Approvals, Purchase Orders)
- One row per RFP (flat table, not grouped)
- Columns: RFP ID, Supplier, PO ref, Project/Service, Tranche label, Amount, Paid, Balance, Due Date, Status badge
- Filters: Status filter (All/Pending/Partially Paid/Fully Paid/Overdue) + Department filter (All/Projects/Services)
- Fully paid RFPs remain visible (user filters to hide if desired)
- Overdue: Red badge replaces status badge + subtle red/pink row tint

### Payment Recording
- "Record Payment" button per RFP row opens modal
- Modal fields: amount (pre-filled with balance), date, method (dropdown), reference number
- Payment methods dropdown: Bank Transfer, Check, Cash, GCash/E-Wallet, Other (shows text input)
- Payment history: expandable below RFP row, chronological list of all payments
- Edit/void policy: Void only (no edit). Voided payments marked cancelled, amount removed from running total. Audit trail maintained.
- NOTE: User wanted to clarify something about payment recording before moving on — ask on resume

### Carrying Forward (from v3.2 scoping)
- RFP IDs: `RFP-[PROJECT CODE]-###` scoped per project code
- RFP data in dedicated `rfps` collection (not embedded on PO)
- Payment status auto-derived: Pending / Partially Paid / Fully Paid (never manually set)
- payment_records as array on RFP doc (recommended for 1-5 records/RFP)
- Security Rules for `rfps` must deploy with first addDoc
- generateSequentialId() won't work for RFP IDs — custom inline generator needed (query rfps where project_code matches)
