<objective>
Add a passive update notification system to the CLMC SPA. A self-contained module polls `HEAD /index.html` every 30 minutes (only while the tab is visible), detects a new Netlify deploy via a changed ETag / Last-Modified header, and slides in a full-width dismissible top strip banner prompting the user to refresh.

Purpose: Active users currently keep running stale app versions after a deploy until they manually hard-refresh. This gives them a passive, non-disruptive nudge with zero build step and zero Firebase usage.

Output: New `app/update-check.js` module; update strip banner markup + boot wiring in `index.html`; strip CSS in `styles/components.css`.
</objective>

  <action>
    Create a new self-contained ES6 module `app/update-check.js` with NO imports (no Firebase, no router, no utils).

    Module-scope state:
    - `let baseline = null;` — holds the first observed version signal (ETag or Last-Modified string).
    - `let pollTimer = null;` — interval handle.
    - Constant `POLL_INTERVAL_MS = 30 * 60 * 1000;` (30 minutes — per Spike 001).

    Private async function `getVersionSignal()`:
    - Calls `fetch('/index.html', { method: 'HEAD', cache: 'no-store' })`.
    - Returns `response.headers.get('ETag') || response.headers.get('Last-Modified') || null`.
    - Wrap in try/catch; on any network error log `console.warn('[UpdateCheck] poll failed', err)` and return `null` (a failed poll must never crash the app or fire a false banner).

    Private async function `poll()`:
    - If `document.visibilityState !== 'visible'` return immediately (skip hidden-tab polls per Spike 001 — do NOT clear the timer).
    - `const signal = await getVersionSignal();`
    - If `signal === null` return (poll failed or header absent — do nothing).
    - If `baseline === null`, set `baseline = signal` (first poll captures baseline) and return — never show the banner on the very first successful poll.
    - If `signal !== baseline`, call `showUpdateBanner()` (the banner re-appears on every detected change even after a prior Dismiss, because Dismiss does not touch `baseline`). Do NOT update `baseline` here — leaving it stale means the user is still on the old version and a subsequent poll keeps the banner shown until they refresh.

    Private function `showUpdateBanner()`:
    - `const slot = document.getElementById('updateStripSlot'); if (slot) slot.classList.add('open');`
    - Log `console.log('[UpdateCheck] new version detected');`.

    Exported function `startUpdateCheck()`:
    - Idempotent guard: if `pollTimer !== null` return (prevent double-start).
    - Register `window.dismissUpdateBanner` (a function that does `const slot = document.getElementById('updateStripSlot'); if (slot) slot.classList.remove('open');` — Dismiss must NOT reset `baseline`).
    - Fire one immediate `poll()` (do not await — fire-and-forget) to capture the baseline on load.
    - `pollTimer = setInterval(poll, POLL_INTERVAL_MS);`

    Note: "Refresh Now" uses an inline `onclick="window.location.reload()"` in the HTML (Task 2) — no module function needed for it. Only `window.dismissUpdateBanner` is registered here.

    Do NOT add a destroy()/teardown export — update-check is a whole-session global concern with no view lifecycle.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Netlify CDN | `HEAD /index.html` response headers (ETag/Last-Modified) are read as a version signal |
| update-check.js → DOM | module toggles a CSS class on `#updateStripSlot` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-94-01 | Denial of Service | poll loop on a failing/unreachable endpoint | mitigate | `getVersionSignal()` wraps fetch in try/catch and returns null on error; a failed poll is a no-op and never throws, never shows a false banner, and the 30-min cadence bounds request volume |
| T-94-02 | Tampering | banner text / button targets | accept | Banner text is a hardcoded string literal and button targets are static (`window.location.reload`, `window.dismissUpdateBanner`); no user input is interpolated into the banner, so no XSS surface |
| T-94-03 | Information Disclosure | HEAD response headers | accept | Only ETag/Last-Modified are read; these are non-sensitive deploy metadata already public to any client requesting the page |
| T-94-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this phase — pure native `fetch`, zero new dependencies; nothing to vet |
</threat_model>
