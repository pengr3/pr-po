**Wired handleMRFScorecardClick toggle handler and filterPRPORecords scorecard AND-filter using Set-based mrf_id lookup against cachedAllPOData**

## Accomplishments

- Added `activeMaterialsFilter` and `activeSubconFilter` module-level state variables (null-initialized, immediately after `activePODeptFilter`)
- Added `renderMRFScorecardActive()` — toggles `project-scorecard-card--active` on matching `data-group`/`data-status` cards using `classList.toggle(cls, bool)`
- Added `handleMRFScorecardClick(group, status)` — toggles the correct state variable (same value → null, new value → set), then calls `renderMRFScorecardActive()` and `filterPRPORecords()`
- Updated `filterPRPORecords()` to build `materialFilterMrfIds` and `subconFilterMrfIds` Sets from `cachedAllPOData`, then AND them with existing dept/search/MRF-status conditions
- Updated `loadPRPORecords()` both paths (cached + non-cached) to route through `filterPRPORecords()` instead of directly assigning `filteredPRPORecords` and calling `renderPRPORecords()`
- Registered `window.handleMRFScorecardClick` in init's MRF Records window block; added `delete window.handleMRFScorecardClick` and state resets (`activeMaterialsFilter = null`, `activeSubconFilter = null`) to `destroy()`

- Default status fallbacks: `'Pending Procurement'` for Materials POs without `procurement_status`; `'Pending'` for Subcon POs — consistent with how `updatePOScoreboards` counts them
- `_prpoRecordsCachedAt = Date.now()` kept immediately before `filterPRPORecords()` in non-cached path so the timestamp is always set, even if `filterPRPORecords` were to throw
