# Phase 91.2: Code Review Report

**Reviewed:** 2026-05-18
**Depth:** standard
**Files Reviewed:** 1
**Status:** issues_found

## Summary

Phase 91.2 adds scorecard-click filtering to the MRF Records tab. Plan 01 notes replacing inline-styled PO scorecard HTML — the diff shows the scorecard HTML in `render()` was unchanged from the baseline; `data-group`, `data-status`, and `onclick` attributes already existed in phase 91.1. The actual changes are all in Plan 02 JS logic: `activeMaterialsFilter`/`activeSubconFilter` state, `handleMRFScorecardClick`, `renderMRFScorecardActive`, and routing both `loadPRPORecords` paths through `filterPRPORecords`.

Three issues were found. One is a blocker: after a tab-switch and return to the records tab, the scorecard visual active state is never restored even though the filter remains active, causing a hidden-filter data bug. Two warnings: a dead variable (`poStatusFilter`) that 91.2 made more salient, and an edge-case null leak in the scorecard Set construction.

---

## Critical Issues

### CR-01: Scorecard active visual state lost after tab switch — filter appears disabled but data remains filtered

**File:** `app/views/procurement.js:448-464`

**Issue:** `renderMRFScorecardActive()` is only called from `handleMRFScorecardClick()`. Per CLAUDE.md, the router calls `render()` + `init()` on every tab switch without calling `destroy()`. This recreates the DOM, stripping all `project-scorecard-card--active` classes from the cards. When `init('records')` then calls `loadPRPORecords()` → `filterPRPORecords()`, the filter state (`activeMaterialsFilter`/`activeSubconFilter`) is still set (module-level, not reset on tab switch), so the table is silently filtered. The user sees an empty or partial table with no active card highlighted, no indication a filter is active, and no obvious way to clear it.

Concrete reproduction:
1. Navigate to Records tab
2. Click "Procuring" scorecard card — card highlights, table filters
3. Click any other tab (e.g. Suppliers)
4. Click back to Records tab
5. Table shows only Procuring MRFs, but no scorecard is highlighted — user cannot tell why rows are missing

**Fix:** Call `renderMRFScorecardActive()` at the end of `filterPRPORecords()` so the visual state is always reconciled from the module-level filter variables:

```javascript
function filterPRPORecords() {
    // ... existing filter logic ...

    prpoCurrentPage = 1;
    renderPRPORecords();
    renderMRFScorecardActive(); // add this line
}
```

## Warnings

### WR-01: `poStatusFilter` dropdown is read but never applied — dead code amplified by 91.2

**File:** `app/views/procurement.js:5229`

**Issue:** `filterPRPORecords()` reads `poStatusFilter` from `#poStatusFilter` (line 5229) but the variable is never referenced in the `filter()` predicate (lines 5242–5268). The "PO Status" dropdown in the UI has no effect. This was pre-existing before 91.2, but 91.2 now adds a second PO-status filter mechanism (the scorecard cards) that does work. Having a visible PO Status dropdown that silently does nothing alongside working scorecard cards is now actively confusing: users will select "Procuring" from the dropdown and see no change, then click the scorecard and see it work, with no explanation for the difference.

**Fix:** Either implement the dropdown filter, or remove the dropdown. To implement it, add a `matchesPOStatus` check using the same pattern as the scorecard filter — look up MRF IDs from `cachedAllPOData` where `procurement_status === poStatusFilter`:

```javascript
const poStatusFilterMrfIds = poStatusFilter
    ? new Set(cachedAllPOData
        .filter(po => (po.procurement_status || (po.is_subcon ? 'Pending' : 'Pending Procurement')) === poStatusFilter)
        .map(po => po.mrf_id).filter(Boolean))
    : null;

// inside .filter(mrf => ...):
const matchesPOStatus = !poStatusFilterMrfIds || poStatusFilterMrfIds.has(mrf.mrf_id);

return matchesDept && matchesSearch && matchesMRFStatus && matchesPOStatus && matchesMaterialsScorecard && matchesSubconScorecard;
```

Alternatively, if the scorecard cards fully replace this dropdown's intent, remove the dropdown and its `id="poStatusFilter"` element.

### WR-02: Scorecard filter Set includes `undefined`/`null` `mrf_id` values — can match unrelated MRF rows

**File:** `app/views/procurement.js:5231-5240`

**Issue:** When building `materialFilterMrfIds` and `subconFilterMrfIds`, the `.map(po => po.mrf_id)` call does not filter out falsy values. If any PO document in Firestore has a missing or null `mrf_id` field (corrupt/legacy data), `undefined` or `null` is added to the Set. The predicate `materialFilterMrfIds.has(mrf.mrf_id)` then matches any MRF that also has a null/undefined `mrf_id`, causing unrelated rows to appear when a scorecard filter is active.

The `loadPRPORecords` path at line 5157 already guards against this with `.filter(Boolean)`:
```javascript
const visibleMrfIds = new Set(allPRPORecords.map(m => m.mrf_id).filter(Boolean));
```
The scorecard filter set construction lacks the same guard.

**Fix:** Add `.filter(Boolean)` to both `.map(po => po.mrf_id)` calls:

```javascript
const materialFilterMrfIds = activeMaterialsFilter
    ? new Set(cachedAllPOData
        .filter(po => !po.is_subcon && (po.procurement_status || 'Pending Procurement') === activeMaterialsFilter)
        .map(po => po.mrf_id)
        .filter(Boolean))  // add this
    : null;
const subconFilterMrfIds = activeSubconFilter
    ? new Set(cachedAllPOData
        .filter(po => po.is_subcon && (po.procurement_status || 'Pending') === activeSubconFilter)
        .map(po => po.mrf_id)
        .filter(Boolean))  // add this
    : null;
```
