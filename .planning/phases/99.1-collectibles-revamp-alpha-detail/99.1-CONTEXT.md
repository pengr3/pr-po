## Phase Boundary

**Surface:** Project **and Service** detail page money-in (collectibles) view ‚Äî the inline "Financial Summary" card + its "Full Breakdown" modal ‚Äî PLUS the service-side port of the Phase 99 billing-request flow and the minimal Finance-side generalization needed to make service billing requests work end-to-end.

**What this phase delivers:**
1. The 025+026 detail-page money-in revamp on **projects** (lifecycle footer, scorecards, fixed modal entry point).
2. **The same revamp mirrored to services** ‚Äî design parity.
3. **The full Phase 99 billing-request capability ported to services** (per user directive 2026-06-06: "what's done on 99 on projects shall be done on services too"). Services currently have collectibles but NO ops-initiated billing-request flow; this phase adds it.
4. The **minimal Finance-side generalization** (finance.js approve bridge + pending banner) so service billing requests are reviewable/approvable ‚Äî without doing the larger Phase 99.2 finance revamp.

### Scope Expansion (user directive, 2026-06-06)
The ROADMAP ¬ß99.1 originally scoped service-detail.js as a simple display "design parity" mirror. The user clarified that the **entire Phase 99 billing-request flow must be mirrored to services**, not just the display. Phase 99 spanned 4 files (firestore.rules, notifications.js, project-detail.js, finance.js); services today have **none** of that billing infrastructure (confirmed by grep: no `currentBillingRequests`, no `ensureBillingRequestsListener`, no `renderOwnBillingRequests`, no `openBillingRequestModal`/`submitBillingRequest`). So this phase now also ports the Phase 99 project-side billing flow into service-detail.js and minimally generalizes finance.js. This is a deliberate, user-approved scope increase over the ROADMAP wording.

**Target files (5 + 1 confirmed-no-change):**
- `app/views/project-detail.js` ‚Äî 99.1 display revamp
- `app/views/service-detail.js` ‚Äî Phase 99 billing-flow port **+** 99.1 display revamp
- `app/expense-modal.js` ‚Äî 99.1 Full-Breakdown modal revamp (already shared; works in both `mode:'project'` and `mode:'service'`)
- `app/views/finance.js` ‚Äî **minimal** generalization of the pending-billing banner + approve bridge to handle service requests (NOT the Phase 99.2 finance revamp)
- `app/notifications.js` ‚Äî BILLING_REQUEST_DECIDED routing for service requests (minor edge)
- `firestore.rules` ‚Äî **NO CHANGE** (the `billing_requests` block at firestore.rules:539-553 is already department-agnostic: create/read=isActiveUser, update/delete=finance roles)

**No Firestore schema migration.** Firestore is schemaless; service billing_requests simply carry `department:'services'` + `service_code`/`service_name` instead of project fields (D-22). All collectibles data already exists (Phase 85).

**Still out of scope (Phase 99.2 owns these):** the Finance Collectibles tab aggregate scorecard, the 10‚Üí5 column table redesign, the ‚â§768px mobile card layout, and the approved-not-filed **orphan-state fix** (making approve+create atomic). ## Implementation Decisions

All decisions below are **LOCKED**. D-01..D-20 are validated in spikes 025/026 + the spec. D-21..D-27 implement the user's full-mirror directive, grounded in the Phase 99 implementation. IDs are referenced by plans' `must_haves`/`truths`.

### Shared Money-In Vocabulary (LOCKED ‚Äî must match Phase 99.2 verbatim)
- **D-01** ‚Äî **Billed** = `billing_requests.amount_requested` WHERE `status='approved'`. What ops requested to bill the client. (Advisory ‚Äî Finance re-derives the authoritative amount at approval, Phase 99 T-99-05.)
- **D-02** ‚Äî **Invoiced** = `collectibles.amount_requested` (FROZEN at collectible creation, Phase 99 D-13). What Finance actually invoiced as a COLL-xxx.
- **D-03** ‚Äî **Collected** = sum of `payment_records[].amount` WHERE `status !== 'voided'` (Phase 85 D-17). Cash actually received.
- **D-04** ‚Äî **Outstanding** = Invoiced ‚àí Collected. Open receivable. Sourced from **collectible docs**, NOT billing_requests.
- **D-05** ‚Äî Reconciliation key between `billing_requests` and `collectibles` is **`project_code + tranche_index`** (services: `service_code + tranche_index`). In-page (already scoped by code) match on `tranche_index` alone.

### Footer Lifecycle Rows (025 ‚Äî LOCKED; applies to BOTH project & service detail)
- **D-06** ‚Äî Replace the status-only `renderOwnBillingRequests()` (project-detail.js:632; net-new on service-detail.js) with **per-tranche lifecycle rows** iterating `collection_tranches` so **every tranche is always a row** (even unfiled).
- **D-07** ‚Äî Lifecycle stages per tranche, derived from billing-request status √ó collectible existence √ó cash %:
  - no request ‚Üí **Not Filed** (gray) ¬∑ `pending` ‚Üí **Pending** (amber) ¬∑ `rejected` ‚Üí **Rejected** (red)
  - approved + no collectible ‚Üí **Approved ‚Äî Not Yet Invoiced** (indigo) ‚Üê key currently-invisible state
  - approved + collectible + 0% ‚Üí **Invoiced ‚Äî Awaiting Payment** (teal)
  - approved + collectible + 0<pct<100 ‚Üí **Billed/Collecting** (progress bar)
  - approved + collectible + 100% ‚Üí **Fully Collected ‚úì** (green)
  - **Services note:** services have no `pending`/`approved`/`rejected` until D-21 ports billing requests; once ported, services use the same stages. A service tranche with a collectible but no billing_request still resolves to the collectible-derived stages.
- **D-08** ‚Äî Unfiled tranches grayed at **45% opacity** with a dashed **"‚Äî Not Filed"** badge.
- **D-09** ‚Äî Partial payments shown as a **note under the Billed badge** ("‚Ç±X of ‚Ç±Y ¬∑ Z%"), NOT a separate stage.
- **D-10** ‚Äî **2-chip card scorecard**: **Collected** + **Outstanding**, both sourced from collectible docs (D-03/D-04). **Project-side landmine:** the current `refreshExpense()` (project-detail.js:1240-1241) sources "filed/totalRequested" from approved **billing_requests**, violating D-04 ‚Äî fix it to source from collectible docs. The **service side already does this correctly** (service-detail.js:984-990); unify the project side to match.
- **D-11** ‚Äî **"Initiate Billing ‚Üí"** entry retained at footer right on projects (existing Phase 99 flow). On **services**, "Initiate Billing ‚Üí" is **added** as part of the D-21 port (it does not exist today).

### Collectibles Listener (025 ‚Äî LOCKED; both detail files)
- **D-12** ‚Äî Add a **`collectibles` listener scoped to `project_code`/`service_code`** so the footer reflects COLL state (raw docs with `payment_records`), not just `billing_requests`. Mirror `ensureBillingRequestsListener()` (project-detail.js:261). **Name collision:** `currentCollectibles`/`currentServiceCollectibles` already exist as *aggregate* objects ‚Äî use a distinct name for the **raw doc array** (e.g. `currentCollectibleDocs`). Recommendation (discretion): a separate named handle (`collectiblesListenerUnsub`). Scope by the **doc field** (`currentProject.project_code`), NOT the URL param (Phase 78 doc-id fallback). Unsubscribe in `destroy()`.
- **D-13** ‚Äî Stage/cash derivation reuses the **exact `deriveCollectibleStatus` formula** (finance.js:110 = expense-modal.js:24): `totalPaid` = non-voided `payment_records[].amount`; `pct = totalPaid / amount_requested`. Do NOT add a third divergent copy or a different denominator/voided rule. Extract a shared `computeTrancheLifecycle(tranche, idx, billingReqs, collectibleDocs)` helper feeding the footer, the modal ghost-rows, and both detail files.
