## Phase Boundary

**Surface:** Project/Service **detail page** money-in (collectibles) view — the inline "Financial Summary" card + its "Full Breakdown" modal. This is **Surface 1** of the two-surface Collectibles Revamp; Surface 2 (the standalone Finance Collectibles tab) is Phase 99.2 and is explicitly **out of scope** here.

**What this phase delivers:** Overhaul the detail-page billing footer from a status-only list into a full per-tranche lifecycle, fix the buried/confusing modal entry point, and add money-in scorecards — so a user reads billing health (Billed → Invoiced → Collected → Outstanding) at a glance, including the currently-invisible "approved but not yet invoiced" state.

**File-independent of Phase 99.2** (which is `finance.js`-only), but MUST share the money-in vocabulary/formulas defined below so the two surfaces never disagree.

<decisions>
## Implementation Decisions

All decisions below are **LOCKED** — validated in spikes 025/026 and the consolidated spec. IDs are referenced by plans' `must_haves`/`truths`.

### Shared Money-In Vocabulary (LOCKED — must match Phase 99.2 verbatim)
- **D-01** — **Billed** = `billing_requests.amount_requested` WHERE `status='approved'`. Meaning: what ops requested to bill the client.
- **D-02** — **Invoiced** = `collectibles.amount_requested` (FROZEN at collectible creation, per Phase 99 D-13). Meaning: what Finance actually invoiced as a COLL-xxx.
- **D-03** — **Collected** = sum of `payment_records[].amount` WHERE `status !== 'voided'`. Meaning: cash actually received. (Voided payments are excluded — they are audit-trail only, per Phase 85 D-17.)
- **D-04** — **Outstanding** = Invoiced − Collected. Meaning: open receivable. Outstanding is sourced from **collectible docs**, NOT billing_requests.
- **D-05** — Reconciliation key between `billing_requests` and `collectibles` is **`project_code + tranche_index`** (both docs carry both fields → client-side `.find()`).

### Footer Lifecycle Rows (025 — LOCKED)
- **D-06** — Replace `renderOwnBillingRequests()` (currently status-only, project-detail.js:632) with **per-tranche lifecycle rows**. **All tranches always visible** (one row per `collection_tranches` entry) — not just filed ones.
- **D-07** — Four always-visible lifecycle stages per tranche: **Not Filed → Pending → Approved → Billed → Collected**. Stage is derived by cross-referencing the tranche's `billing_requests` status against the matching `collectibles` doc (by D-05 key) and its cash %:
  - No request → **Not Filed** (grayed)
  - billing status `pending` → **Pending** (amber "Pending Review")
  - billing status `rejected` → **Rejected** (red)
  - approved + no matching collectible → **Approved — Not Yet Invoiced** (indigo) ← the key currently-invisible state
  - approved + collectible exists + 0% cash → **Invoiced — Awaiting Payment** (teal)
  - approved + collectible exists + 0<pct<100 → **Billed/Collecting** (progress bar)
  - approved + collectible exists + 100% → **Fully Collected ✓** (green)
- **D-08** — Unfiled tranches are **grayed at 45% opacity** with a dashed **"— Not Filed"** badge.
- **D-09** — Partial payments are shown as a **note under the Billed badge** (e.g. "₱X of ₱Y · Z%"), NOT as a separate lifecycle stage (tranche amounts are pre-determined; full payment is the expectation).
- **D-10** — The card carries a **2-chip scorecard**: **Collected** + **Outstanding**, both sourced from collectible docs (D-03/D-04). Replaces the ambiguous current chips.
- **D-11** — **"Initiate Billing →"** entry retained at footer right (existing billing-request submission flow from Phase 99 — unchanged behavior).

### Collectibles Listener (025 — LOCKED)
- **D-12** — Add a **`collectibles` listener scoped to `project_code`** so the footer reflects COLL state (cash received, invoiced amounts), not just `billing_requests`. Follow the existing Phase 99 `billing_requests` listener pattern (`ensureBillingRequestsListener()`, project-detail.js:261). Store the unsubscribe handle and **unsubscribe it in `destroy()`** alongside the existing teardown (project-detail.js:317/344-345).
- **D-13** — Stage derivation reuses the **same status logic as `deriveCollectibleStatus`** (finance.js:110; a local copy already lives in expense-modal.js:24). Do NOT re-derive cash % with a different formula — keep it consistent with D-03.

### Entry Point (026 — LOCKED)
- **D-14** — Add a single **"Full Breakdown →" button in the Financial Summary card header** (right side) that **always refresh-then-opens** the modal. The ~300ms refresh is not perceptible.
- **D-15** — **Remove the hidden click affordance** on the "Projected Expense" number (project-detail.js:544 `onclick="window.showExpenseModal()"`) and the adjacent 🔄 button (project-detail.js:547).
- **D-16** — **Collapse the `window.showExpenseModal` / `window.refreshAndShowExpenseModal` split** (project-detail.js:1902-1903) into one always-refresh path behind the new header button. Clean up the now-unused window function(s) in `destroy()`.

### Full Breakdown Modal Scorecards (026 — LOCKED)
- **D-17** — Modal **header strip = 5 chips**: Expense / Rem. Payable / Billed to Client / Invoiced (COLLs) / Cash Received. Each chip a distinct color family.
- **D-18** — Modal **Collectibles tab = 3-chip summary**: Total Billed / Cash Collected / Outstanding + a short **formula note** explaining Billed (billing_requests) vs Invoiced (collectibles).
- **D-19** — Modal Collectibles tab shows an **"awaiting COLL-xxx" ghost row** for approved-but-not-yet-invoiced tranches (approved billing_request with no matching collectible by D-05).

### Parity (LOCKED)
- **D-20** — `service-detail.js` reaches **design parity** with `project-detail.js` — every footer/scorecard/entry-point change above is mirrored. Follow the established mirror pattern (Phase 96-03 precedent; services use the same `collection_tranches` + `collectibles` shape).

### Claude's Discretion
- Exact DOM structure / CSS class names for lifecycle rows, badges, and chips (match existing card styling in project-detail.js; reuse the design system colors from CLAUDE.md UI section).
- Whether the new `collectibles` listener lives **inside** `ensureBillingRequestsListener()` or as a **separate sibling handle** (D-12) — pick the cleaner option; the open question is non-blocking. Recommendation: separate named handle for clear teardown, but a combined setup is acceptable if both unsubscribe in `destroy()`.
- Helper extraction (e.g. a shared `computeTrancheLifecycle(tranche, billingReqs, collectibles)` function) to avoid duplicating stage logic between the footer, the modal, and service-detail.js — strongly encouraged to keep D-07 consistent in one place.
- Exact label wording on badges/chips as long as vocabulary (D-01..D-04) stays exact.
- Whether to keep a thin backward-compatible alias for removed window functions during transition (prefer clean removal).

</decisions>

<specifics>
## Specific Ideas

**Phase-local requirement IDs** (none exist in REQUIREMENTS.md; planner must define these and put each in a plan's `requirements` frontmatter — mirrors how Phase 99 used BILL-01..06). Map 1:1 to the 6 ROADMAP success criteria:

| Req ID | Success criterion |
|--------|-------------------|
| CDR-01 | Per-tranche lifecycle rows (4 stages, always-visible, 45%-gray unfiled, partial-payment note) — D-06..D-09 |
| CDR-02 | `collectibles` listener scoped to `project_code`, unsubscribed in `destroy()` — D-12/D-13 |
| CDR-03 | 2-chip card scorecard (Collected + Outstanding) + "Initiate Billing →" retained — D-10/D-11 |
| CDR-04 | Single "Full Breakdown →" header button (always refresh-then-open); remove hidden click + collapse show/refresh split — D-14..D-16 |
| CDR-05 | Modal 5-chip header strip + 3-chip Collectibles-tab summary + formula note + "awaiting COLL-xxx" ghost row — D-17..D-19 |
| CDR-06 | `service-detail.js` design parity — D-20 |

**Balance-formula landmine (from spike 026):** the OLD inline card computed remaining from `billing_requests.amount_requested`, while the modal computed from `collectibles.amount_requested`. These diverge if Finance invoices a different amount than requested. The fix (D-01/D-02/D-04/D-10): the **detail-card scorecard and the modal both source Collected/Outstanding from collectible docs**; "Billed" (billing_requests) is surfaced as a *separate* explicit chip in the modal, never conflated with Invoiced. Label both clearly per D-18.

**UAT reality:** zero-build static SPA, no test harness — all acceptance is **browser-UAT-gated**. Plans should budget explicit manual UAT steps (DevTools, multiple tranche/collectible states) especially for the approved-minus-invoiced reconciliation (D-07 indigo state, D-19 ghost row).

</specifics>
