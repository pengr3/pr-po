**1. Create `app/views/mrf-records.js` as a shared rendering module.**

This module encapsulates the MRF Records table rendering logic currently in procurement.js. It does NOT duplicate — procurement.js will import from this module. Following the expense-modal.js precedent (shared between project-detail, service-detail, finance).

The module exports a single primary function:

```javascript
/**
 * Render MRF Records table with search/filter/pagination
 * @param {Object} options
 * @param {string} options.containerId - DOM container ID for the table
 * @param {Function} [options.filterFn] - Optional additional filter (e.g., requestor filter)
 * @param {boolean} [options.showEditControls] - Show PO status dropdowns and edit actions
 * @param {boolean} [options.showTimeline] - Show timeline button column
 * @param {string[]} [options.statusFilter] - MRF statuses to query (defaults to historical statuses)
 * @returns {Object} - Controller object with { load, filter, destroy } methods
 */
export function createMRFRecordsController(options)
```

The controller pattern:
- `load()` — Fetches MRFs from Firestore using `getDocs`, sorts by date, calls render. Uses `getDocs` (one-time fetch), NOT `onSnapshot` (per RESEARCH.md recommendation for "My Requests" — simpler, no listener cleanup risk).
- `filter()` — Applies search, status, and optional custom filter function, re-renders
- `destroy()` — Cleans up any window functions registered by this controller instance

**What to extract from procurement.js:**
- The MRF data-fetching logic from `loadPRPORecords()` (lines 2226-2270): fetching MRFs with status filter, fetching POs for scoreboards
- The filtering logic from `filterPRPORecords()` (lines 2329-2351): search, status, department filter
- The rendering logic from `renderPRPORecords()` (lines 2403-2750): table generation with PR/PO sub-queries
- The pagination logic from `renderPRPOPagination()` (if separate)
- The helper functions: `calculateMRFStatus()`, `renderMRFStatusBadge()`, `updatePOScoreboards()`

**Key considerations:**
- The shared module must import from `firebase.js`, `utils.js`, and `components.js` (for `getMRFLabel`, `getDeptBadgeHTML`, `getStatusClass`)
- The scoreboards are specific to the Procurement view — keep them as an optional feature. The `options` parameter should have `showScoreboards: boolean` to control this.
- The search filter must include `requestor_name` and `service_name` (which Plan 01 adds to procurement.js — this is why Plan 04 depends on Plan 01).
- The `filterFn` option allows "My Requests" to add a `requestor_name === currentUser` filter on top of the base filters.
- Pagination state (currentPage, itemsPerPage) should be instance-scoped, not module-level — this allows Procurement and My Requests to have independent pagination.
- Window functions for `showProcurementTimeline`, `viewPRDetails`, `viewPODetails`, `viewPOTimeline`, `updatePOStatus` must already be registered by the calling view (procurement.js). The shared module should NOT re-register these — it just renders HTML that references them.

**The `options.statusFilter` parameter is important:**
- Procurement MRF Records queries: `where('status', 'in', ['TR Submitted', 'PR Generated', 'PR Rejected', 'Finance Approved', 'PO Issued', 'Delivered', 'Completed'])`
- "My Requests" queries ALL statuses for the user (including 'Pending', 'Approved') — so it passes no status filter or a broader set

**`initMyRequests()` function:**
- Gets the current user via `window.getCurrentUser?.()`
- Imports and creates a controller from mrf-records.js:
  ```javascript
  import('./mrf-records.js').then(async ({ createMRFRecordsController }) => {
      const user = window.getCurrentUser?.();
      myRequestsController = createMRFRecordsController({
          containerId: 'myRequestsContainer',
          showEditControls: false, // Requestors view is read-only for PO status
          showTimeline: true, // Let them see timelines

      });
      await myRequestsController.load();
  });
  ```
- Wire up the search/filter controls in the My Requests HTML to the controller's `filter()` method

**CRITICAL — Router tab switching behavior:**
The router does NOT call `destroy()` when switching between sub-tabs within the same view. So when switching from `form` → `my-requests`:
- `render('my-requests')` is called (replaces DOM)
- `init('my-requests')` is called
- The form's projectsListener and servicesListener are still alive!

Handle this by: In `init()`, BEFORE branching on activeTab, clean up any existing controller or listener from the other sub-tab:
```javascript
export async function init(activeTab = 'form') {
    // Clean up from previous sub-tab
    if (activeTab === 'my-requests') {
        // Unsubscribe form listeners if they exist
        if (projectsListener) { projectsListener(); projectsListener = null; }
        if (servicesListener) { servicesListener(); servicesListener = null; }
        await initMyRequests();
        return;
    } else {
        // Clean up My Requests controller if it exists
        if (myRequestsController) { myRequestsController.destroy(); myRequestsController = null; }
        // ... existing form init ...
    }
}
```

Add `myRequestsController` as a module-level variable.

**4. Update `destroy()` to clean up both sub-tab states:**

export async function destroy() {
