# Cross-AI Plan Review — Phase 87.1

## Gemini Review

This review evaluates the **Phase 87.1: Proposal Lifecycle Integration** implementation plan. The strategy is well-structured, prioritizing foundational changes before branching into parallel UI integrations and final system wiring.

### 1. Summary

The plan is highly coherent and demonstrates a deep understanding of the project's "Pure JS" architecture. By promoting internal `proposals.js` functions to exports, the plan enables code reuse across `project-detail`, `service-detail`, and `home` views without reinventing state transition logic. The wave-based approach correctly sequences the "breaking" refactor (splitting Engagements into a new view) before updating the navigation and router, ensuring the application remains functional during the transition. The inclusion of the `parent_collection` fix in Wave 1 is a critical catch that enables the subsequent Service-detail integration to work correctly.

### 2. Strengths

- **Effective Logic Reuse:** Exporting `_applyProposalStateTransition` and badge renders from `proposals.js` is a surgical way to maintain consistency across the UI while adhering to the module-based architecture.
- **Robust Cleanup:** Every plan explicitly addresses the `destroy()` lifecycle, ensuring that global `window` functions and Firebase listeners are purged, which is vital for preventing memory leaks and event collisions in this SPA.
- **Atomic Parallelism:** The separation of Project and Service detail views into parallel plans (Wave 3) is efficient, as they share logic but target different files and data structures.
- **Safe Dashboard Implementation:** Plan 06 uses `getDocs` instead of `onSnapshot` for the home dashboard. This is a smart performance choice for a high-traffic "Overview" page where real-time proposal updates are less critical than in the main Proposals tab.
- **Defensive UI:** The immediate disabling of "Confirm" buttons in the submission modals (Plans 04/05) is an essential UX pattern to prevent duplicate Firestore writes in a zero-framework environment.

### 3. Concerns

- **[LOW] Circular Dependency Risk:** Plans 04/05 import from `proposals.js`. If `proposals.js` ever imports from `project-detail.js` (e.g., for navigation helpers), this will create a circular dependency. Currently `proposals.js` seems independent, but worth noting.
- **[MEDIUM] Hardcoded Status Lists:** `PROPOSAL_RANGE_STATUSES` is hardcoded in both `project-detail.js` and `service-detail.js`. If the business logic for which projects show a proposal card changes, two files must be updated — increases maintenance overhead and risk of "drift" between Projects and Services.
- **[LOW] Manual Router Gating:** Plan 07 uses a hardcoded role check in `router.js` rather than the `routePermissionMap`. While explained as a way to avoid `hasTabAccess` blocking, it creates a second, divergent way of handling permissions in the app.
- **[MEDIUM] Duplicated Pill Helpers:** Plan 02 copies personnel pill helpers verbatim from `proposals.js`. Any bug fixes or styling changes to the "Personnel" UI must now be applied in two places (`proposals.js` and `engagements.js`).

### 4. Suggestions

- **Centralize Shared Logic:** To avoid duplication in Plan 02 and hardcoding in Plans 04/05, consider creating `app/proposal-utils.js` or moving helpers to `app/utils.js`. Specifically, `PROPOSAL_RANGE_STATUSES` and personnel pill rendering functions are prime candidates for a shared utility module.
- **Standardize Router Permissions:** Try to integrate the `/engagements` route into the existing `routePermissionMap` in `router.js` if possible. If `hasTabAccess` is the blocker, consider refactoring `hasTabAccess` to be more flexible rather than bypassing the standard permission flow in `navigate()`.
- **Home Dashboard Empty State:** In Plan 06, ensure that if a user switches to the "Proposals" sub-tab and then all proposals are closed/filtered out, the UI handles the "Empty" state gracefully (e.g., showing "No active proposals").
- **Explicit Logging:** In Plans 04/05, when `loadProposalCard` fails, ensure the error is logged with the specific ID of the project/service to aid in debugging production issues.

### 5. Risk Assessment

**Overall Risk: LOW**

The plan is technically sound and follows established patterns in the codebase. The dependencies between waves are clearly defined, and the "Threat Models" included in the plans show a proactive approach to security and concurrency. The most significant risks are long-term maintenance (due to code duplication in Plan 02) rather than immediate functional failure. The "Pure JS" constraints are respected, and the manual UAT focus is supported by the clear UI/UX descriptions provided in the tasks.
