## Summary

Phase 87.1 (redesign) restructures the Proposals surface. The full implementation was reverted after a prior attempt due to the CR-01 personnel pill window function name mismatch bug. The new design is architecturally different: instead of a separate #/engagements route, the home page gains three sub-tabs (Overview, Engagements, Proposals) that replace the Proposals top-nav tab. A new shared utility app/proposal-modal.js is extracted from proposals.js so the full proposal detail modal can be opened from home, project-detail, and service-detail without duplicating code.

**Current state after revert (2026-05-20):** proposals.js is the original Phase 87/88/89 monolith (2,385 lines) with render/init/destroy as a routed view. home.js is clean at 206 lines with no sub-tab logic. The Proposals nav link exists in index.html and auth.js. router.js has the /proposals route with a super_admin gate. engagement-create.js exports only createEngagement(). No app/proposal-modal.js exists. No inline proposal card in project-detail.js or service-detail.js. No parent_collection field in proposals.js logic.

**Primary recommendation:** Implement in this order: (1) proposals.js export additions + parent_collection fix, (2) app/proposal-modal.js extraction, (3) engagement-create.js extension with form exports, (4) home.js sub-tab navigation, (5) project-detail.js + service-detail.js inline card, (6) retire Proposals nav (router + auth + index.html). Order matters because home.js imports from proposal-modal.js and engagement-create.js; those must exist first.

## Personnel Pill Window Function Name Map (CR-01 Avoidance)

### How the Current proposals.js Form Works (Post-Revert)

The Create Engagement form uses a **private, local pill implementation** with unsuffixed window function names. The form HTML (render(), lines 143–144) hard-codes:
```
oninput="window.proposalFilterPersonnelDropdown(this.value)"
onfocus="window.proposalShowPersonnelDropdown()"
```

The `proposalFilterPersonnelDropdown()` function generates dropdown item HTML:
```javascript
// proposals.js line 240:
onmousedown="event.preventDefault(); window.proposalSelectPersonnel('${user.id}', '${...}')"
```

The `renderProposalPills()` function generates pill remove buttons:
```javascript
// proposals.js line 197:
onmousedown="event.preventDefault(); window.proposalRemovePersonnel('${user.id}', '${...}')"
```

### Why CR-01 Happened

The prior 87.1 attempt imported `renderProposalPillsFor` from proposals.js — an exported wrapper function that generated HTML calling SUFFIXED names (`window.proposalSelectPersonnelIn`, etc.). The engagements.js in that attempt still registered the UNSUFFIXED names (copying from the proposals.js init pattern), creating the mismatch.

### Why the New Design Avoids CR-01

In the new design, the entire form (HTML + all private helper functions) moves into engagement-create.js as a **unit**. The form HTML continues to call the unsuffixed window functions. The private helper functions continue to generate HTML calling the same unsuffixed names. There is no imported suffixed wrapper anywhere. The names are consistent by construction.

**Register in initEngagementForm():**
```javascript
window.submitNewEngagement = submitNewEngagement;
window.handleEngagementTypeChange = handleEngagementTypeChange;
window.proposalSelectPersonnel = proposalSelectPersonnel;
window.proposalRemovePersonnel = proposalRemovePersonnel;
window.proposalShowPersonnelDropdown = proposalShowPersonnelDropdown;
window.proposalFilterPersonnelDropdown = proposalFilterPersonnelDropdown;
```

**Delete in destroyEngagementForm():**
```javascript
delete window.submitNewEngagement;
delete window.handleEngagementTypeChange;
delete window.proposalSelectPersonnel;
delete window.proposalRemovePersonnel;
delete window.proposalShowPersonnelDropdown;
delete window.proposalFilterPersonnelDropdown;
```

**Lifecycle management:** Call `initEngagementForm()` once in home.js init() (for eligible roles). Call `destroyEngagementForm()` in home.js destroy(). Do NOT re-init/re-destroy on sub-tab switches — the listeners run regardless of which sub-tab is active, which is acceptable since they only read clients/users collections.

As a defensive pattern, `destroyEngagementForm()` should be idempotent (safe to call multiple times) and `initEngagementForm()` should call `destroyEngagementForm()` at the top as a reset guard.

```javascript
export async function openProposalModal(proposalId, context) {
    const snap = await getDoc(doc(db, 'proposals', proposalId));

    window.openCreateProposalModal = openCreateProposalModal;

}
```

## Common Pitfalls

### Pitfall 1: STAGE_ORDER Is Currently a Local Const, Not Module-Level
**What goes wrong:** Implementer tries to add `export` to STAGE_ORDER at line 684 and gets a syntax error because it's inside a function body.
**Why it happens:** STAGE_ORDER is declared with `const` inside `renderProposalDashboard()`, not at module scope.
**How to avoid:** Move `const STAGE_ORDER = [...]` to top of file before adding `export`. Then update renderProposalDashboard() to reference the module-level const (remove the local declaration).
**Warning signs:** SyntaxError on import; "STAGE_ORDER is not exported" at runtime.

### Pitfall 2: PROPOSAL_RANGE_STATUSES Does Not Exist
**What goes wrong:** Importing it from proposals.js yields undefined.
**Why it happens:** It was deleted during the revert. It must be created as new code.
**How to avoid:** Add explicitly as a new `export const PROPOSAL_RANGE_STATUSES = [...]` at module top level.

### Pitfall 3: proposal-modal.js Looks Up proposalsData
**What goes wrong:** openProposalModal() uses `proposalsData.find()` and fails when called from project-detail (proposalsData is empty).
**Why it happens:** Directly copying openProposalDetail() verbatim without adapting to stateless context.
**How to avoid:** openProposalModal() must use `await getDoc(doc(db, 'proposals', proposalId))`.

### Pitfall 4: Home Approval Queue Shows Stale Data After Action
**What goes wrong:** User approves from home queue, the proposal disappears from Firestore but queue HTML is unchanged.
**Why it happens:** getDocs is one-time; no onSnapshot to auto-refresh.
**How to avoid:** After queueConfirmAction() succeeds, re-call `_loadHomeProposalsTab()` to refresh the whole proposals content area. The re-fetch is acceptable since queue actions are infrequent.

### Pitfall 5: cryptoRandomUuid Duplication
**What goes wrong:** proposals.js uses cryptoRandomUuid() in _applyProposalStateTransition (which stays in proposals.js); proposal-modal.js also needs it in saveProposal().
**Why it happens:** The function must exist in both files if it's private.
**How to avoid:** Either duplicate it (two copies, identical function), or move it to app/utils.js as a named export. Moving to utils.js is cleaner.

### Pitfall 6: proposals.js Still Imports Its Own Modal Functions After Removal
**What goes wrong:** References to removed functions cause "not defined" errors in proposals.js callers.
**Why it happens:** The onSnapshot callback calls `renderProposalDashboard()` which may reference functions that moved.
**How to avoid:** renderProposalDashboard() stays in proposals.js. Only verify it does NOT reference any of the moved modal functions.

### Pitfall 7: home.js queueConfirmAction Needs proposalsData Context
**What goes wrong:** home.js approval queue calls queueConfirmAction but queueConfirmAction in proposals.js references `proposalsData` which isn't populated on home.
**Why it happens:** The queue functions in proposals.js use `proposalsData.find()` for lookups.
**How to avoid:** The home Proposals sub-tab should use its own local queue render function that passes the proposal data inline (e.g., encode proposal ID in the button and fetch fresh on confirm). Do NOT import and re-use queueConfirmAction from proposals.js — write a local version in home.js that uses getDoc() for fresh fetch. OR: home.js stores its own fetched proposals array and the queue buttons reference it.

The simplest approach: home.js renders the approval queue with the same HTML pattern but the confirm handler fetches fresh data via getDoc. This avoids coupling to proposals.js module state.
