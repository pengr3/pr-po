## D-06 parent_collection Handling Note

The proposal doc itself carries `parent_collection: 'projects' | 'services'`, set when the engagement was created in `engagement-create.js` (Plan 87.1-03). The inline-card flow handles parent_collection as follows:

1. **loadProposalCard signature** — Accepts `parentCollection` parameter (`'projects'` from project-detail.js, `'services'` from service-detail.js). The Firestore query is the same in both views: `where('project_id', '==', parentDocId)` — proposal docs use `project_id` as the parent reference field regardless of which collection the parent lives in. The parentCollection parameter is held for call-site clarity and potential future expansion (e.g., separate indexes per parent collection).

2. **confirmProposalInlineSubmit write path** — Performs a fresh `getDoc(doc(db, 'proposals', id))` before calling `_applyProposalStateTransition`. The fresh proposal object includes the `parent_collection` field. Inside `_applyProposalStateTransition` (proposals.js Wave 1), the write resolves the target collection via `const parentCollection = proposal.parent_collection || 'projects';` then writes to `doc(db, parentCollection, proposal.project_id)`. This means:
   - A service proposal (parent_collection: 'services') → writes to `services/{project_id}`.
   - A project proposal (parent_collection: 'projects') → writes to `projects/{project_id}` (default for legacy docs).
   - The caller (inline-card confirm handler) doesn't need to know or pass the collection name — the proposal doc is the source of truth.

3. **Safety property** — Even if a future refactor changes the parentCollection parameter accidentally, the write still goes to the correct collection because the resolution happens inside `_applyProposalStateTransition` based on the proposal doc's own field, not on caller input. This is the safer pattern.

The Phase critical anti-pattern callout "D-06 parent_collection: verify your service-detail.js inline-card code passes/preserves parent_collection correctly" is satisfied: the field is preserved through the fresh-getDoc → transition pipeline; no caller-side handling is required.
