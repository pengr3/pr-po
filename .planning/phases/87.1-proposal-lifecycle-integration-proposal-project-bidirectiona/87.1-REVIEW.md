## Critical

### CR-01 — Window function name mismatch breaks personnel pill interactions in engagements.js

**File:** `app/views/engagements.js` (init/destroy) + `app/views/proposals.js` (exported wrappers)
**Confidence:** 97%

`engagements.js init()` registers personnel pill window functions under the **unsuffixed** names:
- `window.proposalSelectPersonnel`
- `window.proposalRemovePersonnel`
- `window.proposalShowPersonnelDropdown`
- `window.proposalFilterPersonnelDropdown`

But the imported `renderProposalPillsFor` and `proposalFilterPersonnelDropdownFor` from proposals.js generate inline `onclick`/`oninput` handlers that call the **suffixed** `*For`/`*In` names:
- `window.proposalSelectPersonnelIn`
- `window.proposalRemovePersonnelFrom`
- `window.proposalShowPersonnelDropdownFor`
- `window.proposalFilterPersonnelDropdownFor`

None of the suffixed names are registered anywhere. Result: clicking a dropdown item, removing a pill, or typing to filter the personnel list does nothing — the pill UI is completely non-functional.

**Fix:** Register the `*For`/`*In` suffixed names in `engagements.js init()` and delete them in `destroy()`. The static search input in `render()` should also use the suffixed names for consistency.

## Warning

### WR-01 — service-detail.js permission/assignment event listener leak on repeated visits

**File:** `app/views/service-detail.js` (permissionsChanged / assignmentsChanged handlers)
**Confidence:** 92%

The `_serviceDetailPermissionHandler` guard assigns the handler reference only on the first visit. On the second visit a new handler is added to `window` but the stored reference is not updated, so `destroy()` removes the first-visit stale reference and the second-visit handler is never removed. Each navigation back to service-detail accumulates another undeletable listener.

`project-detail.js` handles this correctly: it removes the old handler first, then always updates the stored reference.

**Fix:** Mirror `project-detail.js` — remove old handler before re-adding, always update stored reference.

### WR-02 — Dead private pill helper functions in proposals.js are unreachable but risk future breakage

**File:** `app/views/proposals.js` (private renderProposalPills, proposalSelectPersonnel, etc.)
**Confidence:** 85%

The private pill helper functions (`renderProposalPills`, `proposalSelectPersonnel`, `proposalRemovePersonnel`, `proposalFilterPersonnelDropdown`, `proposalShowPersonnelDropdown`) reference window functions that are never registered. They return early harmlessly today because the `proposalPersonnelPillContainer` element no longer exists in the proposals.js DOM (the engagement form was extracted to engagements.js). However, if any code path invokes them, it will throw.

**Fix:** Delete the unreachable private pill helpers and their module-level state variables (`clientsData`, `usersData`, `selectedPersonnel`, `currentEngagementType`) from proposals.js, since the engagement form now lives in engagements.js.

## Info

### IN-01 — home.js proposals getDocs fires for finance/procurement_staff on every home visit

**File:** `app/views/home.js` (init proposals fetch)
**Confidence:** 80%

`init()` unconditionally runs `getDocs(collection(db, 'proposals'))` for all users. `filterProposalsForUser()` returns `[]` for `finance` and `procurement_staff` roles, so the sub-nav never appears — but the Firestore read already happened. Minor billing cost on every home page load for these roles.

**Fix:** Check role eligibility before issuing the query:
```javascript
const role = window.getCurrentUser?.()?.role;
if (role && !['finance', 'procurement_staff'].includes(role)) {
    // existing getDocs block
}
```
