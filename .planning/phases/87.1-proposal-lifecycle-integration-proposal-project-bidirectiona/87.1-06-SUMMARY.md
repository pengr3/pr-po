## Decisions Made

- **Three tasks instead of two.** The plan's `<tasks>` section listed two tasks (router.js; index.html + auth.js). The executor prompt explicitly expanded scope to include a third proposals.js cleanup ("DO remove now-unreachable code: the top-level render() function and its UI; the create-proposal modal that lived in proposals.js (showCreateModal, saveProposal); the clientsData = [] bridging stub from Wave 3 if and only if no remaining code in proposals.js references it; the duplicate proposal detail modal code that was kept until Wave 5"). Added Task 3 to execute that scope. The plan's `files_modified` frontmatter already lists `app/views/proposals.js`, and the plan's `<success_criteria>` line "proposals.js stubs cleaned up" supports this expansion.
- **Stage-card onclick rewritten to window.openProposalModal with && guard.** The pre-cleanup renderStageGroupCard used `onclick="event.preventDefault();window.openProposalDetail(...)"`. That handler was registered in the now-deleted proposals.js init() and pointed to a deleted function. Updated to `onclick="event.preventDefault();window.openProposalModal && window.openProposalModal(...)"` so consumers (home.js, project-detail.js, service-detail.js) that already register `window.openProposalModal = openProposalModal` from proposal-modal.js handle the click correctly. The `&&` guard avoids a ReferenceError if a consumer renders cards before registering. Same pattern applied to the renderApprovalQueue Approve/Reject button onclicks.
- **renderApprovalQueue preserved as export, proposalsData kept as empty `let` binding.** The function references module-internal `proposalsData`. No external module imports renderApprovalQueue today (home.js implements its own local queue per Pitfall 7). The executor preservation list explicitly names renderApprovalQueue, so the export stays. The empty `proposalsData = []` keeps the function callable; it renders the "no proposals awaiting approval" empty state since no listener feeds the array. Documented this with an inline comment block.
- **render/init/destroy stubs kept (not deleted).** proposals.js is statically imported by home.js, proposal-modal.js, project-detail.js, service-detail.js. ESM evaluates the module on first import; the stubs cost essentially nothing in code or load time. Any future dynamic-import caller that invokes render/init/destroy gets a safe no-op instead of an undefined-is-not-a-function crash. Mirrors the Plan 03 decision (87.1-03-SUMMARY.md "Decisions Made" section).
- **Added a docs follow-up commit to clean a stale JSDoc reference.** After the main Task 3 rewrite, a grep for `openProposalDetail` returned 2 matches inside a JSDoc comment block in proposals.js explaining the legacy name. Since the legacy name no longer exists anywhere in code, the comment was misleading. Updated the comment to describe only the post-cleanup reality (window.openProposalModal owned by consuming views, with && safety guards). Committed separately so the Task 3 commit stays focused on the code deletion and the docs commit stays focused on the comment update.
