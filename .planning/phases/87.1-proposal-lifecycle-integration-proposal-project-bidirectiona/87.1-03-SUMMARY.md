## Decisions Made

- **Unsuffixed window function names** — the form HTML and helper-emitted HTML all call unsuffixed names (`window.proposalSelectPersonnel`, etc.). The registrar uses the same names. No suffixed helper variant exists in either source or target module. CR-01 is avoided by structural choice: move the form as a unit, keep emitter and registrar co-located.
- **render() stub vs. removal** — kept the `export function render() { return ''; }` stub because other modules (proposal-modal.js, future Plans 04–06) import from proposals.js. Removing render entirely would not save load time and could surprise external callers. The stub is harmless.
- **clientsData referential-integrity stub** — the plan instructed full removal of `clientsData` listener and declaration, but two legacy modal call sites in proposals.js (`showCreateModal`, `saveProposal`) still reference it. Without a binding, any leftover `window.openCreateProposalModal()` call would crash. Added `const clientsData = []` to keep references valid. Tracked as Rule 1 deviation below.
- **init/destroy stay functional, not stubs** — Plan 03 action step 6 said "Do NOT stub init() or destroy() to empty". Followed that direction: init() retains the proposalsData + projectsData onSnapshot listeners and ~21 Phase 87/89 window function registrations; destroy() retains the symmetric cleanup. Only the engagement-specific lifecycle moved out.
- **Verified plan-checker correction was respected** — the original Plan 03 frontmatter line "initEngagementForm registers all 6 engagement window functions under their unsuffixed names" plus Pitfall 1 in 87.1-RESEARCH.md ("Why CR-01 happened" + "Why the New Design Avoids CR-01") confirm that unsuffixed registration is the correct approach for this plan. The hypothetical CR-01 warning in the executor prompt was a defensive double-check; it does not apply because no suffixed helper variant was introduced in this plan or in either source file.
