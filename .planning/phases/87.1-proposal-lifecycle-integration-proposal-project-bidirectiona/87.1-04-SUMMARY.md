## Accomplishments

- **Three sub-tabs added to the home page** — Overview (existing stats dashboard, default active), Engagements (mounts `renderEngagementForm()` from `engagement-create.js` + calls idempotent `initEngagementForm()`), Proposals (one-time `getDocs(collection(db, 'proposals'))` → role-filter → optional approval queue + stage group cards via `renderStageGroupCard` from `proposals.js`).
- **Role-gated sub-nav rendering** — `getHomeSubTabConfig()` centralizes role logic: `finance` and `procurement_staff` return `showSubNav: false` (their home page is unchanged); `super_admin` + `operations_admin` + `services_admin` get the Engagements tab; all other non-finance/non-procurement roles see Proposals; approval queue further restricted to `super_admin` + `operations_admin` (mirrors Phase 87 D-05 approver gate).
- **filterProposalsForUser** — client-side role filter on `parent_collection`: `super_admin` sees all, ops roles see `parent_collection === 'projects'` (default for legacy docs), services roles see `parent_collection === 'services'`. Unknown role returns empty array (defense in depth — Firestore rules also gate read access).
- **Local home-only approval queue** — `_renderHomeApprovalQueueHtml`, `_openHomeQueueModal`, `_homeQueueConfirmAction` implement the queue UX inside home.js. Buttons call `window.homeQueueOpen{Approve,Reject}Modal('id')`; mini-modal Confirm calls `window.homeQueueConfirmAction('id', 'approve'|'reject')`. Per RESEARCH.md Pitfall 7, this is intentionally NOT a re-use of `proposals.js`'s `renderApprovalQueue` because that function depends on `proposalsData` module state that home does not maintain.
- **Fresh-fetch confirm path** — every queue action revalidates the proposal via `getDoc(doc(db, 'proposals', id))` before calling `_applyProposalStateTransition`. Status guard rejects non-`pending_internal` proposals and refreshes the tab.
- **NOTIF-10 replayed** — on approve/reject, `createNotification` fires for the proposal submitter with `NOTIFICATION_TYPES.PROPOSAL_DECIDED` (link `#/` since /proposals route is being retired in Wave 6).
- **CSS pill tabs** — `.home-sub-nav`, `.home-sub-nav-tabs`, `.home-sub-nav-tab`, `.home-sub-nav-tab--active` appended to `styles/views.css` using `var(--gray-*)` / `var(--primary*)` for theme consistency. Pattern mirrors `.finance-sub-nav-tab`.
- **destroy() symmetry** — clears the 6 home-owned window functions (`switchHomeTab`, `openProposalModal`, `homeQueueConfirmAction`, `homeQueueCancelModal`, `homeQueueOpenApproveModal`, `homeQueueOpenRejectModal`) and calls `destroyEngagementForm()` (which unwinds the 6 engagement window fns + clients/users listeners — owned internally by engagement-create.js). Existing `statsListeners` cleanup preserved.

## CR-01 Compliance Verification

The phase critical anti-pattern callout warned that mounting `renderEngagementForm()` indirectly wires the personnel picker — if `destroyEngagementForm()` is not called on home navigate-away, the 6 engagement window functions persist as stale cross-view leaks.

**This implementation calls `destroyEngagementForm()` from `destroy()` (line 632) wrapped in a try/catch.** The destroy ordering is:

1. Delete the 6 home-owned window functions (switchHomeTab, openProposalModal, 4 homeQueue* fns).
2. Call `destroyEngagementForm()` — engagement-create.js internally clears its 6 engagement window fns under UNSUFFIXED names and unsubscribes its `_formListeners` (clients + users onSnapshot). Verified safe in 87.1-03-SUMMARY.md "CR-01 Verification Result" — name parity matrix all 6 match.
3. Reset `_homeProposalsCache` and `_homeCanApproveQueue`.
4. Unsubscribe `statsListeners` (existing behavior).

home.js never references any of the engagement form's 6 window names directly — engagement-create.js is the canonical owner. The CR-01 class of bug cannot recur because emitter (engagement form HTML), registrar (initEngagementForm), and deleter (destroyEngagementForm) all live in the same module under consistent unsuffixed names.

## Decisions Made

- **No use of proposals.js renderApprovalQueue** (Pitfall 7) — wrote a home-local queue renderer + mini-modal + confirm handler instead. Cost: ~150 extra lines in home.js, ~60 of which are HTML duplication. Benefit: home has zero dependency on proposalsData module state, and the /proposals route can be retired in Wave 6 without breaking the home queue.
- **One-time getDocs instead of onSnapshot** — Context.md D-08 and RESEARCH.md A2 both prescribe one-time fetch; tab switches inside home do NOT re-fetch, and queue actions explicitly re-call `_loadHomeProposalsTab` to refresh. Tradeoff: stale data if another user changes a proposal during this user's home session; mitigated by the refresh-after-action pattern.
- **Fire-and-forget `_loadHomeProposalsTab` in init()** — not awaited, so init() can complete the sub-nav reveal + stats listeners synchronously. Errors are caught inside the function and rendered as a red error card. Alternative (awaiting) would make the entire home view appear delayed for ops/services roles; rejected.
- **NOTIF-10 link `#/` instead of `#/proposals?id=…`** — proposals route is being retired in Wave 6 (Plan 87.1-07). Linking notifications to the soon-dead route would create dead links. `#/` lands the recipient on home where they can navigate to the Proposals sub-tab. The proposals.js path still uses `#/proposals?id=…` because that file is unchanged in this plan and lives until Wave 6.
- **`var(--gray-200)` etc. in .home-sub-nav CSS** — RESEARCH.md showed hardcoded hex (`#e5e7eb`, `#374151`, etc.), but the rest of views.css uses CSS variables wherever they exist (finance-sub-nav uses `var(--gray-200)`, `var(--gray-700)`, `var(--primary)`). Used variables for theme consistency. The hardcoded hex values in RESEARCH.md were illustrative.
- **Stage group "empty state" copy** — when role filter returns proposals but none are in active stages, render a card saying "No active proposals in your scope." instead of leaving the area blank. Context.md "Specifics" section notes the previous attempt's "hide tab until proposals exist" felt broken; explicit empty state is clearer.
