## Decisions Made

- **Preserved existing cryptoRandomUuid body** instead of substituting the alternative one-liner specified in the plan's Task 1 action step. The plan asserted the two implementations were "identical" but they are not — the existing function uses `crypto.randomUUID()` with a `'p87-' + Date.now()…` fallback prefix that is observable in audit_log entry_ids. Replacing it would change the entry_id shape for new writes without changing existing data, creating a mixed-format audit log. Tracked as deviation [Rule 1 - Bug] below.
- **Backward-compatible D-06 default:** chose `proposal.parent_collection || 'projects'` rather than throwing on a missing field. All legacy proposals (Phase 87) lack this field; defaulting preserves their behavior. New writes from Phase 87.1 Wave 3 onward will explicitly set `parent_collection` for service engagements.
- **No render/init/destroy changes:** the Proposals tab continues to render normally via the existing `/proposals` route. Module export additions are non-invasive — they only widen the public API surface. Route retirement happens in Wave 6 (Plan 07 territory).

**1. [Rule 1 - Bug] cryptoRandomUuid body mismatch between plan and existing source**

- **Found during:** Task 1 (Export cryptoRandomUuid from utils.js)
- **Issue:** Plan Task 1 Step 1 specified a body using `crypto.getRandomValues(new Uint8Array(1))` and asserted "This is identical to the private copy in proposals.js."
