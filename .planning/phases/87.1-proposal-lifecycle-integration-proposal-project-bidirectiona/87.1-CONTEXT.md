# Phase 87.1: Proposal Lifecycle Integration — Context

**Gathered:** 2026-05-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 87.1 integrates the proposal lifecycle into project/service records and restructures the navigation surface. Proposal context becomes visible from within project/service detail views. A new dedicated "Engagements" tab handles creating new engagements (extracted from the Proposals tab). The Proposals tab retains the approval queue and proposal dashboard. The home dashboard gains a role-filtered "Proposals" sub-tab visible to all roles that have active proposals.

### In scope
- **Inline proposal card** in project/service detail views — shows the active proposal's title, amount, stage, days-in-stage; "Submit for Approval" action button when status is `draft` or `for_revision`; "View Proposal →" link for all other statuses
- **Services proposal support** — add `parent_collection` field to proposal schema; fix `_applyProposalStateTransition` to write to `services` collection when applicable
- **New "Engagements" tab** — extract Create Engagement form from proposals.js into a dedicated view; visible to Super Admin, Operations Admin, Services Admin
- **Proposals tab refocus** — remove Create Engagement section; retain Approval Queue + Proposal Dashboard
- **Home dashboard "Proposals" sub-tab** — full grouped-by-stage dashboard (same pattern as Proposals tab); all roles see it filtered to their projects/services; Super Admin sees all

### Out of scope (deferred)
- Full proposal lifecycle actions (approve/reject, client decision, loss) from within project/service detail — only Submit for Approval is available there; all other actions stay in the Proposals tab
- Any change to the Proposals tab approval queue or dashboard UI (Phase 89/87 work unchanged)
- Email or push notifications (globally deferred per Phase 83)
- Structured loss-reason taxonomy or analytics (v4.1+)

</domain>

<decisions>
## Implementation Decisions

### D-01 — Inline proposal card in project/service detail
When a project/service status is in the proposal lifecycle range (`'For Proposal'`, `'Proposal for Internal Approval'`, `'Proposal Under Client Review'`, `'For Revision'`, `'Client Approved'`), a proposal card appears inline in the project/service detail view showing:
- Proposal title and ID
- Amount (formatted with `formatCurrency`)
- Current stage badge (reuse existing status badge classes)
- Days in stage (age indicator; orange/warning after 7 days — matches Phase 87 D-11 threshold)
- **"Submit for Approval" button** — visible only when proposal status is `draft` or `for_revision`; opens mini-modal (matching Phase 89 approve/reject pattern) to confirm submission; calls `_applyProposalStateTransition` from proposals.js
- **"View Proposal →" link** — navigates to `#/proposals` for all other statuses

The card must fetch the linked proposal doc on demand (one `getDoc` call when the project detail opens) — do NOT add a second `onSnapshot` listener on the `proposals` collection from within projects.js or services.js.

### D-02 — Services proposal support via parent_collection field
Add `parent_collection: 'projects' | 'services'` as an optional field to the proposal schema (Phase 87 D-02 extended). Existing proposal docs that lack this field implicitly default to `'projects'`.

`_applyProposalStateTransition` in `proposals.js` must be updated:
```javascript
const parentCollection = proposal.parent_collection || 'projects';
if (newProjectStatus && proposal.project_id) {
    batch.update(doc(db, parentCollection, proposal.project_id), { ... });
}
```

The proposal creation form in `proposals.js` must set `parent_collection` when creating a proposal linked to a service engagement. The Proposal Dashboard and Approval Queue are unaffected — they display proposals regardless of parent collection.

### D-03 — New Engagements tab
Create a new `app/views/engagements.js` view (or route) containing the Create Engagement form currently in `proposals.js`. Tab label: "Engagements". Route: `#/engagements`.

**Visibility:** Super Admin + Operations Admin + Services Admin (gated at router level; mirrors pattern in `router.js` for existing role-specific routes).

The Create Engagement form itself is unchanged — same fields, same `createEngagement()` helper, same validation logic. Extract the relevant render/init/destroy sections from `proposals.js` into the new view.

**Nav placement:** Between "Finance" and "Proposals" in the top nav (between the two closest related tabs).

### D-04 — Proposals tab refocus
Remove the Create Engagement section (form + its window functions) from `proposals.js`. The `#proposal-queue-mount` (Approval Queue) and Proposal Dashboard sections remain. Section ordering becomes:
1. Proposal Approval Queue (Phase 89) — unchanged
2. Proposal Dashboard grouped by stage (Phase 87) — unchanged

The tab label ("Proposals") and route (`#/proposals`) are unchanged. Router gate (Super Admin only) is unchanged.

### D-05 — Home dashboard Proposals sub-tab
Add a "Proposals" sub-tab to the home page alongside existing content. The sub-tab shows the same grouped-by-stage view as the Proposals tab dashboard (using the `renderStageGroupCard` pattern from `proposals.js`).

**Role filtering:**
- Super Admin: sees all proposals
- Operations Admin / Operations User: sees proposals where `proposal.parent_collection = 'projects'` AND the project is in their assigned projects
- Services Admin / Services User: sees proposals where `proposal.parent_collection = 'services'` AND the service is in their assigned services
- Finance / Procurement Staff: no proposal sub-tab shown (not in their workflow)

**Implementation approach:** `home.js` adds a one-time `getDocs` query on the `proposals` collection (Firestore Security Rules allow all active users to read proposals per Phase 87 D-10) and filters client-side by the current user's role + assigned projects/services. Do NOT add an `onSnapshot` listener in home.js — the dashboard is a snapshot view, not real-time.

**Sub-tab visibility trigger:** Sub-tab only appears if there is at least one active proposal (status ≠ `'client_approved'` and ≠ `'loss'`) relevant to the current user. If no active proposals: home renders as before with no sub-tab.

### Claude's Discretion
- Exact visual styling of the inline proposal card in project/service detail (match existing modal/card patterns in projects.js/services.js)
- Sub-tab toggle UI on home page (pill tabs, underline tabs — match any existing tab pattern in the app)
- Whether the "Submit for Approval" mini-modal in project/service detail is implemented as a shared function imported from proposals.js or duplicated inline
- Exact nav order within the top nav for the new Engagements link
- Toast copy for proposal card actions

</decisions>

<specifics>
## Specific Ideas

- The "Submit for Approval" button in the inline proposal card should be a clear primary action (blue, labeled "Submit for Approval"); not a secondary action
- Home dashboard sub-tab only appears when the user has at least one active proposal — don't show the tab if it would be empty
- The Engagements tab is the new home for creating engagements across all three types (project, one-time service, recurring service) — no change to the form itself, just a new container view
- The proposal card in project/service detail should look like a lightweight info panel, not a full modal — subtle border, proposal stage badge prominent

</specifics>

<deferred>
## Deferred Ideas

- **Full proposal lifecycle actions from project/service detail** (approve/reject, mark sent to client, client approved, loss) — only Submit for Approval is in scope; all other actions remain in the Proposals tab
- **Real-time proposals listener on home** — snapshot fetch on load is sufficient; real-time updates would require home.js to manage another listener across tab switches
- **Proposal creation from project/service detail** — "Start Proposal" button inline when no proposal exists yet; this would close the loop but adds form complexity; deferred to v4.1+
- **Loss reason analytics / structured loss taxonomy** — v4.1+ per Phase 87 D-06

</deferred>
