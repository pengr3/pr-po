# Phase 87.1: Proposal Lifecycle Integration — Context (Redesign)

**Gathered:** 2026-05-20
**Status:** Ready for planning
**Note:** Full revert of prior 87.1 implementation. This context supersedes the 2026-05-19 version.

<domain>
## Phase Boundary

Phase 87.1 integrates proposals into project/service context and restructures the Proposals surface. The standalone Proposals top-nav tab is **retired**. Instead, the home page gains three sub-tabs (Overview | Engagements | Proposals) that serve as the primary surface for proposal-related work. Project/service detail views get an enriched inline proposal card with a "View Proposal" button that opens the full proposal modal without leaving the record.

### In scope

- **Home page sub-tab navigation** — three sub-tabs on home: Overview (existing stats), Engagements (Create Engagement form), Proposals (Approval Queue + Proposal Dashboard)
- **Retire top-level Proposals nav tab** — remove from nav, remove router registration; `proposals.js` becomes a pure module (no route)
- **Extract proposal modal** — move proposal detail modal code from `proposals.js` into `app/proposal-modal.js` shared utility
- **Extend `app/engagement-create.js`** — add form render/init exports so home.js Engagements sub-tab can call it
- **Services proposal support** — add `parent_collection: 'projects' | 'services'` to proposal schema (carry-forward from prior 87.1 attempt); fix `_applyProposalStateTransition` to write to the correct collection
- **Enriched inline proposal card** in project/service detail — add attachment link + latest comms log entry display + "View Proposal" button (opens full proposal modal from `proposal-modal.js`)

### Out of scope (deferred)

- Full proposal lifecycle actions (approve/reject, send to client, client decision) from within project/service detail — those remain accessible from the home Proposals sub-tab
- Any change to the proposal data schema beyond `parent_collection` (D-02 already locked in Phase 87)
- Email or push notifications (globally deferred per Phase 83)
- Structured loss-reason taxonomy or analytics (v4.1+)

</domain>

<decisions>
## Implementation Decisions

### D-01 — Home page sub-tab navigation

Home page (`home.js`) gains a sub-nav with three tabs:

| Sub-tab | Content | Visible to |
|---------|---------|------------|
| Overview | Existing stats dashboard (unchanged) | All roles |
| Engagements | Create Engagement form (moved from proposals.js) | Super Admin, Operations Admin, Services Admin |
| Proposals | Approval Queue (top, approvers only) + Proposal Dashboard | Super Admin, Ops Admin, Ops User, Services Admin, Services User |

**Finance** and **procurement_staff** do NOT see the Proposals sub-tab. They only see Overview.

**Approval Queue section** within the Proposals sub-tab is visible only to Super Admin + Operations Admin (mirrors Phase 87 D-05 approver gate). It appears at the top of the sub-tab. The Proposal Dashboard (grouped-by-stage) appears below it for all eligible roles.

**Sub-tab rendering rules:**
- Sub-nav is always rendered for roles that have at least one sub-tab beyond Overview
- Finance/procurement_staff: no sub-nav rendered (Overview is the only content)
- Other roles: sub-nav appears with their eligible tabs

### D-02 — Top-level Proposals nav tab retired

- Remove `Proposals` link from `index.html` (desktop + mobile nav)
- Remove `auth.js` Proposals visibility block
- Remove `#/proposals` route from `router.js`
- `proposals.js` becomes a **pure module** (no render/init/destroy as a routed view) — see D-03

**Why:** The Create Engagement form and Proposal Dashboard move to home sub-tabs (D-01). The Approval Queue also moves to the home Proposals sub-tab. There is no longer a standalone Proposals page.

### D-03 — proposals.js restructured to pure module

`proposals.js` is demoted from a routed view to a shared module. It retains all working code but removes the route. Required changes:
- Remove `render()` as a route entry point (the shell HTML is no longer needed as a full page)
- Export functions needed by home.js and project/service-detail.js (see canonical_refs for which functions)
- The approval queue render functions stay in proposals.js and are exported for home.js consumption
- The proposal dashboard functions (`renderStageGroupCard`, `renderAgeBadge`, `getProposalStatusBadge`, `STAGE_ORDER`, `PROPOSAL_RANGE_STATUSES`) are exported — same as prior 87.1 attempt

**Claude's discretion:** whether proposals.js keeps a minimal no-op `render/init/destroy` for safety, or strips them entirely. Researcher should check if any other file imports these lifecycle functions.

### D-04 — Proposal modal extracted to app/proposal-modal.js

Create `app/proposal-modal.js` as a shared utility (mirrors `app/engagement-create.js` pattern). It exports:
- `openProposalModal(proposalId, context)` — opens the full proposal detail modal (audit log, comms log, attachment, all lifecycle actions)
- `closeProposalModal()` — closes and removes it

This shared module is imported by:
- `home.js` — the Proposals sub-tab opens proposal modals when user clicks a proposal
- `app/views/project-detail.js` — "View Proposal" button in the inline card
- `app/views/service-detail.js` — same

The modal code is extracted verbatim from `proposals.js`. All lifecycle action buttons (submit, approve/reject, send to client, client decision) remain in the modal — role-gated as per Phase 87/89 existing logic. No new action restrictions are added in Phase 87.1; the modal just becomes callable from multiple contexts.

### D-05 — Enriched inline proposal card in project/service detail

The inline card in project/service detail shows:

| Field | Notes |
|-------|-------|
| Proposal title + ID | Always shown |
| Amount | `formatCurrency` |
| Stage badge | `getProposalStatusBadge` from proposals.js |
| Days-in-stage | `renderAgeBadge` from proposals.js |
| **Attachment** | If attachment_kind is set: show link (URL) or download filename (file). "No attachment" if absent. |
| **Latest comms log entry** | Most recent entry by `logged_at`: show date + description. "No comms yet" if comms_log is empty. |
| Submit for Approval button | Visible only when proposal.status is `draft` or `for_revision` |
| View Proposal button | Always visible; opens full modal via `openProposalModal()` from proposal-modal.js |

Card fetches via one-time `getDocs` query (not `onSnapshot`) on proposals where `project_id == parentDocId`. Same non-blocking async pattern as before.

### D-06 — Services proposal support (parent_collection)

Add `parent_collection: 'projects' | 'services'` as optional field to proposal schema (defaults to `'projects'` for existing docs). Carry-forward from Phase 87 D-02 extension.

`_applyProposalStateTransition` in `proposals.js` updated to:
```javascript
const parentCollection = proposal.parent_collection || 'projects';
batch.update(doc(db, parentCollection, proposal.project_id), { ... });
```

The Create Engagement form (in Engagements sub-tab) sets `parent_collection` when creating a proposal linked to a service engagement.

### D-07 — Create Engagement form in home Engagements sub-tab

The Create Engagement form (currently in `proposals.js` render HTML + init logic) moves to home's Engagements sub-tab. Implementation approach:
- `app/engagement-create.js` is **extended** to export both the form render HTML and form init/destroy logic (personnel picker setup, client dropdown, engagement type radio handlers)
- `home.js` init/destroy calls the Engagements sub-tab through this extended module
- The form itself is **unchanged** — same fields, same `createEngagement()` write helper, same validation logic, same personnel pill pattern

**Role gate:** Engagements sub-tab visible to Super Admin + Operations Admin + Services Admin. No render for other roles.

### D-08 — Role-filtered Proposal Dashboard on home

Proposals sub-tab role filtering in home.js:
- **Super Admin:** all proposals
- **Operations Admin + Operations User:** proposals where `parent_collection = 'projects'` (researcher must verify: does Ops Admin see all project proposals or only those for assigned projects — check existing `getAssignedProjectCodes` / role-assignment model)
- **Services Admin + Services User:** proposals where `parent_collection = 'services'` (filtered to assigned services if applicable)
- **Finance / procurement_staff:** not shown the sub-tab at all

Uses one-time `getDocs` (not `onSnapshot`) — same decision as prior 87.1 context D-05.

### Claude's Discretion

- Exact sub-nav visual style on home (pill tabs, underline tabs — match existing pattern from finance view `.finance-sub-nav-tab`)
- Whether to show "No proposals in your scope" empty state or just hide the Proposals sub-tab entirely when a user has no active proposals
- Whether proposals.js keeps minimal `render/init/destroy` stubs or strips them entirely
- Toast copy for proposal card actions (Submit for Approval confirmation)
- Card layout for inline proposal card (compact single-row vs stacked — match project/service detail card style)
- Whether `openProposalModal()` accepts a pre-fetched proposal object or a Firestore doc ID (and fetches fresh)

</decisions>

<specifics>
## Specific Ideas

- **home.js sub-nav should be persistent tabs** (always visible when there are eligible sub-tabs for the role) — NOT conditionally appearing based on whether proposals exist. The previous implementation hid the sub-tab until active proposals were detected, which felt broken.
- **Proposals sub-tab on home should default to visible** when the user first lands on home — don't require a click to discover it. The Overview tab is the default active tab, but the sub-nav is always rendered so users can see their options.
- **Inline proposal card** should feel like a lightweight info panel — not a modal trigger, not a full section. The "View Proposal" button is the escape hatch to the full modal.
- **Attachment display:** if `attachment_kind = 'link'`, show a clickable URL label. If `attachment_kind = 'file'`, show the `attachment_filename` as a download link via Firebase Storage. If `attachment_kind = null`, show a muted "No attachment" label.
- **Latest comms entry:** show `comms_log[comms_log.length - 1]` (last array item = most recent if appended in order). Display as: `[date] • [description truncated to ~80 chars]`.

</specifics>

- **Real-time proposals listener on home** — one-time getDocs is sufficient; real-time would add listener management complexity to home.js tab switching
