## Accomplishments

- **New shared utility `app/proposal-modal.js`:** Single file (1,367 lines) containing all proposal-modal code that previously lived inside `app/views/proposals.js` as private helpers. The file exports exactly two functions: `openProposalModal(proposalId, context)` and `closeProposalModal()`. Everything else — the 20+ window-function handlers, the 7 modal HTML builders, the 3 sub-modals (approve/reject, loss, client approved), the attachment widget, the comms log — is private to the module.
- **Fresh-fetch pattern:** `openProposalModal` does `const snap = await getDoc(doc(db, 'proposals', proposalId));`. There is no `proposalsData` array dependency. Every lifecycle handler (submitProposalForApproval, submitProposalApproval, submitMarkSentToClient, submitClientApproved, submitLoss, saveProposalAttachment, removeProposalAttachment, saveCommsEntry) calls a private `_fetchProposalDoc(id)` helper to revalidate state before writing. This makes the utility usable from home.js, project-detail.js, and service-detail.js — none of which have access to proposals.js's onSnapshot-backed module state.
- **Complete window-function lifecycle:** `openProposalModal` registers exactly 20 window functions on entry; `closeProposalModal` deletes exactly 20 on exit. The two sides are kept in lockstep — there is no possibility of a lingering window function after a modal close. Names match the originals in proposals.js so the inline `onclick="window.foo(...)"` strings in the extracted HTML continue to resolve.
- **Lazy dropdown data cache:** The Create/Edit Proposal sub-modal needs lists of active projects and clients for its dropdowns. Rather than holding a Firestore listener (this is a stateless utility), the module does a one-shot `getDocs` on first `openCreateProposalModal()` call and caches the result in `_modalProjectsData` / `_modalClientsData` for the rest of the page session. Filters mirror proposals.js: projects with `project_status === 'Draft'` or `active === false` are excluded; clients require `active === true`.
- **Refresh on transition uses fresh fetch:** `_refreshDetailModalAfterTransition(id)` refetches the proposal from Firestore (no optimistic merge from a missing onSnapshot listener) and rebuilds the modal HTML in place. Audit trail, comms log, attachment state, and action buttons all reflect post-write reality.
- **Imports correctly wired:** From `./firebase.js` — db, collection, doc, getDoc, getDocs, updateDoc, addDoc, query, where, writeBatch, serverTimestamp. From `./utils.js` — showLoading, showToast, formatCurrency, formatTimestamp, escapeHTML, generateProposalId, cryptoRandomUuid. From `./notifications.js` — createNotification, createNotificationForRoles, NOTIFICATION_TYPES. From `./views/proposals.js` — getProposalStatusBadge, renderAgeBadge, getAgeInStageDays, isOverdueInStage, _applyProposalStateTransition (all five exported in Wave 1 plan 87.1-01).
- **proposals.js untouched:** The existing private modal code in proposals.js remains intact. The `/proposals` route is still active until Wave 6 (Plan 87.1-07), so removing the duplicate code now would break the still-routed view. Cleanup of the duplicate is deferred to that wave per the phase's incremental retire-route strategy.

## Decisions Made

- **Fresh-fetch over caching:** Every lifecycle handler revalidates the proposal via `await _fetchProposalDoc(id)` before acting. This costs one extra Firestore read per action button click but eliminates an entire class of stale-state bugs and makes the utility behaviour identical regardless of caller context (home dashboard, project-detail card, service-detail card). The original proposals.js code only got away with `proposalsData.find` because the onSnapshot listener kept that array warm; without that listener, the only correct source is a fresh `getDoc`.
- **Refetch in _refreshDetailModalAfterTransition rather than optimistic merge:** The original `_refreshDetailModalAfterTransition` in proposals.js merges the new audit entry into the in-memory proposalsData entry (because onSnapshot will catch up "soon"). In this utility there's no listener, so the same approach would leave the rebuilt modal showing only the optimistically merged entry — the rest of the doc (e.g., concurrent updates from other users) would not flow through. Refetching is one extra read per transition; given action buttons are not high-frequency, that's a correct trade.
- **Lazy dropdown data load:** The Create/Edit modal is opened far less often than the read-only View modal. Holding a listener for projects + clients all session long would be wasteful; loading on first open and caching for the rest of the session is sufficient. If projects/clients change mid-session, a page reload picks them up.
- **No edits to proposals.js:** Tempting to gut the duplicate code now, but that breaks the still-routed `/proposals` view that uses these functions on its own modal. The plan explicitly scopes Plan 02 to file creation only; proposals.js cleanup is deferred to Wave 6 (Plan 07).
- **CR-01 anti-pattern not applicable:** The modal extracted here does NOT include the Engagements form's personnel picker (which is the source of CR-01 in the prior 87.1 attempt). No `renderProposalPillsFor` calls, no `proposalSelectPersonnelIn` / `proposalRemovePersonnelFrom` window functions are registered in this file. Personnel picker logic belongs to engagement-create.js (Plan 03's responsibility).
