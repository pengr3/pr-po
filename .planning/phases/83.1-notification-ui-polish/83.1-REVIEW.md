# Phase 83.1: Code Review Report

**Reviewed:** 2026-05-26
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Three files reviewed covering the Phase 83.1 SVG icon replacement in the dropdown (`app/notifications.js`), the CSS rule additions for `.notif-type-badge`, dropdown animation, and unread left-border (`styles/components.css`), and the `renderRows()` history view (`app/views/notifications.js`).

The SVG injection into `innerHTML` is the most serious issue: the `svg()` helper injects raw SVG markup via `container.innerHTML`, and the fallback code path passes the literal `n.type` string (user-controlled data from Firestore) as the `icon` when no TYPE_META entry matches — bypassing `escapeHTML` entirely. A secondary critical issue exists in the `absTime` tooltip on line 205 of `app/notifications.js`: `formatTimestamp()` output is placed unescaped into an HTML attribute, breaking attribute safety for any timestamp representation that contains a quote character, and — depending on the timestamp's format — this could be a scripting vector.

Several CSS structural issues will produce visible alignment regressions.

## Critical Issues

### CR-01: Unescaped `n.type` injected into `innerHTML` via fallback icon

**File:** `app/notifications.js:187`

**Issue:** When `TYPE_META[n.type]` has no entry, the fallback object uses `n.type` as the `icon` field:

```javascript
const meta = TYPE_META[n.type] || { label: n.type, icon: '•', color: '#64748b' };
```

The `icon` value is then placed directly into the HTML via `${meta.icon}` at line 201 without `escapeHTML`. For known types, `meta.icon` is the output of the `svg()` helper and contains only hard-coded SVG markup — no user data touches it. However, if a notification document in Firestore has an unknown `type` (e.g., a future enum value, a typo, or a malicious document written by a compromised trigger), `meta.icon` becomes `n.type` (the raw Firestore string), and that string is injected unescaped into `container.innerHTML` at line 215.

Firestore security rules control who can *write* notifications, but this is still a stored XSS path: any process (Phase 84+ trigger sites) that calls `createNotification` with an unrecognised type — or any future type not yet in TYPE_META — produces a row where `n.type` appears verbatim inside innerHTML. A string like `MRF_NEW_TYPE<img src=x onerror=alert(1)>` would execute.

**Fix:** Escape the fallback icon:
```javascript
const meta = TYPE_META[n.type] || {
    label: escapeHTML(n.type || 'Notification'),
    icon: escapeHTML(n.type || '•'),   // ← was `n.type` raw
    color: '#64748b'
};
```
Since the icon for known types is the hard-coded `svg()` string, safe-known-type icons are unaffected. Unknown-type icons will render as the escaped type string rather than SVG, which is acceptable for a fallback.

### CR-02: `absTime` from `formatTimestamp()` injected unescaped into an HTML attribute

**File:** `app/notifications.js:192,205`

**Issue:**

```javascript
const absTime = formatTimestamp(n.created_at);   // line 192
// ...
<div class="notif-row-time" title="${absTime}">   // line 205
```

`formatTimestamp()` returns a locale-formatted date string using `toLocaleDateString`. The format can include commas and spaces, which are benign, but it is never run through `escapeHTML` before being placed in a `title` attribute. If an attacker can craft a `created_at` value that survives the timestamp parse path and is converted to a string containing a double-quote (`"`), the attribute boundary breaks and arbitrary HTML attributes (including `onfocus`, `onmouseover` etc.) can be injected. This is a low-probability but non-zero path because `formatTimestamp` has a fallback branch `date = new Date(timestamp)` for plain objects, which could accept a string-like object from Firestore if the field is malformed.

Additionally, `absTime` is placed in the `.notif-row-time` element's `title` attribute without escape in the same template. The same issue does not apply to `timeStr` (output of `formatRelativeTime`) because its possible outputs are a closed set of string literals, but `absTime` can carry locale-string content.

**Fix:**
```javascript
const absTime = escapeHTML(formatTimestamp(n.created_at));
```

## Warnings

### WR-01: `.notif-row-body`, `.notif-row-content`, and `.notif-row-read-btn` have no CSS rules

**File:** `styles/components.css` (entire file), `app/notifications.js:197,203,208`

**Issue:** The dropdown renderer (`app/notifications.js`) produces elements with classes `.notif-row-body`, `.notif-row-content`, and `.notif-row-read-btn`. None of these classes are defined anywhere in `styles/components.css` or any other stylesheet in the project.

- `.notif-row-body` is the click target for the entire row body (badge + text). Without `display:flex` or `flex:1`, it will not expand to fill the row — the badge and message text will stack vertically in normal block flow rather than sitting side-by-side.
- `.notif-row-content` wraps the message + time text. Without `flex:1;min-width:0` it will not shrink correctly inside `.notif-row-body`, causing overflow with long messages.
- `.notif-row-read-btn` is the checkmark button for unread rows. `.notif-row-mark-read` *is* styled (lines 1988–2001 of components.css), but that class is on the history-page button, not the dropdown button. The dropdown button uses `.notif-row-read-btn`, which gets no styles at all — it will render with browser-default button appearance (no visual consistency, wrong sizing, potentially wrong flex-shrink behavior breaking row alignment).

This is an alignment regression introduced in Phase 83.1 when the inline-styled markup was replaced with class-based markup but the corresponding CSS classes were not added.

**Fix:** Add to `styles/components.css` after `.notif-type-badge svg`:
```css
.notif-row-body {
    display: flex;
    align-items: flex-start;
    gap: 0.5rem;
    flex: 1;
    min-width: 0;
    cursor: pointer;
}

.notif-row-content {
    flex: 1;
    min-width: 0;
}

.notif-row-read-btn {
    background: transparent;
    border: 1px solid transparent;
    color: #64748b;
    cursor: pointer;
    padding: 0.25rem;
    border-radius: 4px;
    flex-shrink: 0;
    font-size: 0.875rem;
    line-height: 1;
}

.notif-row-read-btn:hover {
    background: #fff;
    border-color: #e2e8f0;
    color: #1a73e8;
}
```

### WR-02: `padding-left: calc(1rem - 3px)` on `.notif-row--unread` only compensates left, not right — total horizontal padding changes

**File:** `styles/components.css:1937-1941`

**Issue:**

```css
.notif-row {
    padding: 0.75rem 1rem;        /* 16px left, 16px right */
    border-left: 3px solid transparent;
}

.notif-row--unread {
    border-left: 3px solid #1a73e8;
    padding-left: calc(1rem - 3px);   /* 13px left */
}
```

The intent is to keep the content box the same width when the border changes from transparent to 3 px solid. The calculation is correct: `border-left: 3px` is added, so subtracting 3 px from `padding-left` preserves total left gutter width. However:

1. The base `.notif-row` already declares `border-left: 3px solid transparent`. This means the border slot is already allocated in the layout; adding `border-left: 3px solid #1a73e8` on `.notif-row--unread` does **not** shift content rightward — the 3 px space was already reserved. The `padding-left: calc(1rem - 3px)` therefore under-compensates by 3 px, causing the content inside unread rows to be 3 px closer to the left edge than read rows.

2. Since `border-box` sizing is in effect for most modern browsers, and because `border-left: 3px solid transparent` already occupied 3 px in the box model, unread rows will have `13px` left-padding instead of `16px`, creating a visible left-shift of the badge icon and text on unread rows.

**Fix:** Because the base rule already reserves the border slot with `transparent`, the unread override only needs to change the border color — no padding compensation is needed:
```css
.notif-row--unread {
    background: #f8fbff;
    border-left-color: #1a73e8;   /* change color only — slot already allocated */
}
```

### WR-03: `COLLECTIBLE_CREATED` type missing from `app/views/notifications.js` TYPE_META

**File:** `app/views/notifications.js:43-60`

**Issue:** `app/notifications.js` defines `COLLECTIBLE_CREATED` in both `NOTIFICATION_TYPES` (line 49) and `TYPE_META` (line 90). The history view (`app/views/notifications.js`) has its own local `TYPE_META` table that is missing `COLLECTIBLE_CREATED`. Any notification of that type rendered in the history page will fall through to the `|| { label: notification.type || 'Notification', icon: '•', ... }` fallback at line 274, displaying `COLLECTIBLE_CREATED` as the label and `•` as the icon instead of the correct "New Collectible" / money icon.

This divergence will recur: the two TYPE_META tables are duplicated and will drift with each new type addition.

**Fix (immediate):** Add `COLLECTIBLE_CREATED` to the history view's `TYPE_META`:
```javascript
COLLECTIBLE_CREATED: { label: 'New Collectible', icon: '$', color: '#059669' },
```

**Fix (structural):** Re-export `TYPE_META` from `app/notifications.js` and import it in `app/views/notifications.js` instead of maintaining a parallel table. This is the root cause of the divergence.

### WR-04: Dropdown animation `visibility` delay uses `0ms` on open — no-transition flash on fast re-open

**File:** `styles/components.css:1872,1880`

**Issue:**

```css
.notif-dropdown-menu {
    /* ... */
    visibility: hidden;
    transition: transform 200ms ease-out, opacity 200ms ease-out, visibility 0ms 200ms;
}

.notif-dropdown-menu.open {
    visibility: visible;
    transition: transform 200ms ease-out, opacity 200ms ease-out, visibility 0ms 0ms;
}
```

The `visibility 0ms 200ms` on the closed state correctly delays the `visibility: hidden` until after the fade-out completes (200 ms), so the element is not clipped mid-animation. However, the `pointer-events: none` on the closed state (line 1871) and `pointer-events: auto` on `.open` (line 1879) are also in these rules, but `pointer-events` is not listed in `transition`. That is correct: `pointer-events` cannot be transitioned.

The actual defect: when a user clicks the bell rapidly (open → close → open within 200 ms), the closing animation is interrupted by the re-open. On re-open, `visibility` snaps to `visible` immediately (the `0ms 0ms` shorthand on `.open`), but `opacity` transitions from wherever it was mid-fade back to 1 over 200 ms. This is the intended behavior for CSS transitions and is generally fine. However, during the close animation, `pointer-events: none` is set on the *open* → *closed* transition instantaneously (it has no transition), meaning the dropdown becomes unclickable immediately when `.open` is removed, even while still visible. This creates a 200 ms window where the dropdown is visible but non-interactive, which is a usability defect if the user tries to click a row during the fade-out (e.g., after accidentally clicking outside and quickly clicking back in).

**Fix:** Apply `pointer-events: none` with the visibility trick — keep `pointer-events: auto` during the entire close animation by not toggling it until visibility is hidden. One approach: tie pointer-events to visibility by using a single compound transition, or simply accept the limitation and document it. Alternatively:
```css
.notif-dropdown-menu {
    /* Remove pointer-events from the closed rule — rely on visibility:hidden
       to block interaction during and after the close animation. */
    transition: transform 200ms ease-out, opacity 200ms ease-out, visibility 0ms 200ms;
}
/* Remove pointer-events: none / auto entirely — visibility:hidden already blocks events */
```

## Info

### IN-01: SVG `stroke-width` specified in both `svg()` helper attribute and CSS rule — doubles specificity risk

**File:** `app/notifications.js:70`, `styles/components.css:1962`

**Issue:** The `svg()` helper hard-codes `stroke-width="1.75"` as an SVG presentation attribute on every `<svg>` element. The `.notif-type-badge svg` CSS rule at line 1962 also sets `stroke-width: 1.75`. SVG presentation attributes have lower specificity than CSS declarations, so the CSS value wins — producing the same result as the attribute alone. This duplication is harmless now, but if the CSS value is ever changed without changing the helper (or vice versa), the two values will diverge silently and the CSS value will win unexpectedly.

**Fix:** Remove `stroke-width="1.75"` from the `svg()` helper string and rely solely on the CSS rule, which is already the higher-specificity declaration and the right place to control visual sizing:
```javascript
const svg = (path) => `<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-linecap="round" stroke-linejoin="round">${path}</svg>`;
```

### IN-02: `app/views/notifications.js` `TYPE_META.icon` renders emoji and special chars directly into `innerHTML` without escaping

**File:** `app/views/notifications.js:46-59,296`

**Issue:** The history view's `TYPE_META` uses raw Unicode characters and emoji as icon values (`'✓'`, `'✕'`, `'⏳'`, `'📦'`). These are placed directly into innerHTML via `${meta.icon}` at line 296, without `escapeHTML`. Unlike the dropdown's fallback path (CR-01), this is not a user-data vector — the icons are hard-coded constants, so there is no XSS risk here. However, the inconsistency with the escaping pattern applied elsewhere in the same function (line 275: `escapeHTML(notification.message)`) could lead a future maintainer to assume `meta.icon` is always safe to omit from escaping, and then mistakenly add a dynamic value to TYPE_META without escaping it. This is also the path through which the missing `COLLECTIBLE_CREATED` entry (WR-03) exposes `n.type` unescaped.

**Fix:** Add a comment at the TYPE_META declaration noting that all icon values must be static string literals (no user data), and apply the defensive escaping pattern consistent with CR-01's fix to the fallback at line 274:
```javascript
const meta = TYPE_META[notification.type] || {
    label: escapeHTML(notification.type || 'Notification'),
    icon: escapeHTML(notification.type || '•'),
    color: '#64748b'
};
```
