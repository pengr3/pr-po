### G3.Q2 — Trigger semantics for the auto-advance

| Option | Description | Selected |
|--------|-------------|----------|
| Only after Revision Requested | Sent advances only if proposal is in for_revision (prior Revision Requested) | |

**User's free-text response:** "let's just not automatically update the status, my bad gang. Let's just use the action buttons to move the project status"
**Notes:** Major mid-area pivot. User reversed direction on the whole auto-status idea. Triggered G3.Q3 follow-up to find a corrective path.

### G3.Q3 — Corrective path after the pivot

| Option | Description | Selected |
|--------|-------------|----------|
| Add 'Request Revision' action button | New action at pending_client; Mark Sent extended to for_revision; comms stays a pure log | ✓ |
| Defer the revision cycle entirely | Drop success criterion #3; no in-modal cycle management | |
| Re-extend Mark Sent only | No new button; partial coverage of the cycle | |

**User's choice:** Add 'Request Revision' action button.

**Note:** This pivot supersedes the roadmap's original wording of success criterion #3. CONTEXT D-10 through D-15 captures the corrective design: comms are pure logs; Request Revision + extended Mark Sent buttons handle the cycle through `_applyProposalStateTransition`. Planner should NOT re-introduce any auto-status logic from comms entries.
