# Phase 87.2: Proposal Workflow Polish — Context

**Gathered:** 2026-05-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 87.2 polishes the proposal lifecycle foundation built in Phase 87 and integrated in Phase 87.1. Five integration gaps surfaced in a super_admin walkthrough of an in-flight proposal (PROP-2026-001 on CLMC-SPI-2026004); this phase closes them. No new lifecycle stages, no schema migrations, no new collections — only targeted edits to:

- `app/proposal-modal.js` — Target Client default, expanded action-button visibility, new Request Revision action, audit-trail comms nesting, comms field-name bug fix
- `app/views/proposals.js` — `_applyProposalStateTransition` already supports the new transitions via the existing audit_log + parent_collection plumbing; one new audit action label
- `firestore.rules` — `proposals` block relaxed to allow operations_user/services_user comms writes scoped by field-mask AND parent-assignment
- `app/views/project-detail.js` / `app/views/service-detail.js` — no edits expected (inline card already opens the shared modal)

### In scope

- **G1 (Target Client default + bug fix)** — Auto-default Target Client from the picked project's `client_id` at proposal CREATE time; fix the `client.client_name` → `client.company_name` field-name bug in `saveProposal` so newly created proposals actually persist `target_client_name`
- **G2 (Operations role permissions — UI)** — Extend the action-button visibility gate so operations_user (on assigned projects) and services_user (on assigned services) see the full lifecycle except Approve/Reject
- **G3 (Revision cycle — explicit transitions, not auto-status)** — Add a new "Request Revision" action button (`pending_client` → `for_revision`). Extend "Mark Sent to Client" to also be visible at `for_revision` (so it covers the `for_revision` → `pending_client` resend leg). Comms log entries are pure conversation logs — they no longer attempt to auto-advance status (course-corrected from the roadmap's original phrasing)
- **G4 (Firestore rules)** — Relax the `proposals` update rule so non-admin assigned users can write a narrow field-mask (`comms_log`, `audit_log`, `status`, `current_status_since`, `updated_at`). Existing super_admin + operations_admin blanket update rule stays as-is for backward compat
- **G5 (Audit trail comms nesting)** — `renderAuditTrail` merges audit_log + comms_log; each comms entry nests as an indented child under the most recent preceding `SENT_TO_CLIENT` audit event. Comms entries with no preceding `SENT_TO_CLIENT` nest under the `CREATED` audit entry. Left-column Client Communications section stays as the input surface

### Out of scope (deferred)

- Any change to proposal schema beyond the implicit `audit_log` value-set extension (new `REVISION_REQUESTED` action)
- Adding a new unified `project_status` value (e.g., "Revision Sent" sub-status) — explicitly rejected in G3.Q1; the existing 10-option enum from Phase 81 is preserved
- Legacy backfill for existing prod proposals where `target_client_name` is null due to the company_name bug — user explicitly opted out ("just in prod as of now so no need for legacy"); existing proposals can be corrected manually via the Edit Proposal modal once the save logic is fixed
- Email / push notifications for revision-cycle events (globally deferred per Phase 83)
- Auto-status-advance from comms entries — the original roadmap framing for G3 — explicitly rejected mid-discussion; the corrective is the new "Request Revision" action button + extended "Mark Sent to Client" gate
- Proposal creation from project/service detail (a "Start Proposal" button on the inline card) — still deferred per Phase 87.1 deferred list

</domain>

<decisions>
## Implementation Decisions

### G1 — Target Client default + field-name bug

- **D-01:** At proposal CREATE time, when the user picks a project in the Create Proposal modal (`showCreateModal` / `saveProposal` flow in `app/proposal-modal.js`), the Target Client `<select>` auto-selects the parent project's `client_id` if the project has one. User can still manually override before saving. No force-lock. (G1.Q1 → "Auto-select + fix bug".)
- **D-02:** Fix the field-name bug at `app/proposal-modal.js:632` (and any other call site): change `client?.client_name` → `client?.company_name`. The `clients` collection schema uses `company_name`, not `client_name` — the existing code stored `null` into `target_client_name` for every proposal because the lookup field didn't exist. This is the root cause of "(none)" rendering on PROP-2026-001 and any other recently-created proposal.
- **D-03:** No legacy backfill — existing prod proposals with `target_client_name = null` stay as-is. Users can manually correct via the Edit Proposal modal (which will now save the correct value once D-02 lands). No display-time fallback to parent's client either; the modal displays whatever's stored, and the user repairs it on next edit.
- **D-04:** Implementation hook for the auto-select: a `change` listener on `#proposalCreateProject` reads the selected project's `client_id` from the project dropdown's pre-loaded data (`_modalProjectsData`) and sets `#proposalCreateClient.value` to that ID. On `isEdit = true` and the proposal already has a target_client_id, do NOT overwrite — only auto-select on new creation OR when the user actively changes the project dropdown.

### G2 — Operations role permissions (UI)

- **D-05:** `renderProposalActionButtons` in `app/proposal-modal.js` (currently gates everything behind `canApprove = ['super_admin', 'operations_admin'].includes(cu?.role)`) is refactored to use TWO derived flags:
  - `canApprove` (super_admin + operations_admin OR services_admin for service proposals) — shows Approve / Reject buttons only
  - `canDrive` (super_admin + operations_admin + services_admin + assigned operations_user + assigned services_user) — shows Submit for Internal Approval, Edit Proposal, Mark Sent to Client, Client Approved, Mark as Loss, Request Revision (the new button)
  See D-09 for the services_admin/operations_admin asymmetry by `parent_collection`.
- **D-06:** Department gating by `parent_collection`. For proposals where `parent_collection === 'projects'`, the eligible drivers are operations_admin + assigned operations_user (services_admin/services_user do NOT see the action buttons even if they otherwise have the role). For `parent_collection === 'services'`, eligible drivers are services_admin + assigned services_user. Super_admin sees everything for both. Mirrors the Phase 26-30 department-isolation pattern. (G2.Q2 → "Mirror operations_user".)
- **D-07:** Assignment check at the UI layer reuses existing utilities — `getAssignedProjectCodes` (or the equivalent for services) plus `currentProject.id` comparison. For operations_user invoking from the inline card on project-detail.js (which already has `currentProject`), the assignment is implicit; for invocation from the home Proposals sub-tab the assignment check pulls the parent project/service doc once.
- **D-08:** Edit Proposal is included in the operations_user/services_user driver set — they can edit title/description/amount/target_client of their own proposals while in `draft` or `for_revision` status. This is the "operations_user owns the proposal" stance from G2.Q1 → "Full lifecycle except Approve/Reject".
- **D-09:** Approve / Reject remain admin-only (super_admin + operations_admin for project proposals; super_admin + services_admin for service proposals). The internal-approval gate is the gatekeeper boundary that does NOT relax in this phase.

### G3 — Revision cycle (explicit transitions, NOT auto-status)

- **D-10:** **Comms entries are pure conversation logs.** They do NOT trigger any `proposal.status` or `parent.project_status` change. `saveCommsEntry` continues to append a single entry to `comms_log` and write `updated_at`. (User's mid-discussion course-correction from G3.Q3.)
- **D-11:** **Add new "Request Revision" action button.** Visible to the `canDrive` role set (D-05) when `proposal.status === 'pending_client'`. Click flow:
  1. Open a small confirmation modal that captures an optional comment (mirrors the existing Approve / Reject sub-modal pattern at `app/proposal-modal.js:920+`)
  2. On confirm, call `_applyProposalStateTransition({ proposal, newStatus: 'for_revision', newProjectStatus: 'For Revision', auditAction: 'REVISION_REQUESTED', auditComment })`
  3. `_applyProposalStateTransition` writeBatch atomically flips proposal.status + parent.project_status + appends audit_log entry — already supports this signature today (no changes to that function)
- **D-12:** **Extend "Mark Sent to Client" visibility** to also show at `proposal.status === 'for_revision'` (currently it only shows at `pending_client`). When invoked from `for_revision`, it advances proposal.status → `pending_client` AND parent.project_status → "Proposal Under Client Review" via `_applyProposalStateTransition` (audit action stays `SENT_TO_CLIENT`). Single button handles both the initial send and every resend.
- **D-13:** **New audit action: `REVISION_REQUESTED`.** Adds:
  - Entry in `AUDIT_ACTION_LABELS` (e.g., "Revision Requested")
  - Entry in `AUDIT_ACTION_DOT_COLORS` (suggest amber `#f59e0b` to visually distinguish from the green "Sent to Client" dot)
- **D-14:** **Re-advance target.** A revision-resend goes back to the existing unified `project_status` value `'Proposal Under Client Review'` — NOT a new sub-status. (G3.Q1 → "Back to 'Proposal Under Client Review'".) Phase 81's 10-option enum stays intact; the audit trail makes the resend distinguishable from the first send.
- **D-15:** Roadmap success criterion #3 is interpreted via this corrective: instead of comms entries auto-advancing status, the explicit "Request Revision" button (pending_client → for_revision) and the extended "Mark Sent to Client" gate (for_revision → pending_client) cover the cycle. Audit trail captures both transitions with actor + timestamp + optional comment. The roadmap-as-written phrasing is superseded by this CONTEXT.

### G4 — Firestore rules

- **D-16:** **Relax `proposals` update rule** in `firestore.rules` (currently locked to super_admin + operations_admin). New rule:
  - Existing branch: super_admin + operations_admin can update anything (subject to the existing created_at/created_by immutability checks)
  - **New branch:** operations_user / services_user can update ONLY when BOTH:
    1. **Field-mask:** `request.resource.data.diff(resource.data).affectedKeys().hasOnly(['comms_log', 'audit_log', 'status', 'current_status_since', 'updated_at'])`
    2. **Parent-assignment:** the caller's UID appears in the personnel array of the parent project/service doc (resolved via `get(/databases/$(database)/documents/$(resource.data.parent_collection || 'projects')/$(resource.data.project_id))`)
    3. **Department alignment:** operations_user can only write when `parent_collection == 'projects'`; services_user only when `parent_collection == 'services'`
- **D-17:** **Defense-in-depth:** field-mask AND assignment together. (G4.Q1 → "Field-mask AND parent-assignment".) The success criterion explicitly requires "non-attached users are still denied" — that test is the assignment check. The field-mask additionally prevents an authorized but adversarial user from rewriting `title`, `amount`, or `created_by`.
- **D-18:** **Accept the rule-eval cost** of one extra `get()` per write — same cost shape as Phase 84's `notifications` create rule. Single get() per update.
- **D-19:** **Existing rule preserved:** super_admin + operations_admin blanket update remains in place for backward compat. The new branch is additive — no existing flow regresses.
- **D-20:** **firestore.rules block comment** must document the new branch with the rationale (mirrors the existing Phase 84 NOTIF-12 comment block), so the next auditor understands why operations_user can write a subset of fields.

### G5 — Audit trail comms nesting

- **D-21:** **`renderAuditTrail` extended** in `app/proposal-modal.js` to merge `audit_log + comms_log` into a single timeline structure:
  1. Sort audit_log entries newest-first (existing behavior)
  2. For each comms_log entry, find the most recent `SENT_TO_CLIENT` audit entry whose `ts` precedes the comms entry's `logged_at` — attach as child
  3. Comms entries with no preceding `SENT_TO_CLIENT` (e.g., logged while still in draft / pending_internal / pending_client before first send) attach as children of the `CREATED` audit entry (which is always present and is the earliest entry)
- **D-22:** **Pre-Sent comms placement.** Group under the `CREATED` audit entry at the bottom of the trail. (G5.Q1 → "Group under proposal creation".) Reads as "while drafting/awaiting approval, these conversations happened" — every comms entry has an audit parent; visually consistent; no orphans.
- **D-23:** **Trail sort order.** Trail-level: newest-first (existing). Inside each comms group: OLDEST-first — reads as a forward conversation thread under each "Sent to Client" event. (G5.Q3 → "Oldest-first inside group".)
- **D-24:** **Left-column Client Communications section: KEPT.** It remains the input surface (Add Entry button + form). Comms list also stays in the left column for the chronological view + add-flow co-location. The right-column audit trail adds the nested-children visualization in addition. (G5.Q2 → "Keep — it's the input surface".)
- **D-25:** **Visual treatment for indented comms children** in the trail (Claude's discretion for exact values, must match existing aesthetic):
  - Left padding ~24px relative to parent audit entry
  - Smaller font size (12px vs the existing 13px for audit entries)
  - Distinct dot color — suggest gray (`#94a3b8`) to differentiate from the colored audit-action dots
  - Display: comms type pill (reuse `COMMS_TYPE_META` badge classes) + truncated description (~120 chars) + actor name + relative date
  - Connector line: short stub left-of-dot, indented so it visually attaches under the parent SENT_TO_CLIENT dot
- **D-26:** **No new exports.** `COMMS_TYPE_META` and the merge logic stay private to `app/proposal-modal.js`. No new shared utility is needed for Phase 87.2 — the merge logic is single-use.

### Claude's Discretion

- Exact pixel values, padding, and dot colors for the nested-comms visual treatment in D-25 (the aesthetic must match the existing audit-trail styling at `app/proposal-modal.js:182-200`)
- Confirmation modal copy for the new "Request Revision" button (D-11) — match the tone of existing Approve / Reject sub-modal copy at `app/proposal-modal.js:920+`
- Whether the new audit-action dot color is amber (`#f59e0b`) or another color — D-13 suggests amber but final pick matches the existing `AUDIT_ACTION_DOT_COLORS` palette
- The dropdown auto-select implementation in D-04 — whether `change` listener attaches inline in the modal HTML or wires through an explicit function; planner picks the cleanest pattern given existing modal code shape
- Toast messages for all the new action flows ("Revision requested.", "Resent to client.", etc.) — match the existing toast tone in proposal-modal.js
- The structure of the new field-mask condition in firestore.rules — whether to extract a helper function (e.g., `isProposalCommsOnlyUpdate()`) or inline the predicate; planner picks based on how it composes with the existing rule block
- Whether the new audit-action `REVISION_REQUESTED` is also wired into `notifications.js` (NOTIFICATION_TYPES) for an in-app notification — Phase 83's notification trigger pattern would call for it, but Phase 87.2 success criteria does not explicitly require it. Researcher should investigate whether NOTIF-09/NOTIF-10 already covers it implicitly or if a new NOTIFICATION_TYPE is needed; planner decides scope (lean toward "deferred to v4.1" if a new type is required to avoid scope creep)
