```javascript
//   BRANCH 1 — Admin (existing, preserved): super_admin + operations_admin can update
//     any field (subject to created_at/created_by immutability checks).
//   BRANCH 2 — Attached user (NEW, additive): operations_user or services_user can
//     update ONLY when ALL of:
//       (a) field-mask: affectedKeys().hasOnly(['comms_log', 'audit_log',
//           'status', 'current_status_since', 'updated_at'])
//       (b) department alignment: operations_user requires
//           parent_collection == 'projects' (default when absent);
//           services_user requires parent_collection == 'services'.
//       (c) parent-assignment: caller's UID is in the parent doc's
//           personnel_user_ids array (one get() per write — same cost shape
//           as Phase 84 NOTIF-12 create rule).
//   The field-mask prevents an authorized-but-adversarial caller from rewriting
//   title / amount / target_client_* / created_by / loss_reason / attachment_*.
//   Loss writes (which touch loss_reason) remain admin-only by construction.
```

New `allow update:` predicate (two-branch `||`):

```javascript
// Update branches — see expanded block comment above for the full rationale.
// Both branches preserve the existing created_at + created_by immutability invariants.
allow update: if (
  // BRANCH 1 — Admin (existing, preserved verbatim from Phase 87 D-10).
  hasRole(['super_admin', 'operations_admin'])
  && request.resource.data.created_by == resource.data.created_by
  && request.resource.data.created_at == resource.data.created_at
) || (
  // BRANCH 2 — Attached operations_user / services_user, narrow field-mask
  //            (Phase 87.2 D-16 / D-17 / D-18).
  // Field-mask: only the 5-key allow-list is mutable.
  request.resource.data.diff(resource.data).affectedKeys().hasOnly(['comms_log', 'audit_log', 'status', 'current_status_since', 'updated_at'])
  // Same created_at + created_by immutability as BRANCH 1.
  && request.resource.data.created_by == resource.data.created_by
  && request.resource.data.created_at == resource.data.created_at
  && (
    // Operations branch: parent_collection is 'projects' (defaults when absent).
    (
      isRole('operations_user')
      && (!('parent_collection' in resource.data) || resource.data.parent_collection == 'projects')
      && request.auth.uid in get(/databases/$(database)/documents/projects/$(resource.data.project_id)).data.personnel_user_ids
    )
    ||
    // Services branch: parent_collection is 'services' (explicit).
    (
      isRole('services_user')
      && 'parent_collection' in resource.data
      && resource.data.parent_collection == 'services'
      && request.auth.uid in get(/databases/$(database)/documents/services/$(resource.data.project_id)).data.personnel_user_ids
    )
  )
);
```

`allow create:`, `allow read:`, and `allow delete:` are unchanged.

## Security Design Notes (Threat Model Summary)

- **T-87.2-04-01 (EoP):** An adversarial assigned operations_user could write `status` directly via raw SDK, skipping the client-side state machine. Field-mask restricts to 5 keys only; every legitimate transition appends an audit_log entry — anomalous status flips (no matching audit_log entry) are detectable post-facto. Accepted per D-17.
- **T-87.2-04-02 (Tamper):** Assigned user could replace the whole `audit_log` array. Same residual risk as the notifications block. Detectable via Firebase project-level audit logs and per-entry `actor_id`.
- **T-87.2-04-03 (Info Disclosure):** `get()` reads only `personnel_user_ids` — already readable by any active user via existing `projects.allow read` / `services.allow read`. No new disclosure.
- **T-87.2-04-04 (DoS):** One extra document read per proposal write. At ~1-2 comms entries per proposal per day, negligible.
- **T-87.2-04-05 (Repudiation):** Both branches re-assert `created_by == resource.data.created_by` and `created_at == resource.data.created_at` — authorship attribution cannot be altered.
