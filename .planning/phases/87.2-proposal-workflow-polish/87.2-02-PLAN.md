<objective>
Close Gap G4 — Firestore rules: relax the `proposals` update rule so non-admin assigned users can write a narrow field-mask.

Purpose: operations_user inside a Projects detail view tries to save a comms entry on an attached proposal and the Firestore SDK returns `Missing or insufficient permissions`. The existing rule (lines 558-560) locks updates to super_admin + operations_admin only. This blocks the very role that owns client communications. The fix: add a NEW additive branch to the existing `allow update:` rule on `/proposals/{proposalId}` that grants narrowly-scoped write access to operations_user/services_user when both (a) the field-mask is restricted to a 5-key allow-list AND (b) the caller's UID appears in the parent project/service's `personnel_user_ids` array.

This mirrors the Phase 84 NOTIF-12 field-mask pattern verbatim (line 593-600 of firestore.rules). The existing super_admin + operations_admin blanket branch stays in place — additive change, no regression.

Output: Single-file edit to `firestore.rules`. The new branch is appended to the existing `allow update:` predicate via `|| (...)`. A new block comment documents the rationale per D-20.

This plan runs in parallel with Plan 01 (G1 in `app/proposal-modal.js`) — fully disjoint files, no merge conflict risk.
</objective>

**Existing rule helpers (firestore.rules lines 60-90):**
```
function hasRole(roles)               { return isActiveUser() && getUserData().role in roles; }
function isRole(role)                 { return hasRole([role]); }
function isAssignedToProject(code)    { return getUserData().all_projects == true || code in getUserData().assigned_project_codes; }
function isAssignedToService(code)    { return getUserData().all_services == true || code in getUserData().assigned_service_codes; }
```
These helpers operate on a `code` (project_code / service_code), NOT on a uid. They read the CURRENT USER's assignment list. The new branch needs a different shape: it must check whether the CURRENT USER's UID is in the PARENT DOC's `personnel_user_ids` array — because proposals don't carry `service_code` on the doc, only `project_id` (the parent doc ID) and an optional `project_code`.

**Parent doc shape — confirmed by reading `app/engagement-create.js` lines 60-97:**
```
projects/{id}: { project_code, project_name, client_id, client_code, project_status, personnel_user_ids: [uid1, uid2, ...], ... }
services/{id}: { service_code, service_name, service_type, client_id, client_code, project_status, personnel_user_ids: [uid1, uid2, ...], ... }
```
Both collections use the SAME field name `personnel_user_ids` (an array of UID strings). This is the canonical assignment check at the parent doc level.

**Proposal doc shape (confirmed by reading `app/proposal-modal.js` saveProposal payload, lines 668-689):**
```
proposals/{id}: { proposal_id, project_id, project_code, title, description, amount,
                  target_client_id, target_client_name, status, attachment_*,
                  audit_log[], comms_log[], loss_reason, current_status_since,
                  created_by, created_at, updated_at }
```
Note: there is NO `parent_collection` field stored at saveProposal time (CONTEXT.md confirms it defaults to 'projects' if absent). For the new rule we MUST handle that absence safely.

**Canonical field-mask pattern — mirror this verbatim (firestore.rules lines 593-600, notifications block):**
```
allow update: if isActiveUser()
  && resource.data.user_id == request.auth.uid
  && request.resource.data.user_id == resource.data.user_id    // immutable
  && request.resource.data.type == resource.data.type           // immutable
  && request.resource.data.message == resource.data.message     // immutable
  && request.resource.data.link == resource.data.link           // immutable
  && request.resource.data.created_at == resource.data.created_at
  && request.resource.data.actor_id == resource.data.actor_id;
```
Note: the notifications block uses field-by-field immutability comparisons, NOT `hasOnly([...])`. The CONTEXT.md D-16 explicitly says to use `hasOnly([...])` for the proposals branch — both styles are acceptable in Firestore Security Rules. Use `hasOnly([...])` per CONTEXT — it's more concise for the 5-key allow-list.

**get() call shape (lines 118, 134, 147 of firestore.rules):**
```
get(/databases/$(database)/documents/users/$(userId)).data.role
```
Static path inside `get()` — the collection portion CANNOT be dynamic in a single `get()` call. We must branch on `parent_collection` and emit two separate `get()` calls inside an `||`.

## Trust Boundaries

| Boundary | Description |

| Firestore client SDK → Firestore Security Rules engine | All proposal writes from the browser pass through these rules. The new branch deliberately RELAXES write access for non-admin assigned users, so the security-critical question is whether the relaxation can be abused. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-87.2-04-01 | Elevation of Privilege | New BRANCH 2 — operations_user writing `status: 'client_approved'` from `for_revision` directly | accept | The field-mask allows `status` writes. A malicious assigned operations_user COULD attempt to skip Approve/Reject by writing status directly via raw Firestore SDK. Mitigation: client-side `_applyProposalStateTransition` is the only sanctioned write path and it enforces the state machine. A hand-rolled adversarial write is detectable in audit_log (every legit transition appends an audit_log entry with `action: 'CLIENT_APPROVED'` etc.; a status flip with no matching audit_log entry is anomalous). Accept the residual risk per D-17 — defense-in-depth via field-mask + assignment is sufficient at current scale; full state-machine enforcement in rules is out of scope per CONTEXT. |
| T-87.2-04-02 | Tampering | New BRANCH 2 — assigned user rewriting `audit_log` to remove their own actions | mitigate (partial) | The field-mask allows `audit_log` writes (necessary for `_applyProposalStateTransition` to append entries). A malicious assigned user could replace the whole audit_log array. Mitigation: array-rewrite tampering is detectable post-facto via Firestore audit logs (Firebase project-level audit) and via the per-entry `actor_id` field. Same residual risk as the notifications block currently accepts; not a new attack surface. |
| T-87.2-04-03 | Information Disclosure | get() of parent project/service for assignment check | accept | The `get()` reads only the parent's `personnel_user_ids` — already readable by any active user via the existing `projects.allow read: if isActiveUser()` and `services.allow read: if isActiveUser()` rules. No new disclosure. |
| T-87.2-04-04 | Denial of Service | extra get() per proposal write | accept | One additional document read per proposal update — same cost shape as Phase 84 NOTIF-12 create rule (D-18 explicitly accepts this cost). At current write volumes (~1-2 comms entries per proposal per day) this is negligible. |
| T-87.2-04-05 | Repudiation | created_by / created_at rewrite via BRANCH 2 | mitigate | BRANCH 2 explicitly re-asserts the same `created_by == resource.data.created_by` and `created_at == resource.data.created_at` invariants as BRANCH 1, so neither branch can alter authorship attribution. |
