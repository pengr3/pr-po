## Bug Fix Details (Task 1 ‚Äî D-02)

All six sites shared the same root cause: the `clients` Firestore collection uses `company_name` as the canonical display field, but `proposal-modal.js` was reading `client_name` (nonexistent field) at every interaction point.

**Edit 1D ‚Äî Sort comparator (line 160, `_loadModalDropdownData`):**
- Before: `(a.client_code || a.client_name || '').localeCompare(b.client_code || b.client_name || '')`
- After: `(a.client_code || a.company_name || '').localeCompare(b.client_code || b.company_name || '')`
- Effect: Clients without a `client_code` now sort deterministically by company name instead of collapsing to empty-string comparison.

**Edit 1A ‚Äî Dropdown label with code (line 529, `showCreateModal` clientOptions map):**
- Before: `` `${escapeHTML(c.client_code)} ‚Äî ${escapeHTML(c.client_name || '')}` ``
- After: `` `${escapeHTML(c.client_code)} ‚Äî ${escapeHTML(c.company_name || '')}` ``

**Edit 1B ‚Äî Dropdown label without code (line 530):**
- Before: `escapeHTML(c.client_name || '')`
- After: `escapeHTML(c.company_name || '')`

**Edit 1C ‚Äî `data-name` attribute (line 531):**
- Before: `data-name="${escapeHTML(c.client_name || '')}"`
- After: `data-name="${escapeHTML(c.company_name || '')}"`

**Edit 1E ‚Äî saveProposal write site (line 632):**
- Before: `const clientName = client?.client_name || null;`
- After: `const clientName = client?.company_name || null;`
- Effect: New proposals now persist `target_client_name: "<actual company name>"` to Firestore instead of `null`.

## Listener Block Insertion (Task 2 ‚Äî D-01/D-04)

Inserted immediately after `document.body.insertAdjacentHTML('beforeend', html);` at line 579 (now lines 581-600 after insertion).

Logic: On `#proposalCreateProject` change event, looks up the selected project in `_modalProjectsData`, reads its `client_id`, and sets `#proposalCreateClient.value` to that ID only if the corresponding `<option>` exists in the client dropdown. Clientless projects (null `client_id`) are no-ops. Edit-mode safety is inherent: `change` only fires on user interaction, not on initial render, so existing `target_client_id` values on edit modal open are preserved per D-04.
