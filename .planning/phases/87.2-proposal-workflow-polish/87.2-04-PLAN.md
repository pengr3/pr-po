THREAT MODEL

Trust Boundaries

  - browser SPA --> Firestore writes (proposal + parent doc). The new Request Revision and extended Mark Sent paths both go through the existing _applyProposalStateTransition writeBatch, which is gated by Plan 02's firestore.rules update branch.

STRIDE Threat Register

  - T-87.2-04-01 (Tampering): a malicious operations_user could submit confirmRequestRevision against a proposal where they are not in personnel_user_ids. Disposition: mitigate. The Firestore rule (Plan 02 BRANCH 2) enforces the personnel check at write time — the UI gate is only cosmetic. _applyProposalStateTransition will surface a permission-denied error in the catch block, showToast renders the failure to the user.
  - T-87.2-04-02 (Repudiation): the comment textarea is optional, so an attacker might claim "I didn't request that revision". Disposition: accept. The audit_log entry captures actor_id + actor_name + ts even when comment is null; non-repudiation rests on the actor fields, not the comment. The PROD walkthrough confirmed the previous comms-entry path also had optional content; symmetry maintained.
  - T-87.2-04-03 (DoS): the new sub-modal could be opened repeatedly to spam REVISION_REQUESTED audit entries. Disposition: accept. Same flood-risk as existing Approve/Reject/Loss sub-modals; no per-action rate limit anywhere in the SPA today. At current scale a rate-limit is overkill.
