<objective>
Close Gap G1 — Target Client default + saveProposal field-name bug fix + dropdown display field-name fix + clients sort comparator field-name fix.

Purpose: PROP-2026-001 and every recently-created proposal store `target_client_name: null` because `saveProposal` reads a nonexistent field name (`client?.client_name`) from the `clients` collection (canonical field is `company_name`, confirmed in app/views/clients.js). The SAME defect lives at five READ sites: three in the clients-dropdown render loop in `showCreateModal` (lines 529-531) — `c.client_name` is undefined on every client document, so the rendered `<option>` labels look like "CLIENT-CODE —  " (empty trailing name) and the `data-name` attribute is empty — plus two more in the `_modalClientsData.sort()` comparator at line 160 (`a.client_name`, `b.client_name`) where the fallback after `client_code` collapses to empty-string comparison instead of sorting by company name. Without fixing all five read sites the Plan 01 smoke-test will reasonably look broken (user sees blank labels in a potentially mis-sorted dropdown and concludes the auto-select is wrong when actually the display+sort is). Additionally, when a user creates a proposal from within a project context the Target Client dropdown does not default to the parent project's client — even though the project doc has a `client_id`. All concerns land in the same single-file area (`app/proposal-modal.js` lines ~150-700), so they are bundled into one atomic plan that the user can smoke-test by creating one new proposal end-to-end.

Output: Edits to `app/proposal-modal.js`:
  1. Bug fix at two sites in `_loadModalDropdownData` clients sort comparator (line 160 today): `a.client_name`/`b.client_name` → `a.company_name`/`b.company_name`.
  2. Bug fix at three sites in `showCreateModal` clientOptions render (lines 529-531 today): `c.client_name || ''` → `c.company_name || ''` at all three occurrences (the dropdown label string with code, the dropdown label string without code, and the `data-name` attribute).
  3. Bug fix at the client lookup in `saveProposal` (currently line 632): `client?.client_name || null` → `client?.company_name || null`.
  4. New `change` listener on `#proposalCreateProject` in `showCreateModal` that, on user-triggered project change, looks up the picked project's `client_id` in `_modalProjectsData` and sets `#proposalCreateClient.value` to that ID.

All six read+write sites (one optional-chained write site + three non-optional read sites in the clients dropdown loop + two sort-comparator read sites) share the SAME root cause as D-02 (canonical field is `company_name`, not `client_name`). Bundling them keeps the bug-fix atomic.

Per D-03 (NO legacy backfill): no migration script, no read-time fallback in `buildProposalDetailsBlock`. The display reads whatever's stored; existing "(none)" proposals are corrected manually via the Edit modal once D-02 lands.
