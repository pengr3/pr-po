# Changelog: v3.2 — Supplier Search, Proof of Procurement & Payables Tracking

**Released:** 2026-04-28
**Phases:** 63-82 (28 phases, 55 plans)

---

## New Features

### Supplier Search
- A search bar above the supplier list filters by **supplier name and contact person** in real time (case-insensitive, both fields searched simultaneously)
- Pagination updates to reflect the filtered count — "Showing X of Y" matches the filter, not the total
- Clearing the search restores the full supplier list

### Proof of Procurement
- Procurement can paste any `https://` document link (Google Drive, OneDrive, SharePoint, etc.) onto a PO at **any procurement status**, including after Delivered
- Three-state proof indicators (green checkmark for URL, orange dash for remarks-only, empty circle for none) appear across **MRF Records, My Requests, and Finance PO Tracking**
- Finance sees a clickable "View Document" link in the PO Tracking tab
- Procurement timeline shows a "Proof attached" event when a proof URL is first saved
- Transport Requests now have the same three-state proof indicators, including on Material+TR mixed MRF rows

### Request for Payment (RFP) and Payables Tracking
- Procurement can submit a **Request for Payment** against any PO via the right-click context menu, entering invoice number, amount, payment terms, and due date
- RFPs are auto-assigned PO-scoped IDs in the format `RFP-{PO-ID}-{n}` (eliminates duplicate codes when multiple POs share a project)
- Finance has a new **Payables tab** with two tables:
  - **RFP Processing** — flat list of outstanding RFPs (Fully Paid hidden by default)
  - **PO Payment Summary** — one row per PO with expandable per-tranche sub-rows, paginated at 15 rows per page
- Finance records payments (partial or full) with amount, date, method, and reference; system auto-derives status (**Pending / Partially Paid / Fully Paid / Overdue**) — Finance never manually sets status
- Voided payments are excluded from totals but preserved for audit
- Procurement can **cancel zero-payment RFPs** via right-click on PO IDs and TR badges; cancelled RFPs free up the tranche for re-filing with pre-filled data
- **Delivery fees** can be submitted as standalone RFPs through a simplified modal; PO ID cells show an 8px colored dot (red = unpaid, green = paid) when a delivery fee exists
- **Saved bank accounts** from prior RFPs appear in a dropdown for one-click re-entry on Bank Transfer mode (later evolved into an "Add Alternative Bank" toggle in Phase 76)
- Transport Requests support the full RFP workflow — right-click TR badges in MRF Records to open a pre-filled RFP modal

### Cancel PRs and Restore MRF
- Procurement can **right-click an MRF ID** in MRF Records and see a "Cancel PRs" option for MRFs with PR Generated status
- Cancellation deletes linked Pending/Rejected PRs and restores the MRF to In Progress so items can be revised and re-submitted
- Force-recall path: if any linked PRs are Finance-Approved with POs at Pending Procurement, the POs are voided to Cancelled and all PRs deleted
- If any linked POs have Procuring/Procured/Delivered status, cancellation is **blocked** with an explanatory error

### Financial Breakdown Modal
- The Expense Breakdown modal is renamed to **"Financial Breakdown"** across project and service detail pages
- A new **Payables tab** (third tab after By Category and Transport Fees) shows a sorted, read-only worklist of every payable: one row per PO (Total Payable = total minus delivery fee), one row per Delivery Fee, one row per Finance-Approved TR
- Rows sorted action-needed first: Not Requested → Requested → Partial % → Fully Paid, then by Total Payable descending within each bucket
- Project and Service detail Financial Summary cards show **Paid** and **Remaining Payable** cells with red styling when amount > 0, green when fully paid
- The **Refresh** button on the Financial Summary card now refreshes data AND opens the Financial Breakdown modal in one click

### Create Projects Without a Client
- Projects can now be saved **without selecting a client** (`client_id: null`); the project code is deferred until a client is later assigned
- Clientless projects appear in the project list with em-dash (`—`) in the Code and Client columns
- Clientless projects appear in MRF dropdowns with "(No code yet)" appended to the project name
- Assigning a client triggers a **confirmation modal** showing the about-to-be-generated code and the count of linked records, then runs a batched backfill of `project_code` and `client_code` across all linked MRFs / PRs / POs / TRs / RFPs
- Once issued, `client_id`, `client_code`, and `project_code` are **locked** (no re-assignment), enforced at both the UI and Firestore Security Rules level
- Deep links to clientless projects resolve via Firestore doc ID (`#/projects/detail/{doc_id}`) and continue to work after code issuance

### Home Dashboard Charts
- The home page Project and Service status breakdowns are now rendered as **Chart.js horizontal bar charts** (not text rows)
- Chart card sizes tuned so bars fill the canvas proportionally; legend font/box sized for readability
- Highlighted statuses (For Inspection, For Proposal, Under Client Review, On-going) use muted brand-palette colors; remaining statuses use a graduated slate scale for visual hierarchy

### Delete Rejected MRFs
- A **Delete MRF** button now appears on the MRF Details panel in MRF Processing for rejected MRFs (mirrors the existing Delete TR pattern)
- Single confirm shows the cascade counts (linked PRs, POs, TRs); confirming permanently deletes the MRF and all linked children
- Lightweight cleanup UX: no reason prompt, no audit row — for shedding dead-end requests

## Improvements

### MRF Records / Transport Request Parity
- **PO badges** now show a flush 3px progress bar below each badge filled by RFP payment data (no more badge-fill overlay muddying the text)
- **TR badges** show the same 3px payment progress bar, driven by RFPs keyed to `tr_id`
- POs with no RFPs show an empty progress bar (was previously a misleading full red bar)
- Status badge font color is exclusively controlled by the status-badge CSS class — no overlay interference

### Expense Modal Cleanup
- Total Cost scoreboard card shows only the total amount — removed the misleading "N documents" subtitle
- Category item tables now show **Item Name / Qty / Unit / Unit Cost / Subtotal** instead of just PO IDs (Delivery Fees table retains PO ID + Amount)
- CSV export and Transport Requests table left unchanged

### Finance Tab Mobile Optimization (Phase 73 family)
- At viewports ≤768px, Finance tab card-headers (Material PRs, Transport Requests, Purchase Orders) **stack vertically** — heading above controls — with 44px-min touch targets
- All 5 Finance table groups (Pending Approvals × 2, Purchase Orders, Payables × 2, Project List × 3) render as **vertical card stacks** instead of horizontally-scrolling tables on mobile, via a CSS dual-mode pattern (table and card list both in DOM, media query hides the inactive one)
- PR Details, TR Details, and Financial Breakdown modals collapse to 1-column grid on mobile; wide multi-column tables become horizontally scrollable inside their cards
- Financial Breakdown modal scorecard rows (2/3/2 grid) collapse to 1-column at ≤768px
- Finance sub-tab navigation rebuilt as a single unified **sticky pill bar** with scroll-hide-on-down / show-on-up UX (replaces the desktop `.tab-btn` row + mobile `<select>` dropdown)
- Project List expense tables migrated to the existing `.table-scroll-container` shadow-scroll pattern matching the rest of the app

### Material Request Tab Mobile Optimization (Phase 74)
- The MRF items table is replaced with a **card-per-item layout** at ≤768px (Description / Qty+Unit / Category stacked, remove button top-right, full-width Add button below)
- Cards and underlying table rows stay in sync — adding/removing a card updates the form data
- The 8-column **My Requests table** is replaced with a vertical stack of MRF summary cards (MRF ID, Status badge, Date Needed, 3-dot action menu for Edit/Cancel)
- MRF sub-tab navigation rebuilt as a sticky pill bar matching the Finance pattern
- Desktop layouts (≥769px) fully preserved — all changes CSS media-query gated

### MRF Details Panel
- MRF Details panel now shows **Date Submitted** and **Justification** for existing MRFs (full context for processors)
- **Quantity input column** widened so 5-digit values (e.g., 5000) display and persist correctly
- The native project/service `<select>` in the MRF line-item form is replaced with a **searchable combobox** so large lists are navigable by typing

### Layout & Navigation Polish
- MRF Processing right-panel no longer overflows the viewport at **1366×768** (typical laptop monitor); fixed via `min-width: 0` on grid children and `max-width: 100%` on the items-table wrapper
- Top **navbar** no longer wraps to a second line or stacks the brand vertically between 769px–1400px; fixed with `flex-wrap: nowrap` plus a new compression media query that tightens nav-link padding/gap/font-size
- Supplier column **removed from the PR Details modal items table** (supplier already shown in the PR modal header)

### Unified Project/Service Status (Phase 81)
- The dual `internal_status` + `project_status` fields on Projects and Services are **merged into a single unified `project_status` field** with 10 options:
  - For Inspection, For Proposal, Proposal for Internal Approval, Proposal Under Client Review, For Revision, Client Approved, For Mobilization, On-going, Completed, Loss
- All forms, filters, table columns, CSV exports, validation, and edit-history labels updated to the unified field
- Legacy status values display with a `(legacy)` suffix in tables and filter dropdowns for graceful cleanup
- Home dashboard charts collapse to a single chart per entity (was two: internal + project)
- Edit history retains the old `internal_status` field as **"Internal Status (Legacy)"** for auditor-friendly historical records

### RFP Bank Transfer Polish
- Bank Transfer mode in the RFP modal gained an **"Add Alternative Bank"** button that lets the user manually enter a second bank account (Bank Name, Account Name, Account Number)
- Removing the alternative bank clears all three fields; both PO RFP and TR RFP modals support it (the original saved-bank dropdown was removed in Phase 76 in favor of this simpler manual-entry pattern)

### Active Tranche Display
- The PO Payment Summary "Current Active Tranche" column for partially paid POs now shows payment progress percentage at a glance — e.g., **"Tranche 2 (50%) — 30% Paid"**

### PO Ref Clickability
- **PO IDs in the Finance RFP Processing table are now clickable** — opens a PO details modal showing PO ID, MRF ref, supplier, project, date issued, status, total amount, and line items

### Service Detail Financial Summary Parity
- The Services detail page now has the same **Paid / Remaining Payable** cells, refresh-and-modal behavior, and zero-state visibility as Project detail (added in Phase 72.1)

## Bug Fixes

- **Service Remaining Payable formula** — Was incorrectly calculated as `(poTotal + prTotal) - rfpTotalPaid`, double-counting PR costs that already exist as POs. Fixed to mirror the project-detail formula `(poTotal + trTotal) - rfpTotalPaid` (Phase 75)
- **Remaining Payable in expense modal** — Was using `RFP totalRequested - totalPaid`, causing un-RFP'd PO costs to be hidden from the obligation. Fixed to use `totalCost - totalPaid` (non-voided), and the card now appears even when POs exist with zero RFPs (Phase 69.1)
- **Voided payment exclusion** — Voided payment records are now correctly excluded from Total Paid sums (Phase 69.1)
- **PO Payment Progress Bar** — Was rendering as a fill overlay inside PO badges, muddying the text and showing a misleading full red bar for POs with no RFPs. Replaced with a flush 3px bar below each badge (Phase 66)
- **Procurement view lifecycle cleanup** — `procurement.js destroy()` now resets `rfpsByTR` state and removes `window.showTRRFPContextMenu`, `window.openTRRFPModal`, `window.submitTRRFP` registrations to prevent stale TR-RFP state across view re-entries (Phase 75)
- **Financial Summary stale values** — Navigating between project/service detail pages no longer leaks stale Paid / Remaining Payable values from the previous page (Phase 72)
- **MRF Processing layout overflow** — MRF Details right-panel no longer breaks past the viewport at 1366×768 (Phase 80)
- **Top navbar wrap** — Navbar no longer wraps to a second line or vertically stacks the brand between 769–1400px viewports (Phase 80)
- **Payables UI containment** — Both Payables tables now use `table-scroll-container` for horizontal scroll containment instead of breaking page layout on narrow viewports (Phase 65.8)
- **TR-linked RFPs in Finance Payables** — Fixed blank/broken rendering for RFPs created from TRs (was collapsing under empty-string PO key in `buildPOMap`); now correctly grouped by `tr_id` with `isTR` flag (Phase 67)
- **Active Tranche fallback label** — `deriveStatusForPO` no-firstUnpaid fallback now formats as "NN% Paid" instead of the literal word "Partial" (Phase 71)
