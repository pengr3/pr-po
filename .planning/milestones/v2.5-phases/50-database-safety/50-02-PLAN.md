Firestore schema relationships (from CLAUDE.md and firestore.rules):

**ID formats:**
- MRFs: `mrf_id` = `MRF-YYYY-###`
- PRs: `pr_id` = `PR-YYYY-###`, references `mrf_id`
- POs: `po_id` = `PO-YYYY-###`, references `pr_id` and `mrf_id`
- TRs: `tr_id` = `TR-YYYY-###`, references `mrf_id`
- Suppliers: `supplier_name` (unique)
- Projects: `project_code` (unique), `project_name`
- Services: `service_code` (unique), `service_name`
- Clients: `client_name` or `name` field

**Referential integrity chains:**
1. MRFs → Projects (via `project_code` or `project_name`) OR Services (via `service_code` or `service_name`)
2. PRs → MRFs (via `mrf_id`), PRs → Suppliers (via `supplier_name`)
3. POs → PRs (via `pr_id`), POs → MRFs (via `mrf_id`), POs → Suppliers (via `supplier_name`)
4. TRs → MRFs (via `mrf_id`), TRs → Suppliers (via `supplier_name`)
5. deleted_mrfs → cascaded from mrfs (should have valid structure)

**Required fields by collection:**
- mrfs: mrf_id, status, items_json
- prs: pr_id, mrf_id, supplier_name, total_amount, finance_status, items_json
- pos: po_id, pr_id, mrf_id, supplier_name, total_amount, procurement_status
- transport_requests: tr_id, mrf_id, supplier_name, total_amount, finance_status
- suppliers: supplier_name
- projects: project_code, project_name, status
- services: service_code, service_name, status
- users: email, role, status

**Status enums:**
- MRF status: Pending, Approved, Rejected
- PR/TR finance_status: Pending, Approved, Rejected
- PO procurement_status: Pending Procurement, Procuring, Procured, Delivered
- Project/Service status: active, inactive
- User status: active, inactive

  <action>
Create `scripts/verify-integrity.js` — a Node.js script using Firebase Admin SDK that checks all Firestore data for referential integrity and schema consistency.

**Firebase Admin SDK setup:** Same pattern as backup.js — initialize with service account key via env var or `./serviceAccountKey.json`.

**Script behavior:**

1. **Load all collections** into memory (since the dataset is small — test data):
   - Fetch all documents from: mrfs, prs, pos, transport_requests, suppliers, projects, services, users, clients, role_templates, invitation_codes, deleted_mrfs, deleted_users
   - Build lookup maps: `mrfIds`, `prIds`, `supplierNames`, `projectCodes`, `serviceCodes`, `userIds`

2. **Referential integrity checks** (report as ERRORS):
   - PRs referencing nonexistent `mrf_id` → "PR {pr_id} references MRF {mrf_id} which does not exist"
   - POs referencing nonexistent `pr_id` → "PO {po_id} references PR {pr_id} which does not exist"
   - POs referencing nonexistent `mrf_id` → "PO {po_id} references MRF {mrf_id} which does not exist"
   - TRs referencing nonexistent `mrf_id` → "TR {tr_id} references MRF {mrf_id} which does not exist"
   - PRs/POs/TRs referencing `supplier_name` not in suppliers collection → "PR {pr_id} references supplier '{name}' which does not exist" (report as WARNING since supplier may have been deleted)
   - MRFs with `project_code` that doesn't match any project → WARNING (legacy data may not have project_code)
   - MRFs with `service_code` that doesn't match any service → WARNING

3. **Schema consistency checks** (report as WARNINGS):
   - MRFs missing required fields: `mrf_id`, `status`, `items_json`
   - PRs missing required fields: `pr_id`, `mrf_id`, `supplier_name`, `finance_status`
   - POs missing required fields: `po_id`, `pr_id`, `mrf_id`, `procurement_status`
   - TRs missing required fields: `tr_id`, `mrf_id`, `finance_status`
   - MRFs with invalid status values (not in: Pending, Approved, Rejected)
   - PRs/TRs with invalid finance_status (not in: Pending, Approved, Rejected)
   - POs with invalid procurement_status (not in: Pending Procurement, Procuring, Procured, Delivered)
   - MRFs with `items_json` that fails `JSON.parse()`
   - PRs with `items_json` that fails `JSON.parse()`
   - Projects/Services with invalid status (not in: active, inactive)
   - Users with invalid role or missing email

4. **Orphan detection** (report as INFO):
   - Approved MRFs with no PRs generated → "MRF {mrf_id} is Approved but has no PRs"
   - PRs with finance_status=Approved but no POs → "PR {pr_id} is Approved but has no POs"

5. **Output format:**
```
Data Integrity Verification — clmc-procurement
================================================

Collections loaded:
  mrfs: 25 documents
  prs: 18 documents
  pos: 12 documents
  ...

Referential Integrity
---------------------
  [ERROR] PR PR-2026-005 references MRF MRF-2025-001 which does not exist
  [ERROR] PO PO-2026-003 references PR PR-2025-002 which does not exist
  [WARN]  TR TR-2026-001 references supplier 'Old Supplier Co' not in suppliers collection

Schema Consistency
------------------
  [WARN] MRF MRF-2026-010 missing field: items_json
  [WARN] PO PO-2026-008 has invalid procurement_status: "pending"

Orphan Detection
----------------
  [INFO] MRF MRF-2026-015 is Approved but has no PRs

Summary
-------
  Errors: 2 (referential integrity violations)
  Warnings: 5 (schema inconsistencies)
  Info: 1 (orphaned records)

  Status: ISSUES FOUND — review errors above
```

If no issues: `Status: ALL CLEAR — no integrity issues found`

6. **Exit code:** 0 if no errors (warnings/info are OK), 1 if any errors found.

7. **Optional `--json` flag:** Output results as JSON for programmatic consumption instead of human-readable text.
