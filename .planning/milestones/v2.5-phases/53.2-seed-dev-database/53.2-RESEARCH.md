# Phase 53.2: Seed Dev Database - Research

**Researched:** 2026-03-02
**Domain:** Firestore data seeding, static SPA, dev/prod environment bootstrap
**Confidence:** HIGH

## Summary

Phase 53.1 delivered a fully wired dev Firebase project (`clmc-procurement-dev`) with rules and indexes deployed. However, the dev Firestore database is functionally empty. The app requires specific Firestore configuration documents to be present before any user can log in and work normally. The 53.1 Task 3 summary confirms: "During Task 3 verification, the user seeded `role_templates` to dev Firestore manually via browser console." This was an ad-hoc step, not a structured seed — Phase 53.2 formalises that seeding with a reproducible, committed script.

There are two categories of data the dev database needs: (1) **configuration data** that the app reads on every page load (role_templates — 7 documents, one per role), and (2) **bootstrap data** that enables the first developer login and user onboarding workflow (the Super Admin `users` document and at least one active `invitation_code`). All other collections (mrfs, prs, pos, suppliers, projects, services, clients, etc.) start empty on dev by design — developers create test data as needed during feature work.

The seeding mechanism for this static no-build SPA is a browser console script — the same pattern already used by `scripts/seed-services-role-permissions.js` and `scripts/sync-role-permissions.js`. The script runs while logged into localhost:8000, imports the already-initialised `db` from `app/firebase.js`, and uses `setDoc` with idempotent writes. A new dedicated script at `scripts/seed-dev-database.js` is the right deliverable: it seeds everything from a cold-empty dev database in one pass and is safe to run repeatedly (idempotent).

**Primary recommendation:** Write a single browser-console seed script (`scripts/seed-dev-database.js`) that uses `setDoc` (upsert) to create all 7 `role_templates` documents, plus an active `invitation_code` document. The Super Admin `users` document was already created in 53.1 Task 3 Step 3 — confirm whether it needs to be included or is already present. Run the script as a checkpoint task (human action) from localhost:8000.

### Pattern 1: Browser Console Seed Script (Established Project Pattern)

**What:** A self-contained IIFE (immediately invoked function expression) pasted into DevTools console while the app is running at localhost:8000. It imports `db` from the running app module and performs Firestore writes.

**When to use:** Any time the dev Firestore needs to be bootstrapped from empty. Also serves as recovery when the dev database is accidentally wiped.

**Source of truth:** `scripts/seed-services-role-permissions.js` and `scripts/sync-role-permissions.js` in this project demonstrate the exact pattern.

**Example structure:**
```javascript
// scripts/seed-dev-database.js
// USAGE:
// 1. python -m http.server 8000 (or npx http-server)
// 2. Open http://localhost:8000 and log in as Super Admin dev account
// 3. Open DevTools console (F12)
// 4. Paste this entire file and press Enter

(async () => {
    const { db } = await import('/app/firebase.js');
    const { doc, setDoc, collection, addDoc, serverTimestamp, getDocs } =
        await import('https://www.gstatic.com/firebasejs/10.7.1/firebase-firestore.js');

    // Seed role_templates — 7 documents
    // Seed invitation_codes — 1 active code

    // ... idempotent writes with setDoc ...

    console.log('[SeedDev] Complete');
})();
```

### Pattern 2: Idempotent Writes with setDoc (No Merge)

**What:** Using `setDoc(doc(db, collection, docId), data)` without `{ merge: true }` — this is a full overwrite. Since these are configuration documents with a known full schema, full-overwrite setDoc is safer than merge: it ensures no stale fields from a previous run remain.

**When to use:** Seeding documents that have a known complete shape at seed time. Do NOT use for documents that might have user-generated sub-fields (those need merge).

**Example:**
```javascript
// Source: app/views/role-config.js (handleSave uses batch.set with merge for partial updates;
//         seed scripts use setDoc without merge to establish the full baseline)

await setDoc(doc(db, 'role_templates', 'super_admin'), {
    role_id: 'super_admin',
    role_name: 'Super Admin',
    permissions: {
        tabs: {
            dashboard:   { access: true, edit: true },
            clients:     { access: true, edit: true },
            projects:    { access: true, edit: true },
            services:    { access: true, edit: true },
            mrf_form:    { access: true, edit: true },
            procurement: { access: true, edit: true },
            finance:     { access: true, edit: true },
            role_config: { access: true, edit: true }
        }
    },
    created_at: serverTimestamp()
});
```

### Pattern 4: Invitation Code Schema

**What:** The `invitation_codes` collection stores codes that gate self-service registration. Schema from `app/auth.js` lines 47-51:

```javascript
// invitation_codes collection:
// - code: string (unique, the actual code string users type)
// - status: 'active' | 'used'
// - created_at: Timestamp
// - created_by: string (admin userId — use 'seed-script' for seeded codes)
// - used_at: Timestamp | null
// - used_by: string | null
```

The document ID is auto-generated by Firestore (addDoc). The `code` field is what users type. Use a human-readable dev code like `DEV-SEED-2026`.

### Anti-Patterns to Avoid

- **Copying role definitions from sync-role-permissions.js:** That file has stale role IDs (`finance_user`, `viewer`) and wrong tab sets. The source of truth is `role-config.js`.
- **Using addDoc for role_templates:** Role template document IDs must be the role name (e.g., `super_admin`). addDoc generates a random ID. Always use `setDoc(doc(db, 'role_templates', roleId), ...)`.
- **Not logging results to console:** The developer running the script cannot verify success if the script is silent. Log every setDoc result.
- **Running the script unauthenticated:** Firestore Security Rules require `isRole('super_admin')` to write `role_templates`. The script must run while logged in as the dev Super Admin account.
- **Seeding prod instead of dev:** The yellow DEV ENVIRONMENT banner must be visible before running the script. If no banner is visible, the script will write to production.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Firestore writes | Custom fetch/REST calls | Firebase JS SDK (already imported in app) | SDK handles auth headers, retries, token refresh automatically |
| Multiple writes atomically | Sequential awaits | writeBatch (optional for role_templates) | All-or-nothing semantics; simpler error handling |
| Running as Admin SDK | node script + service account | Browser console script | No service account needed; dev Super Auth handles auth via existing Firebase Auth |

**Key insight:** The existing browser-console-script pattern already used in this project is the right approach. There is no need to set up a Node.js Admin SDK environment for a one-time seed of 7 documents + 1 invitation code.

## Common Pitfalls

### Pitfall 1: Missing `services` Tab in role_templates
**What goes wrong:** The permissions module (`app/permissions.js`) checks `permissions.tabs.services.access`. If the `services` tab is absent from a role template, `hasTabAccess('services')` returns `false` (default fallback), hiding the Services nav item for all roles including super_admin.
**Why it happens:** `scripts/sync-role-permissions.js` is stale and does not include the `services` tab in all roles. Copying from it produces incomplete seed data.
**How to avoid:** Use the TABS array from `app/views/role-config.js` as the authoritative tab list. All 8 tabs must be present in every role template document.
**Warning signs:** Services tab not visible in navigation after login, even for super_admin.

### Pitfall 2: Wrong Role IDs in role_templates
**What goes wrong:** `app/permissions.js` does `doc(db, 'role_templates', user.role)` — the document ID must exactly match the role string stored in the `users` collection. If the document ID is `finance_user` but the user document has `role: 'finance'`, the permission lookup returns null.
**Why it happens:** `scripts/sync-role-permissions.js` uses old role IDs (`finance_user`, `viewer`) from an earlier version of the system.
**How to avoid:** Use ROLE_ORDER from role-config.js: `['super_admin', 'operations_admin', 'operations_user', 'services_admin', 'services_user', 'finance', 'procurement']`. Document IDs must match exactly.
**Warning signs:** Navigation appears without any permission filtering; console shows `[Permissions] Role template not found: finance` when logged in as a finance role user.

### Pitfall 3: Running Script on Production
**What goes wrong:** Script writes role_templates to production `clmc-procurement`, which already has correct data, potentially overwriting customised permissions.
**Why it happens:** Developer forgot to check which environment is active, or started server from wrong directory.
**How to avoid:** Before pasting the script, verify the yellow DEV ENVIRONMENT banner is visible at the bottom of the page. The script should also check `window.location.hostname === 'localhost'` and abort if not.
**Warning signs:** No yellow banner visible on the page. `db` in the browser console resolves to the prod project.

### Pitfall 4: Super Admin users Document Already Exists (or Does Not)
**What goes wrong:** 53.1 Task 3 Step 3 instructed the developer to create the Super Admin users document manually in Firebase Console. If this was done, the seed script must NOT overwrite it (that would reset the name/email). If it was NOT done, the developer cannot log in to run the seed script.
**Why it happens:** 53.1's manual step was not tracked in code — there is no committed record of whether the users document was created.
**How to avoid:** The seed script should check if the Super Admin users document exists before writing it, OR skip the users document entirely (the developer creates it interactively through Firebase Console as part of the checkpoint). The invitation_codes seed is the key enabler — it allows new dev users to register without a pre-existing users document.
**Warning signs:** Login fails on localhost with "Your account information is missing" — the `users` document does not exist for the Firebase Auth user.

### Pitfall 5: Invitation Code Field Mismatch
**What goes wrong:** Registration fails with "Invalid or already used invitation code" even though the code was seeded.
**Why it happens:** `validateInvitationCode()` in `app/auth.js` queries `where('code', '==', code)` AND `where('status', '==', 'active')`. If the seeded document uses a different field name (e.g., `invitation_code` instead of `code`) or a different status value (e.g., `'valid'` instead of `'active'`), the query returns empty.
**How to avoid:** Use the exact schema from app/auth.js line 47-51: field `code` (string), field `status: 'active'`. This query also requires a composite index — but the index was already deployed to dev in Phase 53.1 (`firestore.indexes.json` covers this query).
**Warning signs:** Registration form shows "Invalid or already used invitation code" immediately after entering the correct code.

## Open Questions

1. **Is the Super Admin users document already in dev Firestore?**
   - What we know: 53.1 Task 3 Step 3 instructed the developer to manually create it in Firebase Console. The 53.1 summary says "End-to-end dev environment verified... login succeeds with dev Firebase credentials." Login requires a `users` doc to exist — so it almost certainly exists.
   - What's unclear: What exact fields were populated (did they include `full_name`, `status: 'active'`?).
   - Recommendation: Plan Task 1 should verify the users document exists and has `status: 'active'` and `role: 'super_admin'`. The seed script should NOT overwrite it. If missing, the human must create it via Firebase Console before running the seed script.

2. **Do the role_templates documents already exist from the 53.1 ad-hoc seeding?**
   - What we know: The 53.1 summary says "role_templates seeded to dev Firestore via browser console" during Task 3. But this was ad-hoc without a defined script.
   - What's unclear: Whether all 7 correct role documents exist, whether `services` tab is included, whether role IDs match the current ROLE_ORDER.
   - Recommendation: The seed script uses `setDoc` (full overwrite, not merge) — this is safe to run even if documents already exist. Running it normalises them to the correct schema.

3. **Permission defaults: match prod exactly or use reasonable dev defaults?**
   - What we know: Production role_templates may have been customised via the Role Configuration UI over time.
   - What's unclear: Whether the developer needs an exact copy of prod permissions in dev, or whether default values are sufficient.
   - Recommendation: Use reasonable defaults (documented in Code Examples above). The developer can use the Role Configuration view to adjust permissions after seeding. An exact prod copy is not needed for dev work.
