# Phase 51.1: Data Migration - Context

**Gathered:** 2026-03-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Import real production data (projects, services, clients) into Firestore from CSV files after test data has been wiped. This is a one-time migration script using Firebase Admin SDK, following the same patterns as backup.js/wipe.js from Phase 50-51.

</domain>

<decisions>
## Implementation Decisions

### Data source & format
- User exports Excel to CSV manually before running the script
- One CSV file per collection: projects.csv, services.csv, clients.csv
- File paths passed as CLI arguments (e.g., `node scripts/import.js --projects ./projects.csv --services ./services.csv --clients ./clients.csv`)
- All codes (project_code, service_code, client_code) are already present in the CSV data — script does NOT generate codes
- Status column exists in Excel — script reads and maps it (not hardcoded to 'active')

### Collection scope
- Import 3 collections: projects, services, clients
- Suppliers entered manually through the app later — NOT imported
- MRFs, PRs, POs, transport requests: fresh start — no historical procurement data
- User-project assignments done manually via Admin > Assignments UI — NOT part of import
- Invitation codes managed through the app — NOT part of import
- Users collection already populated (7 users) — untouched by import

### Data relationships & import order
- Projects reference clients via client_code column
- Services reference clients via client_code column
- Import order matters: clients FIRST, then projects + services
- Script must cross-validate all client_code references before writing anything

### Validation & error handling
- Fail the whole import on first bad row — nothing gets written until all data validates
- Dry-run mode (--dry-run): parses all files, validates, shows counts and sample data, writes nothing
- Reject duplicates: fail if any code already exists in Firestore (expects clean post-wipe database)
- Cross-validate references: all client_codes in projects/services must exist in clients CSV

### Run mode & safety
- Dry-run mode like wipe.js for safe preview
- Fail-fast on any validation error (bad data, missing fields, orphaned references, duplicates)
- Same Admin SDK init pattern as backup.js/wipe.js (serviceAccountKey.json already present)

### Claude's Discretion
- CSV parsing library choice
- Exact field mapping from CSV headers to Firestore document fields (discover from CSV headers during planning)
- Batch write strategy (batched commits like wipe.js)
- Output formatting (counts, tables, progress)
- Whether to add a confirmation gate before live import (like wipe.js WIPE prompt)

</decisions>
