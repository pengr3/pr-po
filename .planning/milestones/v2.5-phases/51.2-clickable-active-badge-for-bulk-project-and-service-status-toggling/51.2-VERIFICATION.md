### Human Verification Required

#### 1. View-Only Badge Behavior

**Test:** Log in with a user account that has view-only permission for projects and services (not super_admin or operations_admin/services_admin). Navigate to the Projects list (#/projects) and Services list (#/services).
**Expected:** Active/Inactive badges appear as plain colored labels — no pointer cursor on hover, no scale animation, no confirmation prompt on click.
**Why human:** The `canEditTab()` function reads runtime role state from the auth session. The ternary gate is correct in code but real permission boundary behavior requires a running app with a view-only user session.

#### 2. Badge Click Does Not Trigger Row Navigation

**Test:** Log in as an admin. In the Projects list, click directly on a green "Active" badge. Observe whether a confirmation dialog appears AND whether the browser navigates away.
**Expected:** Confirmation dialog appears ("Deactivate this project?"). The browser stays on the Projects list page. After confirming, the badge updates to "Inactive" in real-time.
**Why human:** `event.stopPropagation()` is present in the template string but the interaction between `<tr onclick>` and the child `<span onclick>` with stopPropagation requires live browser event handling to confirm.

#### 3. Rapid Bulk Toggling Behavior

**Test:** In the Projects list, quickly click the Active/Inactive badge on 3-4 different rows, confirming each dialog rapidly.
**Expected:** Each toggle fires independently. All updates reflect correctly via the onSnapshot listener. No badges get stuck or show incorrect state.
**Why human:** Concurrent async Firestore operations and onSnapshot re-render timing cannot be verified statically.
