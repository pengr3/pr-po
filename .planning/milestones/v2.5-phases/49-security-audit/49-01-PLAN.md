## Goal
All client-side code is hardened against XSS and injection. User-supplied data rendered via innerHTML is escaped with `escapeHTML()`, and no eval/Function patterns exist.

## Context
- 113 innerHTML usages across 17 files — each must be reviewed; user-supplied data escaped with `escapeHTML()`
- No eval() or new Function() patterns found in codebase (confirmed via grep)
- No external sanitization library — lightweight escapeHTML() in utils.js

## Decisions (from CONTEXT.md — non-negotiable)
- Create `escapeHTML()` utility in `app/utils.js` — no external library (no DOMPurify)
- Apply `escapeHTML()` to all user-supplied data before innerHTML insertion
- No refactoring of inline onclick handlers to addEventListener — out of scope
- Per-innerHTML decision on textContent vs escapeHTML — use textContent where content is plain text, escapeHTML for mixed HTML content

1. Add `escapeHTML(str)` function to `app/utils.js`:
   ```javascript
   /**
    * Escape HTML special characters to prevent XSS
    * @param {string} str - String to escape
    * @returns {string} Escaped string safe for innerHTML
    */
   export function escapeHTML(str) {
       if (str == null) return '';
       return String(str)
           .replace(/&/g, '&amp;')
           .replace(/</g, '&lt;')
           .replace(/>/g, '&gt;')
           .replace(/"/g, '&quot;')
           .replace(/'/g, '&#039;');
   }
   ```

2. Review every innerHTML usage across 17 view files + router + auth + expense-modal (113 total). For EACH occurrence, apply one of these treatments:
