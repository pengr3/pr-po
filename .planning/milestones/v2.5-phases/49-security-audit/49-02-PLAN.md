## Goal
Firebase Security Rules are verified to have no authorization gaps across all 12 collections and 2 subcollections. Auth edge cases (session expiry, concurrent sessions, role escalation) are reviewed and any gaps addressed in code.

## Context
- `firestore.rules` has 384 lines covering 12 collections + 2 subcollections (edit_history under projects and services)
- Collections: users, role_templates, invitation_codes, projects, clients, mrfs, prs, pos, transport_requests, suppliers, deleted_mrfs, deleted_users, services
- 2 subcollections: projects/{id}/edit_history, services/{id}/edit_history
- Helper functions: isSignedIn(), getUserData(), isActiveUser(), hasRole(), isRole(), isAssignedToProject(), isLegacyOrAssigned(), isAssignedToService()
- `invitation_codes` has open read/update rules (allow: true) — this is accepted risk per CONTEXT.md (intentional design — codes are random, security is in user approval workflow)
- Auth module (app/auth.js) handles: Firebase Auth state changes, user document creation, role-based navigation, token refresh, deactivation detection
- Application roles: super_admin, operations_admin, operations_user, services_admin, services_user, finance, procurement

## Decisions (from CONTEXT.md — non-negotiable)
- `invitation_codes` open read/update rules documented as accepted risk (intentional design)
- Firebase Security Rules audit across all 12 collections — identify and fix any authorization gaps
- Auth edge cases (session expiry, concurrent sessions, role escalation) — review and address gaps found

<task id="B1" type="auto">
<name>Audit Firebase Security Rules for authorization gaps</name>
<files>
  - firestore.rules
</files>
<action>
1. **Systematic audit of each collection's rules against the principle of least privilege.** For every collection, verify:
   - **Read access**: Only the roles that need the data can read it
   - **Write access**: Only the roles that should modify the data can create/update/delete
   - **No overly permissive rules**: No `allow read: if true` or `allow write: if true` except where explicitly justified
   - **Consistent helper function usage**: All rules use isActiveUser()/hasRole()/isRole() correctly

2. **Collection-by-collection checklist:**

   a. **users** — Verify: get (own doc + admin roles), list (admin roles only), create (own doc + pending status), update (admin roles + own doc), delete (super_admin + deactivated check)

   b. **role_templates** — Verify: read (all active users), write (super_admin only)

   c. **invitation_codes** — Document: read (true) and update (true) as accepted risk. Verify create (super_admin) and delete (super_admin)

   d. **projects** — Verify: read (all active), create/delete (super_admin + operations_admin), update (adds finance). Check edit_history subcollection: read (all active), create (admin roles + finance), update/delete (false)

   e. **clients** — Verify: read (all active), CRUD (super_admin + operations_admin)

   f. **mrfs** — Verify: get (all active), list (scoped by role/project/service), create (dept users + procurement), update (admin roles + finance + procurement), delete (super_admin + operations_admin)

   g. **prs** — Verify: similar scoping to mrfs. Check create roles match workflow (who generates PRs)

   h. **pos** — Verify: create (super_admin + finance only — finance issues POs), update (adds procurement), delete (super_admin only)

   i. **transport_requests** — Verify: similar pattern to mrfs/prs

   j. **suppliers** — Verify: read (all active), CRUD (super_admin + procurement)

   k. **deleted_mrfs** — Verify: read/create (super_admin + operations_admin), update/delete (false — append-only)

   l. **deleted_users** — Verify: get (own doc + super_admin), list (super_admin), create (super_admin), update/delete (false — append-only)

   m. **services** — Verify: get (scoped roles), list (scoped with assignment check for services_user), create/delete (super_admin + services_admin), update (adds services_user with assignment scope). Check edit_history subcollection.

3. **Cross-cutting checks:**
   - Verify no collection is missing rules (Firestore denies by default — missing rules = broken access)
   - Verify all helper functions work correctly together (isActiveUser implies isSignedIn)
   - Check for any rule that allows write without authentication
   - Verify operations_user cannot escalate to operations_admin privileges
   - Verify services_user cannot access operations data and vice versa

4. **Fix any gaps found.** If a rule is too permissive or too restrictive:
   - Too permissive: Tighten to minimum required roles
   - Too restrictive: Widen only if confirmed needed by application workflow
   - Document each change with a comment explaining why

5. **Specific areas of concern to investigate:**
   - Can operations_user delete MRFs they shouldn't? (delete rule allows operations_admin but not operations_user — verify this is correct)
   - Can services_user create PRs? (create rule: super_admin, operations_admin, procurement — services_admin missing? Check if services dept generates PRs)
   - Can finance role access services data? (services collection get rule includes finance)
   - Are there any orphaned rules for collections that no longer exist?
</action>

- No overly permissive rules remain (except documented accepted risks)
- All helper functions are used consistently
- Any gaps found are fixed with explanatory comments
</verify>

<task id="B2" type="auto">
<name>Review auth edge cases and address gaps</name>
<files>
  - app/auth.js
</files>
<action>
1. **Review session handling:**
   - How does the app handle Firebase Auth token expiry? Check if `user.getIdToken(true)` is called on auth state change (line ~211 in auth.js)
   - What happens when the token cannot be refreshed (network error, revoked token)?
   - Is there a periodic token refresh mechanism, or does it rely solely on Firebase SDK auto-refresh?
   - Verify: If token expires during an active session, does the user get gracefully redirected to login?

2. **Review concurrent sessions:**
   - What happens if a user logs in from two browsers/tabs simultaneously?
   - If an admin deactivates a user while they are logged in, how quickly is the session terminated?
   - Check: The onSnapshot listener on the user document (auth.js) should detect status changes in real-time and force logout for deactivated users
   - Verify the deactivation detection code path works correctly

3. **Review role escalation prevention:**
   - Can a user modify their own role via the update rule? Check: users collection update rule allows `request.auth.uid == userId` — this means a user could potentially update their own document including the role field
   - If the self-update rule is too broad, tighten it: either restrict self-update to specific fields (display name, preferences) or add a rule that prevents role changes on self-update
   - Verify: operations_admin cannot promote a user to super_admin
   - Verify: services_admin can only manage services_user roles, not operations roles

4. **Fix any gaps found:**
   - If self-role-escalation is possible via the users collection update rule, add field-level validation in firestore.rules to prevent users from changing their own `role` or `status` fields
   - If deactivation detection has a timing gap, document it as a known limitation
   - If concurrent session handling needs improvement, implement appropriate guards

5. **Specific auth.js code paths to verify:**
   - Auth state change handler (onAuthStateChanged callback)
   - User document real-time listener (onSnapshot on user doc)
   - Deactivation force-logout flow
   - Token refresh on sign-in
   - Redirect behavior for pending/rejected/deactivated users
</action>
<verify>
- Session expiry is handled gracefully (user redirected to login, not left in broken state)
- Concurrent session with deactivation forces logout via onSnapshot listener
- Role escalation is prevented at the Security Rules level (users cannot change their own role/status)
- Any self-update rule in firestore.rules restricts which fields the user can modify
- Auth edge case findings are documented for inclusion in SECURITY-AUDIT.md (Plan C)
</verify>
<done>

- Self-role-escalation blocked in firestore.rules (field-level restriction on role/status for self-update)
