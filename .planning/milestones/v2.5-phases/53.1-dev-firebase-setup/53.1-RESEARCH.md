**Researched:** 2026-03-02

**Confidence:** HIGH

## Summary

This phase adds a second Firebase project (`clmc-procurement-dev`) so developers can work on localhost without touching the production Firestore database. The production app is going live, making dev/prod isolation urgent.

The solution is a runtime hostname detection pattern in `app/firebase.js`. Because this project has no build system, config switching must happen at runtime using `window.location.hostname`. When the hostname is `localhost` or `127.0.0.1`, the app initialises with the dev Firebase config; all other hostnames (Netlify domain) use the prod config. This is an officially documented pattern for the Firebase Web SDK.

Firebase's IndexedDB persistence storage is keyed by `projectId` in its naming convention: `firestore/[persistenceKey]/[projectId]/main`. This means the dev and prod projects are naturally isolated in the browser's IndexedDB even when both are accessed from the same origin. No extra configuration is needed for isolation.

The second concern is the two-terminal requirement: both localhost tabs will hit the same dev Firestore project. The current `persistentSingleTabManager` config will conflict when two browser tabs are open at the same localhost origin. Switching to `persistentMultipleTabManager` for the dev config resolves this. The prod config can keep `persistentSingleTabManager`.

**Primary recommendation:** Extend `app/firebase.js` with a single `isLocal` boolean that selects the dev or prod config at module load time. No external files, no environment variables, no build step required.

### Pattern 1: Runtime Hostname Detection (Official Firebase Pattern)

**What:** Check `window.location.hostname` at module load time to pick the correct config object before calling `initializeApp()`.

**When to use:** Any static SPA needing dev/prod Firebase separation without a build system.

**Example:**
```javascript
// Source: https://firebase.google.com/docs/projects/multiprojects
// "For the Web and Admin SDKs, you can use a runtime check to select
//  development or production configuration variables."

const isLocal = ['localhost', '127.0.0.1'].includes(window.location.hostname);

const prodConfig = {
    apiKey: "AIzaSyAlHcmPmkCk6CKsRbfpHpCheHb2GcLz0Oc",
    authDomain: "clmc-procurement.firebaseapp.com",
    projectId: "clmc-procurement",
    storageBucket: "clmc-procurement.firebasestorage.app",
    messagingSenderId: "946184501660",
    appId: "1:946184501660:web:6559c5de405e72100ab059"
};

const devConfig = {
    apiKey: "<DEV_API_KEY>",
    authDomain: "clmc-procurement-dev.firebaseapp.com",
    projectId: "clmc-procurement-dev",
    storageBucket: "clmc-procurement-dev.firebasestorage.app",
    messagingSenderId: "<DEV_SENDER_ID>",
    appId: "<DEV_APP_ID>"
};

const firebaseConfig = isLocal ? devConfig : prodConfig;
const app = initializeApp(firebaseConfig);
```

### Pattern 2: Multi-Tab Persistence for Dev, Single-Tab for Prod

**What:** Dev needs `persistentMultipleTabManager` because two terminals = two browser tabs on the same localhost. Prod keeps its current `persistentSingleTabManager`.

**Why:** `persistentSingleTabManager` allows only one tab to "own" the IndexedDB persistence at a time. The second tab would fall back to memory-only mode silently, causing confusing behaviour during development. `persistentMultipleTabManager` is the correct setting for multi-tab scenarios.

**Example:**
```javascript
// Source: https://firebase.google.com/docs/firestore/manage-data/enable-offline
const db = initializeFirestore(app, {
    localCache: persistentLocalCache({
        tabManager: isLocal
            ? persistentMultipleTabManager()
            : persistentSingleTabManager()
    })
});
```

### Anti-Patterns to Avoid

- **Hardcoding devConfig as comments only:** If the dev config is commented out, swapping requires editing code in production-deployed files. Use the `isLocal` branch so it is always present but inactive on prod.
- **Using .env files or config.js fetch:** The app has no build system and no server. .env files cannot be injected without a build step. A separate fetched config.js would require an extra HTTP request before Firebase initialises, causing auth flicker.
- **Checking `process.env.NODE_ENV`:** This is undefined in a browser with no build step. Only `window.location.hostname` is reliable here.
- **Sharing one Firebase project with a separate Firestore database named "dev":** Firebase's named-database feature (multi-database) is a paid tier feature and adds complexity. A separate free-tier project is simpler and gives fully independent Auth.

## Common Pitfalls

### Pitfall 1: Dev Config Left with Placeholder Values
**What goes wrong:** Developer forgets to fill in the dev project's real config values. App fails silently or throws "Invalid API key" on localhost.
**Why it happens:** The dev config must be obtained from Firebase Console after creating the second project. The PLAN must include a step to retrieve and fill in the config.
**How to avoid:** PLAN should have a dedicated task: "Create dev Firebase project and copy config values into devConfig."
**Warning signs:** `Firebase: Error (auth/invalid-api-key)` in console on localhost.

### Pitfall 2: Auth Users Do Not Carry Over
**What goes wrong:** Developer tries to log in on localhost with their prod account and gets "user not found."
**Why it happens:** Firebase Auth is per-project. The dev project has its own separate user database. Developer must create a Super Admin account in the dev project.
**How to avoid:** PLAN must include: "Create a Super Admin user in the dev Firebase project via Firebase Console > Authentication."
**Warning signs:** Auth succeeds (Firebase Auth account exists) but Firestore user doc is not found, causing redirect to `/login`.

### Pitfall 3: Security Rules Not Deployed to Dev Project
**What goes wrong:** All Firestore operations return "Missing or insufficient permissions" in dev.
**Why it happens:** A new Firebase project starts with deny-all rules by default. The dev project has never had rules deployed.
**How to avoid:** Deploy `firestore.rules` to the dev project immediately after creating it: `firebase deploy --only firestore:rules --project clmc-procurement-dev`
**Warning signs:** Every Firestore read/write returns permission error despite correct auth.

### Pitfall 4: Firestore Indexes Not Deployed to Dev Project
**What goes wrong:** Compound queries that rely on composite indexes fail with "The query requires an index" in dev.
**Why it happens:** `firestore.indexes.json` has 4 composite indexes. New projects start with no custom indexes.
**How to avoid:** Deploy indexes to dev project: `firebase deploy --only firestore:indexes --project clmc-procurement-dev`
**Warning signs:** Console error "FirebaseError: The query requires an index."

### Pitfall 5: persistentSingleTabManager with Two Dev Tabs
**What goes wrong:** One terminal's browser tab loses Firestore persistence silently. Offline-first cache stops working for that tab.
**Why it happens:** `persistentSingleTabManager` allows exactly one tab to "own" the IndexedDB lock. Second tab falls back to memory-only.
**How to avoid:** Use `persistentMultipleTabManager()` for the dev config. This is already addressed in Pattern 2.
**Warning signs:** Second tab works but real-time listeners fire more slowly or cache data appears stale.

### Pitfall 6: .firebaserc Default Project Gets Changed
**What goes wrong:** Developer runs `firebase use clmc-procurement-dev` to deploy rules and forgets to switch back. Next Netlify deploy targets dev project.
**Why it happens:** `firebase use` writes to `.firebaserc` and changes the `default` alias.
**How to avoid:** Use the `-P` flag in all CLI commands instead of `firebase use`: `firebase deploy --only firestore:rules --project clmc-procurement-dev`. Never change the default. Add `.firebaserc` check to the PLAN's verification steps.
**Warning signs:** `firebase deploy` (without flags) goes to the wrong project.

## Open Questions

1. **Dev project name/ID**
   - What we know: The requirement says `clmc-procurement-dev or similar`
   - What's unclear: The exact project ID the developer will choose in Firebase Console
   - Recommendation: PLAN should have the developer confirm the exact project ID before filling in devConfig. The PLAN task should say "Create dev project, note the exact Project ID."

2. **Firebase Console region for dev project**
   - What we know: Prod is on default region (likely us-central1)
   - What's unclear: Whether the developer needs to match the region exactly
   - Recommendation: Use the same region as prod. It doesn't affect functionality but keeps things consistent.

3. **Invitation codes for dev users**
   - What we know: The app requires invitation codes to register. The invitation_codes Firestore security rule allows open read/update (accepted risk documented in firestore.rules).
   - What's unclear: Whether the developer wants to create test invitation codes in dev or temporarily use a looser initial setup
   - Recommendation: Create at least one invitation code document directly in dev Firestore console, or have the Super Admin create one after first login. The existing rules work correctly for dev â€” no change needed.
