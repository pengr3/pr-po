## Phase Details

### Phase 49: Security Audit
**Goal**: The application is verified safe for production — known XSS vectors addressed, Security Rules cover all collections with no gaps, and CSP headers appropriate for the live environment.
**Depends on**: Phase 48
**Requirements**: SEC-01, SEC-02, SEC-03, SEC-04, SEC-05, SEC-06
**Success Criteria** (what must be TRUE):
  1. All client-side innerHTML rendering of user-supplied data is reviewed and unsafe patterns are documented or fixed
  2. Firebase Security Rules are audited across all 12 collections and any authorization gaps are resolved
  3. No sensitive data (API keys, PII) appears in browser console logs in normal operation
  4. Auth edge cases (session expiry, concurrent sessions, role escalation) are reviewed and any gaps addressed
  5. CSP headers in Netlify config are reviewed and tightened for production with no legitimate functionality blocked
**Plans**: 5 plans
- [x] 49-01-PLAN.md — XSS Protection (escapeHTML utility + innerHTML hardening)
- [x] 49-02-PLAN.md — Firebase Security Rules audit + auth edge cases
- [x] 49-03-PLAN.md — CSP hardening + security audit report
- [x] 49-04-PLAN.md — Console log cleanup (sensitive data exposure)
- [ ] 49-05-PLAN.md — Gap closure: remaining XSS escaping (9 locations)

### Phase 50: Database Safety
**Goal**: A reliable backup and restore workflow exists so data can be exported before the wipe, verified for integrity, and restored if something goes wrong.
**Depends on**: Phase 49
**Requirements**: DBS-01, DBS-02, DBS-03
**Success Criteria** (what must be TRUE):
  1. Running the backup script exports all Firestore collections to local JSON files with no manual steps beyond execution
  2. The integrity check script identifies any orphaned references (e.g., MRFs with no matching project code) and schema inconsistencies across collections
  3. A documented restore procedure exists and has been verified to successfully re-import exported JSON back into Firestore
**Plans**: 2 plans
- [ ] 50-01-PLAN.md — Backup export & restore scripts (Firebase Admin SDK)
- [ ] 50-02-PLAN.md — Data integrity verification script

### Phase 51: Data Wipe
**Goal**: A safe, standalone script can clear all test data from Firestore collections (excluding users) before real data is loaded, with enough protection to prevent accidental execution.
**Depends on**: Phase 50
**Requirements**: WIP-01, WIP-02, WIP-03
**Success Criteria** (what must be TRUE):
  1. Running the wipe script in dry-run mode lists every document that would be deleted, with counts per collection, without deleting anything
  2. Running the wipe script in live mode requires a typed confirmation before any deletion begins
  3. After confirmed execution, all targeted collections (mrfs, prs, pos, transport_requests, suppliers, clients, projects, services, deleted_mrfs, invitation_codes) are empty while the users collection is untouched
