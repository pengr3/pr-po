<objective>
Wire the project-list and project-detail UI to (1) display clientless projects with em-dash in Code/Client columns, (2) resolve clientless project deep links via Firestore doc ID, (3) render an editable client picker on the detail page when no code has been issued, and (4) execute the one-pass batched backfill across MRFs/PRs/POs/TRs/RFPs when a client is assigned, gated by a confirmation modal — with the project doc written LAST in the batch sequence so a partial-batch failure leaves the project un-marked-issued (allowing safe retry).

Purpose: This is the user-facing issuance flow. Without it, clientless projects can be created (Plan 01) and used by the MRF flow (Plan 02) but can never receive a code. This plan closes the loop. The "project doc LAST" write order addresses the REVIEWS.md MEDIUM concern about partial backfill failure — if the browser crashes or batch 2/3 fails to commit, the project still has `project_code: null` so the user simply re-runs issuance, no orphaned/partially-coded children blocking retry.

Output: A user with a clientless project assigns a client, confirms a modal showing "This will generate CLMC-{CLIENT}-2026### and apply it to N existing records", and the children + project doc receive the new code in one chunked atomic-per-batch sequence (children commit before project marker). Em-dash rendering and doc-ID URL routing make clientless projects visible and addressable in the meantime. Project doc carries `is_issued: true` after successful issuance for future security rule / UI use.
</objective>

Firestore writeBatch limit is 500 writes per batch. Typical clientless projects will have very few children (<20), but the implementation MUST chunk into multiple batches if total writes exceed 500.

Children to backfill (each filtered by `where('project_id', '==', currentProject.id)`):
- /mrfs/* — write project_code, client_code
- /prs/* — write project_code, client_code
- /pos/* — write project_code, client_code
- /transport_requests/* — write project_code, client_code
- /rfps/* — write project_code (RFP IDs follow RFP-{PO-ID}-{n} format and don't depend on project_code, but the RFP query in project-detail.js:692 uses project_code so backfill is required for the post-issuance Financial Summary to find them)

Existing clients data shape (from clientsData usage in projects.js):

project-detail.js currently does NOT load clients. We will lazy-load them ONCE when rendering the client-select control on a clientless project.

Atomicity strategy (REVIEWS.md MEDIUM concern fix):
The backfill commits in chunks of 500. Within the writes[] array, the project doc is pushed LAST after all child collection forEach loops. Since chunks are sliced sequentially from writes[0..N-1], the project doc lives in the FINAL chunk — which is committed LAST. If batch K fails for any reason (network, browser close, rule rejection), batches 1..K-1 have committed and batch K..final remain uncommitted. Critically, the project doc commit is in the FINAL batch, so any failure before that means the project still has `project_code: null` and the user can safely re-run runCodeIssuance — children that already received the code will simply be re-written with the same code (idempotent), and children not yet updated will get it on the retry.
