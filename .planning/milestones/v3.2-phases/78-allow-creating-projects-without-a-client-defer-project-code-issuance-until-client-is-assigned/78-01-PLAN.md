<objective>
Relax the project-creation foundation so a project can be saved without a client. Update `addProject()` in `app/views/projects.js` to make `client_id` optional, skip `generateProjectCode()` when no client is selected, and write a project document with `client_id: null`, `client_code: null`, `project_code: null`. Update `firestore.rules` `/projects/{projectId}` update rule to ENFORCE D-12 immutability at the database level — once `project_code` is non-null, any update that changes `project_code`, `client_id`, or `client_code` is rejected server-side.

Purpose: This is the foundation Wave-1 plan that downstream waves depend on. Wave 2 plans (MRF dropdown, projects list, project-detail issuance flow) all rely on the existence of clientless project docs in Firestore. The Firestore rule update closes the security gap noted in REVIEWS.md (Gemini MEDIUM concern): UI-level locking (`saveField` guard) alone is insufficient — a malicious or errant direct Firestore write could overwrite locked fields after issuance. The rule enforces lock at the DB level.

Output: A user (super_admin or operations_admin) can open Add Project, leave the Client dropdown unselected, fill name + statuses + personnel, and create a project. The project shows up in the Firestore `projects` collection with null client/code fields. After Plan 03's runCodeIssuance writes a `project_code`, any subsequent update attempt (via UI bypass or direct Firestore write) that tries to change `project_code`, `client_id`, or `client_code` is rejected with a Firestore permission error.
</objective>
