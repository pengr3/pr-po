# Phase 73.2: Mobile-Optimize Finance Modals — Research

**Researched:** 2026-04-15
**Domain:** CSS responsive design for data-heavy modals — pure JS SPA, no framework
**Confidence:** HIGH — all findings derived from direct codebase inspection

---

## Summary

Phase 73 and 73.1 mobile-optimized the Finance tab content (tab switcher, card layouts for all 5 tables). Phase 73.2 now targets the two modals that open from within the Finance view: the **Financial Breakdown Modal** (`app/expense-modal.js`) and the **PR/TR Details Modal** (`app/views/finance.js`). Both modals have dense grid scorecard layouts and wide multi-column tables inside them that overflow on narrow screens (375–414px). The existing global modal mobile rules (`width: 95%; max-height: 90vh`) and `modal-footer` column-stack are already applied — what is missing is treatment of the **content inside** the modals.

The Financial Breakdown Modal is the more complex target: it contains two rows of inline-styled grid scorecards (2-col Budget row, 3-col Category row, 2-col Projected Cost row) plus sub-tabs with tables inside collapsible category cards. The PR Details Modal has a `.modal-details-grid` (2-col) that collapses cleanly via a CSS-only fix, plus a `.modal-items-table` with 6 columns that needs a scroll wrapper. The TR Details Modal shares the same `#prModal` shell but its body uses raw inline `grid-template-columns: repeat(2, 1fr)` (not the `.modal-details-grid` class) and its items table has no scroll wrapper either.

**Primary recommendation:** Two targeted changes — (1) CSS-only responsive rules for the modal detail grids and (2) wrapping the modal items tables in `overflow-x: auto` divs in the JS template strings. The Financial Breakdown Modal additionally needs its scorecard grids made responsive via CSS classes (replacing or augmenting the inline `display: grid` styles). No data logic changes needed.

- `modal-content`: global CSS gives `width: 90%; max-width: 900px` desktop, `width: 95%; max-height: 90vh` on mobile. OK.
- `modal-footer`: already column-stacks on mobile (components.css L1350). OK.

**PR body content** (finance.js `viewPRDetails()`, lines 3389–3461):

| Element | Current markup | Mobile problem |
|---------|---------------|----------------|
| Detail metadata grid | `<div class="modal-details-grid">` | `.modal-details-grid` CSS: `grid-template-columns: repeat(2, 1fr)` hardcoded — no mobile override. | Items table | `<table class="modal-items-table">` | 6 columns (Item, Category, Qty, Unit Cost, Supplier, Subtotal). No scroll wrapper around it. On 375px the table will overflow the modal body. **Fix needed:** wrap in `overflow-x: auto` div. |

**TR body content** (finance.js `viewTRDetails()`, lines 3539–3621):

| Element | Current markup | Mobile problem |
|---------|---------------|----------------|

### Modal 2: Financial Breakdown Modal (`#expenseBreakdownModal` in expense-modal.js)

**Shell structure** (expense-modal.js line 601, dynamically injected):
```html
<div id="expenseBreakdownModal" class="modal active">
  <div class="modal-content" style="max-width: 900px;">
    <div class="modal-header">...</div>
    <div class="modal-body">...</div>
  </div>
</div>
```
- Has `style="max-width: 900px;"` inline override. Global `width: 95%` mobile rule still applies. OK.
- No `.modal-footer` here — close button is in the header, no action bar. OK.

**Problem areas inside `.modal-body`:**

#### Scorecards Row 1 — Budget + Remaining Budget (line 612)
```html
<div style="display: grid; grid-template-columns: 1fr 1fr; gap: 1rem; margin-bottom: 1.5rem;">
```
2 columns, inline style. On 375px, each card is ~155px wide (after 95% modal width minus padding). Budget values are `font-size: 1.5rem` — at 155px this is readable but cramped. **Recommended fix:** Add class `em-scorecard-row-2` and override to `grid-template-columns: 1fr` at 480px so cards stack. Or use `repeat(auto-fit, minmax(200px, 1fr))` inline — auto-collapses below 400px.

#### Scorecards Row 2 — Category breakdown (line 626)
```html
<div style="display: grid; grid-template-columns: repeat(3, 1fr); gap: 1rem; margin-bottom: 1.5rem;">
```
3 columns. On 375px, each card is ~100px effective content width. The `font-size: 1.5rem` amounts will wrap. **Significant problem.** Fix: `repeat(auto-fit, minmax(140px, 1fr))` inline collapses to 2 or 1 column automatically; or add class and CSS override to `grid-template-columns: 1fr` at 480px.

#### Scorecards Row 3 — Projected Cost + Remaining Payable (line 646, conditional)
```html
<div style="display: grid; grid-template-columns: 1fr 1fr; gap: 1rem; margin-bottom: 2rem;">
```
Same as Row 1 but with `font-size: 2rem` values. More aggressive overflow risk. Fix same as Row 1.

#### Tab Navigation Bar (line 659)
```html
<div style="border-bottom: 2px solid #e5e7eb;">
  <button class="expense-tab active" ...>By Category</button>
  <button class="expense-tab" ...>Transport Fees</button>
  <button class="expense-tab" ...>Payables</button>
</div>
```
3 inline-flex tab buttons. On 375px they may clip if the modal body padding is generous. **Fix:** Add `overflow-x: auto; -webkit-overflow-scrolling: touch; white-space: nowrap` to the tab bar container. Best approach: add class `em-tab-bar` to the div and target via CSS.

#### Category/Transport/Payables tab content — tables inside collapsible cards (lines 449–471, 497–510, etc.)
Inside each `category-card` the `.category-items` div contains a `.modal-items-table`. These tables have 5 columns (Item, Qty, Unit, Unit Cost, Subtotal) or 4 columns (Particulars, Status, Total Payable, Total Paid for Payables tab). No `overflow-x: auto` wrapper. **Fix:** Add `overflow-x: auto; -webkit-overflow-scrolling: touch` to `.category-items` in the CSS.

## Architecture Patterns

### Pattern 1: CSS class override for `.modal-details-grid` (components.css or views.css)

`.modal-details-grid` is defined in `components.css` with `grid-template-columns: repeat(2, 1fr)`. Add a responsive override to the existing `@media (max-width: 768px)` block in `components.css`:

```css
/* In components.css @media (max-width: 768px) block */
.modal-details-grid {
    grid-template-columns: 1fr;
}
```

This is a **global** class used in the PR Details modal. Verify no other modal uses this class in a way that should NOT collapse on mobile. Current search: only `viewPRDetails()` in finance.js uses `.modal-details-grid` (confirmed by grep — no other occurrences in other view files). Safe to make global.

### Pattern 2: TR Details grid — change inline style to use `.modal-details-grid` class (JS change)

The TR details modal (viewTRDetails) uses raw `style="display: grid; grid-template-columns: repeat(2, 1fr)..."` instead of the `.modal-details-grid` class. Fix: replace the inline style with class attribute to get Pattern 1's fix for free.

```javascript
// Before (finance.js viewTRDetails, line 3540):
<div style="display: grid; grid-template-columns: repeat(2, 1fr); gap: 1rem; margin-bottom: 1.5rem;">

// After:
<div class="modal-details-grid">
```

Note: The TR grid items do NOT use `.modal-detail-item` or `.modal-detail-label` / `.modal-detail-value` — they use raw `<div>` with inline label/value styles. To reuse `.modal-details-grid` cleanly, just add the class to the outer container and leave inner divs as-is. The `.modal-details-grid` CSS only defines the grid container properties, not the children.

### Pattern 3: Scroll wrapper for modal items tables (JS change)

Wrap the `<table class="modal-items-table">` in `viewPRDetails` and the bare `<table>` in `viewTRDetails` with a scroll-containing div:

```javascript
// viewPRDetails (finance.js ~line 3432):
// Before:
<table class="modal-items-table">

// After:
<div style="overflow-x: auto; -webkit-overflow-scrolling: touch;">
<table class="modal-items-table">

</table>
</div>

// viewTRDetails (finance.js ~line 3581):
// Before:

  <h4...>Service Items</h4>
  <table style="width: 100%; ...">

// After:
<div style="margin: 1.5rem 0; padding: 1.5rem; background: #f8f9fa; border-radius: 8px;">
  <h4...>Service Items</h4>
  <div style="overflow-x: auto; -webkit-overflow-scrolling: touch;">
  <table style="width: 100%; ...">
  ...
  </table>
  </div>
```

### Pattern 4: Financial Breakdown Modal scorecard grids — add CSS classes, override in media query (expense-modal.js + views.css)

The three scorecard row divs in expense-modal.js use `style="display: grid; grid-template-columns: ..."`. Since these are inline styles, they cannot be overridden by a media query unless the element also carries a CSS class. The fix:

1. Add classes to the scorecard row divs in `expense-modal.js`:
   - Row 1 (Budget/Remaining): add class `em-scorecard-2col`
   - Row 2 (Material/Transport/Subcon): add class `em-scorecard-3col`
   - Row 3 (Projected/Remaining Payable): add class `em-scorecard-2col`

2. Add to `views.css` Finance mobile block:
```css
@media (max-width: 768px) {
    .em-scorecard-2col,
    .em-scorecard-3col {
        grid-template-columns: 1fr !important;
    }
}
```

The `!important` is needed because inline styles override CSS selectors without it.

**Alternative without `!important`:** Replace inline `style="display: grid; grid-template-columns: ..."` with `class="em-scorecard-2col"` and define the base grid in a non-media-query CSS rule. This is cleaner. Since expense-modal.js generates the entire HTML, adding a class and moving grid definitions to CSS is straightforward.

**Recommended approach:** Add class + move grid definition to CSS (avoids `!important`). This matches how Phase 73.1 handled `.fc-card-list` (defined in CSS, not inline).

### Pattern 5: Financial Breakdown Modal tab bar — scroll on narrow screens

Add class `em-tab-bar` to the tab navigation container div in expense-modal.js and add to views.css:

```css
.em-tab-bar {
    overflow-x: auto;
    -webkit-overflow-scrolling: touch;
    white-space: nowrap;
}
.em-tab-bar .expense-tab {
    white-space: nowrap;
    display: inline-block;
}
```

This applies at all viewport widths (no media query needed) — it's a safe scroll-enabling enhancement that only activates when tabs overflow.

### Pattern 6: Financial Breakdown Modal category items table — overflow scroll (views.css)

The `.category-items` div holds `.modal-items-table`. Add to views.css (not inside media query — always safe to scroll):

```css
#expenseBreakdownModal .category-items {
    overflow-x: auto;
    -webkit-overflow-scrolling: touch;
}
```

Or within the 768px block if desktop scrollbar appearance is undesirable.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Modal scorecard responsive layout | JS viewport detection + re-render | CSS `grid-template-columns` override with class | CSS responds to orientation changes automatically |
| Table horizontal scroll in modal | JS scroll listener with shadow | `overflow-x: auto` div + `-webkit-overflow-scrolling: touch` | Native browser scroll, zero JS |
| Tab bar overflow | JS tab "more" dropdown | `overflow-x: auto; white-space: nowrap` on tab container | Native horizontal scroll, simpler |

## Common Pitfalls

### Pitfall 1: `!important` required for inline style overrides

**What goes wrong:** CSS media query rule `.em-scorecard-3col { grid-template-columns: 1fr }` has no effect because the element also has `style="grid-template-columns: repeat(3, 1fr)"`. Inline styles win the specificity battle.

**Why it happens:** CSS specificity — inline `style` attribute always beats class/selector rules, even in a media query.

**How to avoid:** When adding a CSS class to elements that currently have inline grid styles, remove the `grid-template-columns` from the inline style and define it in the CSS class instead. If the inline style MUST stay for other properties, add `!important` only to the specific property being overridden.

**Warning signs:** Grid doesn't change to single column on mobile despite adding the class and CSS rule.

### Pitfall 2: TR Details Modal body does NOT use `.modal-detail-item` children

**What goes wrong:** Planner assumes the TR body can reuse all `.modal-details-grid` child CSS. Tries to add `.modal-detail-item` / `.modal-detail-label` / `.modal-detail-value` classes to TR detail divs and some styling breaks.

**Why it happens:** The TR `viewTRDetails()` template was written with raw inline styles for the child cells (label as `font-size: 0.75rem; font-weight: 600; color: #5f6368;`, value as `<strong>` or plain div). The PR `viewPRDetails()` uses the proper CSS classes for children. They're different.

**How to avoid:** Only add the class to the OUTER container `div`. Do not modify the child div structure inside viewTRDetails — the inner styling works fine even without the classes. Only the outer grid columns need to collapse on mobile.

### Pitfall 3: Financial Breakdown Modal is dynamically injected — CSS targeting by ID works

**What goes wrong:** Someone writes `.modal-body .category-items` thinking it will scope correctly, but this also targets other modals that might have `.category-items` inside `.modal-body`.

**Why it happens:** `.category-items` is a class potentially reused elsewhere; generic scoping via `.modal-body` is too broad.

**How to avoid:** Always scope via `#expenseBreakdownModal .category-items` to target only the Financial Breakdown Modal. The modal is injected with a stable `id="expenseBreakdownModal"`.

### Pitfall 4: `em-scorecard-*` CSS classes need to be defined for desktop too

**What goes wrong:** Classes `em-scorecard-2col` and `em-scorecard-3col` are added to elements but only have rules inside the `@media (max-width: 768px)` block. The desktop layout breaks because the `grid-template-columns` from the inline style was removed.

**Why it happens:** Moving `grid-template-columns` from inline to CSS requires defining it both for desktop (outside media query) and mobile (inside media query).

**How to avoid:** Always define the desktop rule first outside the media query:
```css
.em-scorecard-2col { display: grid; grid-template-columns: 1fr 1fr; gap: 1rem; margin-bottom: 1.5rem; }
.em-scorecard-3col { display: grid; grid-template-columns: repeat(3, 1fr); gap: 1rem; margin-bottom: 1.5rem; }

@media (max-width: 768px) {
    .em-scorecard-2col, .em-scorecard-3col { grid-template-columns: 1fr; }
}
```

Then remove matching properties from the inline `style` attribute on the element.

### Pitfall 5: Financial Breakdown Modal body padding on mobile

**What goes wrong:** The `.modal-body` has `padding: 2rem` (components.css L665). On a 375px viewport with `width: 95%` modal (357px), the body is only 357px. With `2rem` (32px) padding on each side, effective content width = 293px. The 3-column scorecard at 293px would give each card ~91px — very tight even after the fix.

**Why it happens:** The global `2rem` modal body padding is desktop-sized.

**How to avoid:** Add a mobile override in the `@media (max-width: 768px)` block in `components.css`:
```css
.modal-body {
    padding: 1rem;
}
```
This gives 375px * 0.95 = 356px modal, minus 2 * 16px = 324px effective content width. Enough for readable stacked cards.

This is a **global** change — verify it doesn't compress other modal body content unacceptably. All other modals in the app (PR Details, Rejection, Approval) already have content that benefits from the extra padding reduction on mobile.

## Open Questions

1. **Should `.modal-body { padding: 1rem }` be applied globally or only to the Financial Breakdown Modal?**
   - What we know: The global `.modal-body` padding is `2rem` (components.css L665). All Finance modals (PR Details, TR Details, Rejection, Approval) would benefit from padding reduction on mobile.
   - What's unclear: Other views' modals (Procurement MRF detail modal, Projects PO detail modal) — do they need 2rem padding on mobile?
   - Recommendation: Apply globally to `.modal-body` in components.css mobile block — stacking cards with 1rem padding is universally better on mobile. Low risk.

2. **Should the Payables tab in the Financial Breakdown Modal receive the same table scroll treatment as other tabs?**
   - What we know: The Payables tab `#expBreakdownPayablesTab` contains a `.category-card` with `.category-items` div holding a `.modal-items-table` with 4 columns (PARTICULARS, STATUS, TOTAL PAYABLE, TOTAL PAID). The PARTICULARS column contains `PO-ID — Supplier Name` strings which can be long.
   - Recommendation: Yes — the `#expenseBreakdownModal .category-items { overflow-x: auto }` rule (Pattern 6) covers all three tabs uniformly.

3. **Should the Financial Breakdown Modal scorecard grid changes be scoped to a breakpoint smaller than 768px?**
   - What we know: The 3-column scorecard is problematic below ~500px. At 768px (tablet), 3 columns in a 95%-wide modal (729px) gives ~220px per card which is fine.
   - What's unclear: Whether the user wants tablet layout preserved (3-col) and only phone layout collapsed (1-col).
   - Recommendation: Use `max-width: 480px` for the scorecard collapse instead of 768px. This preserves 3-col on tablets (769–480px range stays unchanged) and only collapses on phones. Add to the research plan for the planner.
