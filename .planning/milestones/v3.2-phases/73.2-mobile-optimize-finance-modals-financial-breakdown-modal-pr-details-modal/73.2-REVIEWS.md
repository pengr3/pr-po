# Cross-AI Plan Review: Phase 73.2 - Mobile-optimize Finance modals

## 1. Summary
The implementation plan for Phase 73.2 is well-structured and follows a surgical approach to mobile optimization. By transitioning from brittle inline styles to CSS-based class overrides and implementing horizontal scroll wrappers for data-heavy tables, the plan significantly improves the usability of the Finance module on small viewports. The strategy of using the existing 768px breakpoint aligns with project conventions, and the decoupling of layout logic from template strings in `expense-modal.js` is a positive step toward better maintainability.

## 2. Strengths
*   **Separation of Concerns:** Moving complex grid layouts from inline styles in `expense-modal.js` to CSS classes (`.em-scorecard-2col`, etc.) makes the layout much easier to manage and debug.
*   **Non-Destructive Enhancements:** The use of horizontal scroll wrappers (`overflow-x: auto`) for tables is the correct pattern for data-heavy engineering workflows where collapsing columns into cards would sacrifice readability and data density.
*   **Surgical CSS:** Targeting only the `max-width: 768px` block ensures that the desktop experience remains completely untouched, fulfilling the requirement.
*   **Consistency:** Standardizing the TR Details grid with the `.modal-details-grid` class used by PR Details promotes visual harmony across the Finance module.

## 3. Concerns
*   **Table Content Squeeze (MEDIUM):** Simply wrapping a table in an `overflow-x: auto` div might not trigger scrolling if the table's internal content is allowed to wrap aggressively. Without a `min-width` on the table or `white-space: nowrap` on table headers/cells, the table may simply become extremely tall and narrow on mobile instead of scrolling horizontally.
*   **TR Details Visual Shift (MEDIUM):** The plan notes that `.modal-details-grid` adds `background: #fff`, `padding: 2rem`, `border-radius: 12px`, and `box-shadow`. The original TR inline style lacked these. While this improves consistency, if the TR modal already has a containing card/background, you may end up with "nested cards" (white on white with double padding), which looks amateurish.
*   **Global Modal Padding (LOW):** Reducing `.modal-body` padding from `2rem` to `1rem` is applied globally. While generally appropriate for mobile, it may cause text to hit the edges of the screen in modals that use their own internal padding or margins, specifically older "legacy" modals that might not follow the newer grid patterns.
*   **Touch Targets (LOW):** The tab bar in the Financial Breakdown Modal (`.em-tab-bar`) is becoming scrollable. On mobile, if the tabs are too small or the padding is insufficient, they may be difficult to tap precisely.

## 4. Suggestions
*   **Ensure Table Width:** In both `finance.js` and `expense-modal.js`, ensure the tables inside the scroll wrappers have `width: 100%; min-width: 600px;` (or similar) or apply `white-space: nowrap;` to `<th>` elements to force the horizontal overflow.
*   **Visual Check for TR Modal:** Before applying `.modal-details-grid` to the TR Details modal, verify if the outer container already provides a background and padding. If it does, create a more minimal `.modal-details-grid-lite` that only handles the grid layout without the decorative properties.
*   **Scroll Indicators:** For the horizontally scrollable tables, consider adding a subtle visual cue (like a fading right shadow or a small "Scroll to view more" hint) as users may not immediately realize a table is scrollable if it clips perfectly on a column border.
*   **Tab Padding:** Increase the horizontal padding on `.em-tab-bar .expense-tab` for the mobile view to ensure they meet minimum touch target recommendations (approx. 44px width).
*   **Standardize Scroll Class:** Instead of repetitive inline `style="overflow-x: auto"` wrappers in `finance.js`, define a `.table-responsive` utility class in `components.css` and use that.

## 5. Risk Assessment
**Risk Level: LOW**

**Justification:**
The changes are strictly additive or override-based within a media query, meaning the risk of breaking the core desktop workflow is near zero. The "no build system" nature of the project makes these manual template string updates the standard way to work. The primary risk is purely aesthetic (nested padding or squeezed tables), which can be easily tuned during manual browser testing. Plan 01 and Plan 02 are safe to run in parallel as they have no file-level collisions.
