# Phase 65.8: Fix Payables Tab UI Containment and Add Pagination - Research

**Researched:** 2026-03-25
**Domain:** Finance view UI — table containment, horizontal scroll, and pagination
**Confidence:** HIGH

---

## Summary

The Finance Payables tab currently wraps both Table 1 (RFP Processing) and Table 2 (PO Payment Summary) in `<div class="table-container">` divs, but `.table-container` has no CSS rule defined anywhere in the stylesheet. This means there is no horizontal scroll containment on these tables, so wide tables overflow their container and break the page layout.

The rest of the codebase uses two established container patterns for scrollable tables: `.table-scroll-container` (defined in `components.css` with the shadow-edge scroll technique) for the Supplier table and the Pending Approvals table, and `<div style="overflow-x: auto;">` inline for simpler cases.

Additionally, Table 1 (RFP Processing) has no pagination at all. Table 2 (PO Payment Summary) received pagination in Phase 65.7, and that implementation is the canonical pattern to follow for Table 1.

**Primary recommendation:** Replace `class="table-container"` with `class="table-scroll-container"` on both Payables tables. Then add pagination to Table 1 (RFP Processing) mirroring the Phase 65.7 Table 2 pattern exactly, including state variables, a `changeRFPPage` handler, an `updateRFPPagination` function, and a `<div id="rfpPagination"></div>` placeholder in the HTML.

**changePOSummaryPage handler (finance.js lines 454–469):**
```javascript
function changePOSummaryPage(direction) {
    const poMap = buildPOMap(rfpsData);
    let entryCount = 0;
    poMap.forEach(() => entryCount++);
    const totalPages = Math.ceil(entryCount / poSummaryItemsPerPage);

    if (direction === 'prev' && poSummaryCurrentPage > 1) {
        poSummaryCurrentPage--;
    } else if (direction === 'next' && poSummaryCurrentPage < totalPages) {
        poSummaryCurrentPage++;
    } else if (typeof direction === 'number') {
        poSummaryCurrentPage = direction;
    }
    renderPOSummaryTable();
}
```

## Common Pitfalls

### Pitfall 1: Forgetting to slice `displayed` in renderRFPTable
**What goes wrong:** The `displayed` array in `renderRFPTable()` is currently rendered in full with `.map()`. Adding pagination requires slicing `pageItems = displayed.slice(startIndex, endIndex)` before the map.
**Why it happens:** It's easy to add pagination state but forget to apply the slice to the render.
**How to avoid:** Add slice step immediately after the sort, parallel to the Table 2 pattern at finance.js lines 768–774.

### Pitfall 2: changePOSummaryPage uses buildPOMap for total count; changeRFPPage needs filtered rfpsData count
**What goes wrong:** The Table 2 page handler calls `buildPOMap(rfpsData)` to count total pages. Table 1 operates on a flat filtered `displayed` array — it should derive its total from the filtered RFP count, not the raw `rfpsData.length`.
**Why it happens:** The two tables group data differently.
**How to avoid:** In `changeRFPPage`, compute the filtered count the same way `renderRFPTable` does (apply the same filter/sort logic to get total), then compute `totalPages`. Alternatively, cache the filtered count in a module-level variable updated by `renderRFPTable`.
**Simplest approach:** Store `let rfpFilteredCount = 0;` at module level; set it inside `renderRFPTable` after filtering, before slicing; use it in `changeRFPPage`.

### Pitfall 3: `rfpPagination` div already in HTML vs. created by JS
**What goes wrong:** The Table 2 pagination div is declared in HTML as `<div id="poSummaryPagination"></div>` (line 1790) AND the JS function creates it dynamically if absent. Table 1 needs the same — include the placeholder div in the HTML inside `render()` so the JS find path works correctly.
**How to avoid:** Add `<div id="rfpPagination"></div>` immediately after the RFP `table-container` div in the render function HTML.

### Pitfall 4: Window function not cleaned up
**What goes wrong:** `window.changeRFPPage` persists after leaving the tab, causing stale reference errors on re-entry.
**How to avoid:** Add `delete window.changeRFPPage` to the `destroy()` cleanup block (finance.js ~line 2525).

### Pitfall 5: table-container vs table-scroll-container — class swap only
**What goes wrong:** Developer adds `overflow-x:auto` as inline style instead of swapping to the existing CSS class.
**How to avoid:** Replace `class="table-container"` with `class="table-scroll-container"` on both table wrapper divs. Do not add inline styles.

rfpFilteredCount = displayed.length;

### updateRFPPagination (mirrors updatePOSummaryPagination exactly)
```javascript
function updateRFPPagination(totalPages, startIndex, endIndex, totalItems) {
    let paginationDiv = document.getElementById('rfpPagination');
    if (!paginationDiv) {
        paginationDiv = document.createElement('div');
        paginationDiv.id = 'rfpPagination';
        paginationDiv.className = 'pagination-container';
        const tbody = document.getElementById('rfpTableBody');
        const table = tbody?.closest('table');
        if (table && table.parentNode) {
            table.parentNode.insertBefore(paginationDiv, table.nextSibling);
        }
    }
    if (totalPages <= 1) { paginationDiv.style.display = 'none'; return; }
    paginationDiv.style.display = '';
    // ... same HTML as updatePOSummaryPagination but calls window.changeRFPPage
}
```
