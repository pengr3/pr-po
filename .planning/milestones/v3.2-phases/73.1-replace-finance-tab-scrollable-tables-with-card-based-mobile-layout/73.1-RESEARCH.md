# Phase 73.1: Replace Finance Tab Scrollable Tables with Card-Based Mobile Layout — Research

**Researched:** 2026-04-15
**Domain:** CSS responsive design, mobile-first card layout pattern for data tables — pure JS SPA, no framework
**Confidence:** HIGH — all findings derived from direct codebase inspection

---

## Summary

Phase 73 left the Finance tab with horizontal-scroll tables on mobile (`.table-scroll-container` with sticky first column). That was explicitly scoped as a temporary band-aid pending Phase 73.1, which delivers the correct solution: a card-based layout that replaces the table DOM on small screens. The user confirmed this intent in the Phase 73 verification report.

There are **5 distinct tables** across the 4 Finance sub-tabs that need mobile card treatment:

1. **Material PRs** (Tab 1 — Pending Approvals) — 9 columns, rendered into `#materialPRsBody` by `renderMaterialPRs()`
2. **Transport Requests** (Tab 1 — Pending Approvals) — 9 columns, rendered into `#transportRequestsBody` by `renderTransportRequests()`
3. **Purchase Orders** (Tab 2) — 9 columns, rendered dynamically into `#poList` by `renderPOs()` (regenerates entire container HTML including the `<table>` wrapper)
4. **Project/Service/Recurring Expenses** (Tab 3 — Project List) — 6 columns, rendered into container divs by `renderProjectExpensesTable()` / `renderServiceExpensesTable()` / `renderRecurringExpensesTable()` (each regenerates the container's inner HTML including the table wrapper)
5. **RFP Processing + PO Payment Summary** (Tab 4 — Payables) — 11 and 9 columns respectively, rendered into `#rfpTableBody` and `#poSummaryTableBody`

The project uses pure CSS + inline-style template strings. The canonical mobile breakpoint is `max-width: 768px`. The `views.css` Finance block added in Phase 73 is the right place to extend. No new CSS files may be created.

**Primary recommendation:** For each render function, emit a parallel card-list structure alongside (or instead of) the table, toggled by a CSS `@media (max-width: 768px)` rule that hides `.table-scroll-container` and shows `.fc-card-list`. The card HTML is generated inside the same JS render function using a feature flag determined by viewport — or more elegantly, emit both structures and let CSS hide/show them. The CSS-only toggle is cleanest for this codebase since it requires zero JS viewport detection and responds instantly to orientation changes.

## Architecture Pattern

### The CSS Dual-Mode Pattern (Recommended)

Each render function emits **both** a `.table-scroll-container` (desktop) **and** a `.fc-card-list` (mobile) inside the same container. CSS media query hides the one not in use.

```css
/* views.css — Finance @media block (Phase 73.1 additions) */

/* Desktop: show table, hide cards */
.fc-card-list { display: none; }

@media (max-width: 768px) {
    /* Mobile: hide table, show cards */
    .table-scroll-container { display: none; }
    .fc-card-list { display: flex; flex-direction: column; gap: 0.75rem; padding: 0 0.25rem; }
}
```

**Caveat:** The global `.table-scroll-container { display: none; }` inside the 768px block would break ALL tables in the app. The rule must be scoped:

```css
/* Scope to Finance sections only */
#approvals-section .table-scroll-container,
#pos-section .table-scroll-container,
#projects-section .table-scroll-container,
#payables-section .table-scroll-container { display: none; }
```

This is the correct approach because the Finance section IDs (`#approvals-section`, `#pos-section`, `#projects-section`, `#payables-section`) already exist in the `render()` HTML at finance.js line 1527.

### Alternative: JS Viewport Toggle (Not Recommended)

Detect `window.innerWidth < 768` inside each render function and emit only the card HTML on mobile. This approach is more fragile (requires `resize` event listener, re-renders on orientation change, adds state), and contradicts the project's CSS-first pattern. Do not use.

### Card Structure per Row

Each row becomes a card. The card contains a **label-value grid** for the key fields. Action buttons are full-width at the bottom.

## Render Function Change Pattern

### Pattern A: tbody-only render functions (Material PRs, Transport Requests, RFP Table, PO Summary Table)

These functions write to a `<tbody id="...">` that lives in static HTML. Two changes needed:

1. **In `render()` static template string:** Add `<div class="fc-card-list" id="{name}CardList"></div>` as a sibling immediately after the `<div class="table-scroll-container">` wrapper.

2. **In the render function body:** After building `tbody.innerHTML`, also build and assign `cardListEl.innerHTML`.

```javascript
// Inside renderMaterialPRs() — addition only, no removal:
const cardList = document.getElementById('materialPRsCardList');
if (cardList) {
    if (filtered.length === 0) {
        cardList.innerHTML = `<div class="fc-empty">No pending material PRs${...}</div>`;
    } else {
        cardList.innerHTML = filtered.map(pr => buildMaterialPRCard(pr)).join('');
    }
}
```

The `buildMaterialPRCard(pr)` helper returns a card HTML string. Keeping it as a separate function (not inlined) keeps the render function readable.

### Pattern B: full-container render functions (Purchase Orders, Project/Service/Recurring Expenses)

These functions regenerate the entire container innerHTML including the `<table>` wrapper. They must emit both the table and the card list in one assignment:

```javascript
container.innerHTML = `
    <div class="table-scroll-container">
        <table>...</table>
    </div>
    <div class="fc-card-list">
        ${items.map(item => buildItemCard(item)).join('')}
    </div>
`;
```

This is clean — both structures exist in the DOM simultaneously. CSS determines which is visible. No JS viewport detection needed.

## Common Pitfalls

### Pitfall 1: Scoping `.table-scroll-container { display: none }` Globally

**What goes wrong:** Adding an unscoped `display: none` on `.table-scroll-container` inside `@media (max-width: 768px)` hides all scrollable tables in the app — Procurement, Projects, Suppliers, MRF Records — not just Finance.

**How to avoid:** Always scope to Finance section IDs: `#approvals-section .table-scroll-container`, `#pos-section .table-scroll-container`, etc. These IDs are stable — they exist in the Finance `render()` template string.

**Warning signs:** Procurement table disappears on mobile after deploy.

### Pitfall 2: onclick Strings in Card HTML Use window. Functions That Are Not Yet Registered

**What goes wrong:** Card HTML emits `onclick="window.viewPRDetails(...)"` — if the card is rendered before `attachWindowFunctions()` runs (or if it runs before the Finance tab is initialized), clicking throws `TypeError`.

**How to avoid:** All Finance window functions are already registered in `attachWindowFunctions()` which is called at `init()` time. Card HTML uses the same `window.X(...)` pattern as table rows — identical safety level.

### Pitfall 3: PO Payment Summary Card — togglePOExpand ID Collision

**What goes wrong:** The existing toggle logic uses `#po-expand-{safePoId}` and `#po-chevron-{safePoId}` as element IDs for `<tr>` elements. If the card layout also uses these IDs for `<div>` elements, the JS `togglePOExpand()` function will attempt `row.style.display = 'table-row'` on a `<div>`, which won't work as intended.

**How to avoid:** Use distinct IDs for card expand targets: `#po-card-expand-{safePoId}` and `#po-card-chevron-{safePoId}`. Add a card-specific toggle function (e.g., `window.togglePOCardExpand`) that sets `display: block / none` instead of `table-row`.

**Alternative:** Reuse `togglePOExpand` but rename the card expand ID to avoid collisions, and update the toggle function to detect whether the element is a `<tr>` or `<div>` before setting `display` value.

### Pitfall 4: RFP "Record Payment" Inline onclick Strings

**What goes wrong:** The RFP table rows emit onclick strings like `onclick="window.openRecordPaymentModal('${rfp.id}')"`. The rfp.id can contain characters that break the string if not escaped. The table rows use this pattern without issue because the table is built in a function that already has `escapeHTML` available.

**How to avoid:** Use the same `escapeHTML` call on rfp.id in card onclick strings. The rfp.id values are Firestore document IDs (alphanumeric with hyphens) so in practice this is safe, but keep the pattern consistent.

### Pitfall 5: Proof Indicator onclick in PO Cards — Nested Quote Conflicts

**What goes wrong:** The proof indicator in the PO table uses complex template literals with single and double quotes for the inline `onclick`, `ontouchstart` handlers (lines 3819–3839 in finance.js). These are already complex strings — duplicating them in card HTML increases the chance of quote nesting errors.

**How to avoid:** Extract the proof indicator HTML generation into a named helper function `buildProofIndicator(po)` before writing card HTML. This helper already exists implicitly in the table render — extract it explicitly:

```javascript
function buildProofIndicator(po) {
    // ... same logic as lines 3816-3840 ...
}
```

Call from both the table row and the card.

### Pitfall 6: Expense Cards — Clickable Card vs. Button Inside Card

**What goes wrong:** The Project/Service/Recurring expense rows are `<tr onclick="...">` — clicking anywhere opens the expense modal. If the card has an onClick on the outer `<div>`, and a button inside it also has an onClick, event bubbling can trigger both.

**How to avoid:** For expense cards, make the entire `fc-card` div clickable with `onclick="window.showProjectExpenseModal(...)"` (class `fc-clickable`) and no nested button — consistent with the table row's click-everywhere pattern.

### Pitfall 7: poSummaryPagination div Inserted Relative to the `<table>` Element

**What goes wrong:** `updatePOSummaryPagination()` uses `table.parentNode.insertBefore(paginationDiv, table.nextSibling)` to insert the pagination div after the table (lines 518-524). In the dual-mode layout, the table's parent is the `.table-scroll-container` div, and the card list is a sibling of that. The pagination div would be inserted between the table-scroll wrapper and the card list, which is fine for desktop, but on mobile the table-scroll wrapper is hidden (display:none) and the pagination div still appears.

**How to avoid:** The pagination div already has `id="poSummaryPagination"` and the CSS for it lives outside the Finance-scoped hide rule. The pagination should remain visible on mobile (it controls the page). Ensure the finance-scoped `display:none` only targets `.table-scroll-container` not its siblings. This is correct by the proposed scoping.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead |
|---------|-------------|-------------|
| Card toggle between table/card | JS viewport detection + re-render on resize | CSS `@media` hide/show (no JS) |
| New design tokens/colors | Custom card-specific color variables | Existing CSS variables (`var(--primary)`, `var(--gray-200)`, etc.) and inline colors from the project palette |
| New animation/transitions | JS-driven slide-in | None — match the static rendering pattern of every other Finance component |
| New breakpoint | A 576px or 640px breakpoint for "medium mobile" | Stay with the single 768px breakpoint the project uses everywhere |
