## Spacing Scale

Declared values — all multiples of 4px. The project uses `rem` units; values below show both px equivalent (at 16px root) and rem.

| Token | Value | rem | Usage |
|-------|-------|-----|-------|
| xs | 4px | 0.25rem | Icon gaps, inline badge padding; fc-card-body vertical gap between rows |
| sm | 8px | 0.5rem | Gap between fc-card-row label and value; sub-card internal padding |
| md | 12px | 0.75rem | fc-card-header padding top/bottom; fc-card-actions padding; gap between stacked cards in fc-card-list |
| md+ | 16px | 1rem | fc-card-header/actions horizontal padding; fc-card-body horizontal padding |
| lg | 24px | 1.5rem | Section-level vertical rhythm between card list and toolbar |
| xl | 32px | 2rem | Not used in card components directly |
| 2xl | 48px | 3rem | Not used in card components directly |

Exceptions:
- Touch targets (action buttons in fc-card-actions): minimum height 44px — WCAG 2.5.5 touch target size.
- **Project exception — 0.75rem (12px):** 12px is the established project card padding value per CLAUDE.md ("Consistent spacing (0.75rem, 1rem, 1.5rem, 2rem)") and components.css `.card`. It is not in the standard set {4, 8, 16, 24, 32, 48, 64} but is an approved project exception used consistently throughout existing components. Phase 73.1 card components follow this established value for `fc-card-header`, `fc-card-body`, `fc-card-actions` padding (top/bottom), and `fc-card-list` gap.

Source: RESEARCH.md CSS class definitions; CLAUDE.md "Consistent spacing (0.75rem, 1rem, 1.5rem, 2rem)"; components.css .card (padding: 1.5rem); RESEARCH.md CARD-08 test requirement (min 44px for "Record Payment" button).

## Typography

These are the three type roles used within fc-card components. All are drawn from existing project utility classes — no new font sizes are introduced.

| Role | Size | px equiv | Weight | Line Height | Usage |
|------|------|----------|--------|-------------|-------|
| Card ID | 0.9375rem | 15px | 700 | 1.2 | fc-card-id: PR/PO/TR/RFP identifier in card header |
| Body | 0.875rem | 14px | 400 | 1.5 | fc-card-row value text; all data values |
| Label / Sub-card | 0.75rem | 12px | 700 (label) / 400 (sub-card) | 1.2 (label) / 1.5 (sub-card) | fc-label: uppercased field label; fc-sub-card: tranche rows inside PO Payment Summary expand |

Two weights only: 400 (regular) and 700 (bold).
- Weight 700 is used for: monetary amounts (fc-amount), card ID chips (fc-card-id), and field labels (fc-label). Labels are differentiated from body text by size (0.75rem vs 0.875rem), uppercase transform, and letter-spacing — the shared 700 weight with fc-card-id is acceptable because they appear in distinct zones (header vs. body).
- Weight 400 is used for: all data values (fc-value), sub-card tranche rows (fc-sub-card).

Source: RESEARCH.md CSS class definitions; main.css font-size scale (0.75rem, 0.875rem, 1rem utility classes); main.css body line-height: 1.5, heading line-height: 1.2. Weight consolidated per checker revision: fc-label changed from 600 to 700 to eliminate the third distinct weight.

## Interaction Contract

### 1. Desktop / Mobile Toggle

- Trigger: CSS `@media (max-width: 768px)` — no JavaScript viewport detection.
- Behavior: Tables hidden, card lists shown. Responds instantly to orientation change.
- No re-render needed — both structures live in DOM simultaneously.
- Source: RESEARCH.md Architecture Pattern / "CSS Dual-Mode Pattern (Recommended)".

### 2. Card Actions — All Tabs

Each card's fc-card-actions section contains a single full-width button replicating the table row's action:

| Tab | Card | Button label | onclick target |
|-----|------|-------------|----------------|
| Pending Approvals | Material PR | Review PR | window.viewPRDetails(pr.id) |
| Pending Approvals | Transport Request | Review TR | window.viewTRDetails(tr.id) |
| Purchase Orders | PO | View PO | window.viewPODetails(po.id) |
| Project List | Expense row | (no button — entire card clickable) | window.showProjectExpenseModal(...) |
| Payables — RFP | RFP | Record Payment | window.openRecordPaymentModal(rfp.id) |
| Payables — PO Summary | PO | Show Tranches / Hide Tranches | window.togglePOCardExpand(safePoId) |

Note: "Review PR" and "Review TR" follow the verb + noun CTA convention. "View PO" matches the existing table action label. "Record Payment" matches the existing modal trigger label.

### 3. Expense Cards — Clickable Pattern

Project/Service/Recurring expense cards use `fc-clickable` class. The entire card div carries `onclick="window.showProjectExpenseModal(...)"`. No nested button. This mirrors the `<tr onclick="...">` pattern in the desktop table.

Hover state: border-color changes to #1a73e8, blue focus ring via box-shadow. No background color change (too distracting).

### 4. PO Payment Summary — Expand/Collapse

- Toggle function: `window.togglePOCardExpand(safePoId)` — distinct from the desktop `window.togglePOExpand(safePoId)` which targets `<tr>` elements with `display: table-row`.
- Card expand targets IDs: `#po-card-expand-{safePoId}` (div, not tr).
- Card chevron IDs: `#po-card-chevron-{safePoId}`.
- Toggle sets `display: block / none` (not `table-row`).
- Multiple cards can be expanded simultaneously (matches desktop behavior).
- Source: RESEARCH.md Pitfall 3.

### 5. Overdue Highlight

RFP cards where payment is overdue receive `.fc-overdue` class. This replicates the desktop table row `background: #fef2f2` highlight. Card border changes to #fecaca, card-header background changes to #fee2e2.

### 6. Proof Indicator in PO Cards

The proof indicator HTML (green circle / orange dash / empty circle) is extracted to a named helper function `buildProofIndicator(po)` before use in card HTML, to avoid quote-nesting errors in template literals. Shown as an fc-card-row with label "Proof".

Source: RESEARCH.md Pitfall 5.

## Five-Table Implementation Summary

This section is for the executor's reference — maps each table to its render pattern.

| Table | Render Function | Pattern | Card List ID | Static or Dynamic Container |
|-------|----------------|---------|--------------|----------------------------|
| Material PRs | renderMaterialPRs() | A (tbody-only) | materialPRsCardList | Static in render() |
| Transport Requests | renderTransportRequests() | A (tbody-only) | transportRequestsCardList | Static in render() |
| Purchase Orders | renderPOs() | B (full-container) | Emitted inline in renderPOs() template string | Dynamic |
| Project/Service/Recurring Expenses | renderProjectExpensesTable() etc. | B (full-container) | Emitted inline in each render function | Dynamic |
| RFP Processing | renderRFPTable() | A (tbody-only) | rfpCardList | Static in render() |
| PO Payment Summary | renderPOSummaryTable() | A (tbody-only) | poSummaryCardList | Static in render() |

Pattern A: add `<div class="fc-card-list" id="{name}CardList"></div>` as sibling after `.table-scroll-container` in the static render() template. Populate in render function.

Pattern B: render function emits both `.table-scroll-container` and `.fc-card-list` in a single container.innerHTML assignment.
