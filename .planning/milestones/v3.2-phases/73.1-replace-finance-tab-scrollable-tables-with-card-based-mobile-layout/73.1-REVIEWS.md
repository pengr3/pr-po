# Cross-AI Plan Review — Phase 73.1

## Claude (In-Session) Review

### Summary

The four plans form a well-structured CSS dual-mode solution: emit both `<table>` and `<div class="fc-card-list">` in the DOM simultaneously, let CSS `@media (max-width: 768px)` with Finance-scoped selectors toggle visibility. The wave ordering is sound (Plan 01 CSS foundation → Plans 02+03 parallel → Plan 04 final + checkpoint). The research document is excellent — it pre-identifies every major pitfall and the plans explicitly address them. The plans are detailed to the point of being near-executable specs, which is appropriate for a 3,943-line file with many interdependencies.

There are **three genuine concerns** worth addressing before execution, and a handful of lower-priority observations.

---

### Strengths

- **CSS-only toggle is the right call.** The research correctly rejects JS viewport detection as fragile; the Finance-section-scoped selectors (`#approvals-section .table-scroll-container`, etc.) eliminate global bleed to Procurement/Suppliers tables.
- **`buildProofIndicator` extraction (Plan 02 Task 1)** is smart engineering — the proof indicator has the most complex onclick string in the entire file and extracting it before the card layout prevents duplication of the hardest-to-maintain template literal.
- **Pitfall 3 (ID collision) is handled correctly.** `po-card-expand-{id}` and `po-card-chevron-{id}` distinct from `po-expand-{id}` / `po-chevron-{id}`. `togglePOCardExpand` sets `display: block` on a `<div>` while `togglePOExpand` sets `display: table-row` on a `<tr>`. Both can coexist without interference.
- **Pitfall 7 (pagination visibility) is handled correctly.** `poSummaryPagination` sits outside `.table-scroll-container` so the Finance-scoped hide rule doesn't affect it.
- **Plan 04 human-verify checkpoint** is well-specified — 375px, 320px, and 1280px viewports, cross-tab regression, empty-state parity, and per-tab interaction checks are all enumerated.
- **`buildRecurringExpenseCard` as thin alias** — delegating to `buildServiceExpenseCard` documents the intent clearly and avoids code duplication for identical data shapes.
- **Empty-state parity** addressed consistently across all Pattern A tables; Plan 03 even modifies the early-return branch to populate both tbody and card list before returning.

### Concerns

#### CONCERN 1 — HIGH: Missing `.fc-overdue` highlighting on Material PR and TR cards (Plan 01)

The CSS defines `.fc-card.fc-overdue` with `border-color: #fecaca; background: #fef2f2` (the "red tint" for overdue items). Plan 03 correctly applies this to RFP cards. However, `buildMaterialPRCard` and `buildTRCard` in Plan 01 do **not** compute or apply `.fc-overdue`.

If the desktop Material PRs table highlights overdue rows (PRs where `date_needed` is in the past), mobile cards will fail to match — a visible inconsistency. The research documents this CSS class under "Overdue state (matches desktop table row background:#fef2f2)" with no caveat that it's only for RFPs.

**Before executing Plan 01 Task 2:** Confirm whether `renderMaterialPRs()` / `renderTransportRequests()` apply a red row background when `date_needed < today`. If yes, add overdue detection to both card builders:

```javascript
const isOverdue = mrfCache.get(pr.mrf_id)?.date_needed
    && new Date(mrfCache.get(pr.mrf_id).date_needed) < new Date();
// then: class="fc-card${isOverdue ? ' fc-overdue' : ''}"
```

---

#### CONCERN 2 — MEDIUM: `fc-card-body` gap value conflict between PLAN and RESEARCH

Plan 01 Task 1 declares `.fc-card-body { gap: 0.25rem; }` in the CSS block marked "copy verbatim."  
The RESEARCH.md CSS section shows `.fc-card-body { gap: 0.375rem; }`.  
The UI-SPEC spacing table designates `xs = 0.25rem` and lists it as "fc-card-body vertical gap between rows."

So there are **three different values** in three different artifacts: 0.25rem (Plan), 0.375rem (Research), 0.25rem (UI-SPEC). The Plan and UI-SPEC agree; Research diverges. This is probably a research draft artifact, but it's worth a one-line confirm before Task 1 runs. The UI-SPEC value (0.25rem / 4px) should win per its role as the design contract.

---

#### CONCERN 3 — MEDIUM: `rfp.due_date` rendered raw in RFP cards (Plan 03 Task 1)

`buildRFPCard` emits `${rfp.due_date || 'N/A'}` without calling `formatDate()`.

If `due_date` is stored as a Firestore Timestamp (common for date fields), this renders as `[object Object]`. If it's stored as an ISO string (`"2026-05-01"`), raw display is fine but inconsistent with the date formatting everywhere else.

**Check the `rfps` collection schema for `due_date` field type.** If it's a Timestamp, use `formatDate(rfp.due_date)`. If it's a string, confirm whether it's already user-readable or needs reformatting.

### Suggestions

- **Plan 02 `buildPOCard` — `statusLabel` inconsistency:** The card uses `po.procurement_status || 'Pending'` for the displayed label but `getStatusClass(po.procurement_status || 'Pending Procurement')` for the CSS class. A PO with `null` status would show "Pending" text with the "Pending Procurement" CSS class. Harmonize: use `'Pending Procurement'` for both or `'Pending'` for both.

- **Plan 02 wave-2 parallel edit risk:** Plans 02 and 03 both run in Wave 2 and both modify `app/views/finance.js`. If a GSD executor runs them in parallel sub-agents, there's a merge risk. The plans touch different line regions (02: ~3762; 03: 568, 759, 191) but the file is large. Safe to run parallel — just confirm the executor applies them sequentially to the same file or uses file-locking.

- **Plan 04 `buildProjectExpenseCard` — `escapeHTML` in onclick:** `onclick="window.showProjectExpenseModal('${escapeHTML(proj.projectName)}')"` — `escapeHTML` correctly converts single quotes to `&#39;` which HTML-decodes back to `'` before JS execution (browser behavior). Safe, but worth a quick smoke-test with a project name containing an apostrophe.

- **Plan 01 Task 1 acceptance criteria verify command uses `&amp;` for `&&`:** The `<automated>` element has `&amp;&amp;` which is correct XML escaping, but if these commands are run through a shell directly (not XML-parsed), the executor may need to unescape them. Verify the execution harness handles this correctly.

### Risk Assessment: **LOW–MEDIUM**

The architecture is sound and the research quality is high. The single HIGH concern (overdue state gap) is a UX fidelity issue rather than a functional failure — the app will work, cards will just lack the red tint that desktop table rows have. The two MEDIUM concerns are localized and verifiable before execution. No concerns about performance, security (this is a UI-only change to a read-path render function), or scope creep. Plans are appropriately narrow.

**Recommend:** Resolve Concern 1 (overdue state check) before executing Plan 01 Task 2, and confirm `due_date` field type before Plan 03. Then proceed.
