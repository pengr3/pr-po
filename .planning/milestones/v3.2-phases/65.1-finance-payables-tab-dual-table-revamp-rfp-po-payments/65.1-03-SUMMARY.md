# Phase 65.1 Plan 03: UAT Gap Closure — Table 1 Sort Order and Table 2 Overall Status Summary

One-liner: Status-priority sort in Table 1 (unpaid rows before paid within each PO group) and live PO total_amount feed for Table 2 (prevents false Fully Paid on progress-billing POs).

## What Was Built

Closed two UAT-reported issues in the Finance Payables tab:

**UAT Test 2 — Table 1 Sort Order:**
The RFP table sort was updated from a two-key sort (po_id, tranche_percentage) to a three-key sort inserting a `statusPriority` middle key. Within each PO group, rows now sort: Pending (1) → Overdue (2) → Partially Paid (3) → Fully Paid (4). Rows with equal status sort by tranche_percentage ascending.

**UAT Test 7 — Table 2 Overall Status:**
Progress-billing POs were being falsely marked "Fully Paid" when all billed tranches were paid but future unbilled tranches remained. Fixed by:
1. Adding a `posAmountMap` (Map of po_id → total_amount) kept live via a new `onSnapshot` on the `pos` collection inside `initPayablesTab`.
2. Updating `derivePOSummary` to accept an optional `poTotalAmount` parameter and use it as the authoritative contract value when provided (falling back to RFP sum otherwise).
3. Passing `posAmountMap.get(entry.poId)` from `renderPOSummaryTable` into `derivePOSummary`.

With this fix, a PO with total_amount 1,000,000 where only 300,000 has been billed (and fully paid) now shows "Partially Paid" (remaining = 700,000 > 0), not "Fully Paid".
