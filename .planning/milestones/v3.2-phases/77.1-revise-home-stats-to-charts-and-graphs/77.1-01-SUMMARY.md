## Files Created/Modified

### `index.html` — line 18

Chart.js CDN script tag added immediately after the `signature_pad.umd.min.js` line:

```html
<!-- Chart.js (Phase 77.1) — pinned to v4.4.7 for home page status breakdown charts -->
<script src="https://cdn.jsdelivr.net/npm/chart.js@4.4.7/dist/chart.umd.min.js"></script>
```

**Why `chart.umd.min.js`:** The zero-build static SPA has no bundler. The UMD build exposes `window.Chart` globally, which ES modules can access via `new window.Chart(...)` after the script has executed. The `chart.esm.js` build requires a bundler. The `chart.js` unminified build wastes bandwidth.

**Load order:** Classic `<script>` tags execute synchronously in document order before ES module scripts load, so `window.Chart` is always defined by the time `home.js` imports and calls `renderStatusBreakdown`.

**Lines 206–284: `renderStatusBreakdown`** — 100% stacked bar implementation:

- `toPercent(count)`: `total > 0 ? parseFloat(((count / total) * 100).toFixed(1)) : 0`
- **What happens when total=0:** all statuses → 0%, chart renders an empty bar (no segment visible), no error
- **Dataset structure:** one dataset per status label; `data: [toPercent(count)]`; `labels: ['']` (single row); `stacked: true` on both axes; `x.max: 100`
- **chart._rawCounts:** set to `{ ...countsMap }` on create; updated on every re-render before `chart.update()`. The tooltip callback receives `ctx.chart._rawCounts` — this avoids the closure issue where a callback bound at create-time would only see the initial counts, not live re-render counts
- **Tooltip format:** ` StatusName: {count} ({pct}%)` — shows both human count and percentage
- **Legend:** `position: 'bottom'`, `font.size: 10`, `boxWidth: 10`, `boxHeight: 10`, `padding: 4`, `color: '#64748b'`
- **Scales:** both x and y axes `display: false`, `grid.display: false` — chart is pure visual signal; no axis labels or gridlines

**Lines 504–523: `destroy()`** — extended to tear down chart instances:

```javascript
chartInstances.forEach(chart => {
```

Called in addition to the existing `statsListeners` unsubscribe loop. `cachedStats` intentionally NOT reset (stale-while-revalidate pattern).

## chartInstances Map Lifecycle

| Event | Action |
|-------|--------|
| First `renderStatusBreakdown` call | `new window.Chart(canvas, config)` → `chartInstances.set(containerId, chart)` |
| onSnapshot re-fires (Firestore update) | `existing._rawCounts = {...countsMap}`; update each `dataset.data`; `existing.chart.update()` |
| `destroy()` called (router navigating away) | `chartInstances.forEach(c => c.destroy()); chartInstances.clear()` |
| Re-navigation to home view | `render()` emits fresh canvas elements; next onSnapshot creates new Chart instances |

## 100% Stacked Bar Design (Post-UAT Revision, commit 62291eb)

**What changed from Task 2 (individual bars) to 62291eb (100% stacked):**

The original `renderStatusBreakdown` implementation used a single dataset with one value per status (individual horizontal bars, one row per status). After UAT review, this was revised to a 100% stacked bar approach:

| Aspect | Before (1d9e759) | After (62291eb) |
|--------|-----------------|-----------------|
| Chart layout | One horizontal bar per status (multi-row) | One horizontal bar total, all statuses as segments |
| Data values | Raw counts | Percentages via `toPercent()` helper |
| Dataset structure | 1 dataset, N data values | N datasets, each with 1 data value |
| Tooltip | Shows raw count | Shows `StatusName: count (pct%)` |
| Legend | `display: false` | `display: true`, position bottom, compact |
| CSS height (internal) | 110px | 72px |
| CSS height (project) | 160px | 96px |
| Axis scales | x: beginAtZero, y: ticks | x: stacked, max:100, display:false; y: stacked, display:false |

**Why stacked is better at small card widths:** At the home page card width, individual-bar layouts push 7+ rows into a very tall chart, making the card visually heavy. The stacked bar condenses all statuses into one compact bar, with the legend below showing which color is which status. The 72/96px heights fit naturally in the card without scrolling.

## Decisions Made

- Chart.js v4.4.7 pinned (`chart.umd.min.js`, jsDelivr CDN) — latest stable 4.x patch as of plan creation; UMD required for `window.Chart` global in zero-build SPA
- `chartInstances` Map (not object) — O(1) get/set and clean `forEach+clear` semantics
- `animation: false` for home page performance — avoids animation overhead on every onSnapshot update
- One Chart instance per section (6 total: 2 in Projects, 4 in Services) — each section is independent
- Zero-count statuses shown as zero-length segments (not hidden) — consistent visual baseline; no disappearing/shifting segments
- `chart._rawCounts`: necessary because the tooltip callback is captured at chart creation time; assigning fresh counts to `chart._rawCounts` on every re-render lets the tooltip read live values without recreating the chart
- CSS heights set explicitly on parent `div`, not on `canvas` — Chart.js reads parent `getBoundingClientRect()` when `maintainAspectRatio: false`
