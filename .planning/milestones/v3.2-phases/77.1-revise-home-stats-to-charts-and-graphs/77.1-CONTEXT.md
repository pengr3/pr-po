# Phase 77.1: Revise Home Stats to Charts and Graphs — Context

**Gathered:** 2026-04-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Evolve the 3 stat cards on the Home page (built in Phase 77) by replacing the text-row status breakdowns in the Projects and Services cards with Chart.js horizontal bar charts. The Procurement card is untouched — it keeps its scorecard layout. The 3 navigation cards above the stats are out of scope.

</domain>

<decisions>
## Implementation Decisions

### D-01: Procurement Card — No Change
Keep the Procurement card exactly as Phase 77 left it: 3 large scorecard numbers (Pending MRFs, Pending PRs, Active POs). No chart added.

### D-02: Projects & Services Cards — Horizontal Bar Charts
Replace the `.hs-status-grid` text rows in both Projects and Services cards with Chart.js horizontal bar charts. One chart per section (Internal Status, Project Status). Services card has 4 sections total (One-time Internal, One-time Project, Recurring Internal, Recurring Project).

Charts replace text rows — the number list is gone; the chart is the display.

### D-03: Highlighted vs Monochromatic Statuses
Two visual tiers for bar colors:

**Highlighted statuses** (colored bars — brand palette, muted/classy, not overpowering):
- `For Inspection` (Internal Status)
- `For Proposal` (Internal Status)
- `Under Client Review` (Project Status)
- `On-going` (Project Status)

These 4 statuses get distinct brand-palette colors across both Projects and Services cards.

**All other statuses** (monochromatic — single neutral hue with varying shades):
- `For Internal Approval`, `Ready to Submit` (Internal Status)
- `Pending Client Review`, `Approved by Client`, `For Mobilization`, `Completed`, `Loss` (Project Status)

Monochromatic treatment: a single neutral color (e.g., gray family from the design system) with different shades to distinguish rows, not separate hues.

### D-04: Color Approach
Follow the existing design system color variables (`styles/main.css`). Highlighted bars should draw from the brand palette (`#1a73e8`, `#059669`, `#f59e0b`, `#ef4444` and their lighter variants) but kept muted — avoid full-saturation fills. Monochromatic bars use the gray scale (`#94a3b8`, `#cbd5e1` family). Exact hex values are Claude's discretion — the constraint is classy, on-brand, not overwhelming.

### D-05: Library
**Chart.js via CDN** — load in `index.html` (same CDN pattern already used for Firebase). Pin a specific version for stability. Use Chart.js horizontal bar chart (`indexAxis: 'y'`).

### Claude's Discretion
- Exact Chart.js version to pin
- Canvas sizing / chart height per section
- Whether to hide zero-count bars or show them as zero-length bars
- Whether to animate on first render or disable animations (consider perf on a home page)
- Whether to use one Chart.js instance per section or one per card
- Exact muted color hex values for each highlighted status bar

</decisions>

## Existing Code Insights

### Reusable Assets
- `renderStatusBreakdown(containerId, countsMap)` — currently injects text rows into a container div; will be replaced with Chart.js chart initialization targeting a `<canvas>` element
- `buildStatusBreakdownContainer(containerId, countsMap, rowCount)` — currently returns a `.hs-status-grid` div; will need to return a `<canvas id="${containerId}">` instead
- `cachedStats` count maps (`projectsByInternalStatus`, `projectsByProjectStatus`, etc.) — already in the right shape; Chart.js just reads the same values and counts
- `statsListeners[]` — unchanged; no new Firestore queries needed

### Established Patterns
- Zero-build static site — no bundler; Chart.js must load from CDN in `index.html`
- Firebase already loaded via CDN — same `<script>` tag pattern applies
- Real-time: `onSnapshot` callbacks already compute count maps; Chart.js chart instances need `.update()` calls when data changes (analogous to current `renderStatusBreakdown` calls)

### Integration Points
- `index.html` — add Chart.js CDN `<script>` tag
- `buildStatusBreakdownContainer()` in `home.js` — change output from `.hs-status-grid` div to `<canvas>` element
- `renderStatusBreakdown()` in `home.js` — replace innerHTML injection with Chart.js chart create/update logic
- Chart instance registry needed: a module-level `Map` or object to track active Chart instances by containerId so `destroy()` can clean them up and `onSnapshot` callbacks can call `.update()` instead of recreating charts

<specifics>
## Specific Ideas

- Highlighted status bars: 4 distinct muted brand colors for the 4 highlighted statuses. The same color mapping applies consistently across Projects and Services cards (e.g., "On-going" is always the same color regardless of which card it appears in).
- Monochromatic bars: differentiated only by shade, not hue — conveys "less important" without being invisible.
- The visual hierarchy message: highlighted bars = "these are the statuses you should be watching"; monochromatic bars = "context / reference data."
