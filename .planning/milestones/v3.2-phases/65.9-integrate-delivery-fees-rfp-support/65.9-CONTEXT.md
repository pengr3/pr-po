# Phase 65.9: Integrate Delivery Fees to Enable RFP for Delivery - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Allow delivery fees already captured on PO documents (stored as `delivery_fee` field, set when status → Delivered) to flow through the existing RFP payment workflow. Procurement creates an RFP specifically for the delivery fee; Finance processes it through the existing Payables tab. No changes to how the delivery fee is captured (prompt dialog stays). No new views, no new Firestore collections.

</domain>

<decisions>
## Implementation Decisions

### RFP Trigger
- **D-01:** Extend the existing right-click context menu on PO ID cells to include a second option: "Request Delivery Fee Payment". The existing "Request Payment" (for tranches) stays unchanged.
- **D-02:** "Request Delivery Fee Payment" option is only visible/enabled when `po.delivery_fee > 0`. Hidden when no delivery fee is set.
- **D-03:** Guard against duplicates: if an RFP with `tranche_label='Delivery Fee'` already exists for this PO, the option is disabled (grayed out) with a note "RFP exists" — same deduplication pattern used for tranche RFPs.

### Data Structure
- **D-04:** Delivery fee RFPs stored in the existing `rfps` collection using a synthetic tranche approach: `tranche_label='Delivery Fee'`, `tranche_percentage=null` (or 0), `amount_requested=po.delivery_fee`. No new fields added to the rfps schema.
- **D-05:** A new `openDeliveryFeeRFPModal(poDocId)` function handles the delivery fee RFP creation. It shows a simplified version of the RFP modal (no tranche dropdown since it's always "Delivery Fee"): Supplier, Amount (pre-filled = delivery_fee, read-only), Due Date, Mode of Payment, Bank fields.

### Capture Flow
- **D-06:** The existing `prompt()` dialog that captures delivery fee when marking PO status → Delivered is NOT changed. Fee capture and RFP creation remain separate steps.

### Finance Payables Display
- **D-07:** Delivery fee RFPs appear in Finance Payables exactly the same as regular RFPs. The "Tranche Label" column will show "Delivery Fee" — self-explanatory, no additional markup needed.
- **D-08:** PO payment progress fill (`getPOPaymentFill()`) automatically includes delivery fee RFP payments since they're stored in `rfps` with the same `po_id` and already picked up by `rfpsByPO[]`. No code changes required here.

### Visual Indicator — Delivery Fee Status Dot
- **D-09:** A small colored dot (8px circle) is rendered in the top-right corner of the PO ID cell in PO Tracking, visible only when `po.delivery_fee > 0`.
- **D-10:** Dot colors use existing project palette:
  - Red (`#ef4444`) — delivery fee exists but NOT fully paid via RFP (includes: no RFP created yet, or RFP created but payment not complete)
  - Green (`#059669`) — delivery fee RFP is fully paid
- **D-11:** Hovering the dot shows a tooltip: `"Delivery Fee: PHP {amount} — Not yet paid"` or `"Delivery Fee: PHP {amount} — Paid"`.
- **D-12:** No dot shown on POs where `delivery_fee` is 0 or not set.

### Claude's Discretion
- Exact CSS positioning of the dot within the PO ID cell (use `position:relative` on cell, `position:absolute; top:4px; right:4px` on dot)
- Dot size (8px suggested, adjust if visually cramped)
- Tooltip implementation (native `title` attribute acceptable; styled tooltip if existing tooltip infrastructure exists)
- Whether the dot renders in the PO Tracking table only, or also in PO Payment Summary (Finance) — Claude's call based on what's practical

</decisions>
