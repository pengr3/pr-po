**Change 1: Add PROCUREMENT_STATUS_ORDER constant near the top of the file.**

After the `getPRPOSortIndicator()` function (after line 59), add the ordinal map constant. This maps every possible procurement status string to a numeric ordinal for sorting. The order follows the procurement process flow as specified by the user:

```javascript
// Ordinal order for Procurement Status sorting — matches process flow dropdown order
const PROCUREMENT_STATUS_ORDER = {
    // Material PO statuses (process order)
    'Pending Procurement': 1,
    'Procuring': 2,
    'Procured': 3,
    'Delivered': 4,
    // SUBCON PO statuses (process order)
    'Pending': 1,
    'Processing': 2,
    'Processed': 3
};
```

Note: 'Pending' (SUBCON) maps to ordinal 1, same as 'Pending Procurement' (Material). 'Processing' maps to 2, same as 'Procuring'. 'Processed' maps to 3, same as 'Procured'. This is intentional — both Material and SUBCON process flows share equivalent progression levels, so interleaving them in a unified sort is correct.

**Change 2: Store resolved `_procurement_status` on the MRF object during renderPRPORecords().**

Inside `renderPRPORecords()`, within the `pageItems.map(async (mrf) => { ... })` callback, AFTER the PO data is fully resolved (after the Material PO block ending at line 2775 — after `if (type === 'Material') { ... }`) and BEFORE the "Calculate detailed status" section (line 2777), add code to compute and store a summary procurement status on the MRF record:

```javascript
        // Store resolved procurement status on the MRF record for sorting
        // Use the least-progressed (bottleneck) PO status as summary
        if (poDataArray.length > 0) {
            let minOrdinal = Infinity;
            let summaryStatus = '';
            for (const po of poDataArray) {
                const defaultStatus = po.is_subcon ? 'Pending' : 'Pending Procurement';
                const status = po.procurement_status || defaultStatus;
                const ordinal = PROCUREMENT_STATUS_ORDER[status] || 0;
                if (ordinal < minOrdinal) {
                    minOrdinal = ordinal;
                    summaryStatus = status;
                }
            }
            mrf._procurement_status = summaryStatus;
        } else {
            mrf._procurement_status = '';  // No POs — Transport or no POs yet
        }
```

**IMPORTANT**: The `mrf` variable in the callback is a reference to the object in `pageItems`, which is a slice of `filteredPRPORecords`. But `filteredPRPORecords` is built from `allPRPORecords` via `filterPRPORecords()`. To make the `_procurement_status` field available for sorting (which sorts `allPRPORecords`), we need to store it back on the ORIGINAL record in `allPRPORecords`, not just on the page slice.

Since `filteredPRPORecords` is built by `filterPRPORecords()` which does:
```javascript
filteredPRPORecords = allPRPORecords.filter(...)
```
The `mrf` objects in `filteredPRPORecords` are the SAME object references as in `allPRPORecords` (JavaScript arrays copy references, not objects). So writing `mrf._procurement_status = summaryStatus` on the `pageItems` reference also writes it on the `allPRPORecords` object. No extra lookup needed.

**CAVEAT**: Only the current PAGE's MRFs get `_procurement_status` set during each render. MRFs on other pages will NOT have `_procurement_status` populated until they appear on a rendered page. This means sorting by procurement_status will work best AFTER the user has visited enough pages or after a full render cycle. For first-time sorts, MRFs that have never been rendered will have `_procurement_status` as `undefined`, which the sort function handles by pushing them to the end (null/undefined handling in the sort comparator).

This is an acceptable trade-off because:
1. The async Firestore queries per row make pre-computing all statuses expensive
2. Most users browse through pages sequentially, populating status as they go
3. The null-handling in the sort function gracefully handles unpopulated records

**Change 3: Add custom ordinal sort branch in `sortPRPORecords()` for the procurement_status column.**

Modify the `sortPRPORecords()` function (at lines 2399-2418) to handle `procurement_status` as a special column that uses ordinal lookup instead of string comparison.

Replace the existing `sortPRPORecords` function with:

```javascript
function sortPRPORecords(column) {
    if (prpoSortColumn === column) {
        prpoSortDirection = prpoSortDirection === 'asc' ? 'desc' : 'asc';
    } else {
        prpoSortColumn = column;
        prpoSortDirection = 'asc';
    }
    allPRPORecords.sort((a, b) => {
        let aVal = a[column];
        let bVal = b[column];
        // Custom ordinal sort for procurement status
        if (column === 'procurement_status') {
            aVal = a._procurement_status;
            bVal = b._procurement_status;
            // Empty/null statuses sort to end
            if (!aVal && !bVal) return 0;
            if (!aVal) return prpoSortDirection === 'asc' ? 1 : -1;
            if (!bVal) return prpoSortDirection === 'asc' ? -1 : 1;
            const aOrd = PROCUREMENT_STATUS_ORDER[aVal] || 99;
            const bOrd = PROCUREMENT_STATUS_ORDER[bVal] || 99;
            return prpoSortDirection === 'asc' ? aOrd - bOrd : bOrd - aOrd;
        }
        if (aVal == null) return prpoSortDirection === 'asc' ? 1 : -1;
        if (bVal == null) return prpoSortDirection === 'asc' ? -1 : 1;
        if (typeof aVal === 'string') {
            return prpoSortDirection === 'asc'
                ? aVal.localeCompare(bVal) : bVal.localeCompare(aVal);
        }
        return prpoSortDirection === 'asc' ? aVal - bVal : bVal - aVal;
    });
    filterPRPORecords();
}
```

Key details:
- The `column` parameter is `'procurement_status'` (clicked from the header), but the actual stored field on the MRF object is `_procurement_status` (prefixed with underscore to indicate it's synthetic/computed, not a Firestore field).
- Empty/falsy `_procurement_status` values (no POs, Transport type) sort to the end in ascending order.
- Ordinal lookup via `PROCUREMENT_STATUS_ORDER[status]` gives numeric values 1-4 for process-flow sorting.
- Unknown statuses fall back to ordinal 99 (sort to end).
