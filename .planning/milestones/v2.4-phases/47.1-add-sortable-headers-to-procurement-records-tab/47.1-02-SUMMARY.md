## Accomplishments

- Added PROCUREMENT_STATUS_ORDER constant mapping all 7 procurement status strings to ordinals (Material: 1-4, SUBCON: 1-3) matching process-flow dropdown order
- Implemented bottleneck aggregation in renderPRPORecords() — after PO data is resolved per row, finds the least-progressed PO status and stores it as mrf._procurement_status
- Added custom ordinal sort branch in sortPRPORecords() that uses PROCUREMENT_STATUS_ORDER lookup instead of alphabetical localeCompare
- Added same ordinal sort branch in loadPRPORecords() sort block for consistency
- Made Procurement Status `<th>` sortable with onclick, cursor:pointer, user-select:none, and sort indicator via getPRPOSortIndicator('procurement_status')

## Decisions Made

- PROCUREMENT_STATUS_ORDER maps 'Pending' (SUBCON) and 'Pending Procurement' (Material) both to ordinal 1 — equivalent first-step status in their respective flows, correct to interleave them in a unified sort
- Store-back pattern: writing mrf._procurement_status in the pageItems.map() callback propagates to allPRPORecords because JavaScript arrays hold object references (no extra lookup needed)
- Unknown statuses fall back to ordinal 99 so they sort to end without throwing errors
