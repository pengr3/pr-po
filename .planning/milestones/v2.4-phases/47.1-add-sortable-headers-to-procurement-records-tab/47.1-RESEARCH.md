# Phase 47.1: Add Sortable Headers to Procurement Records Tab - Research

**Researched:** 2026-03-01
**Domain:** Vanilla JS UI enhancement — sortable table headers in procurement.js MRF Records table
**Confidence:** HIGH

## Summary

Phase 47.1 extends the sortable headers pattern established in Phase 47 (finance.js Pending Approvals tables) to the Procurement Records tab (`#/procurement/records`). The Records tab renders a single merged table via `renderPRPORecords()` in `app/views/procurement.js`. This table currently has 8 static `<th>` columns: MRF ID, Project, Date Needed, PRs, POs, MRF Status, Procurement Status, Actions.

The critical architectural difference from the finance.js pattern is that the Records table builds the entire HTML including `<thead>` inside `renderPRPORecords()` — not in the static `render()` function. This means the headers and the sort indicators are regenerated on every render call, which simplifies indicator updates (they're just part of the regenerated HTML string) but also means the sort column headers must be embedded in the `renderPRPORecords()` output, not in `render()`.

Only a subset of columns are sortable candidates. "MRF ID" (`mrf_id`) and "Date Needed" (`date_needed`) map cleanly to fields on the MRF documents stored in `allPRPORecords`. "Project" is a composite display via `getMRFLabel()` spanning multiple fields. "PRs", "POs", and "Procurement Status" are dynamically fetched per row from Firestore and not stored on the MRF document — these cannot be sorted at the source array level. "MRF Status" maps to `mrf.status`. "Actions" is non-sortable.

**Primary recommendation:** Add sort state variables and a `sortPRPORecords()` function that sorts `allPRPORecords` then re-applies filters into `filteredPRPORecords` then calls `renderPRPORecords()`. Sortable columns: MRF ID (`mrf_id`), Date Needed (`date_needed`), MRF Status (`status`). Non-sortable: Project (composite), PRs (dynamically fetched), POs (dynamically fetched), Procurement Status (dynamically fetched), Actions. Sort indicators embed directly in the regenerated `<thead>` HTML in `renderPRPORecords()`.

### Pattern 1: Sort State Variables (existing project pattern)
**What:** Two module-level variables per sortable table: `column` string and `direction` `'asc'|'desc'`.
**When to use:** Any table needing persistent sort across re-renders.
**Example (from finance.js, same pattern used in Phase 47-01):**
```javascript
// At module level (alongside poData, allPRPORecords, etc.)
let prpoSortColumn = 'date_needed';   // default sort column
let prpoSortDirection = 'desc';        // newest first
```

### Pattern 2: Sort Function — Source Array Sort + Filter Re-Apply
**What:** Sort the source array (`allPRPORecords`), then regenerate `filteredPRPORecords` by re-applying filters, then call render.
**When to use:** When a filter function already handles deriving the display set from the source array.
**Example:**
```javascript
function sortPRPORecords(column) {
    if (prpoSortColumn === column) {
        prpoSortDirection = prpoSortDirection === 'asc' ? 'desc' : 'asc';
    } else {
        prpoSortColumn = column;
        prpoSortDirection = 'asc';
    }
    allPRPORecords.sort((a, b) => {
        let aVal = a[column];
        let bVal = b[column];
        if (aVal == null) return prpoSortDirection === 'asc' ? 1 : -1;
        if (bVal == null) return prpoSortDirection === 'asc' ? -1 : 1;
        if (typeof aVal === 'string') {
            return prpoSortDirection === 'asc'
                ? aVal.localeCompare(bVal) : bVal.localeCompare(aVal);
        }
        return prpoSortDirection === 'asc' ? aVal - bVal : bVal - aVal;
    });
    // Re-apply active filters so filteredPRPORecords reflects new order
    filterPRPORecords();   // filterPRPORecords() calls renderPRPORecords() internally
}
window.sortPRPORecords = sortPRPORecords;
```
Note: `filterPRPORecords()` already calls `renderPRPORecords()` at its end (line 2372), so calling `filterPRPORecords()` from the sort function is sufficient to trigger a full re-render with the new sort order applied.

### Pattern 3: Inline Sort Indicators in Regenerated Table HTML
**What:** Since `renderPRPORecords()` regenerates the entire table HTML including `<thead>`, sort indicators can be embedded directly in the template string using the current sort state.
**When to use:** When the entire table (including headers) is regenerated on every render call — not just the tbody.
**Example (in renderPRPORecords() `<thead>` template):**
```javascript
function getSortIndicator(col) {
    if (col === prpoSortColumn) {
        return `<span class="sort-indicator" style="color: #1a73e8; font-size: 0.7rem;">${prpoSortDirection === 'asc' ? ' \u2191' : ' \u2193'}</span>`;
    }
    return `<span class="sort-indicator" style="color: #94a3b8; font-size: 0.7rem;"> \u21C5</span>`;
}

// In the <thead> portion of renderPRPORecords():
`<th onclick="window.sortPRPORecords('mrf_id')" style="padding: 0.75rem 1rem; text-align: center; border-bottom: 2px solid #e5e7eb; font-size: 0.75rem; font-weight: 600; cursor: pointer; user-select: none;">
    MRF ID ${getSortIndicator('mrf_id')}
</th>`
```

This avoids the two-step approach (render then update indicators) used in finance.js where `<thead>` is static. Here, the indicators are always correct at render time.

### Pattern 4: Sort Preservation Across Loads
**What:** `loadPRPORecords()` (called on Refresh and on initial tab load) re-sorts `allPRPORecords` after fetch using a default date sort at line 2263. This default sort must be replaced by the user's current sort state — or the user-chosen sort must be re-applied after the load.
**Issue:** `loadPRPORecords()` currently sorts by `created_at`/`date_submitted` descending (lines 2263-2267). If a user has sorted by MRF ID before clicking Refresh, the refresh resets sort back to date order.
**Resolution options:**
1. After the default sort in `loadPRPORecords()`, immediately re-apply `allPRPORecords.sort()` using `prpoSortColumn`/`prpoSortDirection`. This means the initial default sort (date desc) only applies when sort state is at default. Simplest.
2. Remove the hardcoded sort from `loadPRPORecords()` and rely entirely on `prpoSortColumn`/`prpoSortDirection` — starting with defaults of `date_needed` desc.
**Recommendation:** Option 2 — remove the hardcoded sort from `loadPRPORecords()` and let the sort state variables fully control ordering. Initialize defaults to match the current behavior (newest first). Then sort using the sort state variables before calling `filterPRPORecords()`.

### Anti-Patterns to Avoid
- **Sorting `filteredPRPORecords` directly:** `filterPRPORecords()` regenerates `filteredPRPORecords` from `allPRPORecords` on every filter change, discarding any sort applied to `filteredPRPORecords`. Always sort `allPRPORecords`.
- **Post-render `querySelectorAll` for indicator updates:** Unlike finance.js (where `<thead>` is static), the Records table is fully regenerated. There is no "post-render indicator update" step — indicators are embedded in the template.
- **Making "PRs", "POs", or "Procurement Status" sortable:** These values are fetched from Firestore per row inside `renderPRPORecords()` using async per-MRF queries. The underlying sort field would have to be known before render, but these values don't exist on the MRF document. Sorting by them would require fetching all PR/PO data upfront and pre-computing sort values — excessive complexity for a simple UX enhancement.
- **Making "Project" sortable:** This is a composite display via `getMRFLabel()` that combines `project_name`, `service_name`, and visual badges. While `project_name` or `service_name` are individual sortable fields on the MRF, sorting by `getMRFLabel()` output would require calling that function on every comparison — complex and slow.

## Common Pitfalls

### Pitfall 1: filterPRPORecords() resets page to 1 — breaks sort-after-paginate
**What goes wrong:** `filterPRPORecords()` sets `prpoCurrentPage = 1` (line 2371). Calling it from `sortPRPORecords()` always returns user to page 1.
**Why it happens:** Filter resets pagination as a feature — when you change filters, page 1 is correct. Sort should also reset to page 1 (standard UX for sorting).
**How to avoid:** This is actually correct behavior. Sorting resets to page 1. No special handling needed.

### Pitfall 2: loadPRPORecords() hardcoded sort conflicts with sort state
**What goes wrong:** `loadPRPORecords()` sorts `allPRPORecords` by `created_at`/`date_submitted` descending (lines 2263-2267) before `filteredPRPORecords = [...allPRPORecords]`. If a user sorts by MRF ID then hits Refresh, the sort resets.
**Why it happens:** The initial sort in `loadPRPORecords()` predates sortable headers.
**How to avoid:** Replace the hardcoded sort in `loadPRPORecords()` with a call using `prpoSortColumn`/`prpoSortDirection` state variables.

### Pitfall 3: renderPRPORecords() is async — sort indicator must account for async rendering
**What goes wrong:** `renderPRPORecords()` is declared `async` (line 2518) and awaits `Promise.all(pageItems.map(...))` inside. The container is set to a loading placeholder first (line 2542), then the full table is set on line 2845. Sort indicators embedded in the template will be correct because they're evaluated at template string build time (after the sort has already been applied).
**Why it happens:** Async data fetching for PR/PO sub-documents per page.
**How to avoid:** No special handling needed — template literal with `getSortIndicator()` is evaluated synchronously when the html string is built (line 2823), which happens after the await completes. Indicators will always show the correct sort state.

### Pitfall 4: Empty state returns early before setting container.innerHTML
**What goes wrong:** `renderPRPORecords()` returns early at line 2532 when `filteredPRPORecords.length === 0`, setting a plain text placeholder. The table (including sortable headers) does not render.
**Why it happens:** Empty state check before table build.
**How to avoid:** This is acceptable — when there are no records, there's nothing to sort. The sort state persists in module variables and will be applied when records are loaded. No change needed for empty state.

### Pitfall 5: window function not cleaned up in destroy()
**What goes wrong:** `window.sortPRPORecords` lingers after navigating away, potentially conflicting if another view defines the same name.
**Why it happens:** Missing `delete window.sortPRPORecords` in `destroy()`.
**How to avoid:** Add `delete window.sortPRPORecords` to `destroy()` alongside existing window deletions (lines 564-603). Also add `window.sortPRPORecords = sortPRPORecords` in `attachWindowFunctions()`.

## Open Questions

1. **Should `urgency_level` be a sortable column in the Records table?**
   - What we know: `urgency_level` is a field on MRF documents. Finance tab already sorts by urgency. Records table does not currently display urgency.
   - What's unclear: Whether procurement users want to see urgency in the Records tab at all (it's not a current column).
   - Recommendation: Out of scope for 47.1 — only make existing columns sortable. Adding a new column is a separate UX decision.

2. **Should `date_needed` vs `created_at`/`date_submitted` be used as the default sort?**
   - What we know: The current hardcoded sort uses `created_at || date_submitted`. The column header says "Date Needed" which maps to `date_needed`.
   - What's unclear: Whether some MRFs have null `date_needed` (transport requests use `date_submitted`).
   - Recommendation: Use `date_needed` as default sort since that matches the column header. Null handling (` aVal == null ? 1 : -1`) covers transport requests.
