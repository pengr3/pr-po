## Accomplishments
- Added sort state variables `prpoSortColumn` (default: `date_needed`) and `prpoSortDirection` (default: `desc`) matching existing behavior
- Added `sortPRPORecords()` that sorts `allPRPORecords` source array and calls `filterPRPORecords()` for re-render
- Added `getPRPOSortIndicator()` helper that returns blue directional arrow for active column, gray up-down arrow for inactive
- Replaced static thead in `renderPRPORecords()` with three clickable sortable headers (MRF ID, Date Needed, MRF Status)
- Replaced hardcoded `created_at`/`date_submitted` sort in `loadPRPORecords()` with sort-state-aware sort — sort persists across Refresh

## Decisions Made
- Sort `allPRPORecords` (source array), not `filteredPRPORecords` — consistent with Phase 47-01 pattern and ensures filter re-application correctly inherits sort order
- Embed `getPRPOSortIndicator()` directly in the thead template literal rather than updating DOM post-render — procurement.js `renderPRPORecords()` regenerates the entire table HTML (unlike finance.js where thead is static), so indicators are always correct at render time
- Default sort set to `date_needed desc` to reproduce prior behavior (hardcoded `created_at` sort approximated date-needed ordering)
- `date_needed` is a plain string field (ISO date), so `localeCompare` correctly sorts it — no special date handling needed (matches Phase 47-03 decision for PR/TR date fields)
