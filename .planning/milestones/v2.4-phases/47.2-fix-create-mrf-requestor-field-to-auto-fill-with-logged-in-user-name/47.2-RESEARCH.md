**Researched:** 2026-03-01

**Confidence:** HIGH

## Summary

This is a small, well-scoped bug fix. The Procurement > Create MRF form at `#/procurement` renders a Requestor Name text input that starts empty. The fix is to pre-populate that input with the currently logged-in user's `full_name` at the moment `createNewMRF()` is called.

The user data is already available synchronously via `window.getCurrentUser()`, which is exported from `app/auth.js` and exposed to `window` at module load. The identical fix already exists and works correctly in the standalone MRF form at `#/mrf-form` (`mrf-form.js` lines 368-374). The procurement form just needs the same pattern applied.

**Primary recommendation:** In `createNewMRF()` in `app/views/procurement.js`, set `requestor_name` to `window.getCurrentUser()?.full_name || ''` when building the `currentMRF` object. The existing template string in `renderMRFDetails()` already uses `value="${mrf.requestor_name || ''}"`, so the pre-fill flows naturally with no HTML changes needed.

## Architecture Patterns

### Recommended Project Structure
```
app/views/procurement.js   — one-line change in createNewMRF()

```

### Pattern 1: Pre-populate from getCurrentUser() at object construction time
**What:** When building the `currentMRF` template object in `createNewMRF()`, read the user's `full_name` immediately and embed it in the object. The template string in `renderMRFDetails()` then picks it up via `${mrf.requestor_name || ''}`.
**When to use:** Any form that should be pre-filled with the current user's identity.
**Example (from mrf-form.js lines 367-374 — reference implementation):**
```javascript
// Auto-populate requestor name from current user
const user = window.getCurrentUser?.();
if (user && user.full_name) {
    const requestorInput = document.getElementById('requestorName');
    if (requestorInput) {
        requestorInput.value = user.full_name;
    }
}
```
The procurement.js approach is even simpler because `currentMRF.requestor_name` is already wired into the HTML template, so it suffices to set it on the object:
```javascript
currentMRF = {
    id: null,
    mrf_id: 'NEW',
    request_type: 'material',
    urgency_level: 'Low',
    project_name: '',
    requestor_name: window.getCurrentUser?.()?.full_name || '',  // THE FIX
    date_needed: '',
    delivery_address: '',
    items_json: JSON.stringify([...]),
    status: 'Pending'
};
```

### Pattern 2: Input rendered as editable (not readonly) in procurement form
**What:** Unlike the standalone `mrf-form.js` where the Requestor Name field is `readonly` (styled with `cursor: not-allowed`), in `procurement.js` the Create MRF form renders the input as a plain editable text field. This is intentional — procurement staff may create MRFs on behalf of others.
**Implication:** The fix should pre-fill, not lock. The field stays editable so a procurement user can override the name if needed.

### Anti-Patterns to Avoid
- **Async fetch for user data:** `getCurrentUser()` is synchronous (returns module-level `currentUser` variable updated by onSnapshot). Do not `await getDoc()` here — it's unnecessary overhead.
- **Post-render setTimeout to set input value:** Injecting after render is fragile. Setting `requestor_name` on `currentMRF` before calling `renderMRFDetails()` is the correct, reliable approach.
- **Using `auth.currentUser.displayName`:** Firebase Auth `displayName` is not populated in this project. The user profile is stored in Firestore `users` collection with field `full_name`. Must use `getCurrentUser()` to get the correct value.

## Common Pitfalls

### Pitfall 1: getCurrentUser() returns null
**What goes wrong:** If called before auth observer fires, `getCurrentUser()` returns null.
**Why it happens:** Auth state is async on first page load.
**How to avoid:** Use optional chaining `window.getCurrentUser?.()?.full_name || ''`. If null, the field just stays empty (same as current behavior) — no regression.
**Warning signs:** Requestor field is blank even after the fix. Check if auth observer has fired by checking console for `[Auth] User signed in:` message.

### Pitfall 2: Field is editable and user could clear it
**What goes wrong:** Pre-filled value is cleared by procurement staff.
**Why it happens:** Field is intentionally editable in procurement.js (unlike the readonly field in mrf-form.js).
**How to avoid:** This is by design. The pre-fill is a convenience default. Validation in `saveNewMRF()` (line 1547) already enforces that `requestorName` is not empty — so if the user clears the field, they'll get a toast error and must fill it.

### Pitfall 3: Applying readonly styling unnecessarily
**What goes wrong:** Copying the `readonly` attribute and `cursor: not-allowed` styles from mrf-form.js into procurement.js.
**Why it happens:** The developer sees the reference implementation and copies it verbatim.
**How to avoid:** The procurement Create MRF form uses a plain editable input (line 1051 of procurement.js). Pre-fill only; do NOT add `readonly`.
