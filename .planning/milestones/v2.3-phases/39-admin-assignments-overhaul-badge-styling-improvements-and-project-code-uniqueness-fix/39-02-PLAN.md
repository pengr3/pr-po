  <name>Task 1: Create unified assignments.js module</name>
  <files>app/views/assignments.js</files>
  <action>
Create `app/views/assignments.js` — a new view module following the standard `render()`/`init()`/`destroy()` lifecycle pattern. This replaces both `project-assignments.js` and `service-assignments.js`.

**Module state:**
```javascript
let activeSubTab = 'projects'; // 'projects' | 'services'
let usersData = [];          // Users from Firestore (filtered by assignable roles)
let projectsData = [];       // All projects
let servicesData = [];       // All services
let listeners = [];
let pendingModalCodes = null; // Set of codes currently checked in open modal, or null if modal closed
let currentModalUserId = null;
let currentModalType = null;
```

**render(subTab):**
- Returns HTML with toggle sub-tabs at top: [Projects] [Services] buttons
- Table container below with id `assignmentsTableContainer`
- Sub-tab visibility based on current user role:
  - `super_admin`: both tabs visible
  - `operations_admin`: only Projects tab visible
  - `services_admin`: only Services tab visible
- Default `subTab` = 'projects' (or 'services' if only services tab visible)

**init(subTab):**
- Set `activeSubTab` from parameter
- Register window functions: `window.openManageModal`, `window.saveManageModal`, `window.closeManageModal`, `window.switchAssignmentSubTab`, `window.filterModalItems`
- Set up 3 Firestore `onSnapshot` listeners, push all to `listeners[]`:
  1. `users` collection with `where('status', '==', 'approved')` — all approved users
  2. `projects` collection — all projects (admin can assign any)
  3. `services` collection — all services (admin can assign any)
- Each listener updates its respective data array and calls `renderUsersTable()`

**destroy():**
- Unsubscribe all listeners
- Clear `listeners = []`
- Delete all window functions
- Reset all module state

**renderUsersTable():**
- Guard: if modal is open (`pendingModalCodes !== null`), skip re-render to prevent modal disappearing (Pitfall 2 from research)
- Filter `usersData` by sub-tab:
  - Projects tab: show users with `role` in `['operations_user', 'operations_admin']`
  - Services tab: show users with `role` in `['services_user', 'services_admin']`
- Render table with columns:
  - **Name**: `user.full_name` or `user.display_name` (with email in small gray text below)
  - **Role**: Role label (use existing role label pattern)
  - **Assignment Count**: Call `getAssignmentCount(user, activeSubTab)` — shows count, "None", or "All (legacy)"
  - **Actions**: `<button onclick="window.openManageModal('${user.id}', '${activeSubTab}')">Manage</button>`
- Use consistent table styling (`.table` class pattern from existing views)

**getAssignmentCount(user, type):**
```javascript
if (type === 'projects') {
    if (user.all_projects === true) return 'All (legacy)';
    const codes = user.assigned_project_codes || [];
    return codes.length === 0 ? 'None' : `${codes.length} project(s)`;
} else {
    if (user.all_services === true) return 'All (legacy)';
    const codes = user.assigned_service_codes || [];
    return codes.length === 0 ? 'None' : `${codes.length} service(s)`;
}
```

**openManageModal(userId, type):**
- Find user in `usersData`
- Determine current codes:
  - For projects: if `user.all_projects === true`, pre-populate with ALL project codes from `projectsData` (Pitfall 1: migrate-on-edit). Otherwise use `user.assigned_project_codes || []`
  - For services: if `user.all_services === true`, pre-populate with ALL service codes from `servicesData`. Otherwise use `user.assigned_service_codes || []`
- Set `pendingModalCodes = new Set(currentCodes)`, `currentModalUserId = userId`, `currentModalType = type`
- Render modal HTML:
  - Title: "Manage {Projects|Services} for {user.full_name}"
  - Search input with `oninput="window.filterModalItems(this.value)"`
  - Scrollable list of checkboxes (max-height ~400px, overflow-y auto):
    - For projects: each item shows `project_code - project_name` (or `service_name` if `client_name`)
    - For services: each item shows `service_code - service_name`
    - Checkbox checked if code is in `pendingModalCodes`
    - Checkbox `onchange` toggles the code in `pendingModalCodes` Set
  - Footer: [Cancel] and [Save] buttons
- Use inline modal pattern (window-style, blur backdrop, centered) consistent with existing modals

**filterModalItems(query):**
- Lowercase the query
- Show/hide checkbox items based on whether code or name includes the query

**saveManageModal():**
- Get new codes array from `pendingModalCodes`
- Import `updateDoc`, `doc` from firebase
- Write to Firestore:
  ```javascript
  const field = currentModalType === 'projects' ? 'assigned_project_codes' : 'assigned_service_codes';
  const allFlag = currentModalType === 'projects' ? 'all_projects' : 'all_services';
  await updateDoc(doc(db, 'users', currentModalUserId), {
      [field]: [...pendingModalCodes],
      [allFlag]: false  // Explicit assignments only going forward
  });
  ```
- For projects only: call reverse sync to update project `personnel_user_ids`/`personnel_names` arrays. Import and call `syncPersonnelToAssignments` from `utils.js` (if available) or implement the sync inline — when codes are ADDED, add user to those project docs' personnel arrays; when codes are REMOVED, remove user from those project docs' personnel arrays. Check what `project-assignments.js` does for its sync pattern and replicate it.
- For services: NO reverse sync needed (research confirms this asymmetry is intentional)
- Close modal: `pendingModalCodes = null; currentModalUserId = null; currentModalType = null;`
- Re-render table

**closeManageModal():**
- Reset modal state: `pendingModalCodes = null; currentModalUserId = null; currentModalType = null;`
- Remove modal from DOM
- Re-render table (to pick up any Firestore updates that were blocked during modal)

**switchAssignmentSubTab(subTab):**
- Set `activeSubTab = subTab`
- Update sub-tab button active states
- Call `renderUsersTable()`
