<objective>
Fix two permission guard bugs in service-detail.js that cause edit controls to be visible and operable for read-only roles (services_user).

Purpose: canEditTab() returns undefined while permissions load. The existing guard `canEdit !== false` treats undefined as "allowed" — showing edit inputs to services_user until permissions finish loading. After load, permissionsChanged fires and re-renders, but the brief window allows input interaction. The same bug in saveServiceField() means a fast user could trigger a Firestore write that returns permission-denied. As defense-in-depth, refreshServiceExpense() gains an early-exit guard so it does not flood the console with 403 errors when a services_user visits a service detail page (primary fix for the 403 is in Plan 03 — Firestore rules).

Output: Two one-line changes in service-detail.js fix the race condition. One guard added to refreshServiceExpense().
</objective>
