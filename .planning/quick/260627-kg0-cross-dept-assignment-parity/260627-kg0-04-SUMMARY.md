# Quick 260627-kg0 — UAT follow-ups (2 issues flagged during two-direction UAT)

UAT verdict on the core feature: **APPROVED both directions.** Two issues flagged
and fixed below. Both are **client-only JS** — no firestore.rules change, so no
rules redeploy; just serve the new JS to DEV and re-test the two scenarios.

## Issue 2 (commit 38a3afcb) — assignments lost on role change

**Root cause:** the role-change handler (user-management.js:~1447) rebuilt the new
role's department fields and reset `assigned_*_codes = []`. That cache is a
denormalized copy of `personnel_user_ids` on the project/service docs — clearing it
while leaving the user in `personnel_user_ids` desynced the two, and under the
assignment-driven rules (wave 1) the user's writes to still-assigned work were then
silently denied.

**Fix (D: persist codes, reset all_*):** the handler now PRESERVES
`assigned_project_codes` / `assigned_service_codes` across role changes and only
(re)sets BOTH `all_projects` + `all_services` flags per role. Demotion / cross-dept
switch still strips unrestricted access; specific assignments survive. Also fixes a
latent gap where a cross-dept admin→user switch left the prior department's
`all_* = true`.

**Left intentionally:** the approve-user handler (~768) still initializes
`assigned_*_codes = []` — that path activates a brand-new user with no prior
assignments, so initialization (not persistence) is correct there.

## Issue 1 (commit cabd69ad) — misleading "view-only" banner

**Root cause:** the banner was gated on `canEditTab('projects'/'services')`, a blanket
role-tab flag, but edit capability is now per-assignment.

**Fix (D: assignment-aware banner):**
- detail pages: banner shows only when the user has NO write path to THIS item
  (`canEdit === false && !inPersonnelUserIds`).
- list pages: banner suppressed for a `*_user` holding ≥1 assignment in that area;
  pure viewers still see it.
- `showEditControls` (inline general-field edit gating) left unchanged — the rules
  restrict general-field writes to admins/finance regardless, so the banner now tells
  the truth without enabling fields the rules would deny.

## Issue 2b (commit 85dfa8fa) — cross-dept assignees vanished from the Assignments grid

**Root cause (found on re-test):** the role-change DATA fix (38a3afcb) was correct — the
assignments persisted — but the Assignments admin view (`assignments.js:258`) bucketed users
into the Projects/Services sub-tabs purely by role, so after a role switch the user moved to
the other sub-tab and their prior-department assignments disappeared from the grid (orphaned in
the UI, intact in the data).

**Fix (D: show those with assignments):** each sub-tab now ALSO lists a cross-department *_user
who holds assignments in that department (a services_user with `assigned_project_codes` in the
Projects tab; an operations_user with `assigned_service_codes` in the Services tab). The per-row
count + Manage modal already read the sub-tab's own field, so they work unchanged.

**Known limitation (pre-existing, logged):** the Projects Manage save reverse-syncs project
`personnel_user_ids`; the Services Manage save does not (services were built without that reverse
sync). Managing a cross-dept user's services from this grid updates `assigned_service_codes` only.
