# Quick 260627-kg0 Plan 02 — assignment-driven helpers + procurement union

## What changed

**app/utils.js** — added module-scope `SCOPE_EXEMPT_ROLES = [super_admin,
operations_admin, services_admin, finance, procurement]`. Both
`getAssignedProjectCodes()` / `getAssignedServiceCodes()` rewritten: the old
single-role gate (`role !== 'operations_user'` / `!== 'services_user'`) replaced
with `SCOPE_EXEMPT_ROLES.includes(role) -> null`. Now BOTH `*_user` roles get
their assigned codes; any non-exempt/unknown role falls through to its
assigned-codes array (`[]` default = FAIL-CLOSED, sees nothing). `all_projects` /
`all_services` escape hatches kept (no-op for the cross-dept role).

**app/views/procurement.js** — added `isMrfInAssignedScope(mrf)` union predicate
and replaced all 5 sequential project-then-service scope filters with it:
- loadMRFs onSnapshot, reFilterAndRenderMRFs, reFilterAndRenderPRPORecords,
  loadPRPORecords (cache-hit + fresh-fetch).
`loadServicesForNewMRF` de-roled: dropped `if (role === 'operations_user') return`;
service-MRF availability is now driven by `assigned_service_codes`
(an ops_user with no service assignments hits the `length === 0` early-return).

## Why union (OR), not the old sequential AND

The two old `if (...!== null)` blocks were an implicit AND. Once the helpers
return non-null for BOTH directions on a cross-dept user, that AND would drop the
user's OWN department's rows (a project MRF has `service_code === ''`, so the
service filter removed it). The union `matchesProject || matchesService` is the
correct semantics; for a single-dept user (one non-null scope) it collapses to
that one scope → identical visible set.
