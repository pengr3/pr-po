Purpose: Make the iteration save model match user mental model — load a named iteration, edit it, hit Save, and the same iteration is updated in place. Make the rail's "you are currently working on X" feedback persist across page refreshes.

Make the following coordinated edits to `app/views/project-plan.js`. All edits are scoped to functions touching iteration save/load state — do NOT modify `deleteIteration()`, `restoreIteration()` batch-write logic, `undoIterRestore()`, `dismissUndoToast()`, or any baseline/Gantt/grid code.

1) **Persist last-loaded id on successful restore** — In `restoreIteration(iterationId)`, after the existing `_loadedIterationId = iter.auto ? null : iterationId;` / `_loadedIterationLabel = iter.auto ? '' : (iter.label || '');` block (~line 3721-3722, immediately after STEP 2 batch-commit success), add a non-fatal try/catch that writes `last_loaded_iteration_id` to the project doc:
   - Use `updateDoc(doc(db, 'projects', currentProject.id), { last_loaded_iteration_id: iter.auto ? null : iterationId })`.
   - Wrap in its own try/catch — log to `console.error` on failure but DO NOT throw (the in-memory state is already correct; persistence is a nice-to-have).
   - Place BEFORE the `await loadIterations();` line that follows, so the field is persisted before the rail re-renders.

2) **Read last-loaded id on init** — Add a new module-scope helper `function applyLastLoadedFromProject()`:
   - Reads `currentProject?.last_loaded_iteration_id` (truthy guard).
   - Finds the matching iteration in `_iterations` by id (skip if not found — iteration may have been deleted; clear the stale id by updating the project doc with `last_loaded_iteration_id: null`, non-fatal try/catch).
   - If a matching non-auto iteration is found, sets `_loadedIterationId = iter.id` and `_loadedIterationLabel = iter.label || ''`.
   - If no match (or auto), leaves the module vars at their current null/'' values.
   - This helper is synchronous (no awaits inside) — module state set, no DB writes except the stale-id cleanup which is fire-and-forget (no await, just call `.catch(()=>{})`).
   - Call this helper in `init()` BETWEEN `await loadIterations();` and `renderIterRail();` (the two existing lines ~372-374). Order matters: `_iterations` must be populated before the helper runs, and `renderIterRail()` must run AFTER to paint the badge + strip.

3) **Clear persisted id on auto/null path** — Inside the same try/catch added in step 1, when `iter.auto` is true the value written is `null` — same call already handles it via the ternary. No extra branch needed.

4) **Overwrite-vs-SaveAs branch in `saveIteration(label = null)`** — Restructure the `label === null` branch:
   - Determine `isOverwrite = !!_loadedIterationId` at the top of the null-label branch.
   - When `isOverwrite` is true: SKIP showing the modal entirely. Instead directly call the overwrite path described in step 5 (passing the existing loaded label) and return.
   - When `isOverwrite` is false: show the existing iterSaveModal as today, but change the modal's heading text to `Save Iteration As` and the confirm button text to `Save As` (visual signal to the user that this creates a new entry). Keep the input pre-filled with `Iteration ${_iterSeq + 1}`. Keep the Enter-key handler and the `saveIteration(lbl)` re-entry pattern.

5) **Overwrite path** — Add a new function `async function overwriteLoadedIteration()`:
   - Pre-conditions: `_loadedIterationId` truthy, otherwise no-op + return.
   - Build the same `snapshot` array currently built in `saveIteration(label)` (the 17-field `tasks.map(t => ({ id, task_id, project_id, ... }))` block — extract into the new function or copy verbatim; copy is fine to keep blast radius small).
   - Call `updateDoc(doc(db, 'project_iterations', _loadedIterationId), { tasks: snapshot, saved_at: serverTimestamp() })`. **Do NOT mutate `label`, `project_id`, or `auto` fields** — overwrite only the snapshot and timestamp.
   - After successful updateDoc: call `await loadIterations(); renderIterRail();` (matching existing flow).
   - Toast: `showToast('Iteration "${_loadedIterationLabel}" updated.', 'success')`.
   - Handle the `_pendingLoadAfterSave` chain identically to the existing save path: if set, pop it and call `openIterConfirm(targetId)`.
   - Error path: wrap in try/catch, log to `console.error('[Plan] overwriteLoadedIteration error:', e);`, `showToast('Failed to update iteration.', 'error')`, set `_pendingLoadAfterSave = null` on failure (matching existing pattern at line ~3469).

6) **Wire the overwrite call in the null-label branch** — At the top of `saveIteration(label = null)`'s null-label branch, before the modal code, add:
   ```
   if (_loadedIterationId) {
       return overwriteLoadedIteration();
   }
   ```
   This is the single new branch; the rest of the null-label code (modal creation + Save As confirm button) is reached only when no named iteration is loaded.

7) **Preserve & Load chain in modal path** — The existing modal's confirm button calls `saveIteration(lbl)` which already handles `_pendingLoadAfterSave`. No change needed there. The Save & Load entry point at line ~3633-3634 calls `saveIteration(null)` which now correctly routes to overwrite when a named iteration is loaded — that's the desired behavior per must_haves.
