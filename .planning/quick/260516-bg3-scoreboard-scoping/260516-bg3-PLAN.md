# Plan: Scoreboard scoping fix (Bug 3 of Phase 91 UAT)

## Goal

Stop showing operations_user / services_user scoreboard counts that include POs they cannot see. After this fix, the PO scoreboards on the MRF Records tab must reflect ONLY POs tied to the MRFs the user is scoped to.

## Diagnosis (verified)

`loadPRPORecords` (procurement.js:5034) fetches the entire `pos` collection into a local `allPOData` and passes the full set to `updatePOScoreboards` at line 5117 — no scope is applied to the scoreboard input. Separately, `reFilterAndRenderPRPORecords` (procurement.js:2788) re-scopes `allPRPORecords` on `assignmentsChanged` but never refreshes the scoreboard, so the DOM keeps showing the previous (often wider) counts.

## Diagnosis correction vs. continue-here.md

The continue-here.md instructed building `visibleMrfIds` from `m.id` (Firestore doc ID) and matching against `po.mrf_id`. That is wrong for this codebase. Every PO↔MRF join in procurement.js (lines 915, 3380, 4395, 5313, 5531, plus the explicit `// human ID like "MRF-2026-014"` comment at 3369) uses the **human-readable `mrf_id` string**. POs link to MRFs via `po.mrf_id === mrf.mrf_id`. The Set must therefore be built from `mrf.mrf_id`, not `mrf.id`.

## Approach

1. Add a module-level cache: `let cachedAllPOData = []` near `cachedAllPRPORecords` (procurement.js:31).
2. In `loadPRPORecords` fresh-fetch branch (around lines 5108–5117): save fetched POs to `cachedAllPOData`, derive `visibleMrfIds` from the already-scoped `allPRPORecords` via `mrf.mrf_id`, and pass the filtered set to `updatePOScoreboards`.
3. In `reFilterAndRenderPRPORecords` (procurement.js:2788): after the existing scope filters set `allPRPORecords = scoped`, derive `visibleMrfIds` and call `updatePOScoreboards(cachedAllPOData.filter(po => visibleMrfIds.has(po.mrf_id)))`.
4. Reset `cachedAllPOData = []` alongside the existing `cachedAllPRPORecords = []` resets at procurement.js:2470 and 2500 (destroy + signed-out paths) so stale data does not leak across user switches.

The cache-hit branch of `loadPRPORecords` (5036–5052) intentionally does NOT add a fresh scoreboard call — the most recent scoreboard write was already scoped, and the DOM is stable across tab re-entry. `reFilterAndRenderPRPORecords` covers the only state-change path (assignmentsChanged).

## must_haves

- `cachedAllPOData` declared at module scope alongside `cachedAllPRPORecords`
- Fresh-fetch path in `loadPRPORecords` populates `cachedAllPOData` and passes scoped POs to `updatePOScoreboards`
- `reFilterAndRenderPRPORecords` calls `updatePOScoreboards` with the scoped subset
- Visible-MRF Set is built from `mrf.mrf_id` (human-readable code), not `mrf.id`
- Both reset paths (destroy, signed-out) clear `cachedAllPOData`
- No other call sites of `updatePOScoreboards` touched (loadPOTracking at line 7045 is a separate tab/listener with different scope semantics)
