# Quick Task 20260531-mbl: Multi-baseline selector + Set/Clear toggle

## Boundary

Phase 86.12 shipped a single-baseline overlay. Browser UAT surfaced two UX gaps:

1. **No multi-baseline support** — Clicking "Set Baseline" silently overwrites the visible
   baseline with a new doc. User had no way to compare or pick among saved baselines.
2. **No clear/dismiss path** — Once an overlay was rendered, there was no UI to remove it.

This quick task implements the agreed model (see HANDOFF.json decisions, 2026-05-31):

- Baselines accumulate in Firestore — never overwritten. `Baseline 1`, `Baseline 2`, …
- A selector `<select>` in the toolbar lists all baselines newest-first; picking one drives
  the overlay against that doc's tasks map.
- A single toggle button flips between "Set Baseline" and "Clear Baseline" based on whether
  an overlay is currently active.
  - **Set Baseline** (idle) → `saveBaseline()` creates a new `Baseline N+1` doc, auto-selects
    it, renders the overlay.
  - **Clear Baseline** (active) → hides the overlay, resets selector to "— none —". Does
    **not** delete any Firestore docs.
- No delete-per-baseline UI for now (decision deferred per HANDOFF).

## Tasks

### Task 1 — Data layer: `loadBaselines` (load all)

**Files:** `app/views/project-plan.js`

**Action:**
- Add module state: `_baselines = []` (all baseline docs, ordered desc) and
  `_activeBaselineId = null` (which one drives the overlay; null = no overlay).
- Replace `loadBaseline()` with `loadBaselines()`:
  - `getDocs(query(...baselines, orderBy('created_at','desc')))` — no `limit(1)`.
  - Populate `_baselines` from snap.docs.
  - On initial load, auto-select most-recent: `_activeBaselineId = _baselines[0]?.id ?? null`
    and `_baselineData = _baselines[0] ?? null` for back-compat with existing overlay code.
- Update the `init()` caller (line ~195) to use `loadBaselines()`.
- Update `destroy()` to clear new state: `_baselines = []; _activeBaselineId = null;`
  (line ~369 area).

**Verify:** Load a project page that has 2+ baselines saved. Console:
`_baselines.length === N` and `_activeBaselineId === _baselines[0].id`.

**Done:** Initial render shows the most-recent baseline overlay automatically. No regression
versus current behavior when only 1 baseline exists.

### Task 2 — UI: selector + toggle button + handlers

**Files:** `app/views/project-plan.js`

**Action:**
- In the toolbar `render()` (around line 135), replace the lone Set Baseline button with:
  - `<select id="baselineSelect" class="plan-baseline-select" onchange="window.selectBaseline(this.value)">` (rendered empty by default; populated by `updateBaselineToolbarUI()`).
  - `<button class="plan-export-btn plan-baseline-btn" onclick="window.toggleBaseline()" id="baselineToggleBtn">Set Baseline</button>` (label is updated by `updateBaselineToolbarUI()`).
- Add three new functions:
  - `toggleBaseline()` — if `_activeBaselineId` is set, calls `clearBaseline()`; else calls `saveBaseline()`.
  - `selectBaseline(id)` — sets `_activeBaselineId` and `_baselineData` from `_baselines`; calls `injectBaselineOverlay()` + `renderSlipSummary()` + `updateBaselineToolbarUI()`. Empty `id` = clear path (delegates to `clearBaseline()`).
  - `clearBaseline()` — sets `_activeBaselineId = null` and `_baselineData = null`, removes overlay SVG elements (`.gantt-baseline-outline`, `.gantt-slip-badge`, `.gantt-slip-badge-bg`), calls `renderSlipSummary()` (auto-hides), calls `updateBaselineToolbarUI()`.
  - `updateBaselineToolbarUI()` — rebuilds `<select>` options from `_baselines` (with `— none —` as first option); flips toggle button label between "Set Baseline" / "Clear Baseline" based on `_activeBaselineId`; updates title attribute. Uses `escapeHTML()` on label.
- Wire window exports + destroy cleanup:
  - `window.toggleBaseline = toggleBaseline; window.selectBaseline = selectBaseline; window.clearBaseline = clearBaseline;`
  - `destroy()` deletes the three new window functions (line ~504 area).
- Update `saveBaseline()` (line ~3036) to:
  - Call `await loadBaselines()` (renamed) instead of `loadBaseline()`.
  - After load, set `_activeBaselineId` to the just-written doc's id. Simplest: `loadBaselines()` already auto-selects newest, so this is automatic.
  - Call `updateBaselineToolbarUI()` after the overlay render to refresh the dropdown options + button label.
- Update `renderTaskGrid`/`renderGantt`/`setGanttZoom` paths that call `injectBaselineOverlay()` — none need changes; they read from `_baselineData` which is now driven by `_activeBaselineId`.
- After the toolbar HTML is in the DOM (end of `init()` or after first `renderTaskGrid()`), call `updateBaselineToolbarUI()` to populate the selector.

**Verify:**

2. Pick a different baseline → overlay swaps. Button still "Clear Baseline".
3. Click "Clear Baseline" → overlay disappears, selector resets to "— none —", button flips to "Set Baseline". **No Firestore docs deleted** (refresh — baselines still in dropdown after re-load).
4. Click "Set Baseline" → new `Baseline N+1` doc created, auto-selected, overlay renders. Button flips to "Clear Baseline".
5. Re-run UAT checks 1–7 from `86.12-VERIFICATION.md` (single-baseline behavior must still pass).

**Done:** All four interactions above work; no console errors on view enter/leave; no orphaned window functions after `destroy()`.

## must_haves

- `_baselines` array module state present and populated by `loadBaselines()`.
- `_activeBaselineId` drives both the overlay and the toggle button label.
- Toolbar contains `#baselineSelect` and `#baselineToggleBtn`.
- `window.toggleBaseline`, `window.selectBaseline`, `window.clearBaseline` registered in `init()` and deleted in `destroy()`.
- Clear Baseline does NOT call `deleteDoc` — baselines persist in Firestore.
- All baseline labels rendered via `escapeHTML()` (no XSS regression vs Phase 86.12).

## Risks

- **Scope creep**: Tempting to add "delete this baseline" affordance. Per HANDOFF: explicitly deferred.
- **Concurrent saves**: If two users save baselines simultaneously, count-based label could collide. Phase 86.12 already accepted this; no change.
- **Initial overlay flicker**: Selector populates after `loadBaselines()` resolves; overlay renders inside `renderGantt`. Both already run in `init()` — no new race introduced.
