# Quick Task 20260601-bnm: User-defined baseline names

## Boundary

Phase 86.12 (and quick task `20260531-mbl`) auto-names every baseline `Baseline N`.
User wants the option to give baselines a meaningful name like "Q3 plan freeze" or
"Before scope change".

## Decisions (locked 2026-06-01)

- **Prompt UX:** `window.prompt('Name this baseline:', 'Baseline N')` — native browser
  dialog with the auto-name as default. Cheapest option; zero new HTML/CSS.
- **Required?** Optional with fallback. Empty/whitespace OK → uses `Baseline N`.
  Cancel → aborts the save entirely (standard prompt semantics).
- **Edit after creation?** Out of scope. `firestore.rules:246` enforces baselines
  are immutable (`allow update: if false;`) — changing that is a separate decision.
- **Max length:** Soft cap at 60 characters (substring truncation). Avoids the
  selector dropdown blowing out.

## Tasks

### Task 1 — Prompt user in `saveBaseline()`

**File:** `app/views/project-plan.js` (around line 3036, the `saveBaseline` function)

**Action:** Between the `countSnap` query and the `addDoc` call, replace the
auto-derived `label` with prompt-driven logic:

```js
const countSnap = await getDocs(collection(db, 'projects', currentProject.id, 'baselines'));
const defaultLabel = `Baseline ${countSnap.size + 1}`;
const userInput = window.prompt('Name this baseline:', defaultLabel);
if (userInput === null) return;  // user cancelled — abort save
const label = (userInput.trim() || defaultLabel).substring(0, 60);
```

**Verify:**
1. Click Set Baseline → prompt appears with `Baseline 1` (or next-N) as default.
2. Type `Q3 plan freeze` → click OK → baseline saved with that label; selector
   shows `Q3 plan freeze`; slip summary footer reads `Baseline: Q3 plan freeze`.
3. Click Set Baseline → prompt → click **Cancel** → no Firestore write, no toast,
   no overlay change.
4. Click Set Baseline → prompt → clear text → click OK → falls back to `Baseline N`.
5. Click Set Baseline → type a 200-char string → save → label truncated to 60.

**Done:** All five cases behave as above; no existing baseline-related behavior
regresses (overlay still renders, toggle button still flips, selector still
populates).

## must_haves

- `window.prompt` call sits between `countSnap` and `addDoc` in `saveBaseline()`.
- `null` return from prompt aborts the save (`return;` before any state mutation).
- Empty-string OK falls back to the existing `Baseline N` pattern.
- Label is `.trim()`ed and `.substring(0, 60)`-capped before write.
- No changes to `firestore.rules`, no changes to data shape, no other functions
  touched.

## Risks

- **Cancel semantics:** Users who expect Cancel to mean "use default name" will be
  surprised when nothing happens. Mitigation: prompt copy is "Name this baseline:"
  — the act of clicking Cancel reads as "I changed my mind", not "skip naming".
  If complaints surface, the alternative is to treat Cancel like empty (fallback).
- **`prompt()` UI quality:** Browser native dialogs are utilitarian. If this looks
  cheap in practice, swap to a small modal in a follow-up (decision already
  surfaced and deferred).
- **Length cap:** 60 chars is arbitrary. Picked to avoid breaking `max-width:180px`
  on `.plan-baseline-select`. Could be tighter if labels overflow in practice.
