<decisions_made>

- **Stage-aware finance** — three display states depending on lifecycle position:
  - Pre-contract (Inspection through Client Review): show "Proposed ₱X" or "Pre-contract" label — no billing bar
  - Contracted, billing not started (Client Approved / For Mobilization): show contract value + "Awaiting billings"
  - Active billing (On-going): show contract value + mini billing utilization bar + %
  - Complete: green, "Fully billed · 100% ✓"
  
- **Recommendation: Option D (Attention Feed) + B (Grouped List) as toggle**
  - D ranks projects by urgency signals: stale proposals, quiet on-going projects, contracts not yet billing
  - B kept as "Browse All" for neutral sorted/filtered view
  - Rationale: management wants to know what needs their eyes, not neutral browsing

- **Option E (Swimlane) deferred** — requires stage transition timestamps (when project moved stage to stage), which are not stored today. Would need a data model change to track status change history.

- **Option A (Kanban) not recommended** — construction management prefers lists; horizontal scroll is awkward on shared monitors; column grouping for 10 stages is debatable

- **Layout not limited to tables** — user explicitly opened this up; any paradigm fair game
</decisions_made>

<context>
Pure design spike session. The core insight that drove the work: the current projects/services table (Code, Name, Client, Status, Active, Actions) gives management no quick read on where a project actually stands. The stage-aware finance problem — contract value being meaningless for pre-contract stages — was the key UX gap that pushed the design toward stage-contextual display.

Option D is the strongest answer to the user's stated goal ("management will have a quick thought — oh this is where the project is at") because it surfaces computed urgency signals rather than making management interpret raw status labels. The urgency thresholds (e.g. 14d in Client Review = urgent) would need tuning once the feature is live.
</context>
